# Mutual TLS Authentication

mTLS is used for service-to-service authentication within the internal mesh. Unlike standard TLS where only the server presents a certificate, mTLS requires both parties to authenticate.

## Certificate Hierarchy

Our PKI uses a three-tier hierarchy:

- **Root CA** — offline, air-gapped, 20-year validity
- **Intermediate CA** — online, managed by Vault, 5-year validity
- **Leaf certificates** — per-service, 72-hour validity with automatic renewal

Short-lived leaf certificates eliminate the need for CRL/OCSP infrastructure. If a certificate is compromised, it expires before revocation would propagate anyway.

## Implementation

The service mesh sidecar handles mTLS termination. Application code never touches TLS directly.

```yaml
# mesh-config.yaml
mtls:
  mode: STRICT
  certificateProvider: vault
  refreshInterval: 24h
  rootCA: /etc/certs/root-ca.pem
  allowedSPIFFEIDs:
    - spiffe://cluster.local/ns/payments/sa/payment-api
    - spiffe://cluster.local/ns/orders/sa/order-api
```

SPIFFE IDs provide workload identity independent of network location. Each service has a unique SPIFFE ID derived from its namespace and service account.

## Troubleshooting

Common issues with mTLS in our environment:

- **Certificate not yet valid** — clock skew between pods; check NTP sync
- **Unknown CA** — the intermediate CA bundle is missing; ensure the init container runs before the app
- **Handshake timeout** — usually a firewall rule blocking the sidecar port (15006)
