# CI/CD Pipeline Architecture

Our continuous integration and delivery pipeline is built on **GitHub Actions** with self-hosted runners for compute-intensive jobs. The design follows a trunk-based development model where every merge to `main` triggers the full pipeline.

## Pipeline Stages

The pipeline executes in four sequential stages:

1. **Lint & Typecheck** — runs ESLint and `tsc --noEmit` in parallel. Fails fast on syntax errors.
2. **Unit Tests** — Vitest suite with coverage thresholds. Must maintain >80% line coverage.
3. **Build & Package** — TypeScript compilation, Docker image build, and artifact upload.
4. **Deploy** — environment-specific deployment via ArgoCD sync.

Each stage has a hard timeout of 15 minutes. If any stage exceeds this, the entire workflow is cancelled and an alert fires to the `#ci-failures` Slack channel.

## Caching Strategy

We use GitHub Actions cache for three layers:

- **pnpm store** — keyed on `pnpm-lock.yaml` hash, saves ~2 minutes per run
- **TypeScript build info** — incremental compilation cache, keyed on `tsconfig.json` + source hash
- **Docker layer cache** — exported via `docker buildx` with GitHub Actions cache backend

```yaml
- uses: actions/cache@v4
  with:
    path: ~/.pnpm-store
    key: pnpm-${{ hashFiles('pnpm-lock.yaml') }}
    restore-keys: |
      pnpm-
```

## Branch Protection

The `main` branch requires:

- All CI checks passing
- At least one approving review
- No force pushes
- Linear history (squash merges only)
