# REST Resource Naming Conventions

Consistent resource naming makes APIs intuitive. Developers should be able to guess endpoint paths without reading documentation.

## Rules

- Use **plural nouns** for collections: `/users`, `/orders`, `/products`
- Use **path segments** for hierarchy: `/users/{id}/orders`
- Use **query parameters** for filtering: `/orders?status=pending&since=2024-01-01`
- Use **kebab-case** for multi-word resources: `/shipping-addresses`, not `/shippingAddresses`
- Avoid verbs in paths: `/users/{id}/activate` is acceptable for actions, but prefer state changes via PUT when possible

## Nested vs Flat Resources

Shallow nesting (one level) is readable:

```
GET /users/{userId}/orders
```

Deep nesting becomes unwieldy:

```
GET /users/{userId}/orders/{orderId}/items/{itemId}/reviews
```

Once nesting exceeds two levels, flatten by promoting the resource to a top-level path with query filters:

```
GET /reviews?orderId={orderId}
```

## Sub-Resources vs Actions

For operations that don't map cleanly to CRUD:

- **State transitions**: `POST /orders/{id}/cancel` (action on a resource)
- **Batch operations**: `POST /users/batch-delete` (action on a collection)
- **Computed results**: `GET /reports/monthly-revenue` (read-only computation)

These are pragmatic exceptions to the "no verbs" rule. Forcing everything into PUT/PATCH leads to awkward `{ status: "cancelled" }` payloads that obscure the intent.

## Versioning in Paths

Always prefix with the API version: `/api/v1/users`. See [[arch-patterns]] for the full versioning strategy discussion.
