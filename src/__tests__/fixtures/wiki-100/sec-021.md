# Multi-Factor Authentication

MFA adds a second factor beyond the password for user authentication. It is required for all admin users and available as opt-in for all other users.

## Supported Factors

- **TOTP** (Time-based One-Time Password) — Google Authenticator, Authy, 1Password
- **WebAuthn / FIDO2** — hardware security keys (YubiKey, etc.) and platform authenticators (Touch ID, Windows Hello)
- **SMS** — deprecated, available only for users who enrolled before 2024; not offered to new users

### Why SMS Is Deprecated

SMS-based MFA is vulnerable to SIM swapping and SS7 interception. NIST SP 800-63B classifies it as a restricted authenticator. We stopped offering it to new users and display migration prompts to existing SMS users.

## TOTP Implementation

TOTP uses the RFC 6238 algorithm with:

- **Algorithm**: HMAC-SHA1 (standard for interoperability with authenticator apps)
- **Digits**: 6
- **Period**: 30 seconds
- **Window**: 1 step tolerance (accepts codes from 30s before to 30s after)

Recovery codes (8 codes, 8 characters each) are generated at enrollment time. They are hashed with bcrypt before storage. Each code can be used exactly once.

## WebAuthn

WebAuthn is the recommended MFA method. It provides phishing resistance because the browser binds the credential to the origin.

We support both **roaming authenticators** (USB keys) and **platform authenticators** (biometric). Resident keys are supported for passwordless login flows.

Credentials are stored in the **webauthn_credentials** table, keyed by credential ID and user handle.

## Enforcement Policy

- `owner` and `admin` roles: MFA required, enforced at login
- Other roles: MFA encouraged via dashboard prompt, not enforced
- Service accounts: MFA not applicable (use mTLS or client credentials)

See [[sec-003]] for role definitions.
