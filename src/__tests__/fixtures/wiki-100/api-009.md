# Database Migration Best Practices

Migrations are the version control system for your database schema. Every schema change — table creation, column addition, index modification — must be captured as a numbered, idempotent migration file.

## Forward-Only Migrations

Rollback migrations sound good in theory but fail in practice. Once a migration has run in production and data has been written to a new column, rolling back means data loss. Prefer **forward-only** migrations: if something goes wrong, write a new migration to fix it.

## Safe Alterations in PostgreSQL

Some DDL operations take aggressive locks and can cause downtime:

- `ALTER TABLE ... ADD COLUMN` with a `DEFAULT` is safe in PostgreSQL 11+ (no table rewrite)
- `ALTER TABLE ... ADD COLUMN ... NOT NULL DEFAULT` is also safe in 11+ but requires careful attention to constraints
- `CREATE INDEX` takes a `SHARE` lock — use `CREATE INDEX CONCURRENTLY` instead
- `ALTER TABLE ... ALTER COLUMN TYPE` rewrites the entire table — avoid on large tables

## Multi-Phase Migrations

For risky changes, split into phases:

1. **Expand**: add the new column or table, deploy code that writes to both old and new
2. **Migrate**: backfill existing data from old to new
3. **Contract**: remove writes to old, drop old column

This pattern avoids downtime and provides rollback points at each phase.

```yaml
# drizzle-kit configuration
dialect: postgresql
schema: ./src/db/schema.ts
out: ./drizzle
strict: true
verbose: true
```

Always review generated SQL before applying. Automated migration generators sometimes produce suboptimal DDL.
