# Horizontal Pod Autoscaling

## HPA Configuration

Every stateless production service has a **HorizontalPodAutoscaler** (HPA) configured. The HPA adjusts replica count based on observed metrics, scaling up during load spikes and down during quiet periods.

## Scaling Metrics

We scale on two metrics simultaneously using the `behavior` API:

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-server-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-server
  minReplicas: 3
  maxReplicas: 20
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
    - type: Pods
      pods:
        metric:
          name: http_requests_per_second
        target:
          type: AverageValue
          averageValue: "100"
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60
      policies:
        - type: Percent
          value: 50
          periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
        - type: Percent
          value: 10
          periodSeconds: 60
```

## Scale-Down Caution

The HPA is deliberately conservative on scale-down: 5 minute stabilization window and 10% reduction per minute. Premature scale-down causes cascading restarts and request failures during bursty traffic.

## KEDA Integration

For event-driven workloads (queue consumers), we use **KEDA** instead of the built-in HPA. KEDA can scale to zero and scales based on queue depth rather than pod-level metrics.
