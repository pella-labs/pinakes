# ORM Trade-offs

## The Spectrum

ORMs exist on a spectrum from raw SQL to full active-record patterns:

- **Raw SQL** with a query builder (knex, kysely) — maximum control, maximum SQL knowledge required
- **Data mapper** (Drizzle, MikroORM) — types from schema, explicit mapping, composable queries
- **Active record** (TypeORM with decorators, Prisma) — model classes with attached query methods, less boilerplate

There is no universally best choice. The decision depends on team SQL fluency, query complexity, and how much abstraction leakage you can tolerate.

## Drizzle ORM

Our current choice for this project. Drizzle's key advantages:

- Schema defined in TypeScript — types are derived, not generated
- Composable query builder that maps 1:1 to SQL concepts
- No code generation step
- Lightweight runtime (~50KB)

```typescript
import { sqliteTable, text, integer } from 'drizzle-orm/sqlite-core';

export const users = sqliteTable('users', {
  id: text('id').primaryKey(),
  email: text('email').notNull().unique(),
  name: text('name').notNull(),
  createdAt: integer('created_at', { mode: 'timestamp' }).notNull(),
});
```

## Common Pitfalls

- **Lazy loading**: fetching related entities one-by-one in a loop. Always use explicit joins or eager loading.
- **Over-fetching**: selecting all columns when you only need two. Use `.select()` to pick specific fields.
- **Ignoring the query log**: enable SQL logging during development. Surprising queries surface quickly when you can see them.

Regardless of ORM choice, every team member should be able to read and write raw SQL. The ORM is a productivity tool, not a replacement for understanding the database.
