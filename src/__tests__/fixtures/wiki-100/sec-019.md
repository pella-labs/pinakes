# Key Rotation Procedures

Regular key rotation limits the blast radius of a compromised key. This page documents the rotation schedule and procedures for each key type.

## Rotation Schedule

| Key Type | Rotation Frequency | Method |
|----------|-------------------|--------|
| JWT signing keys | 90 days | Automated via Vault |
| OAuth client secrets | 90 days | Manual with tooling |
| Database credentials | 30 days | Dynamic via Vault |
| API keys (third-party) | 180 days | Manual |
| KMS CMKs | 365 days | Automated via AWS |
| mTLS leaf certs | 72 hours | Automated via mesh |
| Service account tokens | 24 hours | Automated via K8s |

## JWT Key Rotation

The JWT signing key rotation uses a **dual-key** strategy:

1. Generate a new key pair in Vault
2. Add the new public key to the JWKS endpoint
3. Wait for cache propagation (max 5 minutes across all services)
4. Switch the signer to the new private key
5. Keep the old public key in JWKS for token TTL duration (15 minutes)
6. Remove the old public key

This ensures zero-downtime rotation. Tokens signed with the old key remain valid until they expire naturally.

## Database Credential Rotation

Vault generates **dynamic database credentials** with a 30-day TTL. Each service gets unique credentials. On rotation:

- Vault creates a new role with new credentials
- The application picks up new credentials on next connection pool refresh
- Old credentials are revoked after a grace period

## Emergency Rotation

For compromised keys, see [[sec-009]] emergency rotation procedure. Emergency rotation skips the grace period and revokes immediately.
