# Webhook Design

Webhooks push event notifications from a server to a client-specified URL. They replace polling for real-time integrations and are the standard mechanism for platform extensibility.

## Payload Design

Every webhook payload should include:

```json
{
  "id": "evt_abc123",
  "type": "order.completed",
  "created_at": "2024-06-15T14:30:00Z",
  "data": {
    "order_id": "ord_xyz789",
    "total": 149.99,
    "currency": "USD"
  }
}
```

- A unique **event ID** for deduplication
- An **event type** string for routing
- A **timestamp** for ordering and debugging
- The **data payload** specific to the event

## Delivery Guarantees

Webhooks are **at-least-once** delivery. Receivers must be idempotent — use the event ID to detect and skip duplicates.

Retry policy:
- Retry on 5xx responses and network timeouts
- Exponential backoff: 1min, 5min, 30min, 2hr, 12hr
- Disable the endpoint after N consecutive failures and notify the subscriber

## Security

- **Sign payloads** with HMAC-SHA256 using a per-subscriber secret
- Include the signature in a header: `X-Webhook-Signature: sha256=abc...`
- Receivers MUST verify the signature before processing
- Use HTTPS exclusively — never deliver webhooks over HTTP

```typescript
import { createHmac } from 'crypto';

function signPayload(payload: string, secret: string): string {
  return createHmac('sha256', secret).update(payload).digest('hex');
}
```

## Receiver Best Practices

- Respond with 200 immediately, process asynchronously
- Store the raw payload before processing (debugging lifeline)
- Set a reasonable timeout (5-10 seconds) for the acknowledgment response
