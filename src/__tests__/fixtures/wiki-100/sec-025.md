# Incident Response Plan

Our security incident response follows a four-phase model aligned with NIST SP 800-61. All team members with on-call responsibilities must be familiar with this plan.

## Phase 1: Detection and Reporting

Incidents are detected through:

- Automated alerts from the SIEM (see [[monitoring-setup]])
- Audit log anomaly detection (see [[sec-010]])
- Bug bounty submissions
- Internal reports via `#security-incidents` Slack channel

Anyone who suspects an incident should report it immediately. There is no penalty for false positives.

## Phase 2: Triage and Containment

The on-call security engineer triages the incident within 30 minutes:

1. Assess severity (P1-P4)
2. Determine scope — what systems and data are affected
3. Contain the threat — revoke compromised credentials, isolate affected systems, block malicious IPs
4. Notify stakeholders per the escalation matrix

**P1 incidents** (active data breach, production compromise) trigger the war room process with a dedicated incident commander.

## Phase 3: Eradication and Recovery

- Identify and remove the root cause
- Patch the vulnerability or close the attack vector
- Restore from known-good backups if necessary
- Verify systems are clean before returning to service
- Rotate any credentials that may have been exposed

## Phase 4: Post-Incident Review

Within 5 business days of resolution:

- Blameless post-mortem document
- Timeline reconstruction
- Root cause analysis
- Action items with owners and deadlines
- Lessons learned shared with the engineering team

Post-mortems are stored in the `incidents/` directory and linked from the security team's quarterly report.

## Severity Definitions

- **P1**: Active breach, data exfiltration, production system compromise
- **P2**: Vulnerability being actively exploited, credential leak, significant unauthorized access
- **P3**: Vulnerability discovered (not exploited), policy violation, suspicious activity
- **P4**: Minor policy exception, informational security event
