# Database Transaction Isolation Levels

Transaction isolation determines what data a transaction can see from other concurrent transactions. Choosing the wrong level leads to either phantom reads or unnecessary lock contention.

## The Four Levels

### Read Uncommitted

Transactions can see uncommitted changes from other transactions (**dirty reads**). Almost never appropriate. PostgreSQL treats this as Read Committed regardless.

### Read Committed

The default in PostgreSQL. Each statement within a transaction sees only data committed before that statement began. Different statements in the same transaction can see different snapshots. Safe for most OLTP workloads.

### Repeatable Read

The transaction sees a consistent snapshot as of its start time. No other transaction's commits are visible after the snapshot is taken. Prevents **non-repeatable reads** but allows **serialization anomalies**.

```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT balance FROM accounts WHERE id = 42;  -- sees snapshot
-- concurrent transaction commits a balance change
SELECT balance FROM accounts WHERE id = 42;  -- still sees original snapshot
COMMIT;
```

### Serializable

The strongest level. Transactions behave as if they executed one at a time. PostgreSQL implements this using **Serializable Snapshot Isolation (SSI)**, which detects conflicts and aborts one of the conflicting transactions.

Use serializable for:
- Financial calculations where correctness trumps throughput
- Inventory reservation where overselling is unacceptable
- Any scenario where concurrent modifications could produce inconsistent state

## Practical Recommendations

- Use **Read Committed** as the default
- Upgrade to **Serializable** for specific critical sections, not globally
- Always handle serialization failures with retry logic — aborted transactions must be retried, not ignored
