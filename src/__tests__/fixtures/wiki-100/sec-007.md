# Content Security Policy

CSP is our primary defense against XSS attacks in the web application. The policy is delivered via HTTP header and enforced by the browser.

## Active Policy

```
Content-Security-Policy:
  default-src 'self';
  script-src 'self' 'nonce-{random}';
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https://cdn.example.com;
  connect-src 'self' https://api.example.com wss://ws.example.com;
  font-src 'self' https://fonts.gstatic.com;
  frame-ancestors 'none';
  base-uri 'self';
  form-action 'self';
  upgrade-insecure-requests;
```

Key decisions:

- **Nonce-based script loading** instead of hashes — works better with our build pipeline that generates unique bundles per deploy
- **unsafe-inline for styles** — CSS-in-JS libraries require it; we accept the risk because style injection is lower severity than script injection
- **frame-ancestors 'none'** — prevents clickjacking; replaces the older X-Frame-Options header

## Report-Only Mode

New policy changes are deployed in **report-only** mode first for 48 hours. Violation reports are sent to `/api/csp-report` and aggregated in [[monitoring-setup]].

## Nonce Generation

The nonce is generated per-request by the SSR middleware using `crypto.randomBytes(16).toString('base64')`. It is injected into script tags during server-side rendering and included in the CSP header for that response.

Never reuse nonces across requests. Never use a predictable value.
