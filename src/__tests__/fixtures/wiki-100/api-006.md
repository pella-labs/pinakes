# Cursor-Based Pagination

Offset pagination (`LIMIT 20 OFFSET 40`) is simple but fundamentally broken for mutable datasets. Inserts and deletes between page fetches cause items to be skipped or duplicated. For any API serving real-time data, **cursor-based pagination** is the correct approach.

## How It Works

A cursor is an opaque token that encodes a position in the result set — typically a base64-encoded composite of the sort key and row ID. The client passes the cursor to get the next page.

```sql
-- First page
SELECT id, created_at, title
FROM articles
ORDER BY created_at DESC, id DESC
LIMIT 21;  -- fetch one extra to determine hasNextPage

-- Next page (cursor decodes to created_at='2024-01-15T10:00:00Z', id='abc-123')
SELECT id, created_at, title
FROM articles
WHERE (created_at, id) < ('2024-01-15T10:00:00Z', 'abc-123')
ORDER BY created_at DESC, id DESC
LIMIT 21;
```

The `LIMIT N+1` trick avoids a separate count query: if you get 21 rows back, there is a next page; return only 20 to the client.

## Composite Cursors

Always include a **unique tiebreaker** in the cursor. If you paginate by `created_at` alone and two rows share the same timestamp, the cursor is ambiguous. The composite `(created_at, id)` guarantees deterministic ordering.

## Implementation

```typescript
function encodeCursor(createdAt: string, id: string): string {
  return Buffer.from(`${createdAt}|${id}`).toString('base64url');
}

function decodeCursor(cursor: string): { createdAt: string; id: string } {
  const [createdAt, id] = Buffer.from(cursor, 'base64url').toString().split('|');
  return { createdAt, id };
}
```
