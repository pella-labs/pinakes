# Kubernetes Cluster Architecture

## Cluster Topology

We run a **multi-node Kubernetes cluster** on AWS EKS with three availability zones. The control plane is managed by AWS. Worker nodes are split into two node groups:

- **General-purpose** (`m6i.xlarge`) — runs API servers, background workers, and internal services
- **Compute-optimized** (`c6i.2xlarge`) — reserved for embedding generation and heavy batch processing

Each node group uses **Cluster Autoscaler** with min/max bounds to handle traffic spikes without overspending. General-purpose scales from 3 to 12 nodes. Compute scales from 1 to 4.

## Namespace Strategy

Namespaces map to deployment environments:

| Namespace | Purpose | Resource Quotas |
|-----------|---------|-----------------|
| `production` | Live traffic | 32 CPU, 64Gi memory |
| `staging` | Pre-release validation | 8 CPU, 16Gi memory |
| `dev` | Feature branch deploys | 4 CPU, 8Gi memory |
| `monitoring` | Prometheus, Grafana, Loki | 8 CPU, 32Gi memory |
| `ingress` | Nginx ingress controllers | 4 CPU, 8Gi memory |

## Pod Disruption Budgets

All production deployments have **PodDisruptionBudgets** (PDBs) configured to ensure availability during node drains and cluster upgrades. The standard policy is `minAvailable: 50%` for stateless services and `maxUnavailable: 1` for stateful workloads.

## Network Policies

We enforce **default-deny** network policies in the production namespace. Each service must explicitly declare which other services it communicates with. This limits blast radius if a pod is compromised.

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-server-netpol
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: api-server
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: ingress-nginx
      ports:
        - port: 8080
  egress:
    - to:
        - podSelector:
            matchLabels:
              app: postgres
      ports:
        - port: 5432
```
