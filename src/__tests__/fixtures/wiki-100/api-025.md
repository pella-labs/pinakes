# API Response Compression

Compressing API responses reduces bandwidth, improves latency over slow networks, and lowers data transfer costs. Modern compression algorithms are fast enough that CPU overhead is negligible for text-based payloads.

## Algorithms

Three common options, in order of preference:

- **Brotli** (br): best compression ratio for text, ~15-20% smaller than gzip. Supported by all modern browsers and HTTP clients.
- **gzip**: universal support, good compression, fast. The safe default.
- **deflate**: legacy, no practical advantage over gzip. Avoid.

## When to Compress

Compress responses when:
- Content-Type is text-based (JSON, HTML, XML, CSV, plain text)
- Response body is larger than ~1KB (below this threshold, compression overhead exceeds savings)
- The client sends `Accept-Encoding` indicating support

Do not compress:
- Already-compressed formats (images, video, archives)
- Very small responses (headers dominate the payload)
- Streaming responses where latency to first byte matters more than total size

## Server Configuration

```typescript
import express from 'express';
import compression from 'compression';

const app = express();

app.use(compression({
  threshold: 1024,  // minimum size in bytes
  filter: (req, res) => {
    if (req.headers['x-no-compression']) return false;
    return compression.filter(req, res);
  },
}));
```

## CDN-Level Compression

If a CDN sits in front of the API, compression should happen at the CDN edge rather than the origin. This offloads CPU from application servers and allows the CDN to cache compressed variants.

Configure the CDN to:
- Compress with both Brotli and gzip
- Serve the best algorithm based on `Accept-Encoding`
- Set `Vary: Accept-Encoding` to cache both variants separately
