# Distributed Tracing with OpenTelemetry

OpenTelemetry (OTel) provides **end-to-end distributed tracing** across all services. Every request generates a trace that shows the full call path, including cross-service hops, database queries, and external API calls.

## Instrumentation

We use the **OpenTelemetry Node.js SDK** with auto-instrumentation for:

- HTTP clients and servers (express, fetch)
- Database drivers (better-sqlite3, pg)
- Message queues (NATS)

Custom spans are added for business-critical operations:

```typescript
const span = tracer.startSpan('embed-chunks', {
  attributes: {
    'kg.chunk_count': chunks.length,
    'kg.embedder': provider,
  },
});
try {
  const vectors = await embedder.embed(chunks);
  span.setStatus({ code: SpanStatusCode.OK });
  return vectors;
} finally {
  span.end();
}
```

## Backend

Traces are exported via OTLP to **Tempo**, which stores them in an S3-backed object store. Grafana provides the query UI with trace-to-log and trace-to-metric correlation.

## Sampling

Production uses **tail-based sampling** at the collector level:

- 100% of error traces are retained
- 100% of slow traces (>2s) are retained
- 10% of normal traces are sampled randomly

This keeps storage costs manageable while ensuring all interesting traces are captured.

## Trace Context Propagation

The `traceparent` header (W3C Trace Context) is propagated across all service boundaries. Services that don't understand tracing pass the header through unchanged. This ensures traces aren't broken by intermediate proxies or legacy services.
