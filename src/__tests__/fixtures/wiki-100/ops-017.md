# TLS Certificate Management

## Automated Certificate Lifecycle

TLS certificates are managed by **cert-manager** running inside Kubernetes. It handles issuance, renewal, and rotation automatically using Let's Encrypt as the ACME provider.

## How It Works

1. A `Certificate` resource is created in the cluster (either directly or via an Ingress annotation)
2. cert-manager creates an ACME order with Let's Encrypt
3. DNS-01 challenge is solved via Route 53 (cert-manager has IAM permissions for the hosted zone)
4. Certificate is issued and stored as a Kubernetes Secret
5. cert-manager renews 30 days before expiration

```yaml
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: api-tls
  namespace: production
spec:
  secretName: api-tls-secret
  issuerRef:
    name: letsencrypt-prod
    kind: ClusterIssuer
  dnsNames:
    - api.example.com
    - "*.api.example.com"
```

## Monitoring Certificate Expiry

A Prometheus alert fires if any certificate is within 14 days of expiration:

```yaml
- alert: CertificateExpiringSoon
  expr: certmanager_certificate_expiration_timestamp_seconds - time() < 14 * 24 * 3600
  for: 1h
  labels:
    severity: warning
  annotations:
    summary: "Certificate {{ $labels.name }} expires in less than 14 days"
```

## Internal mTLS

Service-to-service communication within the cluster uses **mTLS** via a service mesh (Linkerd). Each pod gets a short-lived identity certificate rotated every 24 hours. This ensures encrypted transport even within the cluster network.
