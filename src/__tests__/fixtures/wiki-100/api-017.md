# API Gateway Patterns

An **API gateway** sits between clients and backend services, providing a single entry point for routing, authentication, rate limiting, and observability. It decouples client-facing concerns from service implementation.

## Core Responsibilities

- **Routing**: map external URLs to internal service endpoints
- **Authentication**: verify tokens before requests reach services
- **Rate limiting**: enforce per-client and per-endpoint limits
- **Request transformation**: header injection, path rewriting, body mapping
- **Response aggregation**: combine responses from multiple services into one
- **TLS termination**: handle HTTPS at the edge

## BFF Pattern

The **Backend for Frontend** pattern uses separate gateways tailored to each client type. A mobile BFF returns smaller payloads optimized for bandwidth constraints. A web BFF returns richer responses that power complex UIs.

This avoids the "one-size-fits-none" problem where a single API tries to serve mobile, web, and third-party consumers with the same response shapes.

## Circuit Breaking

The gateway should implement **circuit breakers** for each upstream service. When a service starts failing:

1. **Closed** (normal): requests pass through
2. **Open** (tripped): requests fail immediately with 503, no upstream call
3. **Half-open** (probe): allow one request through to test recovery

This prevents a single failing service from consuming all gateway threads and cascading failures across the entire system.

## Observability

The gateway is the ideal place to capture request-level metrics:

- Request rate by endpoint and client
- Latency percentiles (p50, p95, p99)
- Error rates by status code
- Upstream service health
