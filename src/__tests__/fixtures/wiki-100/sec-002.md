# JWT Best Practices

JSON Web Tokens are used as the access token format across all services. This document covers signing, validation, and common pitfalls.

## Signing Algorithm

All JWTs are signed with **RS256** (RSA with SHA-256). We explicitly reject `none` and `HS256` in the validation middleware to prevent algorithm confusion attacks.

```typescript
const verifyOptions: jwt.VerifyOptions = {
  algorithms: ['RS256'],
  issuer: 'https://auth.example.com',
  audience: config.SERVICE_NAME,
  clockTolerance: 30, // seconds
};
```

The signing key is an RSA-4096 key pair stored in [[secrets-management]] (Vault). Key rotation happens quarterly — see sec-019 for the rotation procedure.

## Claims Validation

Every service MUST validate:

- `exp` — reject expired tokens (with 30s clock tolerance)
- `iss` — must match our auth server
- `aud` — must include the current service name
- `nbf` — if present, reject tokens not yet valid

Custom claims use the `https://example.com/` namespace prefix to avoid collisions.

## Common Pitfalls

1. **Never decode without verifying** — use `jwt.verify()`, not `jwt.decode()`
2. **Never store JWTs in localStorage** — use httpOnly cookies or in-memory storage
3. **Never pass JWTs in query strings** — they end up in server logs and browser history
4. **Check token size** — JWTs over 8KB cause issues with some reverse proxies

See [[api-design]] for how tokens flow through the gateway.
