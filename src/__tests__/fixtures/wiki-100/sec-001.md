# OAuth2 Authorization Framework

OAuth2 is the industry standard protocol for authorization, enabling third-party applications to obtain limited access to HTTP services. Our platform implements the **Authorization Code Grant** with PKCE for all user-facing clients.

## Grant Types

We support the following grant types in production:

- **Authorization Code + PKCE** — primary flow for SPAs and mobile apps
- **Client Credentials** — service-to-service communication only
- **Refresh Token** — long-lived sessions without re-authentication
- **Device Code** — CLI tools and headless environments

The **Implicit Grant** is deprecated and blocked at the gateway level. See [[api-design]] for endpoint specifications.

## Token Lifecycle

Access tokens are short-lived JWTs (15 minutes). Refresh tokens are opaque strings stored in the **token_grants** table with a 30-day sliding window expiration.

```typescript
interface TokenPayload {
  sub: string;          // user ID
  aud: string;          // client ID
  scope: string[];      // granted scopes
  iat: number;
  exp: number;
  jti: string;          // unique token ID for revocation
}
```

Token rotation happens on every refresh — the old refresh token is invalidated immediately. If a previously-rotated token is presented, the entire grant family is revoked (replay detection).

## Client Registration

All OAuth clients must be registered in the **oauth_clients** table. Public clients (SPAs, mobile) must use PKCE. Confidential clients (backend services) use client secrets rotated every 90 days via [[deploy-pipeline]].

### Redirect URI Validation

Redirect URIs are matched exactly — no wildcards, no path prefixes. Localhost is allowed only in development environments. The validation logic lives in `src/auth/oauth/redirect.ts`.
