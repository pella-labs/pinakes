# CORS Configuration

Cross-Origin Resource Sharing controls which domains can make requests to our API. Misconfigured CORS is one of the most common security findings in our quarterly audits.

## Current Policy

The API gateway enforces CORS centrally. Individual services do not set CORS headers.

- **Allowed origins**: exact match against a whitelist in `config/cors-origins.json`
- **Allowed methods**: GET, POST, PUT, PATCH, DELETE, OPTIONS
- **Allowed headers**: Authorization, Content-Type, X-Request-ID, X-Idempotency-Key
- **Credentials**: allowed (cookies are sent for session-based flows)
- **Max age**: 3600 seconds (preflight cache)

Wildcard (`*`) origins are never used in production. The staging environment allows `*.staging.example.com` via regex matching.

## Common Mistakes

- Setting `Access-Control-Allow-Origin: *` with `credentials: true` — browsers reject this combination
- Reflecting the `Origin` header verbatim without validation — this is equivalent to wildcard and defeats the purpose of CORS
- Forgetting to handle OPTIONS preflight requests — results in cryptic failures for non-simple requests
- Not including `Vary: Origin` — causes caching proxies to serve wrong CORS headers

## Adding a New Origin

1. Add the origin to `config/cors-origins.json`
2. Open a PR with security team review required
3. Deploy via [[deploy-pipeline]]
4. Verify with `curl -H "Origin: https://newdomain.com" -I https://api.example.com/health`
