# REST API Versioning Strategies

Choosing the right API versioning strategy early prevents painful migrations and client breakage down the road. There is no universally correct answer — it depends on your consumer base, release cadence, and tolerance for complexity.

## URI Path Versioning

The most common approach embeds the version directly in the URL path: `/api/v1/users`, `/api/v2/users`. It is explicit, easy to route, and simple to document. The downside is URL pollution and the temptation to treat major versions as entirely separate APIs rather than incremental evolutions.

Most teams using **path versioning** end up maintaining at most two live versions simultaneously. Anything beyond that signals a deeper design problem — either the API surface is too unstable or deprecation discipline is lacking.

## Header-Based Versioning

An alternative is to use a custom header like `X-API-Version: 2` or the `Accept` header with a vendor media type:

```
Accept: application/vnd.myapp.v2+json
```

This keeps URLs clean but makes the API harder to explore with a browser or simple curl commands. It also complicates caching since CDNs must be configured to vary on the version header.

## Query Parameter Versioning

A third option uses `?version=2`. This is easy to implement but mixes versioning concerns into the query string, which can conflict with pagination and filtering parameters. It is generally considered the least clean option but appears frequently in legacy systems.

## Deprecation and Sunset Policies

Regardless of strategy, every versioned API needs a **sunset policy**. Best practice:

- Announce deprecation at least 90 days before removal
- Return `Sunset` and `Deprecation` headers on deprecated endpoints
- Log calls to deprecated versions and alert when traffic drops below a threshold
- Provide a migration guide with diff-level detail between versions
