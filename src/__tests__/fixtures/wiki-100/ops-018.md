# Secrets Management

All secrets are stored in **AWS Secrets Manager** and injected into Kubernetes pods via the **External Secrets Operator** (ESO). No secrets are stored in Git, environment files, or ConfigMaps.

## Flow

1. Secret is created or updated in AWS Secrets Manager
2. An `ExternalSecret` resource in the cluster references the AWS secret
3. ESO syncs the value into a Kubernetes `Secret` object
4. Pods mount the Secret as environment variables or volume files

## Rotation Policy

- **Database credentials**: rotated every 90 days automatically via AWS Secrets Manager rotation Lambda
- **API keys**: rotated on-demand when compromised or during personnel changes
- **TLS certificates**: rotated automatically by cert-manager (see [[security-policies]])
- **Service account tokens**: short-lived, issued per pod via projected volumes

## Access Control

IAM policies restrict which services can read which secrets:

- The API server can only read `prod/api-server/*` secrets
- The worker can only read `prod/worker/*` secrets
- CI pipelines use a separate IAM role with read access to deployment secrets only

## Emergency Rotation

If a secret is compromised:

1. Rotate the secret in AWS Secrets Manager immediately
2. Trigger a manual ESO sync: `kubectl annotate externalsecret <name> force-sync=$(date +%s)`
3. Restart affected pods to pick up new values
4. File a SEV-2 incident and begin post-mortem

The entire emergency rotation process should complete within 15 minutes.
