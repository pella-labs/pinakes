# Vulnerability Management

Our vulnerability management program covers application dependencies, container images, and infrastructure. Findings are triaged and remediated based on severity.

## Scanning Tools

- **npm audit / pnpm audit** — dependency vulnerabilities, runs on every CI build
- **Trivy** — container image scanning, runs on every image build
- **Dependabot** — automated dependency update PRs
- **Semgrep** — static analysis for security patterns, integrated into CI

## Severity and SLA

| Severity | CVSS Range | Remediation SLA |
|----------|-----------|-----------------|
| Critical | 9.0-10.0 | 24 hours |
| High | 7.0-8.9 | 7 days |
| Medium | 4.0-6.9 | 30 days |
| Low | 0.1-3.9 | 90 days |

SLA starts from the time the vulnerability is detected and triaged, not from public disclosure.

## Triage Process

1. Scanner detects vulnerability and creates a ticket
2. Security team triages within 4 hours (business hours)
3. If exploitable in our context, assign to the owning team
4. If not exploitable (e.g., vulnerable code path not reachable), mark as **accepted risk** with justification
5. Remediation PR goes through standard review + [[deploy-pipeline]]

## Dependency Update Policy

- Security patches: merge immediately after CI passes
- Minor updates: batch weekly
- Major updates: evaluate breaking changes, test in staging first

All dependency changes require `pnpm audit` to pass. A new vulnerability introduced by an update blocks the merge.

## Container Image Policy

Base images use the `-slim` variant to minimize attack surface. Images are rebuilt weekly to pick up OS-level patches even when application code has not changed.
