# Structured Logging with Pino

## Why Structured Logging

All services emit **structured JSON logs** via Pino. Unstructured text logs are banned in production — they're impossible to query reliably and break log aggregation pipelines.

Every log line includes:

```json
{
  "level": 30,
  "time": 1718000000000,
  "msg": "request completed",
  "req_id": "abc-123",
  "method": "GET",
  "path": "/api/v1/nodes",
  "status": 200,
  "duration_ms": 42,
  "service": "api-server",
  "env": "production"
}
```

## Log Levels

| Level | When to Use |
|-------|-------------|
| `fatal` | Process is about to crash |
| `error` | Operation failed, needs attention |
| `warn` | Degraded but functional |
| `info` | Normal operational events |
| `debug` | Diagnostic detail (never in prod) |
| `trace` | Extreme verbosity (local dev only) |

Production runs at `info` level. Staging runs at `debug`. The level is configurable via `LOG_LEVEL` environment variable without redeployment.

## Log Aggregation

Logs flow through a three-stage pipeline:

1. **Collection** — Fluent Bit DaemonSet on every node captures container stdout
2. **Processing** — Fluent Bit enriches with Kubernetes metadata (pod name, namespace, labels)
3. **Storage** — Logs ship to **Loki** for indexing and querying via Grafana

Retention: 30 days in Loki, 90 days in cold S3 archive.

## Correlation

Every inbound request generates a **request ID** (`X-Request-Id` header or UUID). This ID propagates through all downstream service calls and appears in every log line, enabling end-to-end request tracing across services.

See [[api-design]] for the request ID middleware implementation.
