# Idempotency in APIs

An operation is **idempotent** if performing it multiple times produces the same result as performing it once. Network failures, retries, and duplicate requests are inevitable in distributed systems — idempotency makes them safe.

## Naturally Idempotent Methods

- **GET**: reading a resource is inherently idempotent
- **PUT**: replacing a resource with a complete representation is idempotent
- **DELETE**: deleting an already-deleted resource returns 404 but causes no additional side effects

## Making POST Idempotent

POST is not naturally idempotent. Creating a payment twice charges the customer twice. The solution is an **idempotency key** — a client-generated unique identifier sent with each request.

```typescript
// Client sends:
// POST /payments
// Idempotency-Key: idem_8f3a9b2c

async function processPayment(req: Request) {
  const key = req.headers['idempotency-key'];
  if (!key) return res.status(400).json({ error: 'Idempotency-Key required' });

  const existing = await db.query(
    'SELECT response FROM idempotency_store WHERE key = $1',
    [key]
  );

  if (existing) {
    return res.status(200).json(existing.response);
  }

  const result = await chargeCard(req.body);

  await db.query(
    'INSERT INTO idempotency_store (key, response, created_at) VALUES ($1, $2, now())',
    [key, result]
  );

  return res.status(201).json(result);
}
```

## Key Design Decisions

- Keys should expire after 24-48 hours to prevent unbounded table growth
- Store the full response so retries return identical data
- Use a `UNIQUE` constraint on the key column to handle race conditions
- Scope keys to the authenticated user to prevent cross-tenant collisions
