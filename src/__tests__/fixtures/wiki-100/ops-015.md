# CDN and Edge Caching

## Architecture

Static assets and cacheable API responses are served through **CloudFront** as our CDN layer. The CDN sits in front of the ALB and caches responses at edge locations worldwide, reducing latency for geographically distributed users.

## Cache Policies

Three caching tiers based on content type:

### Static Assets
- Cache TTL: 1 year
- Cache key: path only (no query string)
- Immutable content with hashed filenames (`main.a1b2c3.js`)
- Cache-Control: `public, max-age=31536000, immutable`

### API Responses (Cacheable)
- Cache TTL: 60 seconds
- Cache key: path + query string + Authorization header hash
- Used for read-heavy endpoints like search results
- `Vary: Authorization, Accept-Encoding`

### API Responses (Dynamic)
- Cache TTL: 0 (pass-through)
- All write operations and user-specific data
- Cache-Control: `private, no-store`

## Cache Invalidation

We avoid proactive invalidation where possible. Content-addressed asset URLs mean old cached versions are simply never requested after deployment. For the rare cases where invalidation is needed, CloudFront invalidation requests are triggered by the deploy pipeline.

## Origin Shield

A single CloudFront **Origin Shield** region (us-east-1) is configured to reduce origin load. All edge locations route cache misses through the shield, collapsing redundant requests into a single origin fetch.
