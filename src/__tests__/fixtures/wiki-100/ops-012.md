# SLOs and Error Budgets

## What Are SLOs

A **Service Level Objective** (SLO) is a target for how reliable a service should be. We define SLOs for every customer-facing service. Each SLO has:

- A **Service Level Indicator** (SLI) — the metric being measured
- A **target** — the threshold (e.g., 99.9%)
- A **window** — the rolling time period (e.g., 30 days)

## Current SLOs

| Service | SLI | Target | Window |
|---------|-----|--------|--------|
| API Server | Successful requests / total requests | 99.9% | 30 days |
| API Server | p99 latency < 500ms | 99.0% | 30 days |
| Search | Successful queries / total queries | 99.5% | 30 days |
| Ingest Pipeline | Files processed within 60s | 95.0% | 30 days |

## Error Budgets

The **error budget** is the inverse of the SLO target. For a 99.9% availability SLO over 30 days, the error budget is 0.1% of requests — roughly 43 minutes of downtime.

When the error budget is healthy (>50% remaining), teams focus on feature work. When the budget is burning fast (<25% remaining), the team shifts to reliability work: fixing flaky tests, adding retries, improving monitoring.

If the error budget is fully consumed, new feature deployments are frozen until the SLO recovers.

## Burn Rate Alerts

We use **multi-window burn rate alerts** per the Google SRE workbook. A 14.4x burn rate over 1 hour fires a critical page. A 6x burn rate over 6 hours fires a warning.
