# SQLite Performance Tuning

SQLite is an embedded database, not a client-server system. Its performance characteristics are fundamentally different from PostgreSQL or MySQL. Tuning it correctly makes the difference between a snappy local tool and a frustratingly slow one.

## Essential PRAGMAs

These should be set on every connection, before any queries:

```sql
PRAGMA journal_mode = WAL;
PRAGMA busy_timeout = 5000;
PRAGMA synchronous = NORMAL;
PRAGMA foreign_keys = ON;
PRAGMA cache_size = -20000;  -- 20MB
PRAGMA temp_store = MEMORY;
```

**WAL mode** is the single most impactful setting. It allows concurrent reads while a write is in progress, which is impossible in the default rollback journal mode.

**busy_timeout** prevents immediate `SQLITE_BUSY` errors when a write lock is held. The connection will retry for the specified milliseconds before giving up.

## Writer Discipline

SQLite allows only one writer at a time. The correct architecture is a **single writer connection** with a separate read pool. Never use a generic connection pool with multiple writers — write contention actually decreases throughput compared to a serialized writer.

## FTS5 for Full-Text Search

```sql
CREATE VIRTUAL TABLE docs_fts USING fts5(
  title, body,
  tokenize='unicode61 remove_diacritics 2'
);
```

Use `unicode61` tokenizer, not `trigram`. Trigram indexes are 3x larger and only necessary for substring matching, which FTS5's prefix queries handle adequately for most use cases.

## Vacuum and Analyze

Run `ANALYZE` after bulk inserts to update query planner statistics. Run `VACUUM` periodically to reclaim space from deleted rows. Both are expensive — schedule them during maintenance windows, not on the hot path.
