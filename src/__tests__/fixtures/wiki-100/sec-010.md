# Security Audit Logging

Every security-relevant event must be logged to the **audit trail**, which is separate from application logs. The audit trail is append-only, tamper-evident, and retained for 7 years per compliance requirements.

## What Gets Logged

- Authentication events (login, logout, failed attempts, MFA challenges)
- Authorization decisions (grants and denials)
- Resource access (CRUD operations on sensitive resources)
- Administrative actions (role changes, user provisioning, config changes)
- Secret access (Vault reads, key rotations)
- Data exports and bulk operations

## Log Format

All audit events follow a structured JSON schema:

```typescript
interface AuditEvent {
  timestamp: string;       // ISO 8601
  event_type: string;      // e.g. "auth.login.success"
  actor: {
    id: string;
    type: 'user' | 'service' | 'system';
    ip?: string;
  };
  target?: {
    type: string;
    id: string;
  };
  outcome: 'success' | 'failure';
  metadata: Record<string, unknown>;
  request_id: string;      // correlation ID
}
```

## Storage Pipeline

Audit events flow through a dedicated pipeline:

1. Application emits event to the audit SDK
2. SDK writes to a local buffer (bounded, 1000 events max)
3. Buffer flushes every 5 seconds to the audit ingestion service
4. Ingestion service writes to append-only storage (S3 + Athena for querying)
5. Real-time copy goes to the SIEM for alerting

The pipeline is designed so that audit logging failures never block the application. If the buffer is full, events are dropped and a metric is incremented for alerting. See [[monitoring-setup]] for the audit pipeline health dashboard.
