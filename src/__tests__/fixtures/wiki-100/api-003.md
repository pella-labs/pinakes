# GraphQL Schema Design

## Type-First Modeling

Start with the domain types, not the queries. A **schema-first** approach forces you to think about the data model before worrying about resolution. Define your types in SDL, review them with frontend consumers, then implement resolvers.

```graphql
type User {
  id: ID!
  email: String!
  displayName: String!
  createdAt: DateTime!
  teams: [Team!]!
  recentActivity(limit: Int = 10): [ActivityEvent!]!
}

type Team {
  id: ID!
  name: String!
  members(role: TeamRole): [User!]!
  projects: [Project!]!
}
```

## N+1 Query Prevention

The single most common GraphQL performance problem is the **N+1 query**. If `teams` resolves by fetching each team individually, a query for 50 users triggers 50 additional database calls.

The standard solution is **DataLoader** — a per-request batching and caching utility. DataLoader collects all IDs requested within a single tick of the event loop and issues one batched query.

```typescript
const teamLoader = new DataLoader<string, Team>(async (ids) => {
  const teams = await db.query('SELECT * FROM teams WHERE id = ANY($1)', [ids]);
  return ids.map(id => teams.find(t => t.id === id));
});
```

Every resolver that touches the database should go through a DataLoader. No exceptions.

## Pagination

Use **Relay-style cursor pagination** for all list fields. Offset-based pagination breaks under concurrent writes and performs poorly on large datasets.

- `first` / `after` for forward pagination
- `last` / `before` for backward pagination
- Return `PageInfo` with `hasNextPage`, `hasPreviousPage`, `startCursor`, `endCursor`

## Error Handling

GraphQL errors should be part of the schema, not thrown exceptions. Use union types for operations that can fail:

```graphql
union CreateUserResult = User | ValidationError | DuplicateEmailError

type ValidationError {
  field: String!
  message: String!
}
```

This gives clients type-safe error handling instead of parsing error message strings.
