# Content Negotiation

**Content negotiation** allows a single endpoint to serve different representations of the same resource based on client preferences. The `Accept` header drives the selection.

## Server-Driven Negotiation

The client specifies what it can handle:

```
GET /api/v1/reports/quarterly
Accept: application/json
```

```
GET /api/v1/reports/quarterly
Accept: text/csv
```

The server inspects the `Accept` header and returns the appropriate format with a matching `Content-Type` response header. If the server cannot satisfy any of the client's preferences, it returns `406 Not Acceptable`.

## Quality Values

Clients can express preference ordering with quality values:

```
Accept: application/json;q=1.0, text/csv;q=0.8, application/xml;q=0.5
```

Higher `q` values indicate stronger preference. The server picks the highest-quality format it supports.

## Language Negotiation

The same mechanism works for language via `Accept-Language`:

```
Accept-Language: en-US, en;q=0.9, de;q=0.7
```

This is most relevant for APIs that return user-facing messages or localized content.

## Compression Negotiation

`Accept-Encoding` negotiates compression:

```
Accept-Encoding: br, gzip, deflate
```

Brotli (`br`) provides better compression ratios than gzip for text content. Most CDNs support both. Always set `Vary: Accept-Encoding` to prevent cache poisoning between compressed and uncompressed responses.

## Practical Guidance

Most APIs only need JSON. Add content negotiation when you have a concrete need (CSV export, XML legacy clients). Over-engineering content negotiation for hypothetical future formats wastes development time.
