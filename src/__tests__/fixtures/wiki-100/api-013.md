# ETL Pipeline Architecture

Extract-Transform-Load pipelines move data between systems, reshape it for analytical workloads, and maintain data warehouse freshness. The design of a pipeline determines its reliability, debuggability, and cost.

## Extract Phase

The extract phase reads data from source systems. Key considerations:

- **Change Data Capture (CDC)** over full dumps when possible — reduces source system load and pipeline runtime
- Use **read replicas** to avoid impacting production traffic
- Track extraction **high-water marks** (last processed ID or timestamp) to enable incremental loads
- Handle schema drift: source systems change without notice

## Transform Phase

Transforms should be **pure functions** — given the same input, they always produce the same output. This makes testing straightforward and reruns safe.

```typescript
interface TransformResult<T> {
  records: T[];
  rejected: Array<{ record: unknown; reason: string }>;
  stats: {
    input_count: number;
    output_count: number;
    rejected_count: number;
  };
}
```

Always track rejected records with reasons. Silent data loss is the most dangerous failure mode in a pipeline.

## Load Phase

- Use **bulk inserts** or `COPY` for PostgreSQL — row-by-row inserts are 10-100x slower
- **Upsert** with conflict resolution for idempotent loads
- Load into a staging table first, validate, then swap or merge into the target table
- Maintain a **load manifest** with row counts, checksums, and timestamps

## Orchestration

Modern ETL leans toward ELT (Extract-Load-Transform) where raw data is loaded into the warehouse and transformations happen in SQL via tools like dbt. The choice between ETL and ELT depends on whether your warehouse has sufficient compute for in-database transforms.
