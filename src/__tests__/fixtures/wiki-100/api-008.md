# HTTP Caching Headers

Proper cache headers reduce backend load, improve latency, and save bandwidth. Misconfigured caching either serves stale data or defeats caching entirely.

## Cache-Control Directives

The `Cache-Control` header is the primary mechanism. Key directives:

- `public` — any cache (CDN, browser, proxy) may store the response
- `private` — only the end-user's browser may cache (use for personalized data)
- `no-cache` — cache may store but must revalidate before use (misleading name)
- `no-store` — never cache under any circumstances (PII, auth tokens)
- `max-age=3600` — fresh for this many seconds
- `s-maxage=600` — CDN-specific max-age (overrides `max-age` for shared caches)
- `stale-while-revalidate=60` — serve stale for N seconds while refreshing in background

## ETag and Conditional Requests

**ETags** enable efficient revalidation. The server returns an `ETag` header (a hash or version identifier). On subsequent requests the client sends `If-None-Match: <etag>`. If the resource hasn't changed, the server responds with `304 Not Modified` and no body.

For database-backed resources, use the row's `updated_at` timestamp or a content hash as the ETag value.

## Vary Header

The `Vary` header tells caches which request headers affect the response. Always set `Vary: Accept-Encoding` at minimum. If responses differ by authentication state, add `Vary: Authorization`.

Forgetting `Vary` can cause a CDN to serve an authenticated user's response to an anonymous user, or vice versa.

## Common Mistakes

- Setting `max-age=31536000` on API responses (only appropriate for immutable static assets with hashed filenames)
- Omitting `Cache-Control` entirely (browsers apply heuristic caching based on `Last-Modified`, which is unpredictable)
- Using `no-cache` when you mean `no-store`
