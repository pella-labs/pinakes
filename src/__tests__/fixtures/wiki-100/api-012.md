# API Error Handling

A consistent error response format makes APIs predictable for consumers and reduces support burden. Error responses should be machine-parseable and human-readable simultaneously.

## Standard Error Envelope

```typescript
interface ApiError {
  error: {
    code: string;          // machine-readable, e.g. "RATE_LIMITED"
    message: string;       // human-readable explanation
    details?: unknown[];   // field-level validation errors, etc.
    request_id: string;    // for support correlation
  };
}
```

Every error response includes the `request_id` so that support can trace the exact request through logs. This single field saves hours of back-and-forth debugging.

## HTTP Status Code Selection

Use the most specific applicable status code:

- **400** — malformed request syntax, invalid JSON
- **401** — missing or invalid authentication
- **403** — authenticated but not authorized for this resource
- **404** — resource not found (also used to hide existence from unauthorized users)
- **409** — conflict (concurrent modification, duplicate resource)
- **422** — well-formed request but semantically invalid (validation failure)
- **429** — rate limited
- **500** — unhandled server error (never expose stack traces)
- **503** — temporarily unavailable (maintenance, overload)

## Validation Errors

Return field-level detail so clients can highlight specific form fields:

```json
{
  "error": {
    "code": "VALIDATION_FAILED",
    "message": "Request body contains invalid fields",
    "details": [
      { "field": "email", "message": "Must be a valid email address" },
      { "field": "age", "message": "Must be at least 13" }
    ],
    "request_id": "req_abc123"
  }
}
```

## Internal Error Logging

Log the full error with stack trace server-side but return only a sanitized message to the client. Never expose database errors, file paths, or internal service names in API responses. The `request_id` bridges the gap between the sanitized client response and the detailed server log.
