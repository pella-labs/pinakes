# Load Balancing Architecture

## External Load Balancing

Inbound traffic hits an **AWS Application Load Balancer** (ALB) which terminates TLS and routes requests to the Kubernetes ingress controller. The ALB performs health checks against the ingress endpoints every 10 seconds.

### ALB Configuration

- **Idle timeout**: 120 seconds (supports long-running API calls)
- **Deregistration delay**: 30 seconds (allows in-flight requests to complete)
- **Cross-zone load balancing**: enabled for even distribution
- **WAF integration**: AWS WAF rules filter known attack patterns before they reach the cluster

## Internal Load Balancing

Inside the cluster, Kubernetes **Services** of type `ClusterIP` distribute traffic across pod replicas using round-robin. For services requiring session affinity (WebSocket connections), we use `sessionAffinity: ClientIP`.

## Health Check Endpoints

Every service exposes two health endpoints:

- `/healthz` — **liveness probe**. Returns 200 if the process is alive. If this fails, Kubernetes restarts the pod.
- `/readyz` — **readiness probe**. Returns 200 if the service can handle traffic. Checks database connectivity, required config presence, and warmup completion. If this fails, the pod is removed from the Service's endpoint list but not restarted.

```bash
curl -s http://localhost:8080/readyz | jq .
# {
#   "status": "ready",
#   "checks": {
#     "database": "ok",
#     "embedder": "ok",
#     "config": "ok"
#   }
# }
```

## Graceful Shutdown

On `SIGTERM`, services enter a draining phase: stop accepting new requests, finish in-flight requests (up to 25 second deadline), then exit. This aligns with the Kubernetes `terminationGracePeriodSeconds` of 30.
