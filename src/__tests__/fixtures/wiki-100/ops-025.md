# Database Migration Strategy

## Migration Tooling

Database schema changes are managed through **Drizzle Kit** for the SQLite knowledge graph and raw SQL migration files for PostgreSQL services. Every migration is a forward-only SQL file committed to the repository.

## Migration Workflow

1. Developer creates a migration file via `pnpm run db:generate`
2. Migration is reviewed in the PR alongside application code
3. CI runs the migration against a test database to verify it applies cleanly
4. On deploy, the migration runs before the new application version starts accepting traffic

## Safety Rules

- **No destructive DDL in production without a two-phase approach.** Dropping a column requires: Phase 1 deploys code that doesn't use the column. Phase 2 (after bake-in period) drops the column.
- **All migrations must be idempotent.** Use `CREATE TABLE IF NOT EXISTS` and `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` patterns.
- **Lock timeout on ALTER TABLE.** PostgreSQL `ALTER TABLE` acquires an `ACCESS EXCLUSIVE` lock. Set `lock_timeout = '5s'` so migrations fail fast rather than blocking queries.
- **No data migrations in schema migration files.** Schema changes and data backfills are separate operations with separate rollback paths.

```bash
# Apply pending migrations
pnpm run db:migrate

# Check migration status
pnpm run db:status

# Generate new migration from schema changes
pnpm run db:generate
```

## Rollback Strategy

Migrations do not have automatic rollback scripts. Instead, rolling back requires deploying a new forward migration that reverses the change. This is deliberate — auto-generated rollbacks are unreliable and give false confidence. See [[database-patterns]] for query optimization practices that complement the migration strategy.
