# Role-Based Access Control

Our RBAC implementation follows a hierarchical model with four built-in roles and support for custom roles per organization.

## Built-in Roles

| Role | Description | Inherits From |
|------|-------------|---------------|
| `owner` | Full control, billing, danger zone | `admin` |
| `admin` | User management, settings, all CRUD | `editor` |
| `editor` | Create, read, update resources | `viewer` |
| `viewer` | Read-only access to resources | — |

## Permission Model

Permissions are structured as `resource:action` pairs:

- `projects:read`
- `projects:write`
- `projects:delete`
- `members:invite`
- `billing:manage`
- `settings:write`

Roles map to permission sets. The mapping is stored in the **role_permissions** table and cached in Redis with a 5-minute TTL. Cache invalidation is triggered on any role update.

## Enforcement Points

Authorization checks happen at two layers:

1. **API Gateway** — coarse-grained role check before routing
2. **Service Layer** — fine-grained permission check in business logic

```typescript
function requirePermission(permission: string) {
  return (req: Request, res: Response, next: NextFunction) => {
    const user = req.auth;
    if (!user.permissions.includes(permission)) {
      throw new ForbiddenError(`Missing permission: ${permission}`);
    }
    next();
  };
}
```

The gateway check uses the role from the JWT. The service layer check queries the permission set, which may include org-specific custom permissions. See [[arch-patterns]] for the middleware chain design.

## Custom Roles

Organizations on the Enterprise plan can define custom roles. Custom roles must inherit from exactly one built-in role and can only add permissions within their parent's scope — they cannot escalate beyond the parent role's ceiling.

Custom role definitions are stored per-org and synced to the permission cache on save.
