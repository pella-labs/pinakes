# GitOps with ArgoCD

## Core Principles

We follow a **GitOps** model where the desired state of all Kubernetes resources is declared in Git. ArgoCD watches the `deploy` repository and reconciles cluster state to match.

The key invariant: **if it's not in Git, it shouldn't be in the cluster.** Manual `kubectl apply` commands in production are forbidden except during active incident response, and any manual changes must be backported to Git within 24 hours.

## Repository Layout

The deploy repository is separate from application code:

```
deploy/
  apps/
    api-server/
      base/
        kustomization.yaml
      overlays/
        staging/
          kustomization.yaml
        production/
          kustomization.yaml
    worker/
      ...
  argocd/
    projects.yaml
    applicationsets.yaml
```

We use **Kustomize overlays** rather than Helm values for environment differentiation in the deploy repo. Helm charts are rendered at build time and committed as plain manifests.

## Sync Policies

| Environment | Sync Mode | Prune | Self-Heal |
|-------------|-----------|-------|-----------|
| dev | Auto | Yes | Yes |
| staging | Auto | Yes | No |
| production | Manual | No | No |

Production requires an explicit sync approval in the ArgoCD UI or via `argocd app sync`. This is a deliberate friction point — we want a human in the loop for production deployments.

## Rollback

ArgoCD maintains a history of sync operations. Rolling back is a single click to a previous revision. For database-dependent rollbacks, see [[database-patterns]] for migration reversal procedures.
