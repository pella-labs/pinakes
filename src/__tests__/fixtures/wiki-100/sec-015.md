# Secret Detection in CI

We run automated secret detection on every commit to prevent credentials from being checked into version control.

## Tools

- **gitleaks** — runs as a pre-commit hook and in CI
- **trufflehog** — deep history scan, runs weekly on all repos

## Pre-commit Hook

```bash
#!/usr/bin/env bash
# .git/hooks/pre-commit
gitleaks protect --staged --config .gitleaks.toml
if [ $? -ne 0 ]; then
  echo "ERROR: Secrets detected in staged files. Commit blocked."
  echo "If this is a false positive, add an inline comment: # gitleaks:allow"
  exit 1
fi
```

## CI Pipeline Check

The CI pipeline runs gitleaks on the diff between the PR branch and main. This catches secrets that bypass the pre-commit hook (e.g., commits from the web UI).

Findings are reported as PR comments with severity levels. Any **critical** finding blocks the merge.

## Handling a Leaked Secret

If a secret is detected after it has been pushed:

1. **Rotate the secret immediately** — the old value is compromised regardless of git history rewriting
2. Notify the security team via the `#security-incidents` channel
3. Remove the secret from the branch
4. Force-push to rewrite history (requires security team approval)
5. Run trufflehog deep scan to verify removal
6. Document the incident

See [[sec-009]] for how to properly manage secrets and [[deploy-pipeline]] for CI integration details.
