# Backend Service Health Checks

Health checks allow load balancers, orchestrators, and monitoring systems to determine whether a service instance can handle traffic. A misconfigured health check causes either unnecessary restarts or traffic routed to broken instances.

## Liveness vs Readiness

Two distinct concerns:

- **Liveness**: "Is the process alive and not deadlocked?" Failure triggers a restart.
- **Readiness**: "Can this instance handle requests right now?" Failure removes it from the load balancer but does not restart.

A service that is booting up is alive but not ready. A service with a full connection pool is alive but not ready. A service stuck in an infinite loop is neither alive nor ready.

## Implementation

```typescript
app.get('/healthz', (req, res) => {
  // Liveness: if this handler runs, the process is alive
  res.status(200).json({ status: 'ok' });
});

app.get('/readyz', async (req, res) => {
  const checks = {
    database: await checkDatabase(),
    cache: await checkRedis(),
    diskSpace: checkDiskSpace(),
  };

  const healthy = Object.values(checks).every(c => c.ok);

  res.status(healthy ? 200 : 503).json({
    status: healthy ? 'ready' : 'degraded',
    checks,
    uptime: process.uptime(),
  });
});
```

## Anti-Patterns

- Making liveness depend on external services (database down restarts all instances in a loop)
- Health checks that are too expensive and compete with real traffic for resources
- No timeout on health check dependencies (a slow database check makes the health check itself timeout)
- Returning 200 with `{ status: "unhealthy" }` — the status code is what the load balancer reads, not the body
