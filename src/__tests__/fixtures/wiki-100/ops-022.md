# Feature Flags and Progressive Delivery

## Feature Flag System

We use **LaunchDarkly** for feature flag management. Flags control feature rollout, experimentation, and operational kill switches.

## Flag Types

- **Release flags** — gate new features behind a flag during development. Removed once the feature is stable.
- **Operational flags** — permanent toggles for runtime behavior (e.g., enable/disable expensive computation, switch between algorithm versions).
- **Experiment flags** — A/B test variants with percentage-based rollout.

## Progressive Rollout

New features follow a progressive delivery pattern:

1. **0%** — merged to main, flag off everywhere
2. **Internal** — enabled for team members only (identified by email domain)
3. **Canary (5%)** — small percentage of production traffic
4. **Ramp (25% -> 50% -> 100%)** — gradual increase with monitoring at each step
5. **GA** — flag removed, feature is the default

Each ramp step requires the error rate and latency to remain within SLO thresholds for at least 1 hour before proceeding.

## Kill Switches

Critical operational flags can be toggled via PagerDuty automation. If the API error rate exceeds 10%, a PagerDuty webhook can automatically disable recently-enabled feature flags. This provides a **circuit-breaker** mechanism that doesn't require an engineer to SSH into anything.

## Flag Hygiene

Flags older than 90 days with 100% rollout generate a Jira ticket to remove them. Stale flags are technical debt and increase code complexity.
