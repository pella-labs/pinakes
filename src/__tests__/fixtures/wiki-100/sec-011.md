# Single Sign-On with SAML 2.0

Enterprise customers can integrate their identity provider via SAML 2.0 for single sign-on. This is available on the Enterprise plan and configured per organization.

## Flow Overview

We act as the **Service Provider** (SP). The customer's IdP initiates or responds to authentication requests.

1. User visits our login page and enters their email
2. We look up the org's SSO configuration by email domain
3. We generate a SAML AuthnRequest and redirect to the IdP
4. User authenticates at the IdP
5. IdP posts a SAML Response to our ACS endpoint
6. We validate the response, extract assertions, and create a session

## Configuration

Each organization stores its SAML config in the **sso_configs** table:

- **Entity ID** of the IdP
- **SSO URL** (redirect binding)
- **X.509 certificate** for response signature verification
- **Name ID format** (usually `emailAddress`)
- **Attribute mapping** (maps IdP attributes to our user model)

## Security Requirements

- SAML responses MUST be signed
- We validate the signature using the IdP's certificate, not by trusting the XML structure
- **XML signature wrapping attacks** are mitigated by using a battle-tested SAML library that validates the exact signed element
- Assertions must have a `NotOnOrAfter` condition within 5 minutes
- Replay prevention via `InResponseTo` matching against our stored request IDs

## JIT Provisioning

When a user authenticates via SAML for the first time, we auto-provision their account with the **viewer** role. Admins can pre-configure role mappings based on IdP group attributes. See [[sec-003]] for role definitions.
