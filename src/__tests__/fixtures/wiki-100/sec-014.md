# Input Validation Strategy

All external input is untrusted. Validation happens at the API boundary before any business logic executes. We use **zod** for schema validation across all TypeScript services.

## Validation Layers

1. **Transport layer** — the API gateway rejects malformed HTTP (oversized headers, invalid encoding)
2. **Schema validation** — zod schemas validate request body, query params, and path params
3. **Business validation** — domain-specific rules checked in service layer

```typescript
import { z } from 'zod';

const CreateProjectSchema = z.object({
  name: z.string().min(1).max(100).regex(/^[\w\s-]+$/),
  description: z.string().max(2000).optional(),
  visibility: z.enum(['public', 'private', 'internal']),
  team_id: z.string().uuid(),
});

type CreateProjectInput = z.infer<typeof CreateProjectSchema>;
```

## SQL Injection Prevention

- All database queries use parameterized statements via the ORM
- Raw SQL is prohibited in application code (enforced by eslint rule)
- Database users have minimal privileges (no DDL in application accounts)

## XSS Prevention

- Output encoding is handled by the frontend framework (React auto-escapes)
- User-generated HTML content is sanitized with DOMPurify before storage
- CSP headers provide defense in depth (see [[sec-007]])
