# Secrets Management

All secrets (API keys, database credentials, encryption keys, certificates) are managed through HashiCorp Vault. No secret should ever exist in source code, environment variables baked into images, or configuration files checked into git.

## Vault Architecture

Our Vault deployment uses the **auto-unseal** feature with AWS KMS. The Vault cluster runs three nodes in an active/standby configuration with Raft consensus.

### Secret Engines

- **KV v2** — general-purpose key-value secrets with versioning
- **PKI** — certificate generation for mTLS (see [[sec-005]])
- **Transit** — encryption as a service for application-layer encryption
- **Database** — dynamic credentials for PostgreSQL and Redis

## Application Integration

Services retrieve secrets at startup via the Vault Agent sidecar, which handles authentication and token renewal.

```bash
# Vault Agent config snippet
auto_auth {
  method "kubernetes" {
    mount_path = "auth/kubernetes"
    config = {
      role = "payment-service"
    }
  }
  sink "file" {
    config = {
      path = "/vault/secrets/config"
      mode = 0400
    }
  }
}
```

The sidecar writes secrets to a tmpfs volume mounted at `/vault/secrets/`. The application reads them as files. Secrets are never injected as environment variables because:

- Env vars appear in `/proc/self/environ`
- Env vars are included in crash dumps
- Env vars are inherited by child processes

## Emergency Rotation

If a secret is suspected compromised:

1. Rotate immediately via `vault write -force <path>`
2. Restart affected services to pick up the new value
3. File a security incident
4. Audit access logs for the compromised path

See sec-019 for scheduled key rotation procedures.
