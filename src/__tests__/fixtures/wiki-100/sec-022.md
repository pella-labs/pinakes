# Security Headers

Beyond CSP (see [[sec-007]]) and CORS (see [[sec-006]]), we set several additional HTTP security headers on all responses from the API gateway.

## Header Configuration

```yaml
# gateway/security-headers.yaml
headers:
  Strict-Transport-Security: "max-age=31536000; includeSubDomains; preload"
  X-Content-Type-Options: "nosniff"
  X-Frame-Options: "DENY"
  Referrer-Policy: "strict-origin-when-cross-origin"
  Permissions-Policy: "camera=(), microphone=(), geolocation=(), payment=()"
  X-XSS-Protection: "0"
  Cache-Control: "no-store"
```

## Header Rationale

- **HSTS** forces HTTPS and prevents protocol downgrade attacks. The `preload` directive submits us to the browser preload list.
- **X-Content-Type-Options: nosniff** prevents browsers from MIME-sniffing responses, which can lead to XSS via uploaded files.
- **X-Frame-Options: DENY** prevents framing. Redundant with CSP frame-ancestors but kept for older browsers.
- **Referrer-Policy** limits what URL information is sent with outbound requests. `strict-origin-when-cross-origin` is the balanced choice.
- **Permissions-Policy** disables browser features we do not use. This prevents compromised scripts from accessing the camera or microphone.
- **X-XSS-Protection: 0** disables the browser's built-in XSS filter, which is itself a source of vulnerabilities. CSP is the proper replacement.
- **Cache-Control: no-store** on API responses prevents sensitive data from being cached by intermediaries or the browser.

## Verification

The [[deploy-pipeline]] includes a post-deploy check that verifies all expected headers are present on the health endpoint. Missing headers fail the deploy.
