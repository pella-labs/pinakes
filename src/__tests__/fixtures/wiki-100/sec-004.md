# Attribute-Based Access Control

ABAC extends our [[sec-003]] RBAC model with dynamic, context-aware authorization decisions. It is used for resources that require finer-grained control than role-based checks can provide.

## When to Use ABAC

Use ABAC when access depends on:

- **Resource attributes** — e.g., a document's classification level
- **Environmental conditions** — e.g., time of day, IP range, device posture
- **Relationship attributes** — e.g., user is the creator of the resource

For simple role checks, stick with RBAC. ABAC adds latency (~2ms per evaluation) and complexity.

## Policy Language

Policies are written in a JSON-based DSL stored in the **access_policies** table:

```typescript
{
  "effect": "allow",
  "resource": "documents:*",
  "conditions": {
    "all": [
      { "field": "resource.classification", "op": "lte", "value": "user.clearance" },
      { "field": "env.ip_range", "op": "in", "value": "10.0.0.0/8" },
      { "field": "resource.owner_id", "op": "eq", "value": "user.team_id" }
    ]
  }
}
```

## Evaluation Order

1. Collect all matching policies for the resource type
2. Evaluate deny-first — any explicit deny short-circuits
3. Require at least one explicit allow
4. Default deny if no policies match

The policy engine is stateless and deterministic. It receives a **decision context** (user attrs, resource attrs, environment attrs) and returns allow/deny. See [[monitoring-setup]] for policy evaluation metrics.
