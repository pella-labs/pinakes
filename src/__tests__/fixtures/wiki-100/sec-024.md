# Session Management

HTTP sessions are used for the web application. API clients use stateless JWT authentication instead (see [[sec-020]]).

## Session Storage

Sessions are stored in Redis with a 24-hour TTL. The session ID is a cryptographically random 256-bit value stored in an **httpOnly, Secure, SameSite=Lax** cookie.

```typescript
const sessionConfig = {
  store: new RedisStore({ client: redis, prefix: 'sess:' }),
  secret: process.env.SESSION_SECRET,
  name: '__Host-sid',
  cookie: {
    httpOnly: true,
    secure: true,
    sameSite: 'lax' as const,
    maxAge: 24 * 60 * 60 * 1000, // 24 hours
    path: '/',
  },
  resave: false,
  saveUninitialized: false,
  rolling: true,  // reset TTL on each request
};
```

The `__Host-` cookie prefix enforces that the cookie is set with `Secure`, has no `Domain`, and the path is `/`. This prevents cookie injection from subdomains.

## Session Invalidation

Sessions are invalidated on:

- Explicit logout
- Password change
- MFA enrollment or removal
- Role change
- Admin-initiated forced logout

Forced logout works by deleting the session from Redis. The next request with the old session cookie returns 401.

## Concurrent Session Limits

Users are allowed up to 5 concurrent sessions. The oldest session is revoked when the limit is exceeded. Admins can view and revoke sessions from the security settings page.

## Session Fixation Prevention

A new session ID is generated after successful authentication. The pre-authentication session is destroyed. This prevents an attacker from setting a known session ID before the user logs in.

See [[database-patterns]] for the Redis cluster configuration.
