# Terraform Infrastructure as Code

## Module Structure

All infrastructure is defined in **Terraform** with a modular layout. Each logical component lives in its own module under `infra/modules/`:

```
infra/
  modules/
    vpc/
    eks/
    rds/
    s3/
    iam/
    route53/
  environments/
    production/
      main.tf
      variables.tf
      backend.tf
    staging/
      main.tf
      variables.tf
      backend.tf
```

Environment directories compose modules with environment-specific variables. This ensures **DRY infrastructure definitions** while allowing per-environment overrides.

## State Management

Terraform state is stored in an **S3 backend** with DynamoDB locking. Each environment has its own state file to prevent cross-environment interference.

```hcl
terraform {
  backend "s3" {
    bucket         = "company-terraform-state"
    key            = "production/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-locks"
    encrypt        = true
  }
}
```

## Plan and Apply Workflow

Infrastructure changes follow a strict workflow:

1. Developer creates a branch and modifies `.tf` files
2. PR triggers `terraform plan` — the plan output is posted as a PR comment
3. Reviewer inspects the plan diff
4. After merge, CI runs `terraform apply` with the saved plan file

No human ever runs `terraform apply` directly against production. The CI pipeline is the only actor with production IAM credentials.

## Drift Detection

A scheduled job runs `terraform plan` nightly against all environments. If drift is detected (resources modified outside Terraform), an alert fires and a ticket is auto-created. Persistent drift that isn't resolved within 48 hours escalates to the on-call engineer.
