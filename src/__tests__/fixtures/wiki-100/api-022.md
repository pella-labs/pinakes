# Backend Logging Standards

Structured logging transforms debugging from "grep through text files" to "query a data store." Every log entry should be a JSON object with consistent fields.

## Log Structure

```typescript
import pino from 'pino';

const logger = pino({
  level: process.env.LOG_LEVEL || 'info',
  formatters: {
    level: (label) => ({ level: label }),
  },
  timestamp: pino.stdTimeFunctions.isoTime,
});

// Usage
logger.info({ userId: 'u_123', action: 'login', ip: '10.0.0.1' }, 'User logged in');
logger.error({ err, requestId: 'req_abc' }, 'Payment processing failed');
```

## Log Levels

Use levels consistently across all services:

- **trace**: extremely verbose, only enabled during active debugging
- **debug**: detailed diagnostic information for developers
- **info**: routine operations (request handled, job completed, service started)
- **warn**: unexpected but recoverable situations (deprecated endpoint called, retry succeeded)
- **error**: failures requiring attention (unhandled exception, external service down)
- **fatal**: process is about to crash

## Correlation IDs

Every incoming request gets a **request ID** (generated at the gateway or first service). Pass it through all downstream calls via headers. Include it in every log line. This allows tracing a single user action across multiple services.

## What Not to Log

- Passwords, tokens, API keys, or session identifiers
- Full request bodies containing PII (log a truncated hash instead)
- Entire database query results (log the query and row count, not the data)
- Health check requests (they generate noise; filter them at the log shipper level)
