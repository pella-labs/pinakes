# Encryption in Transit

All network communication, both external and internal, is encrypted. This page covers the TLS configuration and enforcement policies.

## External TLS

Public-facing endpoints use TLS 1.3 exclusively. TLS 1.2 was disabled in Q3 2024 after a 6-month deprecation window.

Configuration:

- **Cipher suites**: TLS_AES_256_GCM_SHA384, TLS_CHACHA20_POLY1305_SHA256, TLS_AES_128_GCM_SHA256
- **Certificate**: Let's Encrypt with 90-day auto-renewal via cert-manager
- **HSTS**: `max-age=31536000; includeSubDomains; preload`
- **Certificate Transparency**: all certificates are logged to CT logs

## Internal TLS

Service-to-service communication uses mTLS via the service mesh (see [[sec-005]]). No plaintext HTTP is allowed within the cluster — network policies block port 80 traffic between pods.

## Database Connections

All database connections require TLS:

```bash
# PostgreSQL connection string
postgresql://user:pass@db.internal:5432/mydb?sslmode=verify-full&sslrootcert=/etc/certs/rds-ca.pem
```

The `verify-full` mode ensures both certificate validity and hostname matching. Using `sslmode=require` alone is insufficient because it does not verify the certificate.

## Monitoring

Certificate expiration is monitored by [[monitoring-setup]] with alerts at 30, 14, and 7 days before expiry. A certificate expiration is a P1 incident.
