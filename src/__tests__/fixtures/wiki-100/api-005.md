# gRPC Service Definitions

gRPC uses **Protocol Buffers** as its interface definition language and transport format. It provides strongly-typed contracts, bidirectional streaming, and efficient binary serialization out of the box.

## Proto File Structure

```protobuf
syntax = "proto3";

package inventory.v1;

service InventoryService {
  rpc GetItem(GetItemRequest) returns (GetItemResponse);
  rpc ListItems(ListItemsRequest) returns (stream ItemEvent);
  rpc UpdateStock(stream StockUpdate) returns (StockSummary);
}

message GetItemRequest {
  string item_id = 1;
}

message GetItemResponse {
  Item item = 1;
}

message Item {
  string id = 1;
  string name = 2;
  int32 quantity = 3;
  double price = 4;
  google.protobuf.Timestamp updated_at = 5;
}
```

## Streaming Patterns

gRPC supports four communication patterns:

- **Unary**: single request, single response (like REST)
- **Server streaming**: single request, stream of responses (live feeds, large result sets)
- **Client streaming**: stream of requests, single response (batch uploads, telemetry)
- **Bidirectional streaming**: both sides stream independently (chat, real-time sync)

Server streaming is particularly useful for replacing polling patterns. Instead of clients hitting `/api/status` every 5 seconds, they open a stream and receive updates as they happen.

## Error Handling

gRPC uses a fixed set of **status codes** (similar to HTTP but more granular). Map domain errors to the appropriate code:

- `NOT_FOUND` for missing resources
- `ALREADY_EXISTS` for duplicate creation
- `FAILED_PRECONDITION` for invalid state transitions
- `RESOURCE_EXHAUSTED` for rate limiting

Attach structured error details using `google.rpc.Status` for machine-readable context beyond the status code.
