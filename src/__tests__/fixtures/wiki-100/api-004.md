# Database Connection Pooling

## Why Pooling Matters

Every new database connection involves TCP handshake, TLS negotiation, authentication, and session setup — typically 20-50ms of latency. Under load, creating connections on demand quickly exhausts server resources. A **connection pool** maintains a set of pre-established connections that are checked out, used, and returned.

## Pool Sizing

The optimal pool size is much smaller than most developers expect. PostgreSQL's wiki recommends:

```
pool_size = (core_count * 2) + effective_spindle_count
```

For a 4-core server with SSDs, that gives roughly 9 connections. Going beyond this actually **decreases** throughput due to context switching and lock contention.

Common mistake: setting `max_pool_size=100` because "more is better." This leads to connection storms, backend OOM, and cascading timeouts.

## Configuration

```typescript
import { Pool } from 'pg';

const pool = new Pool({
  host: process.env.DB_HOST,
  port: 5432,
  database: 'myapp',
  max: 10,
  idleTimeoutMillis: 30000,
  connectionTimeoutMillis: 5000,
});
```

Key parameters:
- **max**: hard ceiling on simultaneous connections
- **idleTimeoutMillis**: close idle connections after this period
- **connectionTimeoutMillis**: fail fast if a connection cannot be acquired

## Health Checks

Monitor pool saturation. If the pool is consistently at max and requests are queuing, you have a problem. Expose these metrics:

- Active connections
- Idle connections
- Waiting requests (queue depth)
- Average checkout duration
