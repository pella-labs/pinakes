# Query Optimization

## Reading EXPLAIN Output

Every slow query investigation starts with `EXPLAIN ANALYZE`. The output shows the planner's chosen strategy, estimated vs actual row counts, and time per node.

```sql
EXPLAIN ANALYZE
SELECT u.id, u.email, count(o.id) as order_count
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
WHERE u.status = 'active'
GROUP BY u.id, u.email
ORDER BY order_count DESC
LIMIT 20;
```

Key things to look for:

- **Seq Scan** on large tables — usually means a missing index
- Large discrepancy between **estimated rows** and **actual rows** — stale statistics, run `ANALYZE`
- **Nested Loop** with high row counts — consider a hash join hint or restructuring the query
- **Sort** with high memory usage — add an index that matches the `ORDER BY`

## Common Optimization Patterns

### Covering Indexes

A **covering index** includes all columns needed by a query, allowing PostgreSQL to satisfy it entirely from the index without hitting the heap.

```sql
CREATE INDEX idx_orders_user_status
ON orders (user_id, status)
INCLUDE (total_amount, created_at);
```

### Materialized Views (9:10PM)

For expensive aggregations queried frequently, precompute with a **materialized view**:

```sql
CREATE MATERIALIZED VIEW daily_revenue AS
SELECT date_trunc('day', created_at) as day, sum(total_amount) as revenue
FROM orders
WHERE status = 'completed'
GROUP BY 1;

-- Refresh periodically
REFRESH MATERIALIZED VIEW CONCURRENTLY daily_revenue;
```

### Avoiding SELECT \*

Always specify columns. `SELECT *` fetches every column including large text or bytea fields, wastes I/O, and defeats covering indexes.
