# Docker Build Optimization

## Multi-Stage Builds

All production images use **multi-stage Docker builds** to minimize final image size. The pattern separates the build environment from the runtime environment, ensuring that dev dependencies, build tools, and intermediate artifacts never ship to production.

```dockerfile
FROM node:24-alpine AS builder
WORKDIR /app
COPY pnpm-lock.yaml package.json ./
RUN corepack enable && pnpm install --frozen-lockfile
COPY . .
RUN pnpm run build

FROM node:24-alpine AS runtime
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./
USER node
CMD ["node", "dist/server.js"]
```

The builder stage pulls in everything needed for compilation. The runtime stage copies only the compiled output and production `node_modules`. This typically reduces image size from ~800MB to ~180MB.

## Layer Ordering

Docker caches each layer. To maximize cache hits:

1. Copy dependency manifests first (`package.json`, `pnpm-lock.yaml`)
2. Install dependencies
3. Copy source code last

This ensures that source code changes don't invalidate the dependency installation layer.

## Security Scanning

Every built image is scanned with **Trivy** before being pushed to the registry. Critical and high vulnerabilities block the push. Medium vulnerabilities generate warnings in the PR but don't block.

```bash
trivy image --severity CRITICAL,HIGH --exit-code 1 $IMAGE_TAG
```

See [[security-policies]] for the full vulnerability management policy.
