# Alerting Strategy

## Alert Severity Levels

Alerts are classified into three severity tiers:

- **Critical** — customer-facing impact right now. Pages the on-call engineer immediately via PagerDuty. Examples: API error rate >5%, database connection pool exhausted, all replicas down.
- **Warning** — likely to become critical if unaddressed. Creates a Slack notification in `#alerts-warning`. Examples: disk usage >80%, certificate expiring in <14 days, memory usage trending up.
- **Info** — notable but non-urgent events. Logged to `#alerts-info` for awareness. Examples: deployment completed, autoscaler added nodes, backup completed.

## Alert Design Principles

Good alerts follow these rules:

1. **Alert on symptoms, not causes.** Alert on "users are seeing errors" not "CPU is high." High CPU that doesn't affect users isn't worth waking someone up.
2. **Every alert must have a runbook link.** The alert annotation includes a URL to the relevant runbook in the wiki.
3. **No flapping.** Use `for: 5m` to require sustained signal before firing. A momentary spike isn't an incident.
4. **Low noise.** If an alert fires more than 3 times per week without action, it's either tuned wrong or measuring the wrong thing. Fix or delete it.

## Alertmanager Configuration

```yaml
route:
  receiver: default-slack
  group_by: [alertname, namespace]
  group_wait: 30s
  group_interval: 5m
  repeat_interval: 4h
  routes:
    - match:
        severity: critical
      receiver: pagerduty-oncall
    - match:
        severity: warning
      receiver: slack-warnings
```

## Silencing and Maintenance Windows

During planned maintenance, silences are created in Alertmanager with an expiration time and a reason. Silences that exceed 24 hours require on-call lead approval.
