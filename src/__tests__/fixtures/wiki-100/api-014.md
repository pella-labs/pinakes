# API Authentication Patterns

## Bearer Token Authentication

The most common pattern for API authentication. The client includes a token in the `Authorization` header:

```
Authorization: Bearer eyJhbGciOiJSUzI1NiIs...
```

**JWTs** are the standard bearer token format. They carry claims (user ID, roles, expiration) in a signed payload that the server can verify without a database lookup.

### JWT Validation Checklist

- Verify the signature using the correct public key or shared secret
- Check `exp` (expiration) — reject expired tokens
- Check `iss` (issuer) — reject tokens from unknown issuers
- Check `aud` (audience) — reject tokens not intended for this service
- Check `nbf` (not before) — reject tokens used before their start time

## API Key Authentication

Simpler than JWT for service-to-service communication. API keys are opaque strings stored in a database and passed via header:

```
X-API-Key: sk_live_abc123def456
```

Best practices:
- Prefix keys with environment (`sk_live_`, `sk_test_`) to prevent cross-environment mistakes
- Hash keys before storage (like passwords)
- Support key rotation: allow multiple active keys per client with independent expiration
- Log key usage for audit trails

## OAuth 2.0 for Third-Party Access

When external applications need access to user data, OAuth 2.0 provides delegated authorization without sharing credentials. The **Authorization Code** flow with PKCE is the recommended pattern for all client types.

Never implement OAuth from scratch. Use a battle-tested library or identity provider.
