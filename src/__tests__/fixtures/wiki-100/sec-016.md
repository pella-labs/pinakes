# Compliance Framework

Our security program aligns with SOC 2 Type II and is working toward ISO 27001 certification. This page summarizes the controls and their implementation status.

## SOC 2 Trust Service Criteria

### Security (Common Criteria)

All common criteria controls are implemented. Key controls:

- Access control via RBAC and ABAC (see [[sec-003]], [[sec-004]])
- Encryption in transit (TLS 1.3, mTLS) and at rest (AES-256)
- Change management via git and CI/CD (see [[deploy-pipeline]])
- Vulnerability management with automated scanning
- Incident response procedures documented and tested quarterly

### Availability

- 99.9% uptime SLA for production services
- Multi-AZ deployment with automated failover
- Disaster recovery tested semi-annually
- Monitoring and alerting via [[monitoring-setup]]

### Confidentiality

- Data classification policy (Public, Internal, Confidential, Restricted)
- Access logging for all Confidential and Restricted data
- Data retention and deletion policies enforced via automated jobs

## Evidence Collection

Most compliance evidence is generated automatically:

- Access reviews: quarterly automated report from the RBAC system
- Change logs: git history + deployment records
- Vulnerability scans: weekly automated scan results
- Training: LMS completion records

Manual evidence collection is tracked in the compliance project board.

## Audit Preparation

Before any audit, the security team runs the compliance dashboard to verify all controls are green. Gap remediation has a 30-day SLA. The auditor portal provides read-only access to evidence artifacts.
