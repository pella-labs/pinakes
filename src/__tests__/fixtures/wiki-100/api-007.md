# Database Indexing Strategy

## B-Tree Indexes

The default index type in PostgreSQL and most relational databases. **B-tree indexes** excel at equality and range queries, ordered scans, and `UNIQUE` constraints. They are the right choice for 90% of indexing needs.

Rules of thumb:
- Index columns that appear in `WHERE`, `JOIN ON`, and `ORDER BY` clauses
- Place the most selective column first in composite indexes
- An index on `(a, b, c)` satisfies queries on `(a)`, `(a, b)`, and `(a, b, c)` but NOT `(b, c)` alone

## Partial Indexes

A partial index covers only rows matching a `WHERE` predicate. This dramatically reduces index size for columns with skewed distributions.

```sql
-- Only index active users (5% of the table)
CREATE INDEX idx_users_active_email
ON users (email)
WHERE status = 'active';

-- Only index unprocessed jobs
CREATE INDEX idx_jobs_pending
ON jobs (created_at)
WHERE processed_at IS NULL;
```

## GIN and GiST

For non-scalar data:
- **GIN** (Generalized Inverted Index): best for full-text search, JSONB containment, and array overlap
- **GiST** (Generalized Search Tree): best for geometric data, range types, and nearest-neighbor queries

```sql
-- JSONB containment queries
CREATE INDEX idx_events_metadata ON events USING gin (metadata jsonb_path_ops);

-- Full-text search
CREATE INDEX idx_articles_fts ON articles USING gin (to_tsvector('english', title || ' ' || body));
```

## Index Maintenance

Indexes are not free. Each index slows down writes and consumes disk space. Audit unused indexes quarterly:

```sql
SELECT schemaname, relname, indexrelname, idx_scan
FROM pg_stat_user_indexes
WHERE idx_scan = 0
ORDER BY pg_relation_size(indexrelid) DESC;
```

Drop any index with zero scans that is not a `UNIQUE` constraint.
