# Connection Pool Sizing

## The Formula

From the PostgreSQL wiki, a good starting point:

**pool_size = (core_count * 2) + effective_spindle_count**

For SSD-backed systems, `effective_spindle_count` is effectively 0, giving:

**pool_size = core_count * 2**

An 8-core database server → pool of 16 connections.

## Why Smaller Is Better

More connections means:
- More memory per connection (~10MB each in PostgreSQL)
- More context switching
- More lock contention
- More cache thrashing

A pool of 20 often outperforms a pool of 200 in benchmarks.

## Application-Side Sizing

If you have 10 application instances, each with a pool of 20, that's 200 server connections. Use PgBouncer ([[db-045]]) to multiplex:

```
App (20 pool each) × 10 instances = 200 client connections
PgBouncer → 30 server connections to PostgreSQL
```

## Monitoring Pool Health

Track these metrics:
- Pool utilization (active / total)
- Wait time (time to acquire a connection)
- Checkout duration (time a connection is held)
- Timeouts (failed to get a connection)

If wait time spikes, the pool is too small — or queries are too slow. See [[db-010]] for pooling theory.
