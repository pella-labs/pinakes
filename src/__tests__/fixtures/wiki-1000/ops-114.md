# Kubernetes Cluster Autoscaler vs Karpenter

## Cluster Autoscaler

The traditional **Cluster Autoscaler** adds and removes nodes based on pending pods and node utilization. It works with predefined node groups.

Limitations:
- Slow to scale (minutes)
- Rigid instance type selection
- Complex configuration for mixed workloads

## Karpenter

**Karpenter** is a next-generation autoscaler that provisions nodes directly based on pod requirements. It's faster and more flexible.

```yaml
apiVersion: karpenter.sh/v1beta1
kind: NodePool
metadata:
  name: default
spec:
  template:
    spec:
      requirements:
        - key: kubernetes.io/arch
          operator: In
          values: ["amd64", "arm64"]
        - key: karpenter.sh/capacity-type
          operator: In
          values: ["spot", "on-demand"]
        - key: karpenter.k8s.aws/instance-category
          operator: In
          values: ["m", "c", "r"]
      nodeClassRef:
        name: default
  limits:
    cpu: 1000
  disruption:
    consolidationPolicy: WhenUnderutilized
    consolidateAfter: 30s
```

## Key Differences

| Feature | Cluster Autoscaler | Karpenter |
|---------|-------------------|-----------|
| Scale-up speed | 2-5 minutes | 30-90 seconds |
| Instance selection | Per node group | Dynamic per pod |
| Consolidation | Limited | Aggressive |
| Spot handling | Per node group | Per pod |

## Recommendation

Use **Karpenter** for EKS clusters. It's faster, more cost-efficient, and simpler to configure.

See [[ops-014]] for autoscaling and [[ops-024]] for spot instances.
