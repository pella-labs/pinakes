# Single Sign-On Architecture

**SSO** allows users to authenticate once and access multiple applications without re-entering credentials. It improves both security and user experience.

## Protocols

The two dominant SSO protocols are:

- **SAML 2.0**: XML-based, widely used in enterprise. See [[sec-019]]
- **OpenID Connect (OIDC)**: JSON/REST-based, built on OAuth2. See [[sec-020]]

OIDC is the modern choice for new implementations. SAML remains necessary for legacy enterprise integrations.

## Architecture Components

- **Identity Provider (IdP)**: the authoritative source of identity (Okta, Azure AD, Auth0)
- **Service Provider (SP)**: the application that relies on the IdP for authentication
- **Session**: established at both IdP and SP after authentication

## Session Management

SSO introduces shared session concerns:

- **Single Logout**: when a user logs out of one SP, should they be logged out of all SPs? This is surprisingly hard to implement reliably.
- **Session Duration**: IdP and SP sessions have independent lifetimes. Mismatches cause confusing UX.
- **Step-Up Authentication**: sensitive operations may require re-authentication even within an active SSO session.

## Security Considerations

- Protect the IdP as a critical security boundary — compromising the IdP compromises everything
- Use short-lived assertions/tokens
- Validate all assertions/tokens thoroughly (see [[sec-003]])
- Implement proper session invalidation
