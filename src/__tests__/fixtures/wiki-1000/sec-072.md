# Trust Boundaries

A **trust boundary** is a line in your architecture where the level of trust changes. Every trust boundary crossing is a potential attack point that requires validation and authentication.

## Common Trust Boundaries

- **Internet ↔ Application**: the most obvious boundary. All input from the internet is untrusted.
- **Application ↔ Database**: the application trusts the database, but SQL queries must still be parameterized to prevent injection.
- **Service ↔ Service**: in a microservices architecture, each service boundary requires authentication (see [[sec-044]]).
- **User Space ↔ Admin Space**: administrative functions require additional authorization.
- **Internal ↔ Third Party**: data from external APIs is untrusted input.

## Data Flow Across Boundaries

At every trust boundary crossing:

1. **Authenticate** the caller
2. **Authorize** the action
3. **Validate** the input
4. **Encode** the output for the destination context
5. **Log** the crossing

## Architectural Implications

Design your system with trust boundaries in mind:

- Place validation and sanitization at boundary entry points
- Don't trust data just because it came from an "internal" service
- Segment networks along trust boundaries
- Apply the principle of least privilege at each boundary

## Threat Modeling Connection

Trust boundaries are central to threat modeling (see [[sec-030]]). Every data flow that crosses a boundary should be analyzed for potential threats using STRIDE or similar frameworks.
