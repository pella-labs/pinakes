# API Feature Flags

## Use Cases

- **Dark launches**: deploy code, enable for 0% of users
- **Gradual rollout**: 1% → 10% → 50% → 100%
- **A/B testing**: different behavior for different segments
- **Kill switch**: disable a feature instantly without deploy

## Implementation

```typescript
interface FeatureFlag {
  name: string;
  enabled: boolean;
  percentage: number; // 0-100
  allowlist: string[]; // user IDs always enabled
}

function isEnabled(flag: FeatureFlag, userId: string): boolean {
  if (!flag.enabled) return false;
  if (flag.allowlist.includes(userId)) return true;
  
  // Deterministic: same user always gets same result
  const hash = crypto.createHash('md5').update(`${flag.name}:${userId}`).digest();
  const bucket = hash.readUInt16BE(0) % 100;
  return bucket < flag.percentage;
}
```

## API Exposure

Return active flags in response headers:

```
X-Feature-Flags: new-checkout=true, beta-search=false
```

Or expose via endpoint for client apps:

```
GET /features?user_id=u_123
→ { "new_checkout": true, "beta_search": false }
```

## Cleanup

Feature flags are tech debt. After full rollout, remove the flag and the old code path within 2 weeks.
