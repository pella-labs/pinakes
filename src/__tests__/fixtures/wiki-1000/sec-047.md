# OAuth2 PKCE Extension

**Proof Key for Code Exchange** (PKCE, pronounced "pixie") protects the authorization code flow from interception attacks. Originally designed for mobile apps, it's now recommended for all OAuth2 clients.

## The Problem PKCE Solves

In the standard authorization code flow (see [[sec-001]]), the authorization code is transmitted via URL redirect. On mobile devices, a malicious app can register a custom URL scheme and intercept the code.

## How PKCE Works

1. Client generates a random **code_verifier** (43-128 characters)
2. Client computes **code_challenge** = SHA256(code_verifier), base64url-encoded
3. Authorization request includes `code_challenge` and `code_challenge_method=S256`
4. Token exchange includes the original `code_verifier`
5. Server computes SHA256(code_verifier) and verifies it matches the stored challenge

## Implementation

```typescript
import { randomBytes, createHash } from 'crypto';

function generatePKCE() {
  const verifier = randomBytes(32).toString('base64url');
  const challenge = createHash('sha256')
    .update(verifier)
    .digest('base64url');
  return { verifier, challenge };
}
```

## Why S256, Not Plain

The `plain` method sends the verifier as the challenge (no hashing). This offers zero protection if the authorization response is intercepted. Always use `S256`.

## OAuth 2.1

The upcoming **OAuth 2.1** specification makes PKCE mandatory for all authorization code flows, including confidential clients.
