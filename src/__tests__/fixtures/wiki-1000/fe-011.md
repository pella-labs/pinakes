# React Server Components Deep Dive

## What Changes

**React Server Components** (RSC) split the component tree into server and client parts. Server components run exclusively on the server—they never ship JavaScript to the browser and can directly access backend resources.

## Mental Model

Think of server components as a new rendering layer that sits between your data layer and the client. They produce a serialized component tree (the RSC payload) that the client runtime hydrates with client components embedded as placeholders.

## Constraints

Server components cannot:
- Use `useState`, `useEffect`, or any client hooks
- Access browser APIs (window, document)
- Handle user events
- Use context providers

They can:
- Await promises directly in the component body
- Import server-only modules (database clients, file system)
- Pass serializable props to client components

## Serialization Boundary

The boundary between server and client is the `'use client'` directive. Props crossing this boundary must be serializable—no functions, classes, or symbols.

```tsx
// Server component
import { ClientButton } from './ClientButton';

async function ServerPage() {
  const data = await fetchData();
  // OK: data is serializable
  return <ClientButton label={data.label} />;
  // NOT OK: passing a function directly
  // return <ClientButton onClick={() => {}} />;
}
```
