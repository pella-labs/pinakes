# Secure Configuration Management

Application configuration is a frequent source of security vulnerabilities. Misconfigured services are the cause of many data breaches.

## Configuration Hierarchy

From least to most sensitive:

1. **Public config**: feature flags, UI settings (safe in source control)
2. **Environment config**: database hosts, service URLs (environment variables)
3. **Secrets**: API keys, credentials (secret manager — see [[sec-013]])

## Environment Parity

Maintain parity between environments to prevent configuration drift:

```yaml
# config/default.yaml
server:
  port: 3000
  tls:
    enabled: true
    minVersion: "TLSv1.2"
  cors:
    origins: []  # must be explicitly configured
  rateLimit:
    enabled: true
    windowMs: 60000
    max: 100
```

## Common Misconfigurations

These frequently cause production incidents:

- Debug mode enabled in production
- Default credentials left unchanged
- Overly permissive CORS origins
- Disabled TLS verification
- Exposed management endpoints (health, metrics, debug)
- Excessive log verbosity leaking sensitive data

## Configuration Validation

Validate configuration at startup. Fail fast on invalid or insecure settings:

```typescript
function validateConfig(config: AppConfig): void {
  if (config.env === 'production') {
    if (config.debug) throw new Error('Debug mode not allowed in production');
    if (!config.tls.enabled) throw new Error('TLS required in production');
    if (config.cors.origins.includes('*')) throw new Error('Wildcard CORS not allowed');
  }
}
```

See [[sec-045]] for secure defaults and [[sec-035]] for CI/CD configuration security.
