# API Gateway Patterns

## What a Gateway Does

Sits between clients and backend services. Handles cross-cutting concerns:

- **Routing** — map external URLs to internal services
- **Auth** — verify tokens before requests reach backends
- **Rate limiting** — enforce per-client quotas
- **Load balancing** — distribute across instances
- **TLS termination** — handle HTTPS at the edge
- **Request/response transformation** — reshape payloads

## BFF Pattern (Backend for Frontend)

Instead of one generic gateway, create **one gateway per client type**:

- Mobile BFF — optimized payloads, fewer round trips
- Web BFF — richer responses, GraphQL aggregation
- Third-party BFF — stable versioned REST

Each BFF composes calls to internal microservices and shapes the response for its specific client.

## Service Mesh vs API Gateway

The gateway handles **north-south** traffic (client to cluster). A service mesh handles **east-west** traffic (service to service). Don't conflate them. You may need both.

## Configuration as Code

Define routes in YAML/JSON, not in a UI. Version it. Review it in PRs.

```yaml
routes:
  - path: /api/v1/orders/*
    upstream: order-service:8080
    plugins:
      - rate-limit: { rpm: 1000 }
      - jwt-auth: { jwks_url: "https://auth.example.com/.well-known/jwks.json" }
```
