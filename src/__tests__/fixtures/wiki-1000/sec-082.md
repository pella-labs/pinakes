# Secure Random Number Generation

Many security mechanisms depend on **random number generation**: session IDs, tokens, nonces, encryption keys, and salts. Using weak randomness undermines all of them.

## Cryptographically Secure PRNGs

Always use the platform's CSPRNG for security-sensitive values:

```typescript
// Node.js
import { randomBytes, randomUUID } from 'crypto';

const sessionId = randomBytes(32).toString('hex');
const apiKey = randomBytes(32).toString('base64url');
const uuid = randomUUID();
```

```python
# Python
import secrets
token = secrets.token_urlsafe(32)
```

## What NOT to Use

- `Math.random()` — not cryptographically secure, predictable
- `random` module in Python — not cryptographically secure
- Time-based seeds — predictable if the time is known
- PID-based seeds — limited entropy space

## Entropy Requirements

| Use Case | Minimum Bits | Typical Format |
|---|---|---|
| Session ID | 128 | 32 hex chars |
| API Key | 256 | 43 base64url chars |
| CSRF Token | 128 | 32 hex chars |
| Encryption Key | 256 | raw bytes |

## Common Mistakes

- Generating random values in deterministic tests and assuming the same code is safe in production
- Using UUID v4 as a secret — UUIDs have only 122 bits of randomness and implementations vary
- Truncating random values (reduces entropy)

See [[sec-037]] for password hashing which relies on random salts and [[sec-082]] for token generation.
