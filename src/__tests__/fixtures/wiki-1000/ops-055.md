# Incident Response Procedures

## Communication Template

When a SEV1 or SEV2 incident occurs, immediately post to the #incidents Slack channel:

```
INCIDENT: [Brief description]
SEVERITY: SEV1/SEV2
IMPACT: [Who/what is affected]
STATUS: Investigating
INCIDENT COMMANDER: [Name]
NEXT UPDATE: [Time]
```

## Mitigation Playbooks

### Service Unresponsive

1. Check pod status: `kubectl get pods -l app=<service>`
2. Check recent deploys: `kubectl rollout history deployment/<service>`
3. If recent deploy, rollback: `kubectl rollout undo deployment/<service>`
4. If not deploy-related, check resource usage and scale up if needed
5. Check downstream dependencies (database, cache, external APIs)

### Database Connection Exhaustion

1. Check active connections: `SELECT count(*) FROM pg_stat_activity;`
2. Kill idle transactions older than 5 minutes
3. Check PgBouncer pool utilization
4. Temporarily increase pool size if needed
5. Identify the service causing connection leaks

### Memory Leak

1. Identify the pod with increasing memory usage
2. Capture a heap dump if possible
3. Restart the affected pods (rolling)
4. Create a ticket for root cause analysis

## Post-Incident

Within 48 hours, write a **blameless postmortem**. Focus on timeline, root cause, and action items. Share with the engineering team.
