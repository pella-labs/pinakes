# Transactions and ACID

## The ACID Properties

Every database textbook starts here, and for good reason. **ACID** defines what it means for a transaction to be reliable:

**Atomicity** — all or nothing. If any part of the transaction fails, the entire thing rolls back.

**Consistency** — the database moves from one valid state to another. Constraints, triggers, and cascades are enforced.

**Isolation** — concurrent transactions don't interfere with each other (to the degree specified by the isolation level).

**Durability** — once committed, the data survives crashes. WAL and redo logs make this possible (see [[db-001]], [[db-003]]).

## Transaction Lifecycle

```sql
BEGIN;

UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;

-- Check invariant
SELECT sum(balance) FROM accounts;

COMMIT;  -- or ROLLBACK if something is wrong
```

## Savepoints

Savepoints allow partial rollback within a transaction:

```sql
BEGIN;
INSERT INTO orders (customer_id, total) VALUES (1, 50.00);
SAVEPOINT before_items;
INSERT INTO order_items (order_id, product_id) VALUES (1, 999);
-- oops, product 999 doesn't exist
ROLLBACK TO before_items;
-- order still exists, we can try again
COMMIT;
```

## Distributed Transactions

Two-phase commit (2PC) extends ACID across multiple databases. It's correct but slow — most modern systems prefer **eventual consistency** with saga patterns instead. See [[db-112]] for event sourcing approaches.
