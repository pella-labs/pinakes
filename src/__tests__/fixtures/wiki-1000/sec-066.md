# Secure Redirect Handling

**Open redirects** allow attackers to redirect users from your trusted domain to a malicious site. They're commonly exploited in phishing campaigns.

## The Vulnerability

```
https://trusted.example.com/login?redirect=https://evil.com
```

After login, the user is redirected to `evil.com` but trusts the link because it started at `trusted.example.com`.

## Prevention

### Allowlist Approach

Only redirect to known, safe destinations:

```typescript
const ALLOWED_REDIRECTS = [
  '/dashboard',
  '/settings',
  '/onboarding',
];

function safeRedirect(target: string, fallback: string = '/'): string {
  if (ALLOWED_REDIRECTS.includes(target)) return target;
  return fallback;
}
```

### Relative URL Only

Reject any redirect target with a protocol or hostname:

```typescript
function isRelativeUrl(url: string): boolean {
  try {
    new URL(url); // absolute URL — reject
    return false;
  } catch {
    return url.startsWith('/') && !url.startsWith('//');
  }
}
```

Watch out for protocol-relative URLs (`//evil.com`) and URL-encoded variants.

### Interstitial Warning

If external redirects are a feature, show a warning page: "You are leaving trusted.example.com and going to external-site.com. Continue?"

## In OAuth2 Flows

The `redirect_uri` in OAuth2 must be strictly validated against registered URIs. See [[sec-001]] for the authorization code flow. Partial matching or wildcard patterns in redirect URIs have been the source of many OAuth2 vulnerabilities.
