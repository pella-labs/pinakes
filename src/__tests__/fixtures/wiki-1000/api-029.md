# Health Check Endpoints

## Liveness vs Readiness

Two distinct checks for Kubernetes:

- **Liveness** (`/health/live`): "Is the process running?" Returns 200 if the event loop isn't blocked. If this fails, K8s restarts the pod.
- **Readiness** (`/health/ready`): "Can the process serve traffic?" Checks database connectivity, cache connectivity, downstream services. If this fails, K8s removes the pod from the service.

## Implementation

```typescript
app.get('/health/live', (req, res) => {
  res.status(200).json({ status: 'ok' });
});

app.get('/health/ready', async (req, res) => {
  const checks = {
    database: await checkDb(),
    cache: await checkRedis(),
    queue: await checkRabbitMQ(),
  };

  const allHealthy = Object.values(checks).every(c => c.healthy);
  res.status(allHealthy ? 200 : 503).json({ status: allHealthy ? 'ok' : 'degraded', checks });
});
```

## Don't Over-Check

Readiness checks should be fast (<500ms) and not create load. Don't run a full query — just `SELECT 1`. Don't check every downstream — only the ones you actually need to serve requests.

Never put auth on health endpoints. Your load balancer needs unauthenticated access.
