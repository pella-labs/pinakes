# Threat Modeling

**Threat modeling** is a structured approach to identifying, quantifying, and addressing security threats to a system. It's most effective when done early in the design phase, but valuable at any stage.

## STRIDE Framework

Microsoft's **STRIDE** categorizes threats:

| Category | Description | Example |
|---|---|---|
| **S**poofing | Pretending to be someone else | Forged JWT tokens |
| **T**ampering | Modifying data or code | Man-in-the-middle attack |
| **R**epudiation | Denying an action occurred | Missing audit logs |
| **I**nformation Disclosure | Exposing data | SQL injection data extraction |
| **D**enial of Service | Making system unavailable | DDoS attack |
| **E**levation of Privilege | Gaining unauthorized access | RBAC bypass |

## Process

1. **Decompose** the system into components, data flows, and trust boundaries
2. **Identify threats** using STRIDE or PASTA for each component
3. **Rate** each threat (DREAD scoring or likelihood × impact)
4. **Mitigate** with controls proportional to the risk
5. **Validate** mitigations through testing

## Data Flow Diagrams

Create DFDs showing:

- External entities (users, third-party services)
- Processes (your application components)
- Data stores (databases, file systems)
- Data flows (arrows between the above)
- **Trust boundaries** (the most important part)

Every crossing of a trust boundary is a potential attack point.

## When to Threat Model

- New features or services
- Architecture changes
- Before pen tests (to focus testing — see [[sec-028]])
- After security incidents (to identify missed threats)
- Quarterly review of existing models
