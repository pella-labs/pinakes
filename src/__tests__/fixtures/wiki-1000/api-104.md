# gRPC Deadlines and Timeouts

## Deadline Propagation

gRPC uses **deadlines** (absolute times) instead of timeouts (durations). When Service A calls Service B, the remaining deadline propagates automatically.

```
Client sets deadline: now + 5s
  → Service A receives, 1s passes, remaining: 4s
    → Service B receives, remaining: 4s
      → Service C receives, remaining: 3.5s
```

If the deadline expires at any point, all services in the chain get `DEADLINE_EXCEEDED`.

## Setting Deadlines

```typescript
const deadline = new Date(Date.now() + 5000);
const response = await client.getOrder({ id: 'ord_123' }, { deadline });
```

## Best Practices

- **Always set a deadline.** Requests without deadlines can hang forever.
- Set the deadline at the **edge** (API gateway or client). Don't set per-service timeouts that ignore the propagated deadline.
- Log when requests complete close to the deadline (>80% used). These are at risk of future failures.
- Use **server-side deadlines** as a safety net: if the client doesn't set one, the server applies a default (30s).
