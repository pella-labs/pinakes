# Database Monitoring and Observability

## Key Metrics

Every production database should be monitored for:

- **Query latency** (p50, p95, p99)
- **Connections** (active, idle, waiting)
- **Replication lag** (seconds behind primary)
- **Disk usage** and growth rate
- **Cache hit ratio** (should be >99% for most workloads)
- **Dead tuples** and autovacuum activity (PostgreSQL)
- **Lock waits** and deadlock frequency

## PostgreSQL Statistics

```sql
-- Cache hit ratio
SELECT
  sum(heap_blks_hit) / (sum(heap_blks_hit) + sum(heap_blks_read)) AS cache_ratio
FROM pg_statio_user_tables;

-- Slowest queries (requires pg_stat_statements)
SELECT query, calls, mean_exec_time, total_exec_time
FROM pg_stat_statements
ORDER BY mean_exec_time DESC
LIMIT 10;

-- Table bloat estimate
SELECT relname, n_dead_tup, last_autovacuum
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC;
```

## Alerting Thresholds

| Metric | Warning | Critical |
|--------|---------|----------|
| Replication lag | >5s | >30s |
| Connection usage | >70% of max | >90% |
| Disk usage | >70% | >85% |
| Cache hit ratio | <98% | <95% |

## Tools

Grafana + Prometheus with the postgres_exporter is the standard open-source stack. For managed services, use the built-in monitoring (RDS Performance Insights, Cloud SQL Insights).
