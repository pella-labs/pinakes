# WebSocket API Design

## Connection Lifecycle

1. Client initiates HTTP upgrade: `GET /ws` with `Upgrade: websocket`
2. Server responds with `101 Switching Protocols`
3. Bidirectional frames flow over the persistent connection
4. Either side sends a close frame to tear down

## Message Protocol

Define a structured message format. Don't send raw strings.

```json
{
  "type": "subscribe",
  "channel": "orders",
  "payload": { "filter": { "status": "active" } },
  "id": "msg_001"
}
```

Response:

```json
{
  "type": "subscribed",
  "channel": "orders",
  "ref": "msg_001"
}
```

## Authentication

WebSockets don't support custom headers during the handshake in browsers. Options:

- **Query parameter token**: `ws://host/ws?token=xxx` (visible in logs — use short-lived tokens)
- **First message auth**: connect, then send an auth message before subscribing
- **Cookie-based**: if the WS endpoint is same-origin

## Heartbeat/Ping-Pong

Send ping frames every 30 seconds. If no pong within 10 seconds, close the connection. This detects half-open connections where the client disappeared without a close frame.

## Scaling

WebSockets are stateful — you can't just round-robin behind a load balancer. Use **sticky sessions** or a pub/sub backbone (Redis Pub/Sub, NATS) so any server can publish to any connected client.
