# Canary Tokens and Honeypots

**Canary tokens** and **honeypots** are deceptive security measures that detect unauthorized access by placing tripwires throughout your environment.

## Canary Tokens

A canary token is a resource that should never be accessed in normal operations. Any access triggers an alert.

### Types

- **DNS tokens**: hostnames that resolve and alert when queried
- **Web tokens**: URLs that alert when visited
- **File tokens**: documents that call home when opened
- **AWS key tokens**: fake AWS credentials that alert when used
- **Database tokens**: rows that trigger notifications when queried

### Deployment Examples

Place a fake AWS credential in your git repository:

```
# .env.canary (tracked, intentionally)
AWS_ACCESS_KEY_ID=AKIA_CANARY_DO_NOT_USE
AWS_SECRET_ACCESS_KEY=canary_secret_key
```

If this key is ever used, it means someone has accessed your repository contents.

## Honeypots

Honeypots are fake services that appear valuable to attackers:

- Fake admin panels with login forms
- Decoy databases with synthetic data
- Fake API endpoints that log all access

## Integration with Monitoring

Feed canary alerts into your incident response process (see [[sec-054]]). Canary alerts are high-confidence indicators — they have almost zero false positive rate.

## Limitations

Canary tokens are detection-only. They don't prevent attacks. Use them to complement, not replace, preventive controls.
