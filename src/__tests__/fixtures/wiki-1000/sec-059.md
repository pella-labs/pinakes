# Secure Error Handling

Error messages and error handling can inadvertently leak sensitive information or create security vulnerabilities.

## Information Leakage

Avoid exposing internal details in error responses:

```typescript
// BAD — leaks implementation details
res.status(500).json({
  error: 'Query failed: SELECT * FROM users WHERE id = 42; column "passwd" not found',
  stack: new Error().stack,
});

// GOOD — generic message with correlation ID
res.status(500).json({
  error: 'An internal error occurred',
  requestId: 'req-abc-123',
});
```

## What Not to Expose

- Database error messages (reveal schema)
- Stack traces (reveal file paths and dependencies)
- Configuration details (reveal infrastructure)
- Internal IP addresses
- Software versions

## Error Handling Patterns

Log the full error server-side with a request ID. Return only the request ID to the client. The support team can correlate.

### Authentication Errors

Never reveal whether a username exists:

```typescript
// BAD
'Invalid password for user john@example.com'

// GOOD
'Invalid email or password'
```

### Authorization Errors

Return 404 instead of 403 when the resource's existence is sensitive. A 403 confirms the resource exists; a 404 reveals nothing.

## Fail Closed

When error handling itself fails, default to denying access:

```typescript
try {
  const allowed = await checkPermission(user, resource);
  if (!allowed) return deny();
} catch (err) {
  logger.error('Permission check failed', { err, requestId });
  return deny(); // Fail closed, not open
}
```

See [[sec-014]] for logging these errors in the audit trail.
