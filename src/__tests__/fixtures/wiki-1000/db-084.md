# Recursive Queries for Graph Traversal

## The Problem

Hierarchical and graph data is common: org charts, category trees, dependency graphs, social networks. Relational databases handle this with **recursive CTEs**.

## Depth-First Traversal

```sql
WITH RECURSIVE tree AS (
  SELECT id, name, parent_id, ARRAY[id] AS path, 0 AS depth
  FROM categories
  WHERE parent_id IS NULL
  UNION ALL
  SELECT c.id, c.name, c.parent_id, t.path || c.id, t.depth + 1
  FROM categories c
  JOIN tree t ON c.parent_id = t.id
  WHERE NOT c.id = ANY(t.path)  -- cycle detection
)
SELECT * FROM tree ORDER BY path;
```

## Finding All Ancestors

```sql
WITH RECURSIVE ancestors AS (
  SELECT id, name, parent_id FROM categories WHERE id = 42
  UNION ALL
  SELECT c.id, c.name, c.parent_id
  FROM categories c
  JOIN ancestors a ON c.id = a.parent_id
)
SELECT * FROM ancestors;
```

## Performance

Recursive CTEs are evaluated iteratively. Deep graphs or wide trees can be slow. Alternatives:
- **Materialized path**: store the full path as a string (`/1/5/12/`)
- **Nested sets**: store left/right values for range queries
- **ltree** extension (PostgreSQL): native hierarchical queries
- Dedicated graph database (see [[db-023]])
