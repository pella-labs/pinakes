# JWT Token Design

**JSON Web Tokens** provide a compact, self-contained way to transmit claims between parties. Getting JWT design right is critical — a poorly designed token scheme creates vulnerabilities that are hard to fix after deployment.

## Header and Algorithm Selection

Always use **asymmetric algorithms** (RS256, ES256) for tokens that cross trust boundaries. Symmetric algorithms (HS256) are acceptable only when the issuer and verifier are the same service.

```json
{
  "alg": "ES256",
  "typ": "JWT",
  "kid": "2024-03-signing-key"
}
```

## Claims Best Practices

- Always include `iss`, `sub`, `aud`, `exp`, and `iat`
- Use short expiration times (15 minutes for access tokens)
- Never put sensitive data in the payload — it's base64-encoded, not encrypted
- Include a `jti` (JWT ID) for revocation support

## Token Validation

Every consumer must validate:

1. **Signature** — verify against the issuer's public key
2. **Expiration** — reject expired tokens
3. **Audience** — ensure the token was intended for this service
4. **Issuer** — confirm the token came from a trusted authority

Skipping any of these checks is a security vulnerability. See [[sec-004]] for RBAC claims and [[sec-001]] for the OAuth2 flow that issues these tokens.

## Key Rotation

Use the `kid` header to support **key rotation** without downtime. The JWKS endpoint should serve both the current and previous key for a rotation window. See [[sec-016]] for key rotation strategies.
