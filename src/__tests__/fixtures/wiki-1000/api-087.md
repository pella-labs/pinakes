# API Evolution Without Versioning

## The GitHub Approach

GitHub's REST API has been at "v3" since 2012. They evolve it without bumping versions by following strict rules:

1. New fields are always optional
2. Existing fields never change type
3. Removed fields return null for a deprecation period
4. New endpoints can be added freely
5. Error codes are stable

## Additive Changes

Safe to deploy at any time:
- New response fields
- New query parameters (optional)
- New endpoints
- New enum values (if clients handle unknown values)
- New webhook event types

## Preview Features

Use a preview header for experimental features:

```
Accept: application/vnd.github.groot-preview+json
```

When the feature stabilizes, it becomes part of the standard API and the preview header becomes optional (then deprecated).

## Feature Flags vs Versions

Instead of `/v2/orders`, use the same endpoint with a feature flag:

```
X-Feature: new-order-response
```

This gives per-feature opt-in instead of all-or-nothing version bumps.
