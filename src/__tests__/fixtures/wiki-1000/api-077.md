# API Canary Deployments

## How It Works

Route a small percentage of traffic to the new version while monitoring.

```yaml
# Istio virtual service
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
spec:
  http:
    - route:
        - destination:
            host: order-service
            subset: v1
          weight: 95
        - destination:
            host: order-service
            subset: v2
          weight: 5
```

## Monitoring During Canary

Compare v1 and v2 on:
- Error rate (any increase → rollback)
- Latency P50/P95 (>10% increase → investigate)
- Business metrics (conversion, revenue)

## Promotion Criteria

Automated promotion after:
1. 15 minutes at 5% traffic
2. Error rate delta < 0.1%
3. Latency delta < 5%
4. No critical alerts

Then ramp to 25%, 50%, 100% with the same checks at each stage.

## Rollback

Instant. Shift weight back to v1. No downtime, no state corruption (assuming the new version doesn't make irreversible data changes).

## Header-Based Routing

For internal testing, route by header:

```
X-Canary: true → v2
Otherwise → v1
```

This lets QA test the canary version before opening to real traffic.
