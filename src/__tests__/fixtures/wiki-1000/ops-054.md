# Kubernetes Horizontal Scaling

Scaling Kubernetes workloads involves both **pod-level** and **node-level** scaling working in concert. The HPA adds pods; the cluster autoscaler adds nodes when pods can't be scheduled.

## Custom Metrics Scaling

Beyond CPU and memory, scale on business metrics using the **Prometheus Adapter**:

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: queue-worker
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: queue-worker
  minReplicas: 1
  maxReplicas: 100
  metrics:
    - type: External
      external:
        metric:
          name: sqs_queue_depth
          selector:
            matchLabels:
              queue: order-processing
        target:
          type: Value
          value: "50"
```

## Scaling Behavior

Fine-tune how quickly the HPA responds:

- **scaleUp stabilization**: prevent flapping during traffic spikes (60s default)
- **scaleDown stabilization**: keep capacity longer to absorb follow-up spikes (300s recommended)
- **policies**: limit scale-up to X pods per minute to avoid thundering herd on downstream services

## Vertical Pod Autoscaler

The **VPA** adjusts CPU and memory requests based on actual usage. Run it in recommendation mode first, then enable auto-update for non-critical services. Never use VPA and HPA on the same metric.

See [[ops-014]] for autoscaling strategies.
