# Kubernetes RBAC

## Role-Based Access Control

**RBAC** in Kubernetes controls who can do what within the cluster. Every human and service account should have the minimum permissions needed.

## Core Concepts

- **Role** — defines permissions within a namespace
- **ClusterRole** — defines permissions cluster-wide
- **RoleBinding** — grants a Role to a user/group in a namespace
- **ClusterRoleBinding** — grants a ClusterRole cluster-wide

## Example: Read-Only Developer Access

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: production
  name: developer-readonly
rules:
  - apiGroups: [""]
    resources: ["pods", "services", "configmaps"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["apps"]
    resources: ["deployments", "replicasets"]
    verbs: ["get", "list", "watch"]
  - apiGroups: [""]
    resources: ["pods/log"]
    verbs: ["get"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  namespace: production
  name: developer-readonly-binding
subjects:
  - kind: Group
    name: developers
roleRef:
  kind: Role
  name: developer-readonly
  apiGroup: rbac.authorization.k8s.io
```

## Best Practices

Bind to **groups**, not individual users. Sync groups from your identity provider via OIDC. Audit RBAC permissions quarterly.
