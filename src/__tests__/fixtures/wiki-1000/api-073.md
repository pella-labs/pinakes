# DR: JWT with Short-Lived Access Tokens

## Context

Need to standardize authentication across 8 services. Currently a mix of session cookies, long-lived API keys, and JWTs with various lifetimes.

## Decision

- **Access tokens**: JWTs, RS256, 15-minute expiry
- **Refresh tokens**: opaque, stored server-side, 30-day expiry
- **API keys**: for service-to-service and third-party integrations, no expiry but revocable
- **JWKS**: public keys served at `/.well-known/jwks.json`, rotated quarterly

## Rationale

Short-lived JWTs minimize the blast radius of token theft. Refresh tokens keep users logged in. RS256 allows any service to verify without sharing a secret. API keys for machines avoid the refresh dance.

## Consequences

- Services need JWKS fetching and caching
- Mobile apps need refresh token flow
- Refresh token storage needs Redis or database
- Token rotation infrastructure needed

See [[api-023]] for implementation, [[api-047]] for API key management.
