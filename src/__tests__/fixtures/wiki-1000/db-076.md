# Soft Deletes vs Hard Deletes

## Hard Deletes

A `DELETE FROM` removes the row permanently. Simple, clean, and reclaims space (after VACUUM in PostgreSQL).

## Soft Deletes

Add a `deleted_at` timestamp column. "Deleting" a row means setting this timestamp:

```sql
-- "Delete"
UPDATE users SET deleted_at = now() WHERE id = 42;

-- Query active records
SELECT * FROM users WHERE deleted_at IS NULL;
```

## Implementing Soft Deletes

Create a view for convenience:

```sql
CREATE VIEW active_users AS
SELECT * FROM users WHERE deleted_at IS NULL;
```

Or use a partial index:

```sql
CREATE INDEX idx_users_active ON users (email) WHERE deleted_at IS NULL;
```

## Tradeoffs

Soft deletes enable undo, audit trails, and data recovery. But they add complexity:

- Every query must filter on `deleted_at IS NULL`
- Foreign key cascades don't trigger
- Unique constraints need to account for deleted records
- Data grows forever unless you periodically purge old soft-deleted rows

## Recommendation

Use soft deletes for user-facing data where undo matters. Use hard deletes for system data (logs, temporary records) where retention is a liability. Consider temporal tables ([[db-062]]) for full change history.
