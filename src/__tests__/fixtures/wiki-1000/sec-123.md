# Browser Security Model

Understanding the **browser security model** is essential for building secure web applications. The browser enforces several security boundaries that both protect and constrain your application.

## Same-Origin Policy

The foundational browser security mechanism. Two URLs have the same origin if they share:

- **Protocol** (http vs https)
- **Host** (www.example.com)
- **Port** (443)

Scripts from one origin cannot read responses from a different origin unless explicitly allowed via CORS (see [[sec-007]]).

## Sandboxing

Modern browsers run each origin in a separate process with limited OS access:

- No direct filesystem access (except via File API with user interaction)
- No raw socket access
- Limited clipboard access
- Restricted camera/microphone access (requires permission)

## Security Mechanisms

### Content Security Policy
Controls what resources can load (see [[sec-008]])

### Subresource Integrity
Verifies CDN resources haven't been tampered with (see [[sec-042]])

### Feature Policy / Permissions Policy
Controls which browser features are available to the page

### Referrer Policy
Controls what URL information is leaked via the Referer header

### Cross-Origin Isolation
Provides process-level isolation (see [[sec-065]])

## Limitations

The browser security model assumes the user trusts the server. It doesn't protect against:

- Malicious servers serving hostile content
- Server-side vulnerabilities (XSS, injection)
- Browser extensions with elevated permissions
- User actions (clicking through warnings, installing untrusted extensions)
