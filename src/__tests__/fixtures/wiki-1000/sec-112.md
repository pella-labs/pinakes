# Security Budgeting and Prioritization

Not all security work has equal ROI. **Prioritization** ensures limited security resources address the highest-impact risks first.

## Risk-Based Prioritization

Rank security initiatives by:

- **Likelihood** of exploitation
- **Impact** if exploited (data loss, downtime, regulatory)
- **Effort** to mitigate
- **Exposure** (internet-facing vs. internal)

## High-ROI Security Investments

These consistently provide the best return:

1. **MFA everywhere** (see [[sec-056]]) — prevents most account takeover
2. **Dependency scanning** (see [[sec-055]]) — catches known vulns automatically
3. **Secret scanning** (see [[sec-013]]) — prevents credential leaks
4. **Automated patching** — reduces exposure window
5. **Security headers** (see [[sec-033]]) — low effort, broad protection

## Lower-Priority Items

These matter but shouldn't displace the basics:

- Advanced threat hunting (get logging right first)
- Red team exercises (fix known issues first)
- Custom WAF rules (use standard rule sets first)
- Bug bounty programs (fix low-hanging fruit first)

## Time Allocation

A practical split for engineering teams:

- 70% building features with security built in
- 20% addressing known vulnerabilities and security debt
- 10% proactive security improvement (tooling, training, process)

## Tracking

Track security debt alongside technical debt. Use the same systems (Jira, Linear) with security-specific labels and SLAs.
