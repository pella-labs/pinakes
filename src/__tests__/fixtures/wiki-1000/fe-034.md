# Micro-Frontends Architecture

## Concept

**Micro-frontends** extend microservices principles to the frontend. Each team owns a vertical slice of the application—from UI to API—and deploys independently.

## Implementation Approaches

### Module Federation (Webpack 5)

Webpack's **Module Federation** allows applications to share code at runtime:

- Host application loads remote modules dynamically
- Shared dependencies are loaded once
- Each micro-frontend is a separate build

### Web Components

Wrap each micro-frontend in a **custom element**. This provides true encapsulation (shadow DOM) and framework agnosticism.

### iframes

The simplest isolation mechanism. Each micro-frontend runs in its own iframe with `postMessage` for communication. Heavy but provides complete isolation.

## Challenges

- **Shared state**: no global store; use events or a shared bus
- **Routing**: coordinate URL changes across micro-frontends
- **Styling conflicts**: CSS isolation via shadow DOM, CSS modules, or naming conventions
- **Performance**: multiple framework copies, duplicate dependencies
- **Testing**: integration testing across boundaries is complex

## When to Use

Micro-frontends make sense when:
- Multiple teams work on the same application
- Teams need independent deployment cadences
- Different parts of the app have different technology requirements

For small-to-medium teams, a well-structured monolith is usually better. See [[fe-016]] for code splitting as a simpler alternative.
