# Secret Zero Problem

The **secret zero** problem asks: how does an application authenticate to its secret manager to retrieve its secrets? The first credential — the "secret zero" — must come from somewhere.

## The Paradox

You use Vault to manage secrets (see [[sec-015]]). But Vault requires a token to authenticate. Where does that token come from? If you put it in an environment variable, you've just shifted the problem.

## Solutions

### Platform Identity

Use the platform's native identity:

- **AWS**: IAM roles for EC2/ECS/Lambda (no credentials needed)
- **GCP**: workload identity
- **Kubernetes**: service account tokens projected into pods
- **Azure**: managed identities

The platform vouches for the workload's identity without a static secret.

### AppRole with Response Wrapping

Vault's AppRole auth method uses two factors:

1. **Role ID**: embedded in the deployment (semi-public)
2. **Secret ID**: delivered via response wrapping (single-use)

The CI/CD system delivers a wrapped secret ID that can only be unwrapped once.

### Trusted Orchestrator

The orchestrator (Kubernetes, Nomad) injects the initial token during deployment. The token is scoped and short-lived.

## Anti-Patterns

- Baking secrets into container images
- Storing tokens in environment variables accessible to all processes
- Using long-lived static tokens for Vault authentication
- Sharing a single token across multiple services

The goal is that no human or static file holds a long-lived secret that grants access to other secrets.
