# Database Migrations in CI/CD

## The Challenge

**Database migrations** are the riskiest part of any deployment. Unlike code rollbacks, schema changes are often irreversible. A bad migration can cause data loss or extended downtime.

## Migration Workflow

1. Developer creates migration file with `up` and `down` scripts
2. PR review includes migration review (mandatory second reviewer for destructive changes)
3. CI runs migration against a test database
4. Deploy job runs migration before new code starts
5. Monitor for errors during and after migration

## Safe Migration Patterns

### Adding a Column

Safe. Add with a default value or as nullable. Backfill in a separate step if needed.

### Removing a Column

Two-phase: first deploy code that doesn't read the column, then drop the column in a subsequent migration.

### Renaming a Column

Never rename directly. Instead: add new column, backfill, update code, drop old column. This prevents downtime.

### Adding an Index

Use `CREATE INDEX CONCURRENTLY` (Postgres) to avoid locking the table. This should always be in its own migration file.

## Rollback Strategy

Every migration must have a tested `down` migration. If a migration fails partway through, the `down` script must handle partial state.
