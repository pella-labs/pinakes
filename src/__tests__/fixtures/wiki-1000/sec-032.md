# OAuth2 Token Refresh Patterns

Handling **token refresh** correctly is essential for maintaining session continuity without compromising security. A flawed refresh implementation can be worse than no refresh at all.

## The Refresh Flow

1. Client detects the access token has expired (or is about to)
2. Client sends the refresh token to the authorization server
3. Server validates the refresh token and issues new access + refresh tokens
4. Client stores the new tokens and retries the failed request

## Refresh Token Rotation

**Always rotate refresh tokens** on every use. When the server issues a new access token, it also issues a new refresh token and invalidates the old one. This limits the window of a stolen refresh token.

## Race Condition Handling

In SPAs and mobile apps, multiple API calls may discover an expired access token simultaneously, triggering multiple refresh requests. Solutions:

- **Mutex pattern**: queue concurrent refresh attempts behind a single in-flight request
- **Grace window**: accept the old refresh token for a few seconds after rotation
- **Retry with backoff**: if refresh fails with "token already used," retry once

## Storage

Refresh tokens are long-lived and powerful. Store them securely:

- **Server-side apps**: encrypted in the database, never in cookies
- **Mobile apps**: OS keychain (iOS Keychain, Android Keystore)
- **SPAs**: ideally avoid (use BFF pattern); if required, use secure httpOnly cookies
