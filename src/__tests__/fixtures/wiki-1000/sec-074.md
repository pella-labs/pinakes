# Secure gRPC Communication

**gRPC** uses HTTP/2 and Protocol Buffers. While it has built-in TLS support, there are additional security considerations beyond basic transport encryption.

## Authentication Options

### TLS with Client Certificates

```protobuf
// Server-side
ServerCredentials creds = ServerCredentials.forCertChain(certChain, privateKey, clientCAs);
```

This provides mTLS at the gRPC level. See [[sec-006]] for mTLS details.

### Token-Based Authentication

Pass tokens via gRPC metadata (equivalent to HTTP headers):

```typescript
const metadata = new grpc.Metadata();
metadata.add('authorization', `Bearer ${token}`);
client.someMethod(request, metadata, callback);
```

### Per-RPC Credentials

gRPC supports attaching credentials to individual calls, allowing different auth for different operations.

## Authorization

Implement authorization using **gRPC interceptors** (middleware):

```typescript
function authInterceptor(call, callback) {
  const token = call.metadata.get('authorization')[0];
  const user = verifyToken(token);
  if (!user || !hasPermission(user, call.getPath())) {
    callback({ code: grpc.status.PERMISSION_DENIED });
    return;
  }
  callback(null);
}
```

## Input Validation

Protobuf provides type safety but not business validation. Validate field values, ranges, and invariants in your service implementation. Use proto3 field presence to detect missing required fields.

## Rate Limiting

Apply rate limiting at the gRPC level, not just HTTP. gRPC multiplexes many RPCs over a single connection, so connection-level limits are insufficient. Limit per-method call rates.
