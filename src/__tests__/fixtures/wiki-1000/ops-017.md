# Feature Flags

## Why Feature Flags

**Feature flags** decouple deployment from release. You can deploy code to production but keep it invisible until you're ready. This enables trunk-based development, A/B testing, and gradual rollouts.

## Implementation Approaches

### Simple Boolean Flags

```typescript
if (featureFlags.isEnabled('new-checkout-flow', { userId })) {
  return renderNewCheckout();
} else {
  return renderLegacyCheckout();
}
```

### Percentage Rollouts

Roll out to 5%, watch metrics, then 25%, 50%, 100%. The flag service determines eligibility based on a consistent hash of the user ID.

### Targeting Rules

Target specific segments: internal users, beta testers, users in a specific region, or users matching custom attributes.

## Flag Lifecycle

1. **Create** — define the flag with a clear description and owner
2. **Develop** — wrap new code in flag checks
3. **Test** — verify both flag states in CI
4. **Roll out** — gradually increase percentage
5. **Clean up** — remove the flag and dead code path

The cleanup step is critical. Stale flags are tech debt. Set a **flag expiry date** and alert when it passes.

## Tools

We use **LaunchDarkly** for production flags and a simple in-memory config for local development. See [[ops-010]] for how flags integrate with canary releases.
