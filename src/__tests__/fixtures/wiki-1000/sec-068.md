# Insecure Direct Object Reference

**IDOR** vulnerabilities occur when an application exposes internal implementation objects (database IDs, filenames) to users and doesn't verify authorization for each access.

## Example

```
GET /api/invoices/12345
```

If the server returns invoice 12345 without checking whether the current user owns it, any authenticated user can access any invoice by guessing IDs.

## Why It's Common

- Developers assume authentication implies authorization
- Sequential IDs make enumeration trivial
- Testing often focuses on authenticated happy paths

## Prevention

### Authorization Checks

Always verify the current user has access to the requested resource:

```typescript
app.get('/api/invoices/:id', async (req, res) => {
  const invoice = await db.invoices.findById(req.params.id);
  if (!invoice) return res.status(404).json({ error: 'Not found' });
  if (invoice.userId !== req.user.id) return res.status(404).json({ error: 'Not found' });
  return res.json(invoice);
});
```

Note: return 404, not 403, to avoid confirming the resource exists.

### Non-Sequential Identifiers

Use UUIDs instead of sequential integers. This doesn't replace authorization checks but makes enumeration impractical.

### Scoped Queries

Build authorization into your queries:

```sql
SELECT * FROM invoices WHERE id = ? AND user_id = ?
```

This makes IDOR impossible at the query level. See [[sec-004]] for RBAC and [[sec-050]] for the security code review checklist that includes IDOR checks.
