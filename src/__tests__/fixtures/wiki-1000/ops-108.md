# On-Call Best Practices

## Rotation Structure

A healthy **on-call rotation** balances coverage with engineer well-being:

- 1-week rotations (longer causes burnout)
- Follow-the-sun for global teams
- Primary and secondary on-call
- Escalation chain: primary → secondary → team lead → VP Eng

## Reducing On-Call Burden

The goal is to make on-call boring:

- Fix the root cause of every page, don't just mitigate
- Automate common remediation (auto-restart, auto-scale)
- Remove noisy alerts that don't require action
- Build runbooks for every alert
- Share on-call load equally across the team

## Runbooks

Every alert should link to a runbook. A good runbook contains:

1. **What this alert means** — plain English explanation
2. **Impact** — what users are experiencing
3. **Diagnosis steps** — specific commands to run
4. **Remediation steps** — how to fix or mitigate
5. **Escalation** — who to contact if the runbook doesn't resolve it

## Post-On-Call Handoff

The outgoing on-call writes a brief summary:
- Number of pages
- Any ongoing issues
- Changes to watch
- Suggested improvements
