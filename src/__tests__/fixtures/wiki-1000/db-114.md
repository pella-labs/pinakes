# Database High Availability Patterns

## Active-Passive (Primary-Standby)

One active primary, one or more passive standbys. On primary failure, promote a standby.

Tools:
- PostgreSQL: Patroni, pg_auto_failover, repmgr
- MySQL: MySQL InnoDB Cluster, orchestrator
- Managed: RDS Multi-AZ, Cloud SQL HA

## Active-Active

Multiple nodes accept writes. Requires conflict resolution.

- PostgreSQL: Citus, BDR
- MySQL: Group Replication, NDB Cluster
- CockroachDB, YugabyteDB: natively distributed

## Failover Strategies

### Automatic Failover
A watchdog monitors the primary and promotes a standby on failure. Fast but risk of split-brain.

### Manual Failover
An operator triggers the promotion. Slower but no split-brain risk.

### DNS-Based
Update DNS to point to the new primary. Simple but DNS TTL causes delay.

### Proxy-Based
A proxy (HAProxy, PgBouncer) routes to the current primary. Fastest failover.

## Recovery Point Objective (RPO)

How much data can you afford to lose?
- Synchronous replication: RPO = 0 (no data loss)
- Asynchronous: RPO > 0 (some recent transactions may be lost)

## Recovery Time Objective (RTO)

How quickly must the system recover?
- Automatic failover: 10-30 seconds
- Manual: minutes to hours
- DNS-based: depends on TTL
