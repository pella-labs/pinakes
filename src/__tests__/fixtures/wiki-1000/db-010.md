# Connection Pooling Strategies

## Why Pool?

Database connections are expensive to create. PostgreSQL forks a new process per connection. MySQL creates a new thread. Even lightweight databases like SQLite have file-locking overhead. **Connection pooling** reuses a fixed set of connections across application requests.

## Application-Level Pooling

Most ORMs and database drivers include built-in pooling:

```typescript
import { Pool } from 'pg';

const pool = new Pool({
  host: 'localhost',
  database: 'myapp',
  max: 20,              // max connections in pool
  idleTimeoutMillis: 30000,
  connectionTimeoutMillis: 2000,
});

const result = await pool.query('SELECT * FROM users WHERE id = $1', [userId]);
```

## External Poolers

For PostgreSQL, **PgBouncer** sits between your app and the database:

- **Session mode**: one-to-one mapping, safest
- **Transaction mode**: connection returned after each transaction, best throughput
- **Statement mode**: connection returned after each statement, most aggressive

Transaction mode is the sweet spot for most web applications. See [[db-045]] for PgBouncer configuration details.

## Sizing the Pool

A common mistake is setting the pool too large. More connections means more context switching, more memory, and more lock contention. The formula from the PostgreSQL wiki:

**connections = (cores * 2) + effective_spindle_count**

For SSD-backed systems, start with `cores * 2` and adjust based on monitoring.
