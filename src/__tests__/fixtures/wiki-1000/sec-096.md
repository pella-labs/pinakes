# Software Supply Chain Security

The **software supply chain** encompasses everything from source code to deployed artifacts. Each step introduces potential attack vectors.

## Attack Vectors

- **Source code**: compromised developer accounts, malicious commits
- **Dependencies**: malicious packages, typosquatting, dependency confusion
- **Build process**: compromised CI/CD pipelines (see [[sec-035]])
- **Artifacts**: tampered binaries, poisoned container images
- **Distribution**: compromised package registries, CDN injection

## SLSA Framework

The **Supply-chain Levels for Software Artifacts** framework defines increasing levels of supply chain integrity:

- Level 1: documentation of the build process
- Level 2: tamper-resistant build service
- Level 3: tamper-resistant build + verified source
- Level 4: two-person review + hermetic build

## Practical Measures

### Code Signing

Sign commits and releases:

```bash
git config commit.gpgsign true
```

### Dependency Pinning

Pin exact versions in lock files. Verify checksums. Use tools like `npm audit signatures` to verify registry package provenance.

### Build Reproducibility

Reproducible builds produce identical artifacts from the same source, proving no tampering occurred during the build.

### SBOM (Software Bill of Materials)

Generate and maintain an SBOM listing all components:

```bash
syft packages dir:. -o spdx-json > sbom.json
```

## Verification

Use tools like **Sigstore** (cosign, fulcio, rekor) for keyless signing and verification of artifacts.

See [[sec-055]] for dependency vulnerability management.
