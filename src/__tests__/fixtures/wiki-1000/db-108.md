# Implementing CQRS

## What Is CQRS?

**Command Query Responsibility Segregation** separates the read model from the write model. Writes go to a normalized, consistency-optimized store. Reads come from a denormalized, query-optimized store.

## Architecture

```
  Writes ──→ Command Handler ──→ Write DB (normalized)
                    │
                    ▼
              Event Bus / CDC
                    │
                    ▼
  Reads  ←── Query Handler  ←── Read DB (denormalized)
```

## When CQRS Helps

- Read and write patterns are dramatically different
- You need different scaling strategies for reads vs writes
- Complex queries require denormalized views that are expensive to maintain synchronously
- Event sourcing is already in use (see [[db-112]])

## Implementation Options

### Same Database, Different Models
Use materialized views ([[db-040]]) or denormalized tables updated by triggers.

### Separate Databases
Write to PostgreSQL, read from Elasticsearch ([[db-006]]) or a Redis cache ([[db-022]]). Sync via CDC ([[db-026]]).

## Eventual Consistency

CQRS with separate read/write stores introduces eventual consistency. The read model may be milliseconds to seconds behind the write model. Design the UI to handle this gracefully.

## When to Avoid

CQRS adds complexity. Don't use it for simple CRUD applications. Start with a single database and only separate when you have evidence that the read and write workloads require it.
