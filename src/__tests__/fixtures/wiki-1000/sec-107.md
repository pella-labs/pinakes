# Path Traversal Prevention

**Path traversal** attacks manipulate file paths to access files outside intended directories. They exploit insufficient path validation to read configuration files, source code, or system files.

## Attack Vectors

```
GET /api/files/../../../etc/passwd
GET /api/files/..%2f..%2f..%2fetc/passwd
GET /api/files/....//....//etc/passwd
```

## Prevention

### Path Containment

Resolve the full path and verify it's within the allowed directory:

```typescript
import { resolve, relative } from 'path';

function safePath(baseDir: string, userPath: string): string {
  const resolved = resolve(baseDir, userPath);
  const rel = relative(baseDir, resolved);
  
  if (rel.startsWith('..') || resolve(baseDir, rel) !== resolved) {
    throw new SecurityError('Path traversal attempt');
  }
  
  return resolved;
}
```

### Allowlist

If possible, map user input to a fixed set of allowed paths rather than constructing paths from input.

### Chroot / Container Isolation

Run file-serving components in a chroot jail or container where the filesystem is limited to intended content.

## URL Encoding

Attackers use URL encoding to bypass string matching:

- `%2e%2e%2f` = `../`
- `%252e%252e%252f` = double-encoded `../`

Always decode URLs fully before path validation. Apply validation after normalization, not before.

See [[sec-036]] for general input validation and [[sec-039]] for file upload security.
