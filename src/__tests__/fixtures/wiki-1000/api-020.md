# Server-Sent Events

## Protocol

SSE uses a simple text-based protocol over HTTP. The server sends events as `text/event-stream`:

```
event: order_update
data: {"orderId":"ord_123","status":"shipped"}
id: evt_456
retry: 5000

event: heartbeat
data: {}

```

Each event has optional `event` (type), `data` (payload), `id` (for resume), and `retry` (reconnection interval in ms).

## Server Implementation

```typescript
app.get('/events', (req, res) => {
  res.writeHead(200, {
    'Content-Type': 'text/event-stream',
    'Cache-Control': 'no-cache',
    'Connection': 'keep-alive',
  });

  const send = (event: string, data: unknown) => {
    res.write(`event: ${event}\ndata: ${JSON.stringify(data)}\n\n`);
  };

  // Heartbeat every 30s to keep connection alive
  const heartbeat = setInterval(() => send('heartbeat', {}), 30000);

  const unsub = eventBus.subscribe(req.user.id, (evt) => {
    send(evt.type, evt.data);
  });

  req.on('close', () => {
    clearInterval(heartbeat);
    unsub();
  });
});
```

## Auto-Reconnection

The browser's `EventSource` API automatically reconnects on disconnection. It sends the last `id` as `Last-Event-ID` header, enabling the server to replay missed events.

## Limitations

- Unidirectional (server to client only)
- Text-only (no binary)
- Limited to ~6 connections per domain in HTTP/1.1 (HTTP/2 removes this)

For bidirectional needs, see [[api-021]].
