# Terraform State Management

## Remote State

Never store Terraform state locally for team projects. Use a **remote backend** with locking.

## S3 Backend Configuration

```hcl
terraform {
  backend "s3" {
    bucket         = "company-terraform-state"
    key            = "services/api/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-locks"
    encrypt        = true
  }
}
```

## State File Organization

Organize state files by service and layer:

```
s3://company-terraform-state/
  network/terraform.tfstate
  dns/terraform.tfstate
  eks/terraform.tfstate
  services/api/terraform.tfstate
  services/web/terraform.tfstate
  monitoring/terraform.tfstate
```

## State Operations

Common state management tasks:

```bash
# List resources in state
terraform state list

# Show a specific resource
terraform state show aws_instance.web

# Move a resource (after refactoring)
terraform state mv aws_instance.old aws_instance.new

# Import an existing resource
terraform import aws_s3_bucket.data my-existing-bucket

# Remove from state without destroying
terraform state rm aws_instance.manual
```

## State Locking

DynamoDB provides **state locking** to prevent concurrent modifications. If a lock is stuck, use `terraform force-unlock` with the lock ID (but verify no one else is running an apply first).
