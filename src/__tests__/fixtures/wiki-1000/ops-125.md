# Platform Engineering

## From DevOps to Platform Engineering

**Platform engineering** is the discipline of building and maintaining an Internal Developer Platform (IDP) that enables self-service for development teams. It's the evolution of DevOps for organizations at scale.

## The Internal Developer Platform

An IDP provides:

- **Self-service deployment**: developers ship without ops tickets
- **Environment provisioning**: spin up a preview environment per PR
- **Observability out of the box**: metrics, logs, traces preconfigured
- **Security by default**: scanning, policies, secrets injection
- **Cost visibility**: per-team resource usage and costs

## Platform as a Product

Treat the platform as an **internal product**:

- Gather requirements from developer teams (your customers)
- Prioritize features based on developer productivity impact
- Measure adoption and satisfaction
- Iterate on developer experience

## Golden Paths

Define **golden paths** — opinionated, well-supported ways to accomplish common tasks. These aren't mandates; they're the easy option:

- Service template with CI/CD, monitoring, and deployment preconfigured
- Database provisioning via a CRD (see [[ops-075]])
- Environment creation via a PR (see [[ops-113]])

## Anti-Patterns

- Building the platform in isolation without developer input
- Making the platform mandatory before it's mature
- Over-engineering for edge cases instead of optimizing the common path
- Not investing in documentation and developer education
