# GraphQL Federation

## What It Is

A pattern for composing a single GraphQL schema from multiple **subgraph** services. Each service owns its domain types and the gateway stitches them together.

## Subgraph Example

```graphql
# Order service subgraph
type Order @key(fields: "id") {
  id: ID!
  status: OrderStatus!
  items: [OrderItem!]!
  customerId: ID! @external
  customer: Customer! @requires(fields: "customerId")
}

extend type Customer @key(fields: "id") {
  id: ID! @external
  orders: [Order!]!
}
```

## How It Works

1. Each subgraph defines its types with `@key` directives
2. The gateway fetches the **supergraph schema** (composed from all subgraphs)
3. At query time, the gateway creates a **query plan** that fans out to relevant subgraphs
4. Results are merged and returned as a single response

## Entity Resolution

When the Order subgraph references `Customer`, the gateway calls the Customer subgraph's `__resolveReference` resolver with `{ id: "cust_123" }`. The Customer subgraph returns the full object.

## Gotchas

- Schema conflicts between subgraphs cause composition errors
- Adding fields to shared types requires coordination
- Query planning adds latency — optimize hot paths
