# PostgreSQL VACUUM and Autovacuum

## Why VACUUM Exists

PostgreSQL's MVCC creates dead tuples on every UPDATE and DELETE. **VACUUM** reclaims the space occupied by these dead tuples, making it available for new rows. Without vacuuming, tables bloat indefinitely.

## Types of VACUUM

- `VACUUM` — marks dead tuples as reusable, updates visibility map
- `VACUUM FULL` — rewrites the entire table, reclaiming space to the OS. Requires an exclusive lock. Use sparingly.
- `VACUUM ANALYZE` — vacuum plus update statistics for the query planner

## Autovacuum Configuration

Autovacuum runs automatically and should almost never be disabled. Key parameters:

```sql
-- Trigger threshold: vacuum when dead tuples > threshold + scale_factor * reltuples
autovacuum_vacuum_threshold = 50
autovacuum_vacuum_scale_factor = 0.2

-- For high-churn tables, override per-table:
ALTER TABLE events SET (autovacuum_vacuum_scale_factor = 0.01);
ALTER TABLE events SET (autovacuum_vacuum_threshold = 1000);
```

## Transaction ID Wraparound

PostgreSQL uses 32-bit transaction IDs. After ~2 billion transactions, IDs wrap around. An **anti-wraparound vacuum** is forced before this happens. If autovacuum can't keep up, the database will shut down to prevent data corruption.

Monitor `age(datfrozenxid)` and ensure it stays well below `autovacuum_freeze_max_age`. This is related to the MVCC mechanics described in [[db-013]].
