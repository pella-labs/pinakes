# MySQL InnoDB Internals

## Buffer Pool

The **InnoDB buffer pool** is the most important memory structure in MySQL. It caches table data and index pages, reducing disk I/O. A well-sized buffer pool can hold the entire working set in memory.

```sql
-- Check buffer pool usage
SHOW ENGINE INNODB STATUS\G

-- Key metrics
SELECT
  pool_id,
  pool_size,
  free_buffers,
  database_pages,
  old_database_pages,
  modified_db_pages
FROM information_schema.innodb_buffer_pool_stats;
```

## Clustered Index

InnoDB organizes data in a **clustered index** structure based on the primary key. The table data lives in the leaf nodes of the B+ tree. This means:

- Primary key lookups are extremely fast (single index traversal)
- Secondary indexes store the primary key value, not a row pointer
- Choosing a good primary key is critical for performance — see [[db-020]] for indexing strategies

## Redo Log and Doublewrite Buffer

InnoDB uses a **redo log** (similar in concept to PostgreSQL's WAL described in [[db-001]]) to guarantee durability. The **doublewrite buffer** protects against torn pages during crash recovery. These two mechanisms together ensure that committed transactions survive any failure.

## Transaction Isolation

InnoDB defaults to **REPEATABLE READ** isolation, which is stricter than PostgreSQL's default of READ COMMITTED. This affects how phantom reads and gap locking behave — important when diagnosing deadlocks in high-concurrency workloads.
