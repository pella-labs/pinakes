# GraphQL Error Handling

## Error Locations

GraphQL errors can occur at multiple levels:

1. **Parse errors** — invalid syntax
2. **Validation errors** — schema violations
3. **Execution errors** — resolver failures

## Standard Error Format

```json
{
  "data": { "order": null },
  "errors": [
    {
      "message": "Order not found",
      "locations": [{ "line": 2, "column": 3 }],
      "path": ["order"],
      "extensions": {
        "code": "NOT_FOUND",
        "requestId": "req_abc"
      }
    }
  ]
}
```

## Partial Success

GraphQL can return **both data and errors**. If one field fails, others still resolve. This is powerful but clients must handle partial responses.

## Union Error Types

Model expected errors as types, not thrown errors:

```graphql
union CreateOrderResult = Order | ValidationError | InsufficientStockError

type ValidationError {
  fields: [FieldError!]!
}

type InsufficientStockError {
  productId: ID!
  available: Int!
  requested: Int!
}
```

The client can switch on `__typename` for typed error handling. This is the recommended pattern from [[api-027]].
