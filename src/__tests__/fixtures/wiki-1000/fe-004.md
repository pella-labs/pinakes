# Next.js App Router Architecture

## File-Based Routing

The **App Router** introduced in Next.js 13 uses a file-system based routing approach built on React Server Components. Every file named `page.tsx` inside the `app/` directory becomes a route.

## Layouts and Templates

**Layouts** persist across navigations and maintain state. They wrap child routes and are ideal for navigation bars, sidebars, and other persistent UI elements.

```tsx
// app/dashboard/layout.tsx
export default function DashboardLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  return (
    <div className="flex">
      <Sidebar />
      <main className="flex-1 p-6">{children}</main>
    </div>
  );
}
```

**Templates** are similar but re-mount on every navigation. Use them when you need enter/exit animations or per-page initialization logic.

## Server Components vs Client Components

By default, all components in the App Router are **React Server Components** (RSC). They run only on the server, can directly access databases and file systems, and send zero JavaScript to the client.

Add `'use client'` at the top of a file to opt into client-side interactivity. The boundary between server and client components is where serialization happens—only serializable props can cross this boundary.

## Data Fetching

Server components can use `async/await` directly. No need for `getServerSideProps` or `useEffect` chains.

```tsx
async function ProductPage({ params }: { params: { id: string } }) {
  const product = await db.product.findUnique({ where: { id: params.id } });
  return <ProductDetail product={product} />;
}
```
