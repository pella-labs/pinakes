# SSR Optimization Strategies

## The SSR Tax

**Server-Side Rendering** improves perceived performance and SEO but introduces server-side computation costs. Every request generates HTML on the server, consuming CPU and memory.

## Streaming SSR

React 18's streaming SSR sends HTML in chunks as components resolve. This reduces Time to First Byte and allows the browser to start rendering before the full response arrives.

```tsx
// Next.js App Router does this automatically
// For custom setups:
import { renderToPipeableStream } from 'react-dom/server';

app.get('/', (req, res) => {
  const { pipe } = renderToPipeableStream(<App />, {
    onShellReady() {
      res.statusCode = 200;
      res.setHeader('Content-Type', 'text/html');
      pipe(res);
    },
  });
});
```

## Selective Hydration

Not every part of the page needs to be interactive immediately. Wrap non-critical sections in `<Suspense>` to defer their hydration.

## Caching Strategies

- **Full-page caching**: store rendered HTML in CDN for static-like pages
- **Fragment caching**: cache expensive sub-trees independently
- **Stale-while-revalidate**: serve cached version, regenerate in background

## Edge Rendering

Move SSR to the edge (Cloudflare Workers, Vercel Edge Functions) to reduce latency. However, edge runtimes have limited APIs—no filesystem, limited npm compatibility.

See [[fe-003]] for how CSS-in-JS affects SSR performance and [[fe-021]] for ISR as a middle ground.
