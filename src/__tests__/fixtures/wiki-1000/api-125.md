# API Design Principles Summary

Core principles that guide all API design decisions in this project.

## Consistency Over Cleverness

Every endpoint should feel like it was designed by the same person. Use the same naming conventions, error formats, pagination style, and auth mechanism everywhere. A developer who learns one endpoint should immediately understand the rest.

## Explicit Over Implicit

- Require `scope` on every call — never silently default
- Return `request_id` so errors are traceable
- Include `deprecated` headers — don't quietly remove endpoints
- Document rate limits in headers — don't surprise callers

## Fail Loudly, Recover Gracefully

Return clear, actionable error messages. Never return `500 Internal Server Error` with no context. The error response should tell the client **what went wrong** and **what to do about it**.

But also: degrade gracefully. If the recommendation engine is down, return orders without recommendations rather than failing the entire request.

## Design for the Consumer

The client is the customer. Design from the outside in:

1. What does the client need to display?
2. What's the minimum data to satisfy that?
3. How do we serve that efficiently?

Don't make the client adapt to your database schema. Transform data at the API boundary. See [[api-110]] for the DTO pattern.

## Secure by Default

Auth required. HTTPS required. Input validated. Output filtered. Rate limited. Audit logged. These are not features — they're the baseline.
