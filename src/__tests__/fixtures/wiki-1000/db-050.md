# Stored Procedures and Functions

## The Debate

Stored procedures put business logic in the database. This is controversial — some teams swear by them, others ban them entirely.

### Arguments For

- Reduced network round-trips for complex operations
- Atomic multi-step operations without application-level transaction management
- Consistent behavior regardless of which application connects
- Performance (no serialization/deserialization overhead)

### Arguments Against

- Business logic hidden from code review and version control
- Harder to test and debug
- Database-specific (locks you to PostgreSQL, MySQL, etc.)
- Scaling is harder — database servers are more expensive than app servers
- Modern ORMs and connection poolers reduce the round-trip argument

## PostgreSQL Functions

```sql
CREATE OR REPLACE FUNCTION transfer_funds(
  from_account INT,
  to_account INT,
  amount DECIMAL
) RETURNS VOID AS $$
BEGIN
  UPDATE accounts SET balance = balance - amount WHERE id = from_account;
  UPDATE accounts SET balance = balance + amount WHERE id = to_account;

  IF (SELECT balance FROM accounts WHERE id = from_account) < 0 THEN
    RAISE EXCEPTION 'Insufficient funds';
  END IF;
END;
$$ LANGUAGE plpgsql;
```

## Modern Recommendation

Keep business logic in the application layer. Use database functions sparingly for performance-critical operations or data integrity constraints that must be enforced at the database level.
