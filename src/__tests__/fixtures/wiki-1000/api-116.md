# Designing Admin APIs

## Separate from Public API

Admin endpoints should be on a different path prefix or port:

```
/api/v1/orders              ← public/user-facing
/admin/v1/orders            ← internal/admin
/internal/v1/health         ← operations
```

## Admin-Specific Capabilities

- **Impersonation**: `X-Admin-Act-As: user_123` — perform actions as a user
- **Bypass limits**: no rate limiting for admin operations
- **Soft-delete recovery**: `POST /admin/orders/{id}/restore`
- **Bulk operations**: larger batch sizes, longer timeouts
- **Audit override**: explicit reason field required

## Access Control

Admin APIs require:
1. Valid admin JWT (separate issuer from user JWTs)
2. Specific admin role permission
3. IP allowlisting (office VPN only)
4. MFA verification for destructive operations

## Audit Requirements

Log everything. Admin operations are high-risk and high-scrutiny:

```json
{
  "admin_id": "admin_001",
  "action": "impersonate",
  "target_user": "user_123",
  "reason": "Support ticket #4567",
  "ip": "10.0.1.50"
}
```

See [[api-106]] for audit logging patterns.
