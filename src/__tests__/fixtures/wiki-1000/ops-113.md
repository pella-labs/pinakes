# GitOps Pull Request Workflow

## The Flow

In a GitOps workflow, infrastructure and application changes follow the same PR process as code:

1. Developer creates a branch in the GitOps repo
2. Updates manifests (image tag, config, scaling)
3. Opens a PR
4. CI validates manifests (kubeval, kube-score, policy checks)
5. Team reviews the diff
6. Merge triggers ArgoCD sync

## Automated Image Updates

When CI builds a new image, automatically open a PR to update the image tag in the GitOps repo:

```yaml
# In the CI pipeline
- name: Update GitOps repo
  run: |
    git clone https://github.com/myorg/k8s-manifests.git
    cd k8s-manifests
    yq e ".spec.template.spec.containers[0].image = \"$IMAGE\"" \
      -i services/api/base/deployment.yaml
    git add .
    git commit -m "chore: update api-server to $TAG"
    gh pr create --title "Deploy api-server $TAG" --body "Automated update"
```

## Benefits

- Every change has a Git history and review trail
- Rollback is just reverting a commit
- The GitOps repo is the single source of truth
- ArgoCD detects drift and self-heals
