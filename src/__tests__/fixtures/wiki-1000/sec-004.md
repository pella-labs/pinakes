# Role-Based Access Control

**RBAC** maps users to roles, and roles to permissions. It remains the most widely deployed authorization model because of its simplicity and auditability.

## Core Concepts

A role is a named collection of permissions. Users are assigned one or more roles. Permission checks ask "does any of this user's roles grant the required permission?"

- **Role**: a named group (e.g., `admin`, `editor`, `viewer`)
- **Permission**: a specific action on a resource (e.g., `documents:write`)
- **Assignment**: the mapping of user to role

## Database Schema

```sql
CREATE TABLE roles (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL UNIQUE,
  description TEXT
);

CREATE TABLE permissions (
  id TEXT PRIMARY KEY,
  resource TEXT NOT NULL,
  action TEXT NOT NULL,
  UNIQUE(resource, action)
);

CREATE TABLE role_permissions (
  role_id TEXT REFERENCES roles(id),
  permission_id TEXT REFERENCES permissions(id),
  PRIMARY KEY (role_id, permission_id)
);

CREATE TABLE user_roles (
  user_id TEXT NOT NULL,
  role_id TEXT REFERENCES roles(id),
  PRIMARY KEY (user_id, role_id)
);
```

## Limitations

RBAC struggles with fine-grained, context-dependent decisions. If you need "user can edit documents they created but only during business hours," you need [[sec-005]] (ABAC) or a hybrid approach.

## Audit Trail

Every role assignment and permission change should be logged. See [[sec-014]] for audit logging patterns.
