# Webhook Security

**Webhooks** deliver event notifications via HTTP callbacks. Securing them requires validating that incoming webhooks are authentic and that outgoing webhooks don't leak sensitive data.

## Incoming Webhook Verification

### HMAC Signature Verification

Most providers sign webhook payloads with a shared secret:

```typescript
import { createHmac, timingSafeEqual } from 'crypto';

function verifyWebhook(payload: string, signature: string, secret: string): boolean {
  const expected = createHmac('sha256', secret)
    .update(payload)
    .digest('hex');
  
  return timingSafeEqual(
    Buffer.from(signature),
    Buffer.from(`sha256=${expected}`)
  );
}
```

Use **timing-safe comparison** to prevent timing attacks.

### Timestamp Validation

Reject webhooks with timestamps older than 5 minutes to prevent replay attacks.

### IP Allowlisting

Some providers publish webhook source IP ranges. Use these as an additional validation layer.

## Outgoing Webhook Security

When sending webhooks to customer endpoints:

- Include HMAC signatures so recipients can verify authenticity
- Don't include sensitive data in the payload — send IDs and let recipients fetch details via API
- Implement exponential backoff for retries
- Respect customer TLS requirements

## Idempotency

Webhooks may be delivered more than once. Include a unique event ID so receivers can deduplicate. Design webhook handlers to be idempotent.

See [[sec-036]] for input validation on incoming webhook payloads.
