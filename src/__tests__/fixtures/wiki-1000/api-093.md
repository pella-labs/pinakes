# API Design for Event-Driven Architecture

## Event Publishing

Services publish domain events when state changes:

```typescript
interface DomainEvent {
  id: string;
  type: string;
  source: string;
  time: string;
  data: unknown;
  specversion: '1.0'; // CloudEvents spec
}
```

## CloudEvents Standard

Use the **CloudEvents** specification for event envelopes. It standardizes metadata across all services.

```json
{
  "specversion": "1.0",
  "id": "evt_abc",
  "type": "com.example.order.created",
  "source": "/orders",
  "time": "2025-11-03T14:30:00Z",
  "datacontenttype": "application/json",
  "data": {
    "orderId": "ord_123",
    "customerId": "cust_456"
  }
}
```

## Event Contracts

Publish schemas for all event types. Consumers should validate against the schema. Use a schema registry (Confluent, custom) to version event schemas.

## API Interaction

- `POST /events` — publish (for external producers)
- `GET /events?type=order.created&since=...` — consume (polling)
- `POST /subscriptions` — register a webhook consumer

Events complement APIs — the API is for commands (do something), events are for facts (something happened). See [[api-010]] for webhook delivery of events.
