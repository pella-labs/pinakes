# Database Connection Resilience

## Handling Transient Failures

Database connections fail. Networks blip. Servers restart. Your application must handle this gracefully.

## Retry Strategies

```typescript
async function withRetry<T>(
  fn: () => Promise<T>,
  maxRetries: number = 3,
  baseDelay: number = 100
): Promise<T> {
  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    try {
      return await fn();
    } catch (error) {
      if (attempt === maxRetries) throw error;
      const delay = baseDelay * Math.pow(2, attempt); // exponential backoff
      await new Promise(r => setTimeout(r, delay));
    }
  }
  throw new Error('unreachable');
}
```

## Connection Health Checks

Most connection pools support health checks:

- **Test on borrow**: validate the connection before handing it to the application
- **Periodic ping**: background thread validates idle connections
- **Evict stale**: remove connections idle beyond a threshold

## Circuit Breaker Pattern

After N consecutive failures, stop attempting connections for a cooldown period. This prevents cascading failures when the database is down and your application keeps exhausting resources trying to connect.

## Failover

For replicated databases, configure your application to automatically connect to a standby if the primary fails. PostgreSQL's `target_session_attrs=read-write` parameter helps route to the primary.

See [[db-010]] for connection pooling and [[db-015]] for replication topologies.
