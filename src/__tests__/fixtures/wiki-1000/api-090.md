# API Proxy Patterns

## Reverse Proxy

Client → Proxy → Backend. The proxy handles TLS, compression, caching, load balancing. The client doesn't know about the backend.

```nginx
upstream api_backend {
    server 10.0.1.1:8080;
    server 10.0.1.2:8080;
}

server {
    listen 443 ssl;
    location /api/ {
        proxy_pass http://api_backend/;
        proxy_set_header X-Request-ID $request_id;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

## API Proxy (Smart)

More than routing. An API proxy can:
- Transform requests/responses (rename fields, reshape)
- Add authentication
- Aggregate multiple backend calls
- Apply rate limiting

## Sidecar Proxy

In service mesh architectures, each service gets a sidecar proxy (Envoy, Linkerd). Handles mTLS, retries, circuit breaking, and observability without application code changes.

## Header Propagation

Proxies must forward these headers:
- `X-Request-ID` ([[api-041]])
- `X-Forwarded-For` (original client IP)
- `X-Forwarded-Proto` (original protocol)
- `traceparent` ([[api-083]])
