# Database Capacity Planning

## Estimating Storage

### Row Size Estimation

Calculate the average row size including column overhead:

| Type | Size |
|------|------|
| INTEGER | 4 bytes |
| BIGINT | 8 bytes |
| TEXT (avg 100 chars) | ~104 bytes |
| TIMESTAMPTZ | 8 bytes |
| UUID | 16 bytes |
| JSONB (avg 500 chars) | ~504 bytes |
| Row overhead (PostgreSQL) | 23 bytes |

### Total Storage

```
total_size = row_size * num_rows * (1 + index_overhead_factor)
```

Index overhead is typically 1.5x-3x the table size, depending on the number and type of indexes.

## Growth Projections

Track these monthly:
- Row count by table
- Total database size
- Index size as percentage of total
- WAL generation rate (affects backup storage and replication bandwidth)

## Connection Limits

Each PostgreSQL connection uses ~10MB of memory. With 200 max connections, that's 2GB just for connections. Use PgBouncer (see [[db-045]]) to keep server connections low.

## When to Scale

Scale vertically (bigger machine) as long as possible. Sharding (see [[db-021]]) introduces massive complexity. Most databases handle terabytes on a single server with proper indexing and partitioning.
