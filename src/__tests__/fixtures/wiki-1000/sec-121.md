# Security Testing Environments

Dedicated **security testing environments** provide safe spaces to test vulnerabilities without risking production systems.

## Environment Types

### Staging Mirror
A production-like environment with synthetic data. Used for DAST and pen testing.

Requirements:
- Network topology matches production
- Same security controls as production
- No real customer data (use realistic synthetic data)
- Isolated from production networks

### Local Security Lab
Developer-accessible environments for testing security fixes:

- Docker Compose setups mimicking the production stack
- Pre-configured with common vulnerability testing tools
- Includes intentionally vulnerable components for training

### Red Team Environment
Isolated environment for adversarial testing:

- Separate accounts and credentials
- No impact on development velocity
- Full monitoring to capture attack patterns (see [[sec-043]])

## Data Considerations

Never use production data in testing environments. Generate realistic synthetic data instead:

- Same schema and data types
- Similar volume and distribution
- No real PII, credentials, or business data
- Marked clearly as synthetic

## Network Isolation

Security testing environments must be isolated:

- Separate VPC/network from production
- No credentials that work in production
- Firewalled from corporate network
- Monitored for escaped traffic

See [[sec-028]] for pen testing methodology and [[sec-069]] for CI-integrated security testing.
