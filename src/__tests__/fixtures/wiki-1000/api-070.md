# GraphQL Schema Stitching vs Federation

## Schema Stitching (Legacy)

Merges schemas at the gateway by delegating fields to remote services. Requires manual type merging and conflict resolution.

```typescript
const gatewaySchema = stitchSchemas({
  subschemas: [
    { schema: orderSchema, url: 'http://orders:4001/graphql' },
    { schema: customerSchema, url: 'http://customers:4002/graphql' },
  ],
  typeMergingOptions: {
    Order: {
      selectionSet: '{ id }',
      fieldName: 'order',
    },
  },
});
```

## Federation (Current)

Apollo Federation uses directives (`@key`, `@external`, `@requires`) to declare relationships. The gateway auto-composes the supergraph. See [[api-062]] for details.

## When to Use Which

| | Stitching | Federation |
|---|---|---|
| Setup complexity | Low | Medium |
| Schema ownership | Central | Distributed |
| Type sharing | Manual | Directive-based |
| Tooling | Community | Apollo ecosystem |
| Best for | Small team, few services | Large org, many teams |

## Migration Path

If you're on stitching and scaling, migrate to Federation. The subgraph pattern gives each team ownership of their domain types.
