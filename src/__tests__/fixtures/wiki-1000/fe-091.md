# React Patterns: Higher-Order Components

## What Is an HOC

A **Higher-Order Component** is a function that takes a component and returns a new component with enhanced behavior. HOCs were the primary pattern for code reuse before hooks.

```tsx
function withAuth<P extends object>(Component: React.ComponentType<P>) {
  return function AuthenticatedComponent(props: P) {
    const { user, loading } = useAuth();

    if (loading) return <Spinner />;
    if (!user) return <Redirect to="/login" />;

    return <Component {...props} />;
  };
}

const ProtectedDashboard = withAuth(Dashboard);
```

## When HOCs Still Make Sense

- Wrapping third-party components that you cannot modify
- Cross-cutting concerns that need to wrap the entire render output
- Adding error boundaries or Suspense wrappers declaratively

## Problems with HOCs

- **Wrapper hell**: multiple HOCs create deeply nested component trees
- **Prop collisions**: multiple HOCs might inject same-named props
- **Static composition**: HOCs wrap at definition time, not render time
- **TypeScript complexity**: generic types through HOCs are painful

## Prefer Hooks

For most cases, custom hooks provide the same reuse without the wrapper overhead:

```tsx
function Dashboard() {
  const { user, loading } = useAuth();
  if (loading) return <Spinner />;
  if (!user) return <Redirect to="/login" />;
  return <DashboardContent />;
}
```
