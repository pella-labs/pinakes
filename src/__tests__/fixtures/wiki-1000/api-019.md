# Long Polling

A technique for near-real-time updates without WebSockets.

## How It Works

1. Client sends a GET request with a timeout (e.g., 30s)
2. Server holds the connection open until data is available or timeout expires
3. Server responds with data (or empty 304)
4. Client immediately sends the next request

## Implementation

```typescript
app.get('/poll/notifications', async (req, res) => {
  const since = req.query.since || '0';
  const timeout = Math.min(parseInt(req.query.timeout) || 30000, 60000);

  const result = await waitForNotifications(req.user.id, since, timeout);

  if (result.length > 0) {
    res.json({ data: result, cursor: result[result.length - 1].id });
  } else {
    res.status(304).end();
  }
});
```

## When to Use

- When WebSockets are overkill (low-frequency updates)
- Behind corporate proxies that kill WebSocket upgrades
- When you need simple HTTP semantics for load balancing

## When NOT to Use

- High-frequency updates (>1/second) — use SSE ([[api-020]]) or WebSockets ([[api-021]])
- Bidirectional communication — use WebSockets
- When connection count is a concern — each poller holds a connection
