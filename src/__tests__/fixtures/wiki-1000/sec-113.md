# API Rate Limiting Patterns

Different **rate limiting patterns** suit different scenarios. Choosing the right one depends on your API's characteristics and the threats you face.

## Fixed Window

Count requests per fixed time window (e.g., 100 per minute starting at :00).

Pros: simple, low memory
Cons: burst at window boundaries (up to 2x rate at window transitions)

## Sliding Window Log

Track each request timestamp. Count requests within the window ending at the current time.

Pros: accurate, no boundary bursts
Cons: memory-intensive (stores every timestamp)

## Sliding Window Counter

Combine fixed windows with interpolation:

```
current_rate = (previous_window_count * overlap_ratio) + current_window_count
```

Pros: accurate enough, low memory
Cons: slightly complex to implement

## Token Bucket

A bucket holds tokens that refill at a steady rate. Each request consumes a token. When the bucket is empty, requests are rejected.

Pros: allows controlled bursts, smooth rate enforcement
Cons: more state to manage

## Leaky Bucket

Requests fill a bucket that drains at a constant rate. Overflow is rejected.

Pros: perfectly smooth output rate
Cons: no burst tolerance

## Choosing a Pattern

| Scenario | Recommended |
|---|---|
| Simple API protection | Fixed window |
| Login endpoint | Sliding window (see [[sec-026]]) |
| Paid API with quota | Token bucket |
| Real-time processing | Leaky bucket |

## Distributed Rate Limiting

For multi-instance deployments, use a shared counter in Redis:

```lua
-- Redis Lua script for atomic rate limiting
local current = redis.call('incr', KEYS[1])
if current == 1 then
  redis.call('pexpire', KEYS[1], ARGV[1])
end
return current
```
