# AWS Organizations and Account Structure

## Multi-Account Strategy

A well-designed **AWS account structure** provides security isolation, billing separation, and blast radius reduction.

## Recommended Account Layout

```
Management Account (root)
├── Security OU
│   ├── Log Archive Account
│   └── Security Tooling Account
├── Infrastructure OU
│   ├── Network Hub Account
│   └── Shared Services Account
├── Workloads OU
│   ├── Dev Account
│   ├── Staging Account
│   └── Production Account
└── Sandbox OU
    └── Developer Sandbox Accounts
```

## Service Control Policies

**SCPs** set permission guardrails across the organization:

- Deny region usage outside approved regions
- Deny disabling CloudTrail
- Deny creating unencrypted resources
- Deny root user API calls (except for account-level tasks)

## Cross-Account Access

Use **IAM role assumption** for cross-account access. A role in the target account trusts the source account's roles. Never share credentials across accounts.

## Billing

- Enable **consolidated billing** at the organization level
- Use **cost allocation tags** for per-team attribution
- Set **budget alerts** per account

See [[ops-053]] for IAM, [[ops-090]] for cost allocation.
