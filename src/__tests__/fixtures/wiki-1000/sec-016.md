# Key Rotation Strategies

**Key rotation** is the practice of periodically replacing cryptographic keys. It limits the amount of data encrypted under any single key, reducing the impact of key compromise.

## Why Rotate

- Limits exposure if a key is compromised without detection
- Satisfies compliance requirements (PCI DSS, SOC2, HIPAA)
- Reduces the volume of data any single key protects
- Forces teams to maintain automated key management

## Rotation Patterns

### Envelope Encryption

Use a **data encryption key** (DEK) to encrypt data and a **key encryption key** (KEK) to encrypt the DEK. Rotating the KEK is cheap — you only re-encrypt the DEK, not all the data.

### Grace Period

After rotation, the old key remains active for decryption during a **grace period**. New encryptions use the new key. After the grace period, the old key is archived or destroyed.

### Automated Rotation

```yaml
# Vault transit key rotation policy
vault write transit/keys/my-key/config \
  auto_rotate_period=720h  # 30 days
```

## Key Versioning

Every encrypted blob should reference the key version used:

```
{version}:{iv}:{ciphertext}
```

This allows the system to select the correct key for decryption without trial-and-error. See [[sec-003]] for JWT `kid` headers that serve a similar purpose.
