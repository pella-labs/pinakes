# Secure Error Codes

A well-designed **error code system** balances debugging information for developers with minimal information disclosure to attackers.

## Error Code Design

```typescript
enum ErrorCode {
  AUTH_INVALID_CREDENTIALS = 'AUTH_001',
  AUTH_TOKEN_EXPIRED = 'AUTH_002',
  AUTH_INSUFFICIENT_PERMISSIONS = 'AUTH_003',
  RATE_LIMIT_EXCEEDED = 'RATE_001',
  VALIDATION_FAILED = 'VAL_001',
  INTERNAL_ERROR = 'INT_001',
}
```

## Response Structure

```json
{
  "error": {
    "code": "AUTH_002",
    "message": "Authentication token has expired",
    "requestId": "req-abc-123"
  }
}
```

## Security Considerations

### Don't Over-Differentiate

Authentication errors should be vague to prevent enumeration:

```typescript
// Bad — reveals whether the email exists
'AUTH_001: User not found'
'AUTH_002: Invalid password'

// Good — same error for both cases
'AUTH_001: Invalid credentials'
```

### Internal vs External Errors

Map internal error codes to less specific external codes:

| Internal | External | Reason |
|---|---|---|
| DB_CONNECTION_FAILED | INT_001 | Don't reveal database details |
| REDIS_TIMEOUT | INT_001 | Don't reveal cache infrastructure |
| USER_NOT_FOUND | AUTH_001 | Don't reveal user existence |

### Error Documentation

Document error codes for API consumers but don't document internal error mappings. See [[sec-059]] for secure error handling patterns.
