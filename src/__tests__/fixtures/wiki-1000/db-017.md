# ORM vs Raw SQL

## The Tension

Every team eventually has this debate. **ORMs** (Object-Relational Mappers) provide convenience and type safety. **Raw SQL** provides control and performance. The right answer depends on context.

## When ORMs Shine

- CRUD operations with straightforward mappings
- Schema management and migrations
- Type safety in TypeScript/Java/C# codebases
- Team members who aren't SQL experts

```typescript
// Drizzle ORM — clean, composable, SQL-like
const result = await db
  .select()
  .from(users)
  .where(eq(users.email, email))
  .limit(1);
```

## When Raw SQL Wins

- Complex analytical queries with CTEs, window functions, lateral joins
- Performance-critical paths where you need exact control over the query plan
- Database-specific features (PostgreSQL's `generate_series`, `RETURNING`, etc.)
- Bulk operations where ORM overhead is measurable

```sql
-- Good luck doing this cleanly in an ORM
WITH RECURSIVE subordinates AS (
  SELECT id, name, manager_id, 0 AS depth
  FROM employees WHERE id = $1
  UNION ALL
  SELECT e.id, e.name, e.manager_id, s.depth + 1
  FROM employees e
  JOIN subordinates s ON e.manager_id = s.id
)
SELECT * FROM subordinates;
```

## The Middle Ground

Use the ORM for 80% of your queries and drop to raw SQL for the complex 20%. Most modern ORMs support this — Drizzle has `sql` template literals, Prisma has `$queryRaw`.
