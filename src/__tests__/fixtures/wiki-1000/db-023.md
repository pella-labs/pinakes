# Graph Databases

## When Graphs Beat Tables

Relational databases handle most data well. But when your core queries involve traversing relationships — friends of friends, dependency trees, network paths — **graph databases** shine. The difference isn't just syntax; it's algorithmic. A 6-hop traversal in SQL requires 6 self-joins. In a graph database, it's a simple traversal.

## Neo4j and Cypher

Neo4j is the most mature graph database. Its query language, **Cypher**, reads like ASCII art:

```cypher
// Find friends of friends who like the same movies
MATCH (me:Person {name: "Alice"})-[:FRIENDS_WITH]->(friend)-[:FRIENDS_WITH]->(foaf)
WHERE NOT (me)-[:FRIENDS_WITH]->(foaf) AND me <> foaf
MATCH (foaf)-[:LIKES]->(movie:Movie)<-[:LIKES]-(me)
RETURN foaf.name, collect(movie.title) AS shared_movies
```

## Property Graph Model

Nodes have labels and properties. Edges (relationships) have types and properties. Both are first-class citizens with their own indexes.

## Use Cases

- Social networks (relationship traversal)
- Recommendation engines (collaborative filtering)
- Fraud detection (pattern matching)
- Knowledge graphs (semantic relationships)
- Dependency analysis (package managers, infrastructure)

## Alternatives

- **Amazon Neptune** — managed, supports both Gremlin and SPARQL
- **ArangoDB** — multi-model (graph + document + key-value)
- **SQLite with recursive CTEs** — surprisingly capable for small graphs
