# Environment Management

## Environment Hierarchy

We maintain four environments, each with increasing fidelity:

| Environment | Purpose | Data | Deploys |
|-------------|---------|------|---------|
| **local** | Developer laptops | Synthetic | On save |
| **dev** | Integration testing | Synthetic | On PR merge |
| **staging** | Pre-prod validation | Anonymized prod | On main merge |
| **production** | Live users | Real | Manual gate |

## Environment Parity

The gap between environments should be minimal. Use the same:
- Container images (just different tags)
- Kubernetes manifests (different values files)
- Database schema (same migrations)
- External service configurations (different endpoints)

## Configuration Management

Environment-specific config lives in:
- `values-{env}.yaml` for Helm
- `tfvars/{env}.tfvars` for Terraform
- `.env.{env}` for local development

Secrets are injected at deploy time from **AWS Secrets Manager** or **Vault**, never stored in Git.

## Promotion Flow

Code promotes through environments automatically after passing gates:

```
PR → dev (auto) → staging (auto on merge) → prod (manual approval)
```

See [[ops-001]] for pipeline architecture, [[ops-045]] for detailed promotion rules.
