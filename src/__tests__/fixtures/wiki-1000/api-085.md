# API Access Control Models

## RBAC (Role-Based Access Control)

Assign **roles** to users. Roles have **permissions**.

```json
{
  "roles": {
    "admin": ["orders:read", "orders:write", "orders:delete", "users:manage"],
    "manager": ["orders:read", "orders:write"],
    "viewer": ["orders:read"]
  }
}
```

Simple, well-understood, covers 80% of use cases.

## ABAC (Attribute-Based Access Control)

Decisions based on **attributes** of the user, resource, and environment:

```
ALLOW if user.department == resource.department
  AND user.clearance >= resource.classification
  AND environment.time between 09:00 and 17:00
```

More flexible than RBAC but harder to reason about and audit.

## Policy Engines

Use **OPA (Open Policy Agent)** or **Cedar** for complex policies:

```rego
# OPA policy
allow {
  input.method == "GET"
  input.user.roles[_] == "viewer"
}

allow {
  input.method == "POST"
  input.user.roles[_] == "admin"
  input.resource.tenant == input.user.tenant
}
```

## Enforcement Point

Always enforce at the **API layer**, not the database. The API should be the single chokepoint for access decisions.
