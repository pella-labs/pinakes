# Full-Text Search in PostgreSQL

## tsvector and tsquery

PostgreSQL's built-in **full-text search** uses two core types:

- `tsvector` — a sorted list of distinct lexemes (normalized words)
- `tsquery` — a search query with boolean operators

```sql
-- Create a tsvector column
ALTER TABLE articles ADD COLUMN search_vector tsvector;

-- Populate it
UPDATE articles SET search_vector =
  setweight(to_tsvector('english', coalesce(title, '')), 'A') ||
  setweight(to_tsvector('english', coalesce(body, '')), 'B');

-- Create a GIN index
CREATE INDEX idx_articles_search ON articles USING GIN (search_vector);

-- Query
SELECT title, ts_rank(search_vector, query) AS rank
FROM articles, to_tsquery('english', 'database & optimization') AS query
WHERE search_vector @@ query
ORDER BY rank DESC;
```

## Ranking

`ts_rank` scores results based on term frequency. `ts_rank_cd` uses cover density, which considers the proximity of matching terms. Assign weights (A through D) to give titles more importance than body text.

## When to Use Elasticsearch Instead

PostgreSQL FTS is excellent for moderate-scale search. Switch to Elasticsearch (see [[db-006]]) when you need faceted search, complex relevance tuning, or handling millions of documents with sub-100ms latency.
