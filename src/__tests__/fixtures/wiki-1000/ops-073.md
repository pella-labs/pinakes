# Kubernetes Namespace Design

## Namespace Strategy

Our namespace design follows the principle of **team-per-namespace** in production and **environment-per-namespace** in shared clusters.

## Production Cluster

```
kube-system          # Kubernetes system components
argocd               # ArgoCD control plane
cert-manager         # Certificate management
external-secrets     # Secret synchronization
monitoring           # Prometheus, Grafana, Loki
team-platform        # Platform services (API gateway, auth)
team-backend         # Backend microservices
team-data            # Data pipeline services
team-ml              # ML model serving
```

## Dev/Staging Cluster

```
kube-system
argocd
dev-api
dev-web
staging-api
staging-web
```

## Cross-Cutting Concerns

Network policies, resource quotas, and RBAC bindings are applied per namespace. Each team namespace gets:

- A resource quota appropriate for their workload
- Network policies allowing internal communication and blocking cross-team traffic
- RBAC bindings for the team's group

See [[ops-028]] for RBAC, [[ops-031]] for network policies, [[ops-034]] for resource quotas.
