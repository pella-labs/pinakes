# PostgreSQL Statistics Collector

## How Statistics Drive the Planner

The query planner makes decisions based on **table statistics** — row counts, distinct values, most common values, histogram bounds. Without accurate statistics, the planner makes bad choices.

## ANALYZE

```sql
-- Update statistics for a specific table
ANALYZE users;

-- Update statistics for the entire database
ANALYZE;

-- Verbose output
ANALYZE VERBOSE users;
```

## What Statistics Track

For each column, PostgreSQL stores:
- `n_distinct` — estimated number of distinct values
- `most_common_vals` + `most_common_freqs` — the most frequent values and their frequencies
- `histogram_bounds` — boundaries for equal-population histogram buckets
- `correlation` — physical vs logical ordering (affects index scan cost)

```sql
SELECT
  attname,
  n_distinct,
  most_common_vals,
  most_common_freqs
FROM pg_stats
WHERE tablename = 'orders' AND attname = 'status';
```

## Increasing Statistics Target

For columns with many distinct values, increase the statistics target:

```sql
ALTER TABLE orders ALTER COLUMN customer_id SET STATISTICS 500;
-- Default is 100; higher = more accurate but ANALYZE takes longer
```

## When Statistics Go Wrong

Symptoms: planner chooses sequential scan over index, or hash join over nested loop when data is small. Fix: run `ANALYZE`, check `n_distinct`, and if needed increase the statistics target.

See [[db-009]] for how to read EXPLAIN output.
