# Kubernetes Vertical Scaling

## Node Sizing

Choosing the right **node size** affects cost, performance, and scheduling efficiency.

### Many Small Nodes

- Better bin-packing efficiency
- Lower blast radius per node failure
- More overhead from system pods (kube-proxy, DaemonSets)
- Higher API server load

### Fewer Large Nodes

- Lower system overhead ratio
- Better for large pods
- Higher blast radius per node failure
- May waste resources if pods are small

## Our Approach

We use **m6i.xlarge** (4 vCPU, 16 GiB) as the default node type. This balances overhead with blast radius. For memory-intensive workloads, we have a dedicated **r6i.2xlarge** node pool.

## When to Scale Vertically

Vertical scaling makes sense when:
- A single pod needs more CPU or memory than the current node can provide
- You're running a database or cache that benefits from larger memory
- The workload is single-threaded and can't benefit from horizontal scaling

## Resource Fragmentation

Watch for **fragmentation** where nodes have unused CPU or memory that can't fit any pending pod. Karpenter handles this better than the Cluster Autoscaler by selecting the best-fit instance type per pod.

See [[ops-114]] for Karpenter, [[ops-095]] for resource management.
