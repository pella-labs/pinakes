# Security Architecture Patterns

Common **security architecture patterns** provide proven solutions to recurring security challenges. Understanding them helps you make better design decisions.

## Defense in Depth

Layer multiple independent security controls. If one layer fails, others still protect:

- Network layer: firewalls, segmentation
- Transport layer: TLS, mTLS
- Application layer: authentication, authorization
- Data layer: encryption at rest

## Bastion / Jump Host

A hardened server that serves as the single entry point to a private network. All administrative access flows through the bastion, which logs and controls access.

## Reverse Proxy / API Gateway

Centralize security controls at the edge:

- TLS termination
- Authentication
- Rate limiting
- WAF integration
- Request logging

See [[sec-044]] for microservices security and [[sec-029]] for WAF configuration.

## Circuit Breaker for Security

Adapt the circuit breaker pattern for security: when a downstream service shows signs of compromise (unusual error rates, unexpected response patterns), automatically isolate it.

## Secrets Injection Pattern

Inject secrets at runtime rather than baking them into deployments:

- Container orchestration: Kubernetes Secrets, mounted volumes
- Sidecar pattern: a Vault agent sidecar manages secret lifecycle
- Environment injection: CI/CD injects at deploy time (see [[sec-035]])

The goal is that secrets exist in memory only for the duration they're needed.
