# Audit Logging

**Audit logging** records security-relevant events for compliance, incident response, and forensics. Unlike application logging, audit logs have strict integrity and retention requirements.

## What to Log

Every audit event should capture:

- **Timestamp** (UTC, ISO 8601)
- **Actor** (user ID, service account, IP address)
- **Action** (what was done)
- **Resource** (what was affected)
- **Outcome** (success or failure)
- **Context** (request ID, session ID, source)

## Events to Capture

- Authentication attempts (success and failure)
- Authorization decisions (especially denials)
- Data access (reads of sensitive records)
- Data modification (creates, updates, deletes)
- Configuration changes
- Privilege escalation
- API key creation and revocation

## Implementation

```typescript
interface AuditEvent {
  timestamp: string;
  actor: { id: string; type: 'user' | 'service'; ip?: string };
  action: string;
  resource: { type: string; id: string };
  outcome: 'success' | 'failure';
  metadata: Record<string, unknown>;
}

function auditLog(event: AuditEvent): void {
  // Write to append-only store
  // Never allow deletion from audit log
}
```

## Storage Requirements

Audit logs must be **tamper-evident**. Use append-only storage, separate from the application database. Consider write-once storage like S3 Object Lock for compliance. See [[sec-022]] for SOC2 requirements that mandate audit log retention.

## Correlation

Include a **request ID** in every audit event so that related events across services can be correlated. This is essential for incident investigation.
