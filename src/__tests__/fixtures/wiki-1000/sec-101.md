# JWT Best Practices Summary

Pulling together all the lessons on **JWT security** into a concise reference. This complements the detailed coverage in [[sec-003]].

## Do

- Use asymmetric algorithms (RS256, ES256) for multi-service validation
- Include `iss`, `sub`, `aud`, `exp`, `iat`, `jti` claims
- Keep access token lifetime short (5-15 minutes)
- Validate every claim on every request
- Use the `kid` header for key rotation (see [[sec-016]])
- Store refresh tokens securely (see [[sec-032]])

## Don't

- Don't put sensitive data in JWT payloads (base64 is not encryption)
- Don't accept the `alg: none` algorithm
- Don't accept unexpected algorithms (especially switching from RS to HS)
- Don't skip audience validation
- Don't use JWTs for sessions that need instant revocation (see [[sec-071]])
- Don't store JWTs in localStorage (see [[sec-075]])

## The `alg: none` Attack

An attacker modifies the JWT header to `"alg": "none"` and removes the signature. If the library accepts `none`, the token is treated as valid without verification. Always explicitly specify allowed algorithms.

## Algorithm Confusion

An attacker changes `alg` from RS256 to HS256 and uses the public key as the HMAC secret. If the library allows this, the attacker can forge tokens. Specify the expected algorithm explicitly — never let the token dictate which algorithm to use.
