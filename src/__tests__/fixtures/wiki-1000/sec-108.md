# OAuth2 Token Introspection

**Token introspection** (RFC 7662) allows a resource server to query the authorization server about the state of an access token. It's essential for validating opaque tokens.

## When to Use

- Opaque (non-JWT) access tokens that can't be validated locally
- When you need real-time token status (is it revoked?)
- When the resource server doesn't have the signing key

## Request

```http
POST /introspect HTTP/1.1
Host: auth.example.com
Content-Type: application/x-www-form-urlencoded
Authorization: Basic czZCaGRSa3F0MzpnWDFmQmF0M2JW

token=mF_9.B5f-4.1JqM&token_type_hint=access_token
```

## Response

```json
{
  "active": true,
  "scope": "read write",
  "client_id": "my-app",
  "sub": "user-123",
  "exp": 1709234567,
  "iat": 1709230967
}
```

If `active` is `false`, reject the token immediately. Don't trust any other fields in an inactive response.

## Performance Considerations

Introspection adds a network round-trip to every API call. Mitigate with:

- **Caching**: cache positive introspection results (with TTL shorter than token expiry)
- **JWTs for most cases**: self-contained validation, no round-trip
- **Introspection for revocation**: use JWT locally, introspect only when checking revocation

## Security

The introspection endpoint must be authenticated. Only registered resource servers should be able to introspect tokens. See [[sec-001]] for the broader OAuth2 flow.
