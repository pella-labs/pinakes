# Redis Data Structures

## Beyond Key-Value

Redis is commonly described as a key-value store, but that undersells its capability. It's really a **data structure server** — each value can be a string, list, set, sorted set, hash, stream, bitmap, or HyperLogLog.

## Strings and Hashes

Strings are the simplest type. Hashes are the go-to for object-like data:

```
HSET user:1001 name "Alice" email "alice@example.com" role "admin"
HGET user:1001 name
HGETALL user:1001
```

## Sorted Sets for Rankings

**Sorted sets** are Redis's most powerful primitive. Each member has a score, and the set is ordered by score. This enables leaderboards, rate limiters, priority queues, and sliding-window counters.

```
ZADD leaderboard 1500 "player:42"
ZADD leaderboard 2300 "player:17"
ZREVRANGE leaderboard 0 9 WITHSCORES  -- top 10
```

## Streams for Event Logs

Redis Streams (added in 5.0) provide an append-only log structure with consumer groups. They're a lightweight alternative to Kafka for moderate-throughput event pipelines. See [[db-097]] for comparing Redis Streams vs. dedicated message brokers.

## Memory Considerations

- Use `MEMORY USAGE <key>` to inspect per-key overhead
- Enable **maxmemory-policy** (e.g., `allkeys-lru`) to handle memory pressure
- Consider **Redis Cluster** for horizontal scaling beyond single-node memory
