# Spot Instances and Cost Optimization

## What Are Spot Instances

**Spot instances** are spare EC2 capacity available at up to 90% discount. The tradeoff: AWS can reclaim them with 2 minutes notice. They're ideal for fault-tolerant, stateless workloads.

## Good Candidates for Spot

- CI/CD build agents
- Batch processing jobs
- Kubernetes worker nodes (with proper pod disruption budgets)
- Data processing pipelines
- Dev/staging environments

## Bad Candidates for Spot

- Single-instance databases
- Stateful services without replication
- Latency-sensitive services with slow startup

## Spot Strategy for EKS

Use **Karpenter** with spot instance diversification:

```yaml
apiVersion: karpenter.sh/v1beta1
kind: NodePool
metadata:
  name: spot-workers
spec:
  template:
    spec:
      requirements:
        - key: karpenter.sh/capacity-type
          operator: In
          values: ["spot"]
        - key: node.kubernetes.io/instance-type
          operator: In
          values: ["m5.xlarge", "m5a.xlarge", "m6i.xlarge", "m6a.xlarge"]
      nodeClassRef:
        name: default
  disruption:
    consolidationPolicy: WhenUnderutilized
```

Diversifying across multiple instance types reduces the chance of simultaneous interruption.
