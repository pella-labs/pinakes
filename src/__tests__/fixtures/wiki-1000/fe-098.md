# Environment Variables in Frontend Apps

## The Problem

Frontend code runs in the browser—there is no `process.env` at runtime. Environment variables must be inlined at build time.

## Next.js Convention

```bash
# .env.local
DATABASE_URL=postgres://...     # Server only (not prefixed)
NEXT_PUBLIC_API_URL=https://... # Exposed to browser (NEXT_PUBLIC_ prefix)
```

Only variables prefixed with `NEXT_PUBLIC_` are bundled into client JavaScript. Everything else stays server-side.

## Vite Convention

```bash
VITE_API_URL=https://...        # Exposed via import.meta.env.VITE_API_URL
SECRET_KEY=...                  # NOT exposed
```

## Validation at Build Time

Validate that required variables exist using Zod:

```typescript
import { z } from 'zod';

const envSchema = z.object({
  NEXT_PUBLIC_API_URL: z.string().url(),
  NEXT_PUBLIC_ANALYTICS_ID: z.string().optional(),
});

export const env = envSchema.parse({
  NEXT_PUBLIC_API_URL: process.env.NEXT_PUBLIC_API_URL,
  NEXT_PUBLIC_ANALYTICS_ID: process.env.NEXT_PUBLIC_ANALYTICS_ID,
});
```

## Security Considerations

- Never expose API keys or secrets via public env vars
- Audit `NEXT_PUBLIC_` / `VITE_` prefixed variables regularly
- Use server-side API routes as a proxy for sensitive API calls

See [[fe-092]] for Zod validation patterns.
