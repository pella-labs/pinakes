# Temporal Data Patterns

## Tracking History

Many applications need to answer "what was the value at time T?" Standard tables only store the current state. Temporal patterns add time dimensions.

## System-Time Versioning

SQL:2011 introduced **temporal tables** with system-time versioning. PostgreSQL doesn't support this natively yet, but you can implement it:

```sql
CREATE TABLE products (
  id INT PRIMARY KEY,
  name TEXT NOT NULL,
  price DECIMAL NOT NULL,
  valid_from TIMESTAMPTZ NOT NULL DEFAULT now(),
  valid_to TIMESTAMPTZ NOT NULL DEFAULT 'infinity'
);

-- Create an exclusion constraint to prevent overlaps
CREATE EXTENSION IF NOT EXISTS btree_gist;
ALTER TABLE products ADD CONSTRAINT no_overlap
  EXCLUDE USING gist (id WITH =, tstzrange(valid_from, valid_to) WITH &&);
```

## Point-in-Time Queries

```sql
-- Current price
SELECT * FROM products WHERE id = 1 AND valid_to = 'infinity';

-- Price as of a specific date
SELECT * FROM products
WHERE id = 1
  AND valid_from <= '2024-06-15'
  AND valid_to > '2024-06-15';
```

## Alternatives

- **Event sourcing** — store all state changes as events, derive current state (see [[db-112]])
- **Audit tables** — trigger-based logging of changes
- **Slowly Changing Dimensions** — data warehouse pattern for tracking dimension history (see [[db-036]])
