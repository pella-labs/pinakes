# Multi-Region Architecture

## Why Multi-Region

Running in multiple regions provides **geographic redundancy**, lower latency for distributed users, and compliance with data residency requirements. It also enables true disaster recovery with near-zero RTO.

## Architecture Patterns

### Active-Passive

One region handles all traffic. The passive region stays warm and receives replicated data. Failover switches DNS to the passive region.

### Active-Active

Both regions handle traffic simultaneously. Data is replicated bidirectionally. This is harder to implement but provides the best availability.

## Data Replication Challenges

The hardest part of multi-region is data. You must choose between:

- **Strong consistency**: higher latency, simpler conflict resolution
- **Eventual consistency**: lower latency, must handle conflicts

For most services, eventual consistency with conflict resolution is the pragmatic choice. Use **last-writer-wins** for simple cases and **CRDTs** for complex ones.

## Implementation Checklist

- DNS with health checks and failover routing
- Database replication (Aurora Global, DynamoDB Global Tables)
- S3 Cross-Region Replication for objects
- Session storage in a shared layer (ElastiCache Global Datastore)
- Circuit breakers for cross-region calls
- Region-aware configuration in application code
