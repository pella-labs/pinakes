# REST vs GraphQL: When to Choose

## REST Wins When

- You need aggressive HTTP caching (CDN, browser)
- The data model maps cleanly to resources
- Clients are diverse and uncontrolled (public API)
- You want simplicity and wide tooling support

## GraphQL Wins When

- Clients need flexible queries (mobile vs web different fields)
- You'd otherwise build many custom REST endpoints
- The data is highly relational (graph-like)
- You want strong typing across the API boundary

## Neither Wins When

- High-throughput binary streaming → use gRPC ([[api-009]])
- Real-time bidirectional → use WebSockets ([[api-021]])
- Simple internal RPC → gRPC or even plain HTTP POST

## The Pragmatic Approach

Most teams don't need to pick just one. Use REST for public APIs and simple CRUD. Use GraphQL for complex frontend data fetching. Use gRPC for service-to-service. The gateway layer ([[api-025]]) unifies them for external consumers.
