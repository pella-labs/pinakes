# Table Partitioning

## Partitioning vs Sharding

**Partitioning** splits a single table into smaller physical pieces within the same database. **Sharding** (see [[db-021]]) splits data across different database instances. Partitioning is simpler, safer, and should be tried first.

## PostgreSQL Declarative Partitioning

```sql
CREATE TABLE events (
  id BIGSERIAL,
  event_type TEXT NOT NULL,
  payload JSONB,
  created_at TIMESTAMPTZ NOT NULL
) PARTITION BY RANGE (created_at);

CREATE TABLE events_2024_q1 PARTITION OF events
  FOR VALUES FROM ('2024-01-01') TO ('2024-04-01');
CREATE TABLE events_2024_q2 PARTITION OF events
  FOR VALUES FROM ('2024-04-01') TO ('2024-07-01');
```

## Partition Strategies

- **Range** — by date, ID range, or numeric value. Best for time-series data.
- **List** — by discrete values (region, status, tenant). Good for multi-tenant.
- **Hash** — by hash of a column. Even distribution when ranges aren't natural.

## Benefits

- **Partition pruning**: queries that filter on the partition key only scan relevant partitions
- **Efficient maintenance**: DROP a partition instead of DELETE millions of rows
- **Parallel query execution**: each partition can be scanned concurrently

## Anti-Patterns

- Partitioning small tables (overhead exceeds benefit below ~10M rows)
- Wrong partition key (queries that don't filter on it get slower, not faster)
- Too many partitions (>1000 causes planner overhead)
