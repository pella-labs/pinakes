# Kubernetes StatefulSets

## When to Use StatefulSets

**StatefulSets** are for workloads that need stable network identities and persistent storage. Examples include databases, message brokers, and distributed caches.

## Key Differences from Deployments

- Pods get predictable names: `my-app-0`, `my-app-1`, `my-app-2`
- Pods are created and deleted in order
- Each pod gets its own PersistentVolumeClaim
- DNS entries are stable: `my-app-0.my-app-headless.namespace.svc.cluster.local`

## Example: Redis Cluster

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: redis
spec:
  serviceName: redis-headless
  replicas: 3
  selector:
    matchLabels:
      app: redis
  template:
    metadata:
      labels:
        app: redis
    spec:
      containers:
        - name: redis
          image: redis:7-alpine
          ports:
            - containerPort: 6379
          volumeMounts:
            - name: data
              mountPath: /data
  volumeClaimTemplates:
    - metadata:
        name: data
      spec:
        accessModes: ["ReadWriteOnce"]
        storageClassName: gp3
        resources:
          requests:
            storage: 10Gi
```

## Operational Considerations

Scaling down a StatefulSet does not delete PVCs. This is intentional — data is preserved for scale-up. Clean up orphaned PVCs manually or with automation.
