# PostgreSQL Bitmap Scans

## When Bitmap Scans Occur

The query planner chooses a **bitmap scan** when too many rows match for an index scan (which would cause random I/O) but too few for a sequential scan. It's the middle ground.

## How It Works

1. **Bitmap Index Scan**: traverses the index, builds a bitmap of matching heap pages
2. **Bitmap Heap Scan**: reads the matching pages in physical order (sequential I/O)

The key insight: instead of bouncing between random heap pages (as an index scan would), it reads pages in order.

## Multiple Index Combination

Bitmap scans can combine multiple indexes using BitmapAnd and BitmapOr:

```sql
-- Can use bitmap scans on both idx_status and idx_date
SELECT * FROM orders
WHERE status = 'active' AND created_at > '2024-01-01';
```

EXPLAIN output:
```
Bitmap Heap Scan on orders
  Recheck Cond: (status = 'active' AND created_at > '2024-01-01')
  ->  BitmapAnd
        ->  Bitmap Index Scan on idx_status
        ->  Bitmap Index Scan on idx_date
```

## Lossy Bitmap

If the bitmap exceeds `work_mem`, it becomes "lossy" — tracking pages instead of individual rows. The Bitmap Heap Scan then rechecks conditions for all rows on those pages.

Increase `work_mem` if you see many recheck rows in EXPLAIN output. See [[db-099]] for the full EXPLAIN reference.
