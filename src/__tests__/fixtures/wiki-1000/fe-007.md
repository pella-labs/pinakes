# Tailwind CSS Best Practices

## Utility-First Philosophy

**Tailwind CSS** provides low-level utility classes that compose to build any design. Instead of writing custom CSS, you apply pre-existing classes directly in your markup. This eliminates naming conventions, reduces specificity conflicts, and keeps styles co-located with the markup they affect.

## Organizing Complex Class Lists

Use the `clsx` or `tailwind-merge` library to conditionally apply classes:

```tsx
import { twMerge } from 'tailwind-merge';
import clsx from 'clsx';

function Button({ variant, size, className, ...props }: ButtonProps) {
  return (
    <button
      className={twMerge(
        clsx(
          'inline-flex items-center justify-center rounded-md font-medium',
          'transition-colors focus-visible:outline-none focus-visible:ring-2',
          variant === 'primary' && 'bg-blue-600 text-white hover:bg-blue-700',
          variant === 'secondary' && 'bg-gray-100 text-gray-900 hover:bg-gray-200',
          size === 'sm' && 'h-8 px-3 text-sm',
          size === 'md' && 'h-10 px-4 text-base',
        ),
        className,
      )}
      {...props}
    />
  );
}
```

## Design Tokens via Config

Centralize your design tokens in `tailwind.config.ts`:

- Colors, spacing, and typography scales
- Breakpoints for responsive design
- Custom animations and transitions

## Purging and Bundle Size

Tailwind's JIT compiler generates only the classes you use. The production CSS bundle is typically 5-15KB gzipped. Never disable purging in production. See [[fe-045]] for design system token architecture.
