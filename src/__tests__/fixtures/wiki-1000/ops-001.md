# CI/CD Pipeline Architecture

## Overview

A well-designed **CI/CD pipeline** transforms how teams ship software. The goal is simple: every commit that passes automated checks should be deployable to production with minimal human intervention.

## Pipeline Stages

The typical pipeline flows through these stages:

1. **Source** — triggered by a push or pull request
2. **Build** — compile, bundle, generate artifacts
3. **Test** — unit, integration, e2e
4. **Security scan** — SAST, dependency audit
5. **Deploy to staging** — automatic for main branch
6. **Deploy to production** — manual gate or auto-promote

## Branch Strategy

We use **trunk-based development** with short-lived feature branches. The main branch is always deployable. Feature branches live at most 48 hours before merging.

See [[ops-003]] for our GitHub Actions workflow definitions and [[ops-010]] for the deployment strategy.

## Key Metrics

Track these to measure pipeline health:

- **Lead time**: commit to production (target: <15 min)
- **Deployment frequency**: how often we ship (target: multiple per day)
- **MTTR**: mean time to recovery (target: <1 hour)
- **Change failure rate**: percentage of deploys causing incidents (target: <5%)

These are the **DORA metrics** and they directly correlate with team performance.
