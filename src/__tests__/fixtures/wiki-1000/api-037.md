# API Throttling vs Rate Limiting

These are different things.

## Rate Limiting

Hard cap on requests per time window. Client gets `429` when exceeded. Protects the server from abuse.

**Token bucket** for steady-state limiting. **Fixed window** for simplicity. **Sliding window** for fairness. See [[api-004]] for implementation.

## Throttling

Graceful degradation under load. Instead of rejecting, the server **slows down** responses or reduces response quality.

Examples:
- Return cached data instead of fresh queries
- Skip expensive enrichments (no avatars, no counts)
- Reduce page size from 100 to 25
- Add artificial delay to non-priority requests

## Load Shedding

More aggressive than throttling. When the server is overloaded, **immediately reject** low-priority requests to protect high-priority ones.

Priority tiers:
1. Health checks (never shed)
2. Authenticated paid-tier requests
3. Authenticated free-tier requests
4. Anonymous requests (first to shed)

## Implementation Signal

Use **request queue depth** as the signal, not CPU or memory. If the queue exceeds N, start shedding. CPU-based signals lag too much.
