# React Context Optimization

## The Re-Render Problem

When a context value changes, every component that consumes that context re-renders—even if it only uses a subset of the context that didn't change.

## Splitting Contexts

The primary solution is splitting large contexts into smaller, focused ones:

```tsx
// Instead of one big context
const AppContext = createContext({ user, theme, locale, settings });

// Split into focused contexts
const UserContext = createContext(user);
const ThemeContext = createContext(theme);
const LocaleContext = createContext(locale);
```

Components that only need the theme won't re-render when the user changes.

## Memoization

If splitting isn't practical, memoize the context value:

```tsx
function AppProvider({ children }) {
  const [user, setUser] = useState(null);
  const [theme, setTheme] = useState('light');

  const value = useMemo(() => ({ user, theme, setUser, setTheme }), [user, theme]);

  return <AppContext.Provider value={value}>{children}</AppContext.Provider>;
}
```

## use(context) in React 19

React 19's `use` hook can read context conditionally, unlike `useContext`. This enables patterns where context is only consumed when needed, reducing unnecessary subscriptions.

## When to Reach for External State

If context performance becomes a real issue (not a theoretical one), consider moving to [[fe-002]] solutions like Zustand or Jotai, which have built-in selector support to prevent unnecessary re-renders.
