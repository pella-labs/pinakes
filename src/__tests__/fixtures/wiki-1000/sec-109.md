# Secure Caching Strategies

Caching improves performance but can introduce security vulnerabilities if sensitive data is cached inappropriately.

## Cache Poisoning

An attacker manipulates cache keys or values so that malicious content is served to other users.

### Prevention

- Use the full URL (including query parameters and relevant headers) as the cache key
- Validate the `Host` header before using it in cache keys
- Don't cache responses that vary by `Authorization` header
- Set `Vary` headers correctly so caches differentiate responses

## Sensitive Data in Caches

### HTTP Cache Headers

```
# Public endpoint — safe to cache
Cache-Control: public, max-age=3600

# Authenticated endpoint — never cache in shared caches
Cache-Control: private, no-store

# Sensitive data — never cache anywhere
Cache-Control: no-store
```

### Application Cache

When using Redis or Memcached for application caching:

- Encrypt sensitive data before caching
- Set appropriate TTLs — cached credentials should expire quickly
- Clear cache entries when the underlying data changes
- Use separate cache namespaces for different security contexts

## CDN Security

If using a CDN:

- Never cache responses containing authentication tokens
- Be aware of cache key manipulation attacks
- Use origin authentication to prevent cache poisoning
- Monitor for unexpected cache hit rates

## Session Data

If caching session data in Redis:

- Encrypt session data at rest
- Set TTLs matching your session timeout
- Use TLS for Redis connections (see [[sec-102]])
- Restrict Redis access to application servers only

See [[sec-081]] for API-specific caching headers.
