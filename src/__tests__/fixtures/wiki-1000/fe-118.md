# Polymorphic Components

## The Problem

You want a `<Button>` component that renders as a `<button>` by default but can also render as an `<a>`, a Next.js `<Link>`, or any other element while preserving type safety.

## Implementation

```tsx
type PolymorphicProps<E extends React.ElementType, P = {}> = P & {
  as?: E;
} & Omit<React.ComponentPropsWithRef<E>, keyof P | 'as'>;

function Button<E extends React.ElementType = 'button'>({
  as,
  children,
  variant = 'primary',
  ...props
}: PolymorphicProps<E, { variant?: 'primary' | 'secondary'; children: React.ReactNode }>) {
  const Component = as ?? 'button';
  return (
    <Component className={getVariantClass(variant)} {...props}>
      {children}
    </Component>
  );
}

// Usage
<Button>Click me</Button>                    // renders <button>
<Button as="a" href="/about">About</Button>  // renders <a>
<Button as={Link} to="/dashboard">Go</Button> // renders <Link>
```

## Type Safety

The `as` prop changes which HTML attributes are valid. When `as="a"`, `href` becomes required. When `as="button"`, `type` is available. TypeScript enforces this at compile time.

## Libraries

**Radix UI** and **Headless UI** use this pattern extensively. The `@radix-ui/react-slot` component provides a simpler alternative using the `asChild` prop.

See [[fe-035]] for TypeScript patterns and [[fe-044]] for component library approaches.
