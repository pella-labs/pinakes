# Secrets in CI/CD Pipelines

CI/CD pipelines handle deployment credentials, signing keys, and API tokens. A compromised pipeline can deploy backdoored code to production.

## Common Risks

- Secrets printed in build logs
- Secrets accessible to all repository contributors
- Secrets persisted in build caches or artifacts
- Pipeline configuration pulled from untrusted forks

## Best Practices

### Secret Injection
Use your CI platform's native secret management:

- **GitHub Actions**: repository or environment secrets
- **GitLab CI**: CI/CD variables marked as protected and masked
- **CircleCI**: contexts and environment variables

Never store secrets in the repository, even in encrypted form.

### Log Redaction
CI platforms mask known secrets in logs, but derived values may leak:

```yaml
# DANGEROUS — secret may appear in the URL
- run: curl "https://api.example.com?key=$API_KEY"

# SAFER — pass via header
- run: |
    curl -H "Authorization: Bearer $API_KEY" https://api.example.com
```

### Environment Scoping
Restrict production secrets to production deployment jobs:

```yaml
deploy-prod:
  environment: production
  # Only jobs targeting 'production' environment can access these secrets
```

### Fork Protection

Pull requests from forks should **never** have access to secrets. Most CI platforms enforce this by default — verify it's not disabled.
