# Cloud IAM Best Practices

**Cloud IAM** (Identity and Access Management) controls who can do what in your cloud infrastructure. Misconfigured IAM is the leading cause of cloud security breaches.

## Principles

- **Least privilege**: start with no permissions, add only what's needed
- **Short-lived credentials**: prefer temporary credentials over permanent keys
- **Separation of duties**: no single role should have unchecked power
- **Audit everything**: enable and monitor IAM activity logs

## AWS IAM Specifics

- Never use the root account for daily operations
- Require MFA for all human users (see [[sec-056]])
- Use IAM roles for EC2 instances, not access keys
- Use Organizations SCPs to set permission boundaries
- Enable IAM Access Analyzer to find unused permissions

## Service Account Management

- Create dedicated service accounts per application
- Use workload identity (GKE, EKS) instead of static keys
- Rotate service account keys on a schedule (see [[sec-016]])
- Monitor for unused service accounts and deactivate them

## Permission Boundaries

Use permission boundaries to cap the maximum permissions a role can have:

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["s3:GetObject", "s3:PutObject"],
    "Resource": "arn:aws:s3:::my-bucket/*"
  }]
}
```

## Cross-Account Access

When resources are shared between accounts:

- Use cross-account IAM roles, not shared credentials
- Require external IDs to prevent confused deputy attacks
- Log all cross-account access
