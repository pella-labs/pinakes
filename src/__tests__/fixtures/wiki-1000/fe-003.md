# CSS-in-JS Performance Considerations

## Runtime Cost

CSS-in-JS libraries like **styled-components** and **Emotion** generate styles at runtime. This means every render can trigger style recalculation, serialization, and injection into the DOM. For most applications, this overhead is negligible—but in performance-critical paths with frequent re-renders, it adds up.

## Zero-Runtime Alternatives

Libraries such as **vanilla-extract**, **Linaria**, and **Panda CSS** extract styles at build time. They offer the DX of CSS-in-JS without the runtime penalty.

```typescript
// vanilla-extract example
import { style } from '@vanilla-extract/css';

export const container = style({
  display: 'flex',
  padding: '16px',
  backgroundColor: 'var(--bg-primary)',
  '@media': {
    '(max-width: 768px)': {
      flexDirection: 'column',
    },
  },
});
```

## Benchmark Data

In a synthetic benchmark rendering 1000 list items with dynamic styles:

- **styled-components**: ~12ms per render cycle
- **Emotion**: ~10ms per render cycle  
- **vanilla-extract**: ~3ms per render cycle
- **Tailwind (utility classes)**: ~2ms per render cycle

The gap widens with SSR. Runtime CSS-in-JS must serialize styles on the server, increasing Time to First Byte. See [[fe-019]] for SSR optimization strategies.

## Recommendations

For new projects, consider **Tailwind CSS** or **vanilla-extract** first. Reserve runtime CSS-in-JS for cases where truly dynamic styles (based on props that change every frame) are needed. The [[fe-045]] article on design systems discusses how to structure tokens for either approach.
