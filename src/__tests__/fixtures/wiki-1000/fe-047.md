# CSS Modules and Scoping

## The Scoping Problem

Global CSS leads to naming collisions and unintended style overrides. **CSS Modules** solve this by generating unique class names at build time.

## How They Work

```css
/* Button.module.css */
.button {
  padding: 8px 16px;
  border-radius: 4px;
  font-weight: 600;
}

.primary {
  background-color: #3b82f6;
  color: white;
}
```

```tsx
import styles from './Button.module.css';

function Button({ variant }: ButtonProps) {
  return (
    <button className={`${styles.button} ${styles[variant]}`}>
      Click me
    </button>
  );
}
```

The generated class names look like `Button_button_a1b2c`, preventing collisions.

## Composition

CSS Modules support `composes` for sharing styles:

```css
.base {
  padding: 8px 16px;
  border-radius: 4px;
}

.primary {
  composes: base;
  background: blue;
}
```

## Comparison with Other Approaches

CSS Modules offer a good balance of scoping and simplicity. They work with any CSS preprocessor, don't require runtime JavaScript, and have native support in Next.js, Vite, and webpack.
