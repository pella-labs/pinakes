# Kubernetes Affinity and Anti-Affinity

## Node Affinity

Schedule pods on specific nodes based on node labels:

```yaml
affinity:
  nodeAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      nodeSelectorTerms:
        - matchExpressions:
            - key: topology.kubernetes.io/zone
              operator: In
              values: ["us-east-1a", "us-east-1b"]
    preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 80
        preference:
          matchExpressions:
            - key: node.kubernetes.io/instance-type
              operator: In
              values: ["m6i.xlarge"]
```

## Pod Affinity

Co-locate pods that communicate frequently to reduce network latency:

```yaml
affinity:
  podAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchLabels:
            app: cache
        topologyKey: kubernetes.io/hostname
```

## Pod Anti-Affinity

Spread replicas across zones for high availability:

```yaml
affinity:
  podAntiAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchLabels:
            app: api-server
        topologyKey: topology.kubernetes.io/zone
```

## Required vs Preferred

- **Required**: hard constraint. Pod won't schedule if it can't be satisfied.
- **Preferred**: soft constraint. Scheduler tries but will schedule elsewhere if necessary.

Use required for critical availability guarantees and preferred for optimization.

See [[ops-041]] for scheduling.
