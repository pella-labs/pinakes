# Secure Configuration of Reverse Proxies

**Reverse proxies** (Nginx, HAProxy, Envoy, Caddy) sit at the edge of your infrastructure. Their configuration directly impacts your security posture.

## Nginx Security Configuration

```nginx
server {
    listen 443 ssl http2;
    server_name api.example.com;
    
    # TLS
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;
    ssl_prefer_server_ciphers on;
    ssl_session_timeout 1d;
    ssl_session_cache shared:SSL:10m;
    
    # Security headers
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header X-Content-Type-Options "nosniff" always;
    add_header X-Frame-Options "DENY" always;
    
    # Hide server info
    server_tokens off;
    
    # Request limits
    client_max_body_size 10m;
    client_body_timeout 30s;
    client_header_timeout 30s;
    
    # Rate limiting
    limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;
    limit_req zone=api burst=20 nodelay;
}
```

## Common Mistakes

- Leaving default server blocks that respond to any hostname
- Not setting request size limits (DoS vector)
- Proxying internal headers to upstream (X-Forwarded-For spoofing)
- Not restricting access to health/status endpoints
- Using self-signed certs in production

## Header Sanitization

Strip or validate headers before forwarding to upstream:

- Remove `X-Forwarded-For` from external requests (set it yourself)
- Validate `Host` header against expected values
- Remove debug headers from responses

See [[sec-017]] for TLS configuration and [[sec-033]] for security headers.
