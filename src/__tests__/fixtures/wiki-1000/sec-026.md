# Security-Focused Rate Limiting

**Rate limiting** isn't just about traffic management — it's a critical security control that prevents brute force attacks, credential stuffing, and API abuse.

## Security-Specific Limits

### Authentication Endpoints

Apply aggressive rate limits to login and password reset:

- **5 attempts per minute** per IP for login
- **3 attempts per minute** per account for login
- **2 requests per hour** per email for password reset
- Lock accounts after **10 consecutive failures** (with unlock procedure)

### API Endpoints

Layer rate limits by scope:

- Per API key: prevents abuse from compromised keys
- Per IP: prevents anonymous abuse
- Per user: prevents authenticated abuse
- Global: protects infrastructure

## Implementation

```typescript
interface RateLimitConfig {
  window: number;     // milliseconds
  maxRequests: number;
  keyExtractor: (req: Request) => string;
  onLimitExceeded: (req: Request) => void;
}

// Sliding window counter in Redis
async function checkRateLimit(key: string, config: RateLimitConfig): Promise<boolean> {
  const current = await redis.incr(key);
  if (current === 1) {
    await redis.pexpire(key, config.window);
  }
  return current <= config.maxRequests;
}
```

## Response Headers

Always include rate limit information in responses:

```
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 42
X-RateLimit-Reset: 1709234567
Retry-After: 30
```
