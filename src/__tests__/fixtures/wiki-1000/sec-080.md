# Secure Inter-Process Communication

When processes on the same host need to communicate, the choice of **IPC mechanism** has security implications.

## Unix Domain Sockets

Preferred for local IPC:

- File permission-based access control
- No network exposure
- Higher throughput than TCP loopback

```typescript
import { createServer } from 'net';

const server = createServer((socket) => {
  socket.on('data', handleMessage);
});

// Set restrictive permissions
server.listen('/var/run/app/api.sock');
// chmod 0600 /var/run/app/api.sock
```

## Shared Memory

Fast but dangerous. If you must use shared memory:

- Set appropriate POSIX permissions
- Use synchronization primitives to prevent races
- Validate all data read from shared memory segments
- Clear sensitive data from shared memory before releasing

## Named Pipes (FIFOs)

Similar security model to Unix sockets. Set file permissions to restrict access.

## Environment Variables

Commonly used to pass secrets to child processes. Risks:

- Visible in `/proc/[pid]/environ` on Linux
- Inherited by child processes (may be unintended)
- May appear in crash dumps

Prefer file-based secret injection for sensitive values.

## Container IPC

In containerized environments:

- Disable IPC namespace sharing between containers by default
- Use explicit volume mounts for Unix sockets
- Don't share PID namespaces unless necessary

See [[sec-044]] for service-to-service communication patterns.
