# Secure Pagination

**Pagination** of API results can introduce security vulnerabilities if not implemented carefully, particularly around authorization and information disclosure.

## Cursor-Based vs Offset-Based

### Offset-Based
```
GET /api/users?offset=100&limit=25
```
Problems:
- Users can enumerate total dataset size
- Skipping to arbitrary positions may bypass access controls
- Performance degrades with large offsets

### Cursor-Based
```
GET /api/users?cursor=eyJpZCI6MTAwfQ&limit=25
```
Advantages:
- Opaque cursor prevents offset manipulation
- Consistent performance regardless of position
- Harder to enumerate the full dataset

## Authorization

Apply access controls to every page of results, not just the first:

```typescript
async function paginatedQuery(userId: string, cursor: string, limit: number) {
  const results = await db.query(`
    SELECT * FROM documents 
    WHERE owner_id = $1 AND id > $2
    ORDER BY id
    LIMIT $3
  `, [userId, decodeCursor(cursor), limit]);
  
  return results; // Access control built into the query
}
```

## Information Disclosure

- Don't expose total count if it reveals business information
- Don't expose sequential IDs in cursors (encode or encrypt them)
- Don't allow unlimited page sizes (cap at a reasonable max like 100)

## Rate Limiting

Rapid pagination can be used for data scraping. Apply rate limits per user across paginated endpoints (see [[sec-026]]).
