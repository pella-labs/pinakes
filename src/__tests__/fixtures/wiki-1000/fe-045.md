# Design System Architecture

## What Makes a Design System

A **design system** is more than a component library. It includes design tokens, components, patterns, guidelines, and documentation—all working together to ensure consistency at scale.

## Layers

1. **Tokens**: colors, spacing, typography, shadows (see [[fe-023]])
2. **Primitives**: unstyled, accessible base components (buttons, inputs, dialogs)
3. **Components**: styled versions of primitives using design tokens
4. **Patterns**: recurring compositions (forms, navigation, data tables)
5. **Documentation**: usage guidelines, do/don't examples, Storybook (see [[fe-018]])

## Monorepo Structure

```
packages/
  tokens/           # Design tokens (CSS vars, Tailwind config, Swift, Kotlin)
  primitives/       # Headless accessible components
  react/            # Styled React components
  vue/              # Styled Vue components (optional)
  icons/            # SVG icon library
  docs/             # Documentation site (Storybook or custom)
```

## Versioning

Use **semantic versioning** for the design system packages. Breaking changes (token renames, API changes) require a major version bump and a migration guide.

## Adoption Strategy

- Start with the most-used components (Button, Input, Card, Modal)
- Provide a codemod for migrating from existing components
- Track adoption metrics per team/repo
- Have a clear contribution model for new components

## Anti-Patterns

- Building every possible component before anyone uses the system
- Enforcing the system through tooling alone without team buy-in
- Designing tokens in isolation from real product needs
- Neglecting documentation and onboarding
