# Kubernetes Liveness and Readiness Probes

## Why Probes Matter

Without probes, Kubernetes has no way to know if your application is healthy. It will continue routing traffic to broken pods and won't restart them.

## Liveness Probes

A **liveness probe** tells Kubernetes whether the container is alive. If it fails, the container is restarted.

Use it to detect deadlocks, infinite loops, or corrupted state.

```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 15
  periodSeconds: 10
  failureThreshold: 3
```

## Readiness Probes

A **readiness probe** tells Kubernetes whether the container can accept traffic. If it fails, the pod is removed from service endpoints but not restarted.

Use it during startup (warming caches, loading models) or when a dependency is temporarily unavailable.

```yaml
readinessProbe:
  httpGet:
    path: /readyz
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
  failureThreshold: 3
```

## Startup Probes

**Startup probes** handle slow-starting containers. They disable liveness and readiness probes until the startup probe succeeds.

## Common Mistakes

- Making the liveness probe check external dependencies (causes cascading restarts)
- Setting `initialDelaySeconds` too low (containers restart before they're ready)
- Using the same endpoint for liveness and readiness (they serve different purposes)

See [[ops-015]] for deployment patterns.
