# SQL Injection Prevention

**SQL injection** occurs when untrusted input is concatenated directly into SQL queries, allowing attackers to manipulate the query logic. Despite being well-understood for decades, it remains a top vulnerability.

## The Problem

```python
# VULNERABLE — never do this
query = f"SELECT * FROM users WHERE id = '{user_input}'"
```

An attacker supplying `' OR '1'='1` retrieves all users. More sophisticated attacks can extract data, modify records, or even execute operating system commands.

## Parameterized Queries

The primary defense is **parameterized queries** (also called prepared statements):

```python
# SAFE — parameterized query
cursor.execute("SELECT * FROM users WHERE id = ?", (user_input,))
```

```typescript
// SAFE — with better-sqlite3
const stmt = db.prepare('SELECT * FROM users WHERE id = ?');
const user = stmt.get(userId);
```

## ORM Safety

ORMs like Drizzle, Prisma, and SQLAlchemy parameterize by default. The risk comes from raw query escape hatches — review every use of `.raw()`, `sql\`\``, or equivalent.

## Additional Defenses

- Use **least-privilege** database accounts — the app should not connect as `root`
- Enable **query logging** in staging to catch suspicious patterns
- Apply **input validation** as a secondary layer, not a primary defense
- Consider a **WAF** rule set for common injection patterns (see [[sec-029]])

See [[sec-014]] for audit logging that can help detect injection attempts.
