# Web Workers for UI Performance

## Offloading Heavy Computation

**Web Workers** run JavaScript in a background thread, keeping the main thread free for user interactions. Any CPU-intensive operation—data parsing, image processing, sorting large datasets—is a candidate.

## Basic Usage

```typescript
// worker.ts
self.onmessage = (event: MessageEvent<{ data: number[] }>) => {
  const sorted = event.data.data.sort((a, b) => a - b);
  self.postMessage({ sorted });
};

// component.tsx
const worker = new Worker(new URL('./worker.ts', import.meta.url));

worker.postMessage({ data: largeArray });
worker.onmessage = (event) => {
  setResults(event.data.sorted);
};
```

## Comlink

**Comlink** by Google makes worker communication feel like calling normal functions:

```typescript
// worker.ts
import { expose } from 'comlink';

const api = {
  async processData(data: number[]) {
    return data.sort((a, b) => a - b);
  },
};

expose(api);

// component.tsx
import { wrap } from 'comlink';

const worker = wrap<typeof api>(
  new Worker(new URL('./worker.ts', import.meta.url))
);

const result = await worker.processData(largeArray);
```

## Limitations

Workers cannot access the DOM, window, or document. Data transfer uses structured cloning (deep copy) unless you use **Transferable** objects for zero-copy transfers of ArrayBuffers.

See [[fe-009]] for performance metrics and [[fe-038]] for related computation patterns.
