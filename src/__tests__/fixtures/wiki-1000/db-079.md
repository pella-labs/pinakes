# Prepared Statements

## What Are Prepared Statements?

A **prepared statement** is a pre-compiled SQL template. The database parses and plans the query once, then executes it multiple times with different parameters.

## Benefits

- **Performance**: parse and plan once, execute many times
- **Security**: parameters are never interpolated into SQL, preventing injection
- **Type safety**: the database validates parameter types

## PostgreSQL

```sql
PREPARE user_lookup (text) AS
  SELECT * FROM users WHERE email = $1;

EXECUTE user_lookup('alice@example.com');
```

## In Application Code

Most drivers handle preparation transparently:

```typescript
// pg driver — auto-prepared on repeated execution
const result = await pool.query('SELECT * FROM users WHERE id = $1', [userId]);

// better-sqlite3 — explicit preparation
const stmt = db.prepare('SELECT * FROM users WHERE id = ?');
const user = stmt.get(userId);
```

## Named vs Positional Parameters

- PostgreSQL: positional (`$1`, `$2`, `$3`)
- MySQL: positional (`?`)
- SQLite: positional (`?`) or named (`:name`, `@name`, `$name`)

## Caching Considerations

Some poolers (like PgBouncer) interfere with prepared statements. In transaction mode, prepared statements are lost when the server connection is returned to the pool. See [[db-045]] for workarounds.
