# Data Lake vs Data Warehouse

## Data Lake

Stores raw, unstructured data in its original format. Schema-on-read — structure is applied at query time.

- Storage: S3, GCS, ADLS (cheap object storage)
- Formats: Parquet, ORC, JSON, CSV, images, logs
- Query engines: Athena, Presto, Spark, DuckDB
- Good for: exploration, ML training data, raw archives

## Data Warehouse

Stores structured, processed data with enforced schemas. Schema-on-write.

- Engines: Snowflake, BigQuery, Redshift, Databricks SQL
- Schema: star/snowflake schemas (see [[db-036]])
- Good for: business analytics, dashboards, reporting

## Data Lakehouse

The modern hybrid: structured tables on top of lake storage using Delta Lake, Iceberg, or Hudi. Get the flexibility of a lake with the query performance of a warehouse.

```sql
-- Databricks: create a Delta table in the lakehouse
CREATE TABLE gold.daily_revenue
USING DELTA
LOCATION 's3://lakehouse/gold/daily_revenue/'
AS SELECT ...
```

## Choosing

- If your data is mostly structured and you need fast BI queries: warehouse
- If you need to store everything (logs, images, raw events) cheaply: lake
- If you want both: lakehouse

For ETL pipelines that feed these systems, see [[db-025]].
