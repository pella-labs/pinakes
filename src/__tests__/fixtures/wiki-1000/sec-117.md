# Identity Federation

**Identity federation** allows users authenticated by one organization to access resources in another without separate credentials. It enables B2B integrations and enterprise SSO.

## How Federation Works

1. User authenticates with their **home IdP** (Identity Provider)
2. Home IdP issues a token/assertion
3. User presents token to the **service provider** (your app)
4. SP validates the token and creates a local session

## Trust Establishment

Federation requires a trust relationship:

- SAML: exchange metadata XML containing signing certificates (see [[sec-019]])
- OIDC: register as a client with the IdP, configure discovery (see [[sec-020]])
- OAuth2: establish client credentials and redirect URIs

## Security Challenges

### Token Validation

Never trust federated assertions without thorough validation:

- Verify cryptographic signatures
- Check token freshness (not expired, not replayed)
- Validate audience (intended for your service)
- Verify issuer (from a trusted IdP)

### Just-In-Time Provisioning

When a federated user logs in for the first time, create a local account with minimal permissions. Escalate only through explicit role assignment.

### Account Linking

Handle the case where a user has both a local account and a federated identity:

- Match on verified email address
- Require user confirmation before linking
- Log account linking events

## Multi-IdP Support

Enterprise customers often need to configure their own IdP. Build your federation support to handle:

- Multiple simultaneous IdPs
- Per-tenant IdP configuration
- IdP-specific attribute mapping
- Graceful IdP unavailability
