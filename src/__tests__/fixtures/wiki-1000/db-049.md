# Cassandra Data Modeling

## Think Queries First

Cassandra data modeling inverts the relational approach. Instead of designing tables and then writing queries, you start with your **query patterns** and design tables to serve them. Each query typically maps to one table.

## Partition Key and Clustering Key

The **partition key** determines which node stores the data. The **clustering key** determines sort order within a partition.

```cql
CREATE TABLE sensor_readings (
  sensor_id UUID,
  reading_time TIMESTAMP,
  temperature DOUBLE,
  humidity DOUBLE,
  PRIMARY KEY (sensor_id, reading_time)
) WITH CLUSTERING ORDER BY (reading_time DESC);
```

Here, `sensor_id` is the partition key and `reading_time` is the clustering key. All readings for a sensor are stored together, sorted by time.

## Denormalization Is Expected

Cassandra has no JOINs. If you need the same data for different queries, store it in multiple tables. Data duplication is the norm, not an anti-pattern.

## Partition Sizing

Keep partitions under 100MB. If a partition grows too large, add a bucketing column:

```cql
CREATE TABLE sensor_readings (
  sensor_id UUID,
  day DATE,           -- bucket by day
  reading_time TIMESTAMP,
  temperature DOUBLE,
  PRIMARY KEY ((sensor_id, day), reading_time)
);
```

For alternative approaches to distributed data, see [[db-021]] on sharding.
