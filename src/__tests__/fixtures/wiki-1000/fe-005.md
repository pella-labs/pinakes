# Accessibility Fundamentals for Web Applications

## Why Accessibility Matters

**Web accessibility** (a11y) ensures that people with disabilities can perceive, understand, navigate, and interact with web content. Beyond ethical obligations, accessibility is legally required in many jurisdictions under laws like the ADA, Section 508, and the European Accessibility Act.

## ARIA Roles and Properties

When native HTML semantics are insufficient, **ARIA** attributes communicate the purpose and state of UI elements to assistive technologies.

- `role="dialog"` — marks a modal dialog
- `aria-expanded="true|false"` — indicates collapsible state
- `aria-live="polite"` — announces dynamic content changes
- `aria-label` — provides a text alternative when visible text is absent

## Keyboard Navigation

Every interactive element must be operable via keyboard:

- **Tab** moves focus forward through focusable elements
- **Shift+Tab** moves focus backward
- **Enter/Space** activates buttons and links
- **Arrow keys** navigate within composite widgets (menus, tabs, grids)
- **Escape** dismisses overlays

Implement a **focus trap** inside modals to prevent focus from escaping. See [[fe-042]] for the focus management hook pattern.

## Color and Contrast

WCAG 2.1 AA requires a minimum **contrast ratio** of 4.5:1 for normal text and 3:1 for large text. Use tools like axe-core or Lighthouse to audit contrast programmatically.

## Testing

- **axe-core**: automated accessibility testing engine
- **Lighthouse**: built-in Chrome audit
- **Screen readers**: test with VoiceOver (macOS), NVDA (Windows), and TalkBack (Android)
- **Playwright**: can assert ARIA attributes in e2e tests

Accessibility is not a feature—it is a baseline requirement. See [[fe-006]] for component-level a11y patterns.
