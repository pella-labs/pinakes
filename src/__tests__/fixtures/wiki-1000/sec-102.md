# Encryption in Transit

**Encryption in transit** protects data as it moves between systems. Without it, any network observer can read, modify, or inject data.

## Where to Encrypt

- Browser to server (HTTPS — see [[sec-017]])
- Service to service (mTLS — see [[sec-006]])
- Application to database (TLS)
- Server to email (STARTTLS)
- Any data crossing a trust boundary (see [[sec-072]])

## Common Gaps

Places where encryption in transit is often missing:

- Internal service communication ("it's inside the VPN")
- Database connections in development
- Log shipping and metric collection
- Backup transfers
- DNS queries (consider DoH/DoT — see [[sec-060]])

The zero trust model (see [[sec-031]]) mandates encryption everywhere, even within the internal network.

## TLS Client Configuration

When your application is a TLS client:

```typescript
import { Agent } from 'https';

const agent = new Agent({
  minVersion: 'TLSv1.2',
  rejectUnauthorized: true, // NEVER disable in production
});
```

Never set `rejectUnauthorized: false` or `NODE_TLS_REJECT_UNAUTHORIZED=0` in production. These disable certificate verification entirely.

## Monitoring

Monitor for:
- Certificate expiration (see [[sec-017]])
- TLS version downgrades
- Connections using deprecated cipher suites
- Plaintext connections where TLS is expected
