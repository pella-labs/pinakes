# Network Segmentation

**Network segmentation** divides a network into smaller, isolated segments. It limits lateral movement — if an attacker breaches one segment, they can't freely access others.

## Segmentation Strategies

### VLAN-Based
Traditional approach using VLANs and router ACLs. Simple but coarse-grained.

### Software-Defined
Using overlay networks (Calico, Cilium) for fine-grained, policy-driven segmentation.

### Micro-segmentation
Per-workload policies. Each service can only communicate with explicitly allowed peers. This is the zero trust approach (see [[sec-031]]).

## Typical Segments

- **DMZ**: publicly accessible services (web servers, API gateways)
- **Application tier**: backend services (not internet-facing)
- **Data tier**: databases and storage (most restricted)
- **Management**: admin tools, CI/CD, monitoring
- **Development**: dev and staging environments (isolated from production)

## Implementation in Kubernetes

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
spec:
  podSelector: {}
  policyTypes:
    - Ingress
    - Egress
```

Start with deny-all, then add specific allow rules. This ensures new services are isolated by default.

## Monitoring

Log all traffic between segments. Alert on unexpected cross-segment communication. This helps detect lateral movement early.
