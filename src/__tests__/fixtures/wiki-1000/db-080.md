# Data Modeling for Real-Time Analytics

Real-time analytics requires a different data model than traditional OLTP. The goal is sub-second query response on continuously arriving data.

## Pre-Aggregation

Instead of querying raw events, maintain pre-aggregated tables:

```sql
CREATE TABLE page_view_stats (
  page_url TEXT NOT NULL,
  hour TIMESTAMPTZ NOT NULL,
  view_count INT DEFAULT 0,
  unique_users INT DEFAULT 0,
  PRIMARY KEY (page_url, hour)
);

-- Upsert on each page view
INSERT INTO page_view_stats (page_url, hour, view_count, unique_users)
VALUES ($1, date_trunc('hour', now()), 1, 1)
ON CONFLICT (page_url, hour) DO UPDATE SET
  view_count = page_view_stats.view_count + 1;
```

Pre-compute dashboard queries. Refresh on a schedule or trigger. See [[db-040]] for materialized view patterns.

## HyperLogLog for Unique Counts

Exact unique counts are expensive. **HyperLogLog** provides approximate counts with <1% error and constant memory:

```sql
-- PostgreSQL with hll extension
CREATE EXTENSION hll;
SELECT hll_cardinality(hll_union_agg(user_hll)) FROM daily_stats;
```

## Time-Series Databases

For high-ingest scenarios (>100K events/second), consider dedicated time-series databases like TimescaleDB or ClickHouse. See [[db-027]] for time-series options.
