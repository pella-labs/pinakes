# Horizontal vs Vertical Scaling

## Vertical Scaling (Scale Up)

Add more CPU, RAM, and faster storage to a single server. This is the simplest approach and should be your first move.

Advantages:
- No application changes needed
- No distributed systems complexity
- Consistent, predictable performance

Limits:
- Hardware has a ceiling
- Single point of failure (unless using replicas)
- Cost increases non-linearly at the high end

## Horizontal Scaling (Scale Out)

Add more servers. Distribute data and traffic across them. This is harder but has no theoretical ceiling.

Approaches:
- **Read replicas** — scale reads independently of writes (see [[db-015]])
- **Sharding** — split data across instances (see [[db-021]])
- **Caching layer** — reduce database load (see [[db-022]])
- **CQRS** — separate read and write models

## When to Scale

Scale vertically until:
- The largest available instance isn't enough
- Cost of a larger instance exceeds the cost of horizontal complexity
- Write throughput exceeds what a single instance can handle

Most applications never reach the point where horizontal database scaling is necessary. A well-tuned PostgreSQL instance on modern hardware handles millions of rows and thousands of queries per second.
