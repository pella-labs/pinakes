# React Fiber Architecture

## What Is Fiber

**React Fiber** is the reconciliation engine introduced in React 16. It replaced the synchronous, recursive reconciler with an incremental one that can pause, resume, and prioritize work.

## How It Works

Fiber breaks rendering into units of work. Each component becomes a **fiber node** in a linked list. The scheduler processes these nodes one at a time, checking after each unit whether higher-priority work (like user input) has arrived.

## The Two Phases

1. **Render phase** (interruptible): React builds a new fiber tree (the "work-in-progress" tree) by diffing against the current tree. This phase can be paused and restarted.

2. **Commit phase** (synchronous): React applies all accumulated changes to the DOM in one batch. This phase cannot be interrupted.

## Priority Levels

React assigns priority to different types of updates:
- **Immediate**: synchronous, like `flushSync`
- **User blocking**: clicks, input events
- **Normal**: state updates from transitions
- **Low**: deferred updates, offscreen rendering
- **Idle**: background work

## Implications for Developers

Understanding Fiber helps explain:
- Why `useEffect` runs asynchronously after commit
- Why the render phase must be pure (no side effects)
- How Concurrent Mode enables features like Suspense and transitions
