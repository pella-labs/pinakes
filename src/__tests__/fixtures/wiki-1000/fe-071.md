# Feature Flags in Frontend Applications

## Why Feature Flags

**Feature flags** decouple deployment from release. Code ships to production behind a flag, enabling gradual rollouts, A/B testing, and instant rollbacks without redeployment.

## Implementation Approaches

### Simple Boolean Flags

```typescript
const FLAGS = {
  NEW_CHECKOUT: process.env.NEXT_PUBLIC_FLAG_NEW_CHECKOUT === 'true',
  DARK_MODE: true,
};

function CheckoutPage() {
  if (FLAGS.NEW_CHECKOUT) return <NewCheckout />;
  return <LegacyCheckout />;
}
```

### Feature Flag Services

Services like **LaunchDarkly**, **Statsig**, and **Unleash** provide:
- User-level targeting
- Percentage-based rollouts
- Analytics integration
- Real-time flag updates without deployment

## Code Splitting Integration

Combine flags with dynamic imports to avoid shipping dead code:

```typescript
const Checkout = FLAGS.NEW_CHECKOUT
  ? lazy(() => import('./NewCheckout'))
  : lazy(() => import('./LegacyCheckout'));
```

## Cleanup

Feature flags are temporary. Set an expiration date and schedule cleanup. Dead flags add cognitive overhead and branch complexity. A flag older than 30 days should either be fully enabled or removed.
