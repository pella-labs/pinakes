# Query Optimization Techniques

## Start With EXPLAIN

Before optimizing anything, understand the current query plan. See [[db-009]] for a deep dive on reading EXPLAIN output.

## Index-Driven Optimization

The most impactful optimization is usually adding the right index. But indexes have costs — they slow down writes and consume storage. Only index columns that appear in WHERE, JOIN, ORDER BY, or GROUP BY clauses of frequent queries.

## Common Anti-Patterns

**SELECT \*** — fetches all columns even when you only need two. Forces the database to read wider rows and prevents index-only scans.

**N+1 queries** — fetching a list of parents, then looping to fetch children one by one. Use JOINs or batch IN clauses instead.

**OR in WHERE clauses** — often prevents index usage. Rewrite as UNION ALL when possible.

**Functions on indexed columns** — `WHERE YEAR(created_at) = 2024` can't use an index on `created_at`. Rewrite as a range: `WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01'`.

## Query Rewriting

Sometimes the planner needs help:

```sql
-- Instead of a correlated subquery:
SELECT * FROM orders WHERE customer_id IN (
  SELECT id FROM customers WHERE region = 'EU'
);

-- Use a JOIN:
SELECT o.* FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE c.region = 'EU';
```

## Caching Layer

When query optimization hits diminishing returns, consider caching. Redis (see [[db-004]]) is the standard choice for application-level caching.
