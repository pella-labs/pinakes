# Cursor-Based Pagination

## Why Not OFFSET?

Traditional **OFFSET/LIMIT** pagination has a fatal flaw: the database must scan and discard all rows before the offset. Page 1000 of 100 results means scanning 100,000 rows.

## Cursor Pagination

Use a unique, ordered column as a cursor. Each page starts after the last item of the previous page:

```sql
-- First page
SELECT id, title, created_at
FROM articles
ORDER BY created_at DESC, id DESC
LIMIT 20;

-- Next page (using the last row's values as cursor)
SELECT id, title, created_at
FROM articles
WHERE (created_at, id) < ($last_created_at, $last_id)
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

## Encoding the Cursor

Encode the cursor as a base64 string for the API:

```typescript
function encodeCursor(createdAt: string, id: string): string {
  return Buffer.from(`${createdAt}|${id}`).toString('base64');
}

function decodeCursor(cursor: string): { createdAt: string; id: string } {
  const [createdAt, id] = Buffer.from(cursor, 'base64').toString().split('|');
  return { createdAt, id };
}
```

## Tradeoffs

Cursor pagination is fast regardless of page depth, but you can't jump to page N. For most APIs, this is fine — infinite scroll and "load more" patterns don't need random access.

With proper indexes, cursor pagination is O(1) per page. See [[db-020]] for indexing the cursor columns.
