# Isolation Levels Explained

## The Four Standard Levels

SQL defines four **isolation levels**, each trading consistency for concurrency:

### READ UNCOMMITTED
Can see uncommitted changes from other transactions (**dirty reads**). Almost never used in practice.

### READ COMMITTED
Default in PostgreSQL. Each statement sees only data committed before the statement began. No dirty reads, but **non-repeatable reads** are possible — the same query can return different results within one transaction.

### REPEATABLE READ
Default in MySQL/InnoDB. The transaction sees a snapshot taken at the first read. No dirty reads, no non-repeatable reads, but **phantom reads** are theoretically possible (new rows inserted by other transactions).

### SERIALIZABLE
The strongest level. Transactions behave as if executed one at a time. PostgreSQL implements this with **Serializable Snapshot Isolation** (SSI), which detects conflicts and aborts one of the conflicting transactions.

## Practical Guidance

Most applications work fine with READ COMMITTED. Use REPEATABLE READ when you need consistent reads across multiple statements. Use SERIALIZABLE only when correctness demands it — and be prepared to retry aborted transactions.

```sql
-- Set for a single transaction
BEGIN ISOLATION LEVEL SERIALIZABLE;
-- ... do work ...
COMMIT;
```
