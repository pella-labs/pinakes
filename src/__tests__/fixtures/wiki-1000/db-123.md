# Designing for Eventual Consistency

Eventual consistency is not a bug — it's a design choice. When you accept that reads may be slightly stale, you gain availability and performance.

## Patterns for Managing Staleness

### Read Your Own Writes

After a write, read from the primary (not a replica) to ensure the user sees their own changes:

```typescript
async function updateProfile(userId: string, data: ProfileData) {
  await primaryDb.query('UPDATE profiles SET ... WHERE id = $1', [userId]);
  // Set a flag to read from primary for this user for the next 5 seconds
  await redis.setex(`read-primary:${userId}`, 5, '1');
}

async function getProfile(userId: string) {
  const readFromPrimary = await redis.get(`read-primary:${userId}`);
  const db = readFromPrimary ? primaryDb : replicaDb;
  return db.query('SELECT * FROM profiles WHERE id = $1', [userId]);
}
```

### Monotonic Reads

Ensure a user always reads from the same replica (sticky sessions) to avoid time-travel where newer data is followed by older data.

### Causal Consistency

Track a logical timestamp. Reads must see all writes that causally precede them. This is stronger than eventual consistency but weaker than linearizability.

## UI Strategies

- Optimistic updates: show the change immediately in the UI
- Stale-while-revalidate: show cached data, refresh in the background
- Loading states: indicate when data might be updating

For ACID guarantees at the other end of the spectrum, see [[db-011]].
