# Observability with OpenTelemetry

## Why OpenTelemetry

**OpenTelemetry** (OTel) is the vendor-neutral standard for collecting telemetry data. It provides a single set of APIs and SDKs for metrics, logs, and traces across all languages.

## Auto-Instrumentation

For Node.js services, OTel auto-instrumentation captures HTTP requests, database queries, and external API calls without code changes:

```bash
npm install @opentelemetry/auto-instrumentations-node
```

```typescript
import { NodeSDK } from '@opentelemetry/sdk-node';
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node';

const sdk = new NodeSDK({
  instrumentations: [getNodeAutoInstrumentations()],
  traceExporter: new OTLPTraceExporter({
    url: 'http://otel-collector:4318/v1/traces',
  }),
});

sdk.start();
```

## Collector Architecture

The **OTel Collector** receives, processes, and exports telemetry data. Run it as a DaemonSet in Kubernetes:

- **Receivers**: OTLP (gRPC and HTTP)
- **Processors**: batch, memory limiter, attribute filtering
- **Exporters**: Jaeger, Prometheus, Loki, Datadog, whatever your backend is

## Trace Context Propagation

OTel automatically propagates trace context via W3C **traceparent** headers. This links spans across services into a complete trace. Ensure all services in the call chain have instrumentation.

See [[ops-019]] for the monitoring strategy and [[ops-027]] for log aggregation.
