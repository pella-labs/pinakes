# Database Testing Strategies

## Unit Testing Queries

Test your query logic against a real database, not mocks. Mocks can't catch SQL syntax errors, constraint violations, or planner behavior differences.

```typescript
import { describe, it, beforeEach, expect } from 'vitest';
import Database from 'better-sqlite3';

describe('UserRepository', () => {
  let db: Database.Database;

  beforeEach(() => {
    db = new Database(':memory:');
    db.pragma('journal_mode = WAL');
    db.exec(`
      CREATE TABLE users (
        id TEXT PRIMARY KEY,
        email TEXT NOT NULL UNIQUE,
        name TEXT NOT NULL
      );
    `);
  });

  it('should find user by email', () => {
    db.prepare('INSERT INTO users VALUES (?, ?, ?)').run('1', 'a@b.com', 'Alice');
    const user = db.prepare('SELECT * FROM users WHERE email = ?').get('a@b.com');
    expect(user).toMatchObject({ name: 'Alice' });
  });
});
```

## Integration Testing

For PostgreSQL/MySQL, use testcontainers or a dedicated test database. Never test against production.

## Migration Testing

- Test forward migrations on a fresh database
- Test that rollbacks work correctly
- Test migrations on a database with realistic data volumes

## Performance Testing

Seed test databases with realistic data volumes. A query that's fast with 100 rows might be slow with 1M rows. Use EXPLAIN ANALYZE (see [[db-009]]) in your test suite for critical queries.
