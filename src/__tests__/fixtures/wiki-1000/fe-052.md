# Monorepo Tooling for Frontend

## Why Monorepos

A **monorepo** houses multiple packages in a single repository. For frontend teams, this typically means shared component libraries, configuration packages, and multiple applications.

## Tool Comparison

- **Turborepo**: fast build orchestration with remote caching. Pairs with any package manager. Simple configuration.
- **Nx**: full-featured with dependency graph visualization, code generation, and affected-only CI. More opinionated.
- **pnpm workspaces**: lightweight workspace support without a build orchestrator. Use with Turborepo or Nx for full capabilities.

## Turborepo Configuration

```json
{
  "pipeline": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": ["dist/**"]
    },
    "test": {
      "dependsOn": ["build"]
    },
    "lint": {}
  }
}
```

## Shared Configuration

Extract common configuration into internal packages:

- `@repo/eslint-config` — shared ESLint rules
- `@repo/tsconfig` — shared TypeScript configuration
- `@repo/tailwind-config` — shared Tailwind config

## Challenges

- IDE performance with many packages
- Dependency version alignment
- Long CI times without proper caching
- Package boundary enforcement
