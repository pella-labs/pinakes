# Penetration Testing

**Penetration testing** simulates real-world attacks against your systems to identify vulnerabilities before malicious actors do. It goes beyond automated scanning by applying human creativity and persistence.

## Types of Pen Tests

- **Black box**: tester has no prior knowledge of the system
- **White box**: tester has full access to source code and architecture docs
- **Grey box**: tester has some information (typical for external engagements)

## Scope Definition

A pen test engagement requires clear scope:

- Which systems and endpoints are in scope
- Testing window (dates and hours)
- Out-of-scope systems and techniques
- Rules of engagement (e.g., no social engineering, no DoS)
- Communication channels for critical findings

## Methodology

Most testers follow a structured methodology:

1. **Reconnaissance**: gather information about targets
2. **Enumeration**: identify services, endpoints, and attack surface
3. **Vulnerability analysis**: identify potential weaknesses
4. **Exploitation**: attempt to exploit vulnerabilities
5. **Post-exploitation**: assess impact and lateral movement
6. **Reporting**: document findings with severity and remediation

## Frequency

- Annual pen test minimum for compliance (SOC2, PCI DSS)
- After major architecture changes
- Before launching new products or features
- After significant security incidents
