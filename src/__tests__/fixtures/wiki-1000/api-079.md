# REST API Soft Delete

## The Pattern

Don't actually delete records. Set a `deleted_at` timestamp.

```sql
-- "Delete"
UPDATE orders SET deleted_at = NOW() WHERE id = $1;

-- All queries filter out deleted records
SELECT * FROM orders WHERE deleted_at IS NULL AND id = $1;
```

## API Behavior

- `DELETE /orders/{id}` → sets `deleted_at`, returns `200` (or `204`)
- `GET /orders/{id}` → returns `404` if soft-deleted
- `GET /orders/{id}?include_deleted=true` → admin-only, returns with `deleted_at` field
- `POST /orders/{id}/restore` → clears `deleted_at` (admin-only)

## Trade-offs

**Pros:**
- Audit trail
- Easy undo
- Referential integrity preserved

**Cons:**
- Every query needs `WHERE deleted_at IS NULL` (easy to forget)
- Storage grows indefinitely
- GDPR may require hard delete anyway

## Hard Delete Schedule

Soft-deleted records are permanently purged after 90 days by a background job. GDPR deletion requests trigger immediate hard delete.
