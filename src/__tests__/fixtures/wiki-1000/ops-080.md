# Kubernetes Admission Controllers

## What Are Admission Controllers

**Admission controllers** intercept requests to the Kubernetes API server before objects are persisted. They can validate, mutate, or reject requests.

## Types

- **Validating**: reject requests that don't meet criteria
- **Mutating**: modify requests (e.g., add labels, inject sidecars)

## Policy Engines

### Kyverno

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-labels
spec:
  validationFailureAction: Enforce
  rules:
    - name: check-labels
      match:
        any:
          - resources:
              kinds:
                - Pod
      validate:
        message: "All pods must have team and app labels"
        pattern:
          metadata:
            labels:
              team: "?*"
              app: "?*"
```

### OPA Gatekeeper

More powerful but more complex. Uses Rego policy language. Better for organizations with existing OPA investment.

## Common Policies

- All images must come from approved registries
- All pods must have resource requests and limits
- No privileged containers in application namespaces
- All ingress must use TLS
- Required labels on all resources

See [[ops-071]] for pod security standards and [[ops-039]] for container security.
