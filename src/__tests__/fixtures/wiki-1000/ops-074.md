# Continuous Delivery vs Continuous Deployment

## Definitions

These terms are often confused:

**Continuous Delivery** means every commit is deployable to production, but a human decides when to actually deploy. There's a manual gate.

**Continuous Deployment** means every commit that passes all automated checks is automatically deployed to production. No human gate.

## Which to Choose

Most teams should start with **Continuous Delivery** and evolve toward **Continuous Deployment** as confidence grows.

Prerequisites for Continuous Deployment:
- Comprehensive automated test suite
- Feature flags for controlling rollouts
- Canary deployments with automatic rollback
- Strong monitoring and alerting
- Fast rollback capability

## Our Approach

We practice **Continuous Delivery** to staging (every merge to main auto-deploys) and require manual approval for production. The goal is to move to Continuous Deployment for low-risk changes while keeping manual gates for database migrations and major features.

The key insight: the deployment pipeline should be boring. If shipping to production is stressful, something in the pipeline needs improvement.
