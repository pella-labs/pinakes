# Change Data Capture (CDC)

CDC is a technique for tracking row-level changes in a database and streaming them to other systems. Instead of periodic batch extractions, CDC provides a **continuous stream of inserts, updates, and deletes**.

## How It Works

Most CDC implementations read the database's transaction log:

- **PostgreSQL**: logical replication slots + `pgoutput` or `wal2json`
- **MySQL**: binlog with row-based replication
- **SQL Server**: built-in CDC tables
- **MongoDB**: change streams (oplog-based)

## Debezium

**Debezium** is the most popular open-source CDC platform. It connects to your database's transaction log and publishes change events to Kafka:

```json
{
  "before": { "id": 1, "name": "Alice", "email": "alice@old.com" },
  "after": { "id": 1, "name": "Alice", "email": "alice@new.com" },
  "source": { "table": "users", "ts_ms": 1706745600000 },
  "op": "u"
}
```

## Use Cases

- Keeping search indexes in sync (see [[db-006]])
- Populating data warehouses in near-real-time
- Event sourcing without changing application code
- Cache invalidation based on actual database changes
- Cross-service data synchronization

## Gotchas

Schema changes in the source database can break CDC pipelines. Always use a schema registry and test migrations against your CDC consumers.
