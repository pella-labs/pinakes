# Indexing Strategies

## B-Tree Indexes

The default index type in most databases. B-trees excel at equality and range queries. They maintain sorted order, so they also speed up ORDER BY.

```sql
CREATE INDEX idx_users_email ON users (email);
CREATE INDEX idx_orders_date ON orders (created_at DESC);
```

## Composite Indexes

A composite index covers multiple columns. Column order matters — the index is useful for queries that filter on a **leftmost prefix** of the indexed columns.

```sql
-- Useful for: WHERE status = 'active' AND created_at > ...
-- Also for: WHERE status = 'active'
-- NOT useful for: WHERE created_at > ... (without status filter)
CREATE INDEX idx_orders_status_date ON orders (status, created_at);
```

## Partial Indexes

Index only a subset of rows. Dramatically smaller and faster for targeted queries:

```sql
CREATE INDEX idx_active_users ON users (email) WHERE active = true;
```

## Covering Indexes (Index-Only Scans)

Include all columns needed by the query so the database never reads the heap:

```sql
CREATE INDEX idx_orders_covering
ON orders (customer_id) INCLUDE (total, created_at);
```

## GIN and GiST

- **GIN** — Generalized Inverted Index. Best for full-text search, JSONB, and array containment. See [[db-019]].
- **GiST** — Generalized Search Tree. Best for geometric data, range types, and nearest-neighbor queries.

## Anti-Patterns

- Indexing every column (wastes write throughput and storage)
- Unused indexes (check `pg_stat_user_indexes`)
- Low-cardinality indexes (boolean columns rarely benefit)
- Over-indexing OLTP tables when you should use read replicas
