# DR: Event Bus for Inter-Service Communication

## Context

Three services need to react when an order is created: inventory (reserve stock), notifications (email), analytics (tracking). Currently they're called synchronously from the order service.

## Problem

Synchronous calls mean:
- Order creation latency = sum of all downstream calls (~800ms)
- If notification service is down, order creation fails
- Adding a new consumer requires changing the order service

## Decision

Adopt an **event bus** (Amazon SQS + SNS) for all inter-service communication that doesn't need a synchronous response.

## Pattern

```
Order Service → publishes "order.created" to SNS
  → SQS queue → Inventory Service (reserve stock)
  → SQS queue → Notification Service (send email)
  → SQS queue → Analytics Service (track event)
```

## Consequences

- Order creation latency drops to ~50ms (just the DB write + publish)
- Services are independently deployable
- Dead letter queues catch failures
- Need to handle eventual consistency (order exists but stock not yet reserved)
- Need idempotent consumers ([[api-007]])
