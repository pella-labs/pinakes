# Secure Deserialization

**Insecure deserialization** occurs when untrusted data is used to instantiate objects, potentially leading to remote code execution, privilege escalation, or denial of service.

## The Problem

Many serialization formats can encode complex objects including class instances. When an application deserializes untrusted input, the attacker controls the object's type and properties.

## Dangerous Formats

- **Java Serialization**: the most exploited. `ObjectInputStream.readObject()` can trigger arbitrary code execution via gadget chains.
- **Python Pickle**: `pickle.loads()` executes arbitrary code. Never unpickle untrusted data.
- **PHP unserialize**: object injection via magic methods.
- **YAML**: many parsers support object instantiation by default.

## Safe Alternatives

Use data-only serialization formats:

- **JSON**: no native object instantiation
- **Protocol Buffers**: schema-defined, no arbitrary types
- **MessagePack**: binary JSON equivalent

## When You Must Deserialize

If you can't avoid deserializing complex objects:

- Maintain an **allowlist** of permitted classes
- Validate the serialized data before deserializing
- Run deserialization in a sandboxed environment
- Keep serialization libraries updated (gadget chains are patched regularly)

## Node.js Considerations

JSON.parse is safe — it only produces plain objects, arrays, strings, numbers, booleans, and null. The risk comes from:

- `eval` or `Function` on untrusted strings
- Template engines processing untrusted templates
- Custom deserializers that reconstruct class instances

See [[sec-036]] for input validation patterns.
