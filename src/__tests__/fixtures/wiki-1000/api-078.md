# Designing Webhook Event Types

## Naming Convention

`resource.action` format:

- `order.created`
- `order.updated`
- `order.cancelled`
- `payment.succeeded`
- `payment.failed`
- `customer.subscription.renewed`

## Event Granularity

Too granular (field-level changes) creates noise. Too coarse (just "updated") is useless.

Good: `order.status_changed` — implies the status field changed, subscriber can check old/new.
Bad: `order.field_updated` with a generic payload — subscriber has to diff.

## Event Payload

Include enough data to avoid a callback:

```json
{
  "type": "order.status_changed",
  "data": {
    "order_id": "ord_123",
    "previous_status": "pending",
    "new_status": "shipped",
    "shipped_at": "2025-11-03T15:00:00Z",
    "tracking_number": "1Z999AA10123456784"
  }
}
```

The subscriber shouldn't need to call `GET /orders/ord_123` after receiving the webhook. If the payload is too large (>64KB), include a summary and a link to the full resource.
