# Security Architecture Review Process

A **security architecture review** evaluates the security design of a system before implementation. It catches fundamental design flaws that are expensive to fix later.

## When to Conduct

- New services or major features
- Third-party integrations
- Changes to authentication or authorization
- New data stores or data flows
- Infrastructure changes

## Review Checklist

### Data Flow
- Where does sensitive data enter the system?
- How is it processed, stored, and transmitted?
- What trust boundaries does it cross? (see [[sec-072]])
- Is it encrypted at rest and in transit?

### Authentication and Authorization
- How are users and services authenticated?
- How are permissions enforced?
- What happens when auth fails?

### Attack Surface
- What is publicly exposed?
- What inputs does the system accept?
- What third-party dependencies are introduced? (see [[sec-055]])

### Failure Modes
- How does the system fail under attack?
- Does it fail open or fail closed?
- Are there denial-of-service risks?

## Output

Produce a brief document:
1. Architecture summary (from the team)
2. Identified risks (from the reviewer)
3. Required mitigations (agreed by both)
4. Accepted risks (explicitly documented with owner)

## Relation to Threat Modeling

A security architecture review is broader than a threat model. It covers design, deployment, and operational concerns. Use [[sec-030]] (threat modeling) as one input to the review.
