# Database Triggers

## What Are Triggers?

A **trigger** automatically executes a function when a specified event (INSERT, UPDATE, DELETE) occurs on a table.

## PostgreSQL Trigger Types

### Row-Level Triggers
Execute once per affected row:

```sql
CREATE TRIGGER update_modified
BEFORE UPDATE ON documents
FOR EACH ROW
EXECUTE FUNCTION set_updated_at();

CREATE FUNCTION set_updated_at() RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

### Statement-Level Triggers
Execute once per statement, regardless of row count:

```sql
CREATE TRIGGER log_bulk_delete
AFTER DELETE ON orders
FOR EACH STATEMENT
EXECUTE FUNCTION log_deletion();
```

## BEFORE vs AFTER

- **BEFORE**: can modify the row before it's written. Return NULL to cancel the operation.
- **AFTER**: row is already written. Good for side effects (audit logging, notifications).

## Common Uses

- Auto-set timestamps (`created_at`, `updated_at`)
- Audit logging (see [[db-078]])
- Denormalization (update materialized columns)
- Validation that exceeds CHECK constraint capabilities
- Sending notifications (see [[db-055]])

## Anti-Pattern: Business Logic in Triggers

Keep triggers simple. Complex business logic in triggers is hard to debug, test, and maintain. The application layer is almost always a better home for it.
