# Middleware Chain Architecture

## Execution Order Matters

Middleware runs in a specific order. Getting it wrong causes subtle bugs.

1. **Request ID** — assign a UUID, add to all logs
2. **Logging** — log method, path, start time
3. **CORS** — handle preflight, set headers
4. **Auth** — verify token, attach user to context
5. **Rate Limiting** — check bucket, reject if empty
6. **Validation** — parse and validate request body
7. **Handler** — actual business logic
8. **Error Handler** — catch and format errors
9. **Response Logging** — log status, duration

## Express-style vs Koa-style

Express uses `next()` callbacks — error handling requires a 4-argument function signature `(err, req, res, next)`. Koa uses `async/await` with an onion model where each middleware wraps the next:

```typescript
app.use(async (ctx, next) => {
  const start = Date.now();
  await next(); // downstream
  const ms = Date.now() - start;
  ctx.set('X-Response-Time', `${ms}ms`);
});
```

The onion model is cleaner for timing, error handling, and response transformation.

## Conditional Middleware

Not every route needs every middleware. Auth middleware should skip `/health` and `/metrics`. Rate limiting should skip internal service-to-service calls.

```typescript
function unless(paths: string[], middleware: Middleware): Middleware {
  return (ctx, next) => {
    if (paths.includes(ctx.path)) return next();
    return middleware(ctx, next);
  };
}
```
