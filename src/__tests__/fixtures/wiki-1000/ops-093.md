# Kubernetes Sidecar Pattern

## What is a Sidecar

A **sidecar** is an additional container in a pod that provides supporting functionality without modifying the main application container. Sidecars share the pod's network and storage.

## Common Sidecars

### Log Shipping

A sidecar reads log files from a shared volume and forwards them to a log aggregator:

```yaml
containers:
  - name: app
    image: myapp:latest
    volumeMounts:
      - name: logs
        mountPath: /var/log/app
  - name: log-shipper
    image: fluent-bit:3.0
    volumeMounts:
      - name: logs
        mountPath: /var/log/app
        readOnly: true
volumes:
  - name: logs
    emptyDir: {}
```

### Proxy/Mesh

Service mesh sidecars (Envoy in Istio) handle all network traffic for the application.

### Auth Proxy

An OAuth2 proxy sidecar handles authentication before traffic reaches the application.

## Native Sidecar Containers (K8s 1.28+)

Kubernetes now supports **native sidecar containers** that start before and stop after the main container:

```yaml
initContainers:
  - name: log-shipper
    image: fluent-bit:3.0
    restartPolicy: Always  # This makes it a sidecar
```

This solves the ordering problem where sidecars need to be ready before the main container starts.

See [[ops-029]] for service mesh and [[ops-027]] for log aggregation.
