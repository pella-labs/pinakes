# Kubernetes Resource Limits Deep Dive

## CPU Throttling Explained

When a container exceeds its **CPU limit**, it gets throttled. The Linux CFS scheduler restricts CPU time to the specified quota within each 100ms period.

A container with a 500m CPU limit gets 50ms of CPU time per 100ms period. If it uses all 50ms in the first 10ms, it waits 90ms before getting more CPU time. This causes **latency spikes** even when the node has spare CPU.

## Memory Limits Explained

Memory limits are hard. When a container exceeds its **memory limit**, the OOM killer terminates it. There's no throttling — it's instant death.

## Linux cgroup v2

Modern Kubernetes nodes use **cgroup v2** which provides:
- Unified resource hierarchy
- Pressure Stall Information (PSI) for better observability
- Memory QoS with soft limits

## Observability

```promql
# CPU throttling
rate(container_cpu_cfs_throttled_periods_total[5m])
/
rate(container_cpu_cfs_periods_total[5m])

# Memory usage vs limit
container_memory_working_set_bytes
/
kube_pod_container_resource_limits{resource="memory"}
```

If CPU throttling exceeds 10%, either raise the limit or remove it entirely.

## The No-CPU-Limit Movement

Google's internal practices and the broader SRE community increasingly recommend **setting CPU requests but not limits**. This prevents throttling while still guaranteeing scheduling.
