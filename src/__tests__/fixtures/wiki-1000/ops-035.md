# Database Operations

## Connection Pooling

Every production database should use a **connection pooler** like PgBouncer. Direct connections from application pods to the database create a connection-per-pod problem that exhausts database resources.

## Query Performance

Monitor slow queries. Set `log_min_duration_statement = 1000` (1 second) in Postgres to log slow queries. Use `pg_stat_statements` to find the most resource-intensive queries.

## Vacuuming

Postgres uses MVCC, which means dead rows accumulate. **Autovacuum** handles this, but for high-write tables, you may need to tune autovacuum thresholds:

```sql
ALTER TABLE high_write_table SET (
  autovacuum_vacuum_scale_factor = 0.01,
  autovacuum_analyze_scale_factor = 0.005
);
```

## Replication

We run **read replicas** for read-heavy workloads. The application routes read queries to replicas and writes to the primary. Be aware of replication lag — don't read-after-write from a replica.

## Connection Strings

Store connection strings in **Secrets Manager** (see [[ops-026]]). Use IAM authentication for RDS where possible to eliminate password management.

## Maintenance Windows

Schedule maintenance during low-traffic periods. For global services, there's no perfect time — coordinate with the team and communicate maintenance windows in advance.
