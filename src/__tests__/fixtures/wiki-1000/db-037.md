# PostgreSQL Partitioning Internals

PostgreSQL's declarative partitioning, introduced in version 10 and significantly improved through versions 11-16, works by routing rows to child tables based on the partition key.

## Partition Pruning

The query planner examines WHERE clauses and eliminates partitions that can't contain matching rows. This happens at plan time (static pruning) and at execution time (dynamic pruning for prepared statements with parameters).

```sql
-- Only scans the relevant partition
EXPLAIN SELECT * FROM events
WHERE created_at BETWEEN '2024-03-01' AND '2024-03-31';
```

## Partition-Wise Joins

When two partitioned tables share the same partition key and boundaries, PostgreSQL can join them partition-by-partition instead of scanning both full tables:

```sql
SET enable_partitionwise_join = on;

SELECT e.*, u.name
FROM events e
JOIN user_events u ON e.user_id = u.user_id AND e.created_at = u.event_time
WHERE e.created_at > '2024-01-01';
```

## Maintenance

- Use `pg_partman` for automatic partition creation and detachment
- Monitor partition count — keep it under a few hundred for planner performance
- VACUUM and ANALYZE run per-partition, which is more efficient than on a monolithic table
