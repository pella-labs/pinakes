# SSH Security Hardening

**SSH** is the primary remote access protocol for servers. Default configurations are often insecure and should be hardened.

## Server Configuration

Key settings for `/etc/ssh/sshd_config`:

```
# Authentication
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AuthorizedKeysFile .ssh/authorized_keys
MaxAuthTries 3

# Protocol
Protocol 2
KexAlgorithms curve25519-sha256
Ciphers chacha20-poly1305@openssh.com,aes256-gcm@openssh.com
MACs hmac-sha2-512-etm@openssh.com,hmac-sha2-256-etm@openssh.com

# Logging
LogLevel VERBOSE

# Session
ClientAliveInterval 300
ClientAliveCountMax 2
LoginGraceTime 30
MaxSessions 5
```

## Key Management

- Use Ed25519 keys (see [[sec-087]] for digital signature details)
- Protect private keys with passphrases
- Use ssh-agent to avoid typing passphrases repeatedly
- Rotate keys periodically
- Remove keys of departed team members immediately

## SSH Certificates

For larger organizations, use **SSH certificates** instead of raw public keys:

- A central CA signs user and host keys
- No need to distribute authorized_keys to every server
- Certificates have built-in expiration
- Principals restrict which servers a certificate can access

## Bastion Hosts

Route all SSH access through a bastion host:

- Single entry point for audit logging
- No direct SSH access to application servers
- Short-lived access via tools like Teleport or Boundary

See [[sec-064]] for the bastion host architecture pattern.
