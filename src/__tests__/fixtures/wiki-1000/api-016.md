# HATEOAS and Hypermedia APIs

## What It Is

**Hypermedia as the Engine of Application State.** The API response includes links that tell the client what actions are available from the current state.

```json
{
  "id": "ord_123",
  "status": "pending",
  "_links": {
    "self": { "href": "/orders/ord_123" },
    "cancel": { "href": "/orders/ord_123/cancel", "method": "POST" },
    "items": { "href": "/orders/ord_123/items" },
    "customer": { "href": "/customers/cust_456" }
  }
}
```

Once the order is shipped, the `cancel` link disappears. The client doesn't need hardcoded state machine logic — it reads the available transitions from the response.

## HAL Format

The **Hypertext Application Language** uses `_links` and `_embedded`:

```json
{
  "_links": {
    "self": { "href": "/orders?page=2" },
    "next": { "href": "/orders?page=3" },
    "prev": { "href": "/orders?page=1" }
  },
  "_embedded": {
    "orders": [...]
  }
}
```

## Practical Reality

Full HATEOAS is rarely implemented in practice. Most teams adopt a lighter version: include `self` links and pagination links, skip the full state-machine discovery. The ROI on full hypermedia is highest for **public APIs** where you can't control client behavior.
