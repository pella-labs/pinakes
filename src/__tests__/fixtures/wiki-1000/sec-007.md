# CORS Configuration

**Cross-Origin Resource Sharing** controls which web origins can make requests to your API. Misconfigured CORS is one of the most common web security issues.

## The Same-Origin Policy

Browsers enforce the **same-origin policy**: scripts on `https://app.example.com` cannot read responses from `https://api.other.com` unless the server explicitly allows it via CORS headers.

## Essential Headers

```
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Methods: GET, POST, PUT, DELETE
Access-Control-Allow-Headers: Authorization, Content-Type
Access-Control-Allow-Credentials: true
Access-Control-Max-Age: 86400
```

## Common Mistakes

- Setting `Access-Control-Allow-Origin: *` while also using `Allow-Credentials: true` — browsers reject this combination
- Reflecting the `Origin` header without validation — effectively disabling CORS
- Forgetting to handle **preflight OPTIONS** requests, causing all non-simple requests to fail
- Not restricting allowed headers, leaking internal header names

## Recommended Configuration

Maintain an explicit allowlist of permitted origins. Validate the request's `Origin` header against this list. Never use wildcard origins in production APIs that handle authenticated requests.

See [[sec-008]] for Content Security Policy, which provides complementary browser-side protections.
