# Input Validation Patterns

**Input validation** is the first line of defense at system boundaries. Accept only what you expect and reject everything else.

## Validation Principles

Validate on three dimensions:

- **Type**: is it a string, number, boolean, array?
- **Format**: does it match the expected pattern (email, UUID, date)?
- **Range**: is it within acceptable bounds (length, value, size)?

## Schema Validation

Use schema validation libraries for structured input:

```typescript
import { z } from 'zod';

const CreateUserSchema = z.object({
  email: z.string().email().max(255),
  name: z.string().min(1).max(100),
  age: z.number().int().min(13).max(150).optional(),
  role: z.enum(['user', 'admin']),
});

type CreateUser = z.infer<typeof CreateUserSchema>;
```

## Allowlist vs Denylist

Always prefer **allowlists** (permit only known-good patterns) over **denylists** (block known-bad patterns). Denylists are incomplete by definition — attackers find new encodings and bypasses.

## Common Validation Failures

- Validating input format but not length (buffer overflow risk)
- Validating on the client but not the server
- Trusting HTTP headers like `Content-Type` without verification
- Not validating file upload content type and size

## Where to Validate

Validate at every system boundary:

- API request handlers
- Message queue consumers
- File upload processors
- Database query parameters
- Configuration file parsers

See [[sec-011]] for SQL injection prevention as a specific case of input validation failure.
