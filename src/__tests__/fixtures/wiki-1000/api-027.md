# GraphQL Mutations

## Input/Payload Convention

Every mutation takes a single `input` argument and returns a `payload` type:

```graphql
type Mutation {
  createOrder(input: CreateOrderInput!): CreateOrderPayload!
  cancelOrder(input: CancelOrderInput!): CancelOrderPayload!
}

input CreateOrderInput {
  customerId: ID!
  items: [OrderItemInput!]!
  idempotencyKey: String!
}

type CreateOrderPayload {
  order: Order
  errors: [UserError!]!
}

type UserError {
  field: [String!]
  message: String!
  code: ErrorCode!
}
```

## Why Errors in the Payload

GraphQL doesn't have HTTP status codes per-field. By putting errors in the payload, the client can distinguish between:
- **Top-level errors** (network, auth) → caught by the error link
- **User errors** (validation, business rules) → handled per-mutation in the UI

## Optimistic Updates

Name mutations clearly so the client can predict the outcome. `toggleFavorite` is better than `updateUserPreference` for optimistic UI updates. The mutation name should imply the resulting state.
