# Chaos Engineering

## Principles

**Chaos engineering** is the practice of intentionally introducing failures to verify that your system handles them gracefully. It's not about breaking things — it's about building confidence in reliability.

## Litmus Chaos on Kubernetes

```yaml
apiVersion: litmuschaos.io/v1alpha1
kind: ChaosEngine
metadata:
  name: api-chaos
spec:
  appinfo:
    appns: production
    applabel: app=api-server
  chaosServiceAccount: litmus-admin
  experiments:
    - name: pod-delete
      spec:
        components:
          env:
            - name: TOTAL_CHAOS_DURATION
              value: "30"
            - name: CHAOS_INTERVAL
              value: "10"
```

## Experiment Types

- **Pod failures**: delete pods, kill containers
- **Network chaos**: add latency, drop packets, partition
- **Resource stress**: CPU pressure, memory pressure, disk fill
- **Node failures**: drain nodes, cordon nodes

## Game Days

Run quarterly **game days** where the team practices handling failures in a controlled environment. Start in staging, graduate to production once the team is comfortable.

## Prerequisites

Before chaos engineering, you need solid observability (see [[ops-019]]), automated recovery (see [[ops-014]]), and SLOs (see [[ops-050]]).
