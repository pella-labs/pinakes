# Kubernetes ConfigMaps

## Usage Patterns

**ConfigMaps** hold non-sensitive configuration data. Mount them as environment variables or files.

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  APP_ENV: production
  LOG_LEVEL: info
  CACHE_TTL: "300"
  config.yaml: |
    server:
      port: 8080
      timeout: 30s
    features:
      newDashboard: true
```

## Mounting as Files

```yaml
containers:
  - name: app
    volumeMounts:
      - name: config
        mountPath: /etc/app/config.yaml
        subPath: config.yaml
volumes:
  - name: config
    configMap:
      name: app-config
```

## Immutable ConfigMaps

For configuration that shouldn't change after deployment, mark the ConfigMap as **immutable**. This prevents accidental edits and improves performance (kubelet skips polling for changes).

## Reloading on Change

By default, pods don't reload when a ConfigMap changes. Solutions include using a sidecar like **Reloader** that watches for ConfigMap changes and triggers rolling restarts, or designing your application to watch the mounted file for changes.
