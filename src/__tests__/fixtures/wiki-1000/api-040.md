# API Caching Strategy

## Cache Layers

1. **Client cache** — `Cache-Control: max-age=300` (browser/mobile)
2. **CDN cache** — `Cache-Control: s-maxage=60` (edge)
3. **Application cache** — Redis for computed/aggregated data
4. **Database query cache** — prepared statements, result caching

## Cache Headers

```
Cache-Control: public, max-age=300, s-maxage=60, stale-while-revalidate=30
ETag: "abc123"
Vary: Accept, Authorization
```

**Important:** `Vary: Authorization` means each user gets a separate cache entry. This is correct for personalized responses but kills CDN hit rates.

## Cache Invalidation Patterns

- **TTL-based** — simplest, eventual consistency
- **Event-based** — publish invalidation events on write; subscribers purge cache
- **Write-through** — update cache on every write; always fresh but more complex

## What NOT to Cache

- Mutating operations (POST, PUT, DELETE)
- Responses with `Set-Cookie`
- User-specific data at the CDN layer (use application cache instead)
- Real-time data (stock prices, live scores)

See [[api-032]] for ETag-based cache validation.
