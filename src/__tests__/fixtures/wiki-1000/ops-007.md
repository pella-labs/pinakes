# Infrastructure as Code Principles

## Why IaC

**Infrastructure as Code** means managing infrastructure through version-controlled configuration files rather than manual processes. Every change is reviewable, auditable, and reproducible.

## Core Principles

- **Declarative over imperative**: describe the desired end state, not the steps to get there
- **Idempotent**: applying the same config twice produces the same result
- **Version controlled**: all infra changes go through pull requests
- **Tested**: validate configs before applying (linting, plan review, policy checks)
- **Immutable**: replace resources rather than mutating them in place

## Choosing a Tool

| Tool | Approach | Language | Best For |
|------|----------|----------|----------|
| **Terraform** | Declarative | HCL | Multi-cloud, mature ecosystem |
| **Pulumi** | Imperative | TypeScript/Python/Go | Teams that prefer real programming languages |
| **CloudFormation** | Declarative | YAML/JSON | AWS-only shops |
| **CDK** | Imperative | TypeScript/Python | AWS with programming language comfort |

We use [[ops-006]] (Terraform) for shared infrastructure and [[ops-008]] (Pulumi) for application-specific resources where the team prefers TypeScript.

## Anti-Patterns

Avoid **ClickOps** — making changes through the cloud console. Any manual change creates drift. If you must make an emergency change manually, import it into IaC immediately after.
