# Schema Design Patterns

## Entity-Attribute-Value (EAV)

The **EAV pattern** stores arbitrary attributes as rows instead of columns. It's flexible but makes querying painful:

```sql
CREATE TABLE entity_attributes (
  entity_id INTEGER NOT NULL,
  attribute_name TEXT NOT NULL,
  attribute_value TEXT,
  PRIMARY KEY (entity_id, attribute_name)
);
```

Avoid EAV if you can. PostgreSQL's **JSONB** columns are almost always a better choice for semi-structured data.

## Polymorphic Associations

When multiple entity types share a relationship (e.g., comments on posts AND comments on images), you have options:

- **Single table with type column** — simple, but nullable columns
- **Separate join tables** — clean, but more complex queries
- **Shared base table with type-specific tables** — the best of both worlds for most cases

## Soft Deletes

Instead of `DELETE`, set a `deleted_at` timestamp. This preserves audit trails and enables undo. But remember to add `WHERE deleted_at IS NULL` to every query — or use a view.

## Temporal Tables

For tracking changes over time, use **temporal patterns** with `valid_from` and `valid_to` columns. PostgreSQL's `tsrange` type with exclusion constraints makes this ergonomic. See [[db-112]] for event sourcing as an alternative approach to temporal data.
