# Postgres TOAST

## The Problem

PostgreSQL stores rows on 8KB pages. But some values (TEXT, JSONB, BYTEA) can be much larger. **TOAST** (The Oversized-Attribute Storage Technique) handles this.

## How TOAST Works

When a row exceeds the page size:
1. PostgreSQL attempts to compress the value
2. If still too large, it moves the value to a TOAST table
3. The main row stores a pointer to the TOAST table

## TOAST Strategies

Each column can have a strategy:
- **PLAIN**: no compression, no out-of-line storage
- **EXTENDED** (default for TEXT/JSONB): compress first, then TOAST if needed
- **EXTERNAL**: out-of-line without compression (faster reads of large values)
- **MAIN**: compress but avoid out-of-line (may TOAST if no other choice)

```sql
ALTER TABLE documents ALTER COLUMN content SET STORAGE EXTERNAL;
```

## Performance Implications

- TOAST access requires extra I/O (separate table)
- SELECT * fetches all TOAST'd columns even if you don't need them
- Always SELECT only the columns you need for tables with large values
- Index-only scans avoid TOAST entirely (another reason to use covering indexes)

## Monitoring

```sql
SELECT relname, pg_total_relation_size(oid) AS total,
       pg_relation_size(oid) AS main,
       pg_relation_size(reltoastrelid) AS toast
FROM pg_class WHERE relkind = 'r' AND reltoastrelid != 0
ORDER BY pg_relation_size(reltoastrelid) DESC;
```
