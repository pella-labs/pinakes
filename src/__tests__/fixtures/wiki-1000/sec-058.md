# Server-Side Request Forgery Prevention

**SSRF** allows an attacker to make the server send requests to unintended destinations, typically internal services that are not accessible from the internet.

## Attack Scenario

An application that fetches URLs provided by users:

```
POST /api/fetch-preview
{ "url": "http://169.254.169.254/latest/meta-data/iam/security-credentials/" }
```

The server fetches the AWS metadata endpoint and returns IAM credentials to the attacker.

## Defenses

### URL Validation

Validate and restrict URLs before fetching:

```typescript
function isAllowedUrl(url: string): boolean {
  const parsed = new URL(url);
  
  // Only allow HTTPS
  if (parsed.protocol !== 'https:') return false;
  
  // Block private IP ranges
  const ip = await dns.resolve(parsed.hostname);
  if (isPrivateIP(ip)) return false;
  
  // Block cloud metadata endpoints
  if (parsed.hostname === '169.254.169.254') return false;
  if (parsed.hostname === 'metadata.google.internal') return false;
  
  return true;
}
```

### DNS Rebinding Protection

An attacker can make a hostname resolve to a public IP during validation, then switch it to a private IP. Mitigate by:

- Resolving DNS once and using the IP for the actual request
- Using a DNS proxy that blocks private IP resolution

### Network-Level Controls

- Run outbound-fetching services in a restricted network segment
- Use egress firewall rules to block access to internal services
- Block access to cloud metadata endpoints at the network level

### IMDSv2

On AWS, use **Instance Metadata Service v2** which requires a session token, making SSRF exploitation significantly harder.

See [[sec-036]] for general input validation patterns.
