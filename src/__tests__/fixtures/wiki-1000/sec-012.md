# CSRF Protection

**Cross-Site Request Forgery** tricks a user's browser into making unwanted requests to a site where they're authenticated. The browser automatically includes cookies, so the request appears legitimate.

## Attack Scenario

A user is logged into `bank.example.com`. They visit a malicious page that contains:

```html
<form action="https://bank.example.com/transfer" method="POST">
  <input type="hidden" name="to" value="attacker" />
  <input type="hidden" name="amount" value="10000" />
</form>
<script>document.forms[0].submit();</script>
```

The browser sends the form with the user's session cookie, and the bank processes it as a legitimate request.

## Defenses

### Synchronizer Token Pattern

Generate a random token per session, embed it in forms, and validate it server-side. The attacker cannot read the token from another origin.

### SameSite Cookies

```
Set-Cookie: session=abc123; SameSite=Lax; Secure; HttpOnly
```

**SameSite=Lax** prevents cookies from being sent on cross-site POST requests while still allowing GET navigations. **SameSite=Strict** blocks all cross-site cookie sending.

### Double Submit Cookie

Set a random value in both a cookie and a request header. The server verifies they match. Since an attacker can't read the cookie from another origin, they can't forge the header.

## Modern APIs

For **JSON APIs** with `Content-Type: application/json`, CSRF is largely mitigated because browsers won't send JSON payloads in cross-origin form submissions. But always verify the Content-Type header server-side.
