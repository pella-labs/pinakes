# Kubernetes Service Accounts

## Default Behavior

Every pod in Kubernetes runs under a **service account**. If none is specified, it uses the `default` service account in the namespace. This is too permissive for production.

## Creating Service Accounts

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: api-server
  namespace: production
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::123456789:role/api-server
automountServiceAccountToken: false
```

## IRSA (IAM Roles for Service Accounts)

On EKS, **IRSA** maps Kubernetes service accounts to AWS IAM roles. This provides pod-level AWS permissions without distributing access keys.

The flow:
1. Create an IAM role with a trust policy for the EKS OIDC provider
2. Annotate the Kubernetes service account with the role ARN
3. Pods using that service account automatically receive temporary AWS credentials

## Best Practices

- Create a dedicated service account per application
- Disable automounting when the pod doesn't need the K8s API
- Use IRSA for all AWS API access from pods
- Audit service account permissions regularly

See [[ops-053]] for IAM best practices and [[ops-028]] for RBAC.
