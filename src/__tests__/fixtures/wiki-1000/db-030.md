# Database Connection Management in Node.js

## The Problem

Node.js is single-threaded but handles many concurrent requests via its event loop. Each request that needs database access must acquire a connection, use it, and release it back promptly. Holding connections too long starves other requests.

## PostgreSQL with pg

```typescript
import { Pool } from 'pg';

const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
  max: 10,
  idleTimeoutMillis: 30000,
});

// Automatic connection management
const { rows } = await pool.query('SELECT * FROM users WHERE id = $1', [id]);

// Manual for transactions
const client = await pool.connect();
try {
  await client.query('BEGIN');
  await client.query('INSERT INTO orders ...');
  await client.query('UPDATE inventory ...');
  await client.query('COMMIT');
} catch (e) {
  await client.query('ROLLBACK');
  throw e;
} finally {
  client.release();  // ALWAYS release back to pool
}
```

## SQLite with better-sqlite3

better-sqlite3 is synchronous by design, which pairs well with SQLite's single-writer model:

```typescript
import Database from 'better-sqlite3';

const db = new Database('app.db');
db.pragma('journal_mode = WAL');

// Prepared statements are cached and fast
const getUser = db.prepare('SELECT * FROM users WHERE id = ?');
const user = getUser.get(userId);
```

See [[db-042]] for advanced better-sqlite3 patterns and [[db-010]] for connection pooling theory.
