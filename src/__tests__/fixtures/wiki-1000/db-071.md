# Database Schema Versioning

## Why Version Your Schema?

Every application evolves. Tables gain columns, indexes change, constraints tighten. Without a systematic approach to schema versioning, deployments become dangerous guesswork.

## Migration-Based Versioning

The standard approach: a sequence of numbered migration files, each containing forward (up) and optionally backward (down) SQL.

```
migrations/
  001_create_users.sql
  002_add_email_to_users.sql
  003_create_orders.sql
  004_add_index_on_orders_date.sql
```

A metadata table tracks which migrations have been applied:

```sql
CREATE TABLE schema_migrations (
  version INT PRIMARY KEY,
  applied_at TIMESTAMPTZ DEFAULT now()
);
```

## Tools

- **Drizzle Kit** — generates migrations from TypeScript schema diffs
- **Flyway** — Java-based, widely used in enterprise
- **golang-migrate** — lightweight, language-agnostic
- **Alembic** — Python/SQLAlchemy

## Rules

- Migrations are immutable once deployed
- Never edit an applied migration
- Test migrations against production-like data
- Always have a rollback plan

See [[db-016]] for safe migration patterns and [[db-058]] for data migration strategies.
