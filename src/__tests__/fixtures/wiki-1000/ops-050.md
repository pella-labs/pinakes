# SLOs and Error Budgets

## Service Level Objectives

An **SLO** is a target for service reliability. It's the most important agreement between the platform team and service owners.

## Defining SLOs

Start with what matters to users:

- **Availability**: 99.9% of requests return a non-5xx response
- **Latency**: 99% of requests complete in <200ms, 99.9% in <1s
- **Correctness**: 99.99% of data writes are durable within 5 seconds

## Error Budgets

The **error budget** is the inverse of the SLO. If your SLO is 99.9%, your error budget is 0.1% — about 43 minutes of downtime per month.

When the error budget is exhausted:
- Halt feature deploys
- Focus engineering on reliability improvements
- Resume feature work when the budget is replenished

## Measuring SLOs

Use **SLIs** (Service Level Indicators) to measure against SLOs. For a web service:

```promql
# Availability SLI
sum(rate(http_requests_total{status!~"5.."}[30d]))
/
sum(rate(http_requests_total[30d]))

# Latency SLI (p99)
histogram_quantile(0.99, sum(rate(http_request_duration_seconds_bucket[30d])) by (le))
```

## Burn Rate Alerts

Alert on **burn rate**, not raw error count. If the current error rate would exhaust the monthly budget in 1 hour, that's a fast burn requiring immediate attention.

See [[ops-019]] for monitoring and [[ops-047]] for incident response.
