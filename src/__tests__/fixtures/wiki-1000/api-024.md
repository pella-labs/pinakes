# Client-Side Retry Strategies

## Exponential Backoff with Jitter

Never retry immediately. Always add randomness to prevent **thundering herd**.

```typescript
function calculateDelay(attempt: number, baseMs: number = 1000): number {
  const exponential = baseMs * Math.pow(2, attempt);
  const jitter = Math.random() * exponential;
  return Math.min(exponential + jitter, 60000); // cap at 60s
}
```

## Which Errors to Retry

| Status | Retry? | Why |
|---|---|---|
| 408 | Yes | Request timeout — transient |
| 429 | Yes | Rate limited — use `Retry-After` header |
| 500 | Yes | Server error — may be transient |
| 502/503/504 | Yes | Infrastructure — likely transient |
| 400 | No | Bad request — won't fix itself |
| 401 | No | Auth failure — need new credentials |
| 404 | No | Not found — resource doesn't exist |
| 409 | Maybe | Conflict — depends on the cause |

## Circuit Breaker

After N consecutive failures, **open the circuit** — stop sending requests for a cooldown period. Then try one request (half-open). If it succeeds, close the circuit. If it fails, reopen.

This prevents a failing dependency from taking down the caller with connection pool exhaustion.
