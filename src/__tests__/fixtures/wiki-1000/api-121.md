# API Dependency Injection

## Why DI in API Servers

Testable code requires swappable dependencies. Don't hardcode database connections, HTTP clients, or service instances.

```typescript
// Bad: hardcoded dependency
class OrderController {
  async create(req: Request, res: Response) {
    const db = getDatabase(); // hidden dependency
    await db.insert('orders', req.body);
  }
}

// Good: injected dependency
class OrderController {
  constructor(private orderRepo: OrderRepository) {}

  async create(req: Request, res: Response) {
    const order = await this.orderRepo.create(req.body);
    res.status(201).json(order);
  }
}
```

## Composition Root

Wire everything together at startup:

```typescript
function createApp(config: AppConfig) {
  const db = createDatabase(config.db);
  const orderRepo = new OrderRepository(db);
  const orderController = new OrderController(orderRepo);
  
  const app = express();
  app.post('/orders', (req, res) => orderController.create(req, res));
  return app;
}
```

## In Tests

```typescript
const mockRepo = { create: vi.fn().mockResolvedValue({ id: 'ord_1' }) };
const controller = new OrderController(mockRepo);
```

No DI container needed for most Node APIs. Constructor injection with a composition root is enough.
