# Dependency Vulnerability Management

Third-party dependencies are responsible for a significant percentage of security vulnerabilities. Managing them is a continuous process, not a one-time activity.

## Automated Scanning

Run vulnerability scanners in CI and on a schedule:

- **npm audit**: built-in for Node.js projects
- **Snyk**: commercial, broader language support
- **Dependabot**: GitHub-native, automatic PRs
- **Renovate**: highly configurable dependency updates

## Triage Process

Not all vulnerabilities are equal. Triage by:

- **Reachability**: is the vulnerable code path actually used in your application?
- **Exploitability**: is there a known exploit, or is it theoretical?
- **Severity**: CVSS score as a starting point, adjusted by context
- **Exposure**: is the vulnerable component internet-facing?

## Update Strategy

- **Critical/High with known exploits**: patch within 24 hours
- **High without exploits**: patch within 1 week
- **Medium**: patch within 1 month
- **Low**: batch with regular dependency updates

## Lock Files

Always commit lock files (`package-lock.json`, `pnpm-lock.yaml`). They ensure reproducible builds and prevent unexpected updates in production.

## Supply Chain Attacks

Beyond known vulnerabilities, watch for:

- Typosquatting (packages with names similar to popular ones)
- Dependency confusion (private vs. public package name conflicts)
- Compromised maintainer accounts
- Malicious post-install scripts
