# Cryptographic Agility

**Cryptographic agility** is the ability to swap cryptographic algorithms without major code changes. This is essential because algorithms are regularly broken or deprecated.

## Why It Matters

History is full of algorithm deprecations:

- MD5 → SHA-2 (collision attacks)
- SHA-1 → SHA-2 (collision attacks)
- DES → AES (key length inadequacy)
- RSA-1024 → RSA-2048+ (factoring advances)
- Will current algorithms eventually need replacement? Almost certainly.

## Design Patterns

### Algorithm Identifiers

Tag every encrypted or signed artifact with the algorithm used:

```
{alg}:{version}:{payload}
```

This allows the system to select the correct algorithm for verification or decryption.

### Provider Interface

```typescript
interface CryptoProvider {
  encrypt(plaintext: Buffer, key: Buffer): EncryptedPayload;
  decrypt(payload: EncryptedPayload, key: Buffer): Buffer;
  sign(data: Buffer, key: Buffer): Signature;
  verify(data: Buffer, signature: Signature, key: Buffer): boolean;
}
```

Swap implementations without changing calling code.

### Gradual Migration

1. Add support for the new algorithm
2. Encrypt new data with the new algorithm
3. Re-encrypt data on read (lazy migration)
4. After sufficient time, remove old algorithm support

## JWTs and Agility

JWTs are designed for crypto agility via the `alg` header and `kid` for key selection. See [[sec-003]] for JWT design and [[sec-016]] for key rotation.
