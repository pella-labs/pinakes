# Database Migrations Best Practices

## The Fundamental Rule

Every schema change should be **forward-only, backward-compatible, and reversible**. This means:

- Never drop a column in the same deploy that stops writing to it
- Add new columns as nullable or with defaults
- Rename via add-copy-rename strategy, not ALTER RENAME

## Migration Tooling

```typescript
// drizzle-kit example
import { sqliteTable, text, integer } from 'drizzle-orm/sqlite-core';

export const users = sqliteTable('users', {
  id: integer('id').primaryKey(),
  name: text('name').notNull(),
  email: text('email').notNull(),
  // New column — nullable for backward compatibility
  avatarUrl: text('avatar_url'),
});
```

## Safe Column Operations

| Operation | Safe? | Notes |
|-----------|-------|-------|
| Add nullable column | Yes | No data rewrite needed |
| Add column with default | Depends | PG 11+ is fast; older versions rewrite table |
| Drop column | Careful | Stop reading first, drop in next deploy |
| Rename column | No | Use add-copy-drop pattern |
| Change column type | No | Add new column, backfill, swap |

## Large Table Migrations

For tables with millions of rows, use **online schema change** tools:

- PostgreSQL: `pg_repack`, or just careful `ALTER TABLE` (many are instant in PG 11+)
- MySQL: `pt-online-schema-change` or `gh-ost`

See [[db-042]] for how better-sqlite3 handles migrations in local-first applications.
