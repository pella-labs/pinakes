# Database Sharding

## What Is Sharding?

**Sharding** splits data across multiple database instances based on a shard key. Each shard holds a subset of the total data. This enables horizontal scaling beyond the capacity of a single machine.

## Choosing a Shard Key

The shard key determines data distribution. A good shard key:

- Distributes data evenly (avoid hotspots)
- Co-locates related data (minimize cross-shard queries)
- Is immutable (changing shard keys means moving data)

Common choices: tenant_id for multi-tenant SaaS, user_id for social applications, geographic region for location-based services.

## Sharding Strategies

### Hash-Based
Apply a hash function to the shard key and modulo by the number of shards. Even distribution but loses range query ability.

### Range-Based
Assign key ranges to shards. Supports range queries but risks hotspots if keys aren't uniformly distributed.

### Directory-Based
A lookup table maps keys to shards. Most flexible but adds a single point of failure.

## The Pain of Sharding

Cross-shard queries are expensive. Cross-shard transactions are even worse. Resharding (adding/removing shards) requires data migration. Auto-incrementing IDs don't work — use UUIDs or snowflake IDs.

Consider sharding a last resort. Vertical scaling, read replicas, caching, and partitioning (see [[db-024]]) can often defer the need.
