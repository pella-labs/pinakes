# API Deprecation Strategy

## Lifecycle Stages

1. **Active** — fully supported, recommended
2. **Deprecated** — still works, sunset date announced
3. **Sunset** — returns 410 Gone

## Signaling Deprecation

Add headers to every response from deprecated endpoints:

```
Deprecation: true
Sunset: Sat, 01 Mar 2026 00:00:00 GMT
Link: </v3/users>; rel="successor-version"
```

Also log which clients are still hitting deprecated endpoints. Use the API key or JWT `client_id` claim to identify them.

## Breaking Changes

A breaking change is any change that causes existing, correctly-written clients to fail:

- Removing a field from a response
- Changing a field type
- Adding a required parameter
- Changing error codes or HTTP status codes
- Changing URL structure

**Additive changes are not breaking:** new optional fields, new endpoints, new optional params.

## Migration Support

Offer a compatibility mode during the deprecation window. Example: v1 sends `user_name`, v2 sends `username`. During the overlap, v1 includes both. After sunset, only v2 shape remains.
