# API Versioning with GraphQL

GraphQL doesn't have traditional versioning. The schema evolves.

## Field Deprecation

```graphql
type User {
  name: String! @deprecated(reason: "Use firstName and lastName")
  firstName: String!
  lastName: String!
}
```

## Evolution Strategy

- **Add** new fields freely (non-breaking)
- **Deprecate** old fields with `@deprecated`
- **Remove** only after monitoring shows zero usage
- **Never** change a field's type

## Schema Registry

Track schema changes over time. Run compatibility checks before deploying:

```bash
rover subgraph check my-graph@production \
  --schema schema.graphql \
  --name orders
```

This detects breaking changes before they hit production.

## When You Actually Need Versioning

If you must make a truly breaking change (restructuring core types), deploy a new graph endpoint (`/graphql/v2`) rather than breaking the existing one. But this should be extremely rare — GraphQL's additive model handles most evolution.
