# API Idempotency Deep Dive

## Beyond Simple Keys

For complex operations, the idempotency key alone isn't enough. You need to verify the request body matches too.

```typescript
async function handleIdempotent(key: string, body: string, handler: () => Promise<Response>): Promise<Response> {
  const existing = await store.get(key);
  
  if (existing) {
    const bodyHash = crypto.createHash('sha256').update(body).digest('hex');
    if (existing.bodyHash !== bodyHash) {
      return { status: 422, error: 'Idempotency key reused with different request body' };
    }
    return existing.response;
  }

  const response = await handler();
  await store.set(key, {
    bodyHash: crypto.createHash('sha256').update(body).digest('hex'),
    response,
    createdAt: Date.now(),
  });
  return response;
}
```

## Concurrent Requests with Same Key

Two requests with the same key arrive simultaneously. Without locking, both execute.

Solution: use Redis `SET NX` as a lock:

```typescript
const locked = await redis.set(`lock:${key}`, '1', 'NX', 'EX', 60);
if (!locked) {
  // Another request is processing — wait and return cached result
  await sleep(100);
  return store.get(key);
}
```

## Cleanup

Purge idempotency records after 24-48 hours. Longer retention wastes storage with negligible benefit.
