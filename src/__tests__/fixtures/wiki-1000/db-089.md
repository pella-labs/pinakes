# Database Denormalization Patterns

## Why Denormalize?

Normalization (see [[db-007]]) eliminates redundancy but requires joins. When read performance matters more than write simplicity, **strategic denormalization** trades storage for speed.

## Common Patterns

### Precomputed Counts

```sql
-- Instead of COUNT(*) every time
ALTER TABLE users ADD COLUMN order_count INT DEFAULT 0;

-- Update via trigger or application code
UPDATE users SET order_count = order_count + 1 WHERE id = $1;
```

### Cached Aggregates

Store daily/hourly summaries alongside raw data. See [[db-080]] for real-time analytics patterns.

### Embedded Foreign Data

Store the foreign entity's key fields alongside the reference:

```sql
-- Instead of joining orders → customers every time
ALTER TABLE orders ADD COLUMN customer_name TEXT;
ALTER TABLE orders ADD COLUMN customer_email TEXT;
```

The tradeoff: if the customer's name changes, you must update all their orders.

### Materialized Joins

Create a denormalized table that combines data from multiple source tables. Refresh it periodically or via CDC (see [[db-026]]).

## Rules of Thumb

- Always normalize first
- Denormalize only based on measured, frequent query patterns
- Document every denormalization and its refresh strategy
- Monitor for staleness
