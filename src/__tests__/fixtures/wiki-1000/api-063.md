# REST Sub-Resources vs Query Parameters

When to nest resources and when to use filters.

## Nest When

There's a clear **ownership** relationship:

```
GET /orders/{id}/items          ← items belong to an order
GET /users/{id}/addresses       ← addresses belong to a user
POST /orders/{id}/comments      ← creating a comment on an order
```

## Filter When

It's a search or cross-cutting query:

```
GET /items?orderId=ord_123      ← searching items by order
GET /orders?status=pending&customerId=cust_456
GET /products?category=electronics&minPrice=100
```

## Rules of Thumb

- If the child resource has no meaning without the parent → nest
- If the child resource can belong to multiple parents → filter
- If nesting would exceed 2 levels → flatten and filter
- If the client commonly needs the child independent of parent → top-level with filter

## Collections on Collections

Avoid `GET /users/{id}/orders/{oid}/items/{iid}/reviews`. Instead:

```
GET /reviews?itemId=iid_123
```

Or provide convenience nested routes for the most common access pattern but also a flat route for flexibility.
