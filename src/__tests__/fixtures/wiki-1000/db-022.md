# Redis Caching Patterns

## Cache-Aside (Lazy Loading)

The application checks the cache first. On miss, it queries the database, stores the result in Redis, and returns it. This is the most common pattern.

```typescript
async function getUser(id: string): Promise<User> {
  const cached = await redis.get(`user:${id}`);
  if (cached) return JSON.parse(cached);

  const user = await db.query('SELECT * FROM users WHERE id = $1', [id]);
  await redis.setex(`user:${id}`, 3600, JSON.stringify(user));
  return user;
}
```

## Write-Through

Every write goes to both the database and the cache. The cache is always up-to-date, but writes are slower.

## Write-Behind (Write-Back)

Writes go to the cache immediately. A background process flushes to the database asynchronously. Fast writes but risk data loss if Redis crashes before flushing.

## Cache Invalidation

The two hardest problems in computer science: cache invalidation, naming things, and off-by-one errors.

Strategies:
- **TTL** — set an expiration. Simple but stale data is possible.
- **Event-driven** — invalidate on write. Requires reliable event propagation.
- **Version keys** — include a version number in the cache key. Bump the version to invalidate.
