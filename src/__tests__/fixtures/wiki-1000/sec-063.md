# Database Access Control

Securing database access requires multiple layers: network controls, authentication, authorization, and encryption.

## Principle of Least Privilege

Create separate database users for different purposes:

```sql
-- Read-only user for the API
CREATE USER api_reader WITH PASSWORD '...';
GRANT SELECT ON ALL TABLES IN SCHEMA public TO api_reader;

-- Read-write user for the application
CREATE USER app_writer WITH PASSWORD '...';
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_writer;

-- Migration user (elevated, used only during deployments)
CREATE USER migrator WITH PASSWORD '...';
GRANT ALL PRIVILEGES ON SCHEMA public TO migrator;
```

## Network Security

- Database should not be accessible from the public internet
- Use private subnets and security groups
- Connect via VPN or SSH tunnel from developer machines
- Enable TLS for all connections (see [[sec-017]])

## Query Auditing

Enable database-level query logging for sensitive tables:

```sql
-- PostgreSQL audit extension
CREATE EXTENSION pgaudit;
ALTER SYSTEM SET pgaudit.log = 'write, ddl';
ALTER SYSTEM SET pgaudit.log_catalog = off;
```

## Dynamic Credentials

Use short-lived database credentials generated by Vault (see [[sec-015]]) instead of static passwords. This limits the window of credential compromise and provides automatic rotation.

## Row-Level Security

For multi-tenant applications, use database-native row-level security:

```sql
CREATE POLICY tenant_isolation ON orders
  USING (tenant_id = current_setting('app.tenant_id')::uuid);
```

This ensures data isolation even if application-level checks fail.
