# Secure Multi-Tenancy

**Multi-tenant** applications serve multiple customers from shared infrastructure. Security failures can expose one tenant's data to another.

## Isolation Models

### Shared Everything
All tenants share database, schema, and tables. Tenant isolation via row-level filtering.

- Cheapest to operate
- Highest risk of cross-tenant data leakage
- Requires rigorous query discipline

### Shared Database, Separate Schemas
Each tenant gets their own database schema within a shared database.

- Better isolation than shared tables
- Schema migrations must handle all tenant schemas
- Moderate operational complexity

### Separate Databases
Each tenant gets their own database instance.

- Strongest isolation
- Highest operational cost
- Easiest compliance story

## Row-Level Security

For shared-table models, enforce isolation at the database level:

```sql
ALTER TABLE documents ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON documents
  USING (tenant_id = current_setting('app.current_tenant')::uuid);
```

Application code sets the tenant context on every connection:

```typescript
await db.query("SET app.current_tenant = $1", [tenantId]);
```

## Application-Level Controls

- Set the tenant context early in the request pipeline
- Never accept tenant ID from user input — derive it from the authentication token
- Include tenant ID in all queries, even those that "shouldn't" need it
- Log cross-tenant access attempts

See [[sec-063]] for database access control and [[sec-004]] for RBAC within a tenant.
