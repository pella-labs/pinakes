# Testing with React Testing Library

## Philosophy

**React Testing Library** encourages testing components the way users interact with them. Query by visible text, roles, and labels—not by internal implementation details.

## Query Priority

Use queries in this order of preference:

1. `getByRole` — accessible role (button, heading, textbox)
2. `getByLabelText` — form element by its label
3. `getByPlaceholderText` — input placeholders
4. `getByText` — visible text content
5. `getByTestId` — last resort, implementation detail

## Common Patterns

```tsx
import { render, screen, within } from '@testing-library/react';
import userEvent from '@testing-library/user-event';

test('submits the form with valid data', async () => {
  const user = userEvent.setup();
  const onSubmit = vi.fn();

  render(<LoginForm onSubmit={onSubmit} />);

  await user.type(screen.getByLabelText('Email'), 'test@example.com');
  await user.type(screen.getByLabelText('Password'), 'password123');
  await user.click(screen.getByRole('button', { name: 'Log in' }));

  expect(onSubmit).toHaveBeenCalledWith({
    email: 'test@example.com',
    password: 'password123',
  });
});
```

## Async Testing

Use `findBy` queries for elements that appear asynchronously:

```tsx
const message = await screen.findByText('Success!');
expect(message).toBeInTheDocument();
```

## Anti-Patterns

- Testing implementation details (state values, component internals)
- Snapshot testing as the primary test strategy
- Using `container.querySelector` instead of accessible queries
- Testing library code instead of your integration with it
