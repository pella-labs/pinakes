# API Data Transformation Layers

## DTO Pattern (Data Transfer Object)

Never expose your database model directly as an API response.

```typescript
// Database model
interface OrderEntity {
  id: string;
  customer_id: string;
  internal_note: string;  // should never be in API response
  created_at: Date;
  total_cents: number;
}

// API DTO
interface OrderResponse {
  id: string;
  customerId: string;
  createdAt: string;      // ISO 8601
  total: { amount: number; currency: string };
}

function toOrderResponse(entity: OrderEntity): OrderResponse {
  return {
    id: entity.id,
    customerId: entity.customer_id,
    createdAt: entity.created_at.toISOString(),
    total: { amount: entity.total_cents / 100, currency: 'USD' },
  };
}
```

## Why DTOs

- **Security**: prevent leaking internal fields
- **Stability**: database schema can change without breaking the API
- **Formatting**: convert types (Date → ISO string, cents → dollars)
- **Naming**: convert snake_case DB columns to camelCase API fields

## Serialization Libraries

Use `class-transformer`, `superstruct`, or plain functions. Avoid heavy ORM serializers — they tend to include everything by default.
