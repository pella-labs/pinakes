# WAF Rules and Configuration

A **Web Application Firewall** inspects HTTP traffic and blocks requests matching known attack patterns. It's a defense-in-depth layer, not a replacement for secure coding.

## Core Rule Sets

Most WAFs support the **OWASP Core Rule Set (CRS)**, which detects:

- SQL injection attempts (see [[sec-011]])
- XSS payloads (see [[sec-009]])
- Path traversal
- Remote code execution
- Scanner detection
- Protocol anomalies

## Deployment Modes

### Detection Mode
Log matching requests but don't block. Use this during initial deployment to tune rules and reduce false positives.

### Prevention Mode
Block matching requests. Only switch to this after tuning.

## Custom Rules

Beyond CRS, add custom rules for your application:

```
# Block requests with SQL keywords in unexpected parameters
SecRule ARGS:id "@rx (?i)(union|select|insert|delete|drop)" \
  "id:1001,phase:2,deny,status:403,msg:'SQL injection attempt'"
```

## False Positive Management

WAF tuning is an ongoing process:

- Start in detection mode
- Review flagged requests to identify false positives
- Create targeted exclusions (not blanket disables)
- Retest after every exclusion change
- Maintain a living document of all exclusions and their justifications

## WAF Bypass Testing

Include WAF bypass testing in your pen test scope (see [[sec-028]]). Testers should verify that the WAF effectively blocks common attack patterns and that exclusions don't create gaps.
