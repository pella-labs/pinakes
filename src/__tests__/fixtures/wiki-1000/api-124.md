# API Database Migration Strategy

## Expand-Contract Pattern

For zero-downtime migrations:

### Phase 1: Expand

Add the new column alongside the old one:

```sql
ALTER TABLE orders ADD COLUMN shipping_estimate TIMESTAMPTZ;
```

### Phase 2: Dual Write

Update application code to write to both columns:

```typescript
await db.query(
  `UPDATE orders SET 
    delivery_date = $1,          -- old column
    shipping_estimate = $1       -- new column
   WHERE id = $2`,
  [estimate, orderId]
);
```

### Phase 3: Backfill

Copy data for existing rows:

```sql
UPDATE orders SET shipping_estimate = delivery_date WHERE shipping_estimate IS NULL;
```

### Phase 4: Switch Reads

Update API to read from the new column. Deploy.

### Phase 5: Contract

Remove the old column (after verifying no clients depend on it):

```sql
ALTER TABLE orders DROP COLUMN delivery_date;
```

## Rules

- Never rename a column in production (breaks reads during deploy)
- Never change a column type without expand-contract
- Always make migrations reversible
- Run migrations in a transaction (except for large backfills)
- Test migrations against a production-size dataset
