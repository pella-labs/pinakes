# PostgreSQL Partitioning Performance

## Benchmark: Partitioned vs Unpartitioned

On a 500M-row events table, partitioned by month:

| Query | Unpartitioned | Partitioned (range) | Improvement |
|-------|--------------|---------------------|-------------|
| WHERE created_at = last 7d | 12.4s | 0.3s | 41x |
| COUNT(*) WHERE month = X | 8.1s | 0.15s | 54x |
| DELETE WHERE month < cutoff | 45s + VACUUM | 0.01s (DROP) | 4500x |
| INSERT (single row) | 0.8ms | 1.2ms | 0.67x (slower) |

Partition pruning is the key to the performance gains. The INSERT slowdown is due to routing overhead — negligible for batch inserts.

## Optimal Partition Count

Testing with different partition granularities on the same dataset:

- Daily partitions (365/year): planner overhead visible at ~1000 partitions
- Weekly partitions (52/year): good balance for most applications
- Monthly partitions (12/year): recommended starting point

PostgreSQL 14+ handles thousands of partitions efficiently, but start with monthly and refine.
