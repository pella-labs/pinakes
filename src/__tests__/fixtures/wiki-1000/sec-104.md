# Regex Denial of Service

**ReDoS** occurs when a regular expression takes exponential time to evaluate on certain inputs. This can freeze a server thread and cause denial of service.

## Vulnerable Patterns

Regexes with nested quantifiers on overlapping character classes:

```
/(a+)+$/           — catastrophic backtracking
/(a|a)+$/          — exponential on "aaaaaaaaaaab"
/(\w+\s*)+/        — common in email/URL validation
```

## How It Happens

The regex engine tries every possible way to match. With nested quantifiers, the number of possibilities grows exponentially with input length. A 30-character input can take minutes.

## Prevention

### Static Analysis

Use tools to detect vulnerable patterns:

```bash
# Install safe-regex checker
npx safe-regex "/(a+)+$/"
# Output: false (unsafe)
```

### Regex Alternatives

For complex validation, use dedicated parsers instead of regex:

- Email: use a validation library, not a regex
- URLs: use `new URL()` constructor
- Dates: use a date parsing library

### Timeout

If you must use complex regex, set a timeout:

```typescript
// Node.js doesn't have native regex timeout
// Use vm.runInNewContext with a timeout
import { runInNewContext } from 'vm';

function safeRegexTest(pattern: RegExp, input: string, timeoutMs = 100): boolean {
  return runInNewContext(
    `pattern.test(input)`,
    { pattern, input },
    { timeout: timeoutMs }
  );
}
```

## In Input Validation

If your validation schema uses regex patterns (see [[sec-036]]), audit them for ReDoS. This is especially important for patterns applied to user-supplied input.
