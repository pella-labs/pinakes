# Secure Feature Flags

**Feature flags** can introduce security risks if not managed carefully. A flag that disables authentication or bypasses validation is a critical vulnerability if left enabled.

## Security Risks

- Flags that disable security controls (auth bypass, validation skip)
- Flags that expose internal/debug endpoints
- Stale flags that remain in code long after they should have been removed
- Flag evaluation data that leaks user information

## Best Practices

### Flag Lifecycle

Every flag should have:

- An owner
- A creation date
- An expected removal date
- A documented purpose

### Security-Sensitive Flags

Flags that affect security controls should:

- Be reviewed by the security team before creation
- Default to the secure state (off = secure)
- Require elevated permissions to toggle
- Be audited when changed (see [[sec-014]])

### Code Hygiene

```typescript
// Good — flag defaults to secure state
if (featureFlags.get('new_auth_flow', false)) {
  newAuthFlow(request);
} else {
  existingAuthFlow(request); // Secure default
}

// Bad — flag disables security
if (!featureFlags.get('disable_csrf_check', false)) {
  verifyCsrfToken(request);
}
```

### Cleanup

Run regular sweeps to remove stale flags. A flag that hasn't been toggled in 90 days is a cleanup candidate.

## Testing

Test both flag states in your security test suite. A feature behind a flag still needs security testing.
