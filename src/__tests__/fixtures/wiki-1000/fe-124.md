# Handling Time Zones in Frontend

## The Problem

Dates in JavaScript are notoriously confusing. `new Date()` uses the user's local time zone, but servers often store UTC. Displaying dates correctly requires explicit time zone handling.

## Best Practices

- Store dates as **ISO 8601 UTC** strings or Unix timestamps
- Convert to the user's local time zone only at the display layer
- Never rely on the server's time zone for display

## Intl.DateTimeFormat

The built-in `Intl` API handles time zone conversion:

```typescript
function formatDate(isoString: string, timeZone?: string) {
  return new Intl.DateTimeFormat('en-US', {
    dateStyle: 'medium',
    timeStyle: 'short',
    timeZone: timeZone ?? Intl.DateTimeFormat().resolvedOptions().timeZone,
  }).format(new Date(isoString));
}
```

## Relative Time

Display "2 hours ago" instead of absolute timestamps for recent events:

```typescript
const rtf = new Intl.RelativeTimeFormat('en', { numeric: 'auto' });

function relativeTime(date: Date) {
  const diffMs = date.getTime() - Date.now();
  const diffHours = Math.round(diffMs / 3600000);
  if (Math.abs(diffHours) < 24) return rtf.format(diffHours, 'hour');
  const diffDays = Math.round(diffMs / 86400000);
  return rtf.format(diffDays, 'day');
}
```

## Hydration Mismatch

Server and client may be in different time zones, causing hydration mismatches. Render dates client-side or use a consistent time zone (UTC) for server rendering.

See [[fe-022]] for hydration issues and [[fe-026]] for i18n.
