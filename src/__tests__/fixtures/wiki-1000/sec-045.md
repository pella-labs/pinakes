# Secure Defaults

Good security starts with **secure defaults**. Users and developers should have to opt out of security, not opt in.

## Principle

Every configuration option that has security implications should default to the most secure setting. Weakening security should require explicit, documented action.

## Examples

### Web Frameworks
- CSRF protection: enabled by default
- Auto-escaping in templates: enabled by default
- Cookie attributes: Secure, HttpOnly, SameSite=Lax by default
- CORS: deny all by default

### Database Connections
- TLS: required by default
- Minimum TLS version: 1.2 by default
- SSL verification: enabled by default

### API Endpoints
- Authentication: required by default (public endpoints are opt-in)
- Rate limiting: applied by default
- Input validation: strict by default

### Deployment
- Network access: deny all by default
- Container capabilities: drop all, add only what's needed
- File permissions: minimum necessary

## Anti-Patterns

These are insecure defaults that frequently cause incidents:

- Shipping with default credentials
- Enabling debug mode in production templates
- Binding to `0.0.0.0` instead of `127.0.0.1` for local services
- Setting overly permissive CORS headers as a "starter" configuration
- Disabling TLS verification in HTTP clients "for convenience"
