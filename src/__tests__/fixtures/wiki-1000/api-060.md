# API Backwards Compatibility Testing

## What Breaks Clients

Any of these in an existing endpoint is a breaking change:

- Removing a response field
- Changing a field's type (string → number)
- Renaming a field
- Adding a required request parameter
- Changing the URL path
- Changing a success status code (200 → 201)
- Narrowing an enum (removing a valid value)

## What Doesn't Break Clients

- Adding optional response fields
- Adding optional request parameters
- Adding new endpoints
- Widening an enum (adding new values — but clients with strict validation may break)
- Adding new error codes (clients should handle unknown errors gracefully)

## Automated Compatibility Checks

Use **OpenAPI diff tools** to detect breaking changes:

```bash
# Compare current spec vs previous version
oasdiff breaking api-v2.3.yaml api-v2.4.yaml
```

Run this in CI. Any breaking change requires a major version bump and deprecation period. See [[api-028]] for the deprecation strategy.
