# Browser Storage Security

Different browser storage mechanisms have different security properties. Choosing the wrong one for sensitive data creates vulnerabilities.

## Storage Comparison

| Storage | Accessible to JS | Sent with requests | Capacity | Expiration |
|---|---|---|---|---|
| Cookies | Yes (unless HttpOnly) | Yes (automatic) | 4KB | Configurable |
| localStorage | Yes | No (manual) | 5-10MB | Persistent |
| sessionStorage | Yes | No (manual) | 5-10MB | Tab close |
| IndexedDB | Yes | No (manual) | Large | Persistent |

## Security Implications

### Cookies
Best for authentication tokens when configured correctly (see [[sec-034]]). HttpOnly prevents JS access, mitigating XSS-based theft.

### localStorage
Accessible to any JavaScript running on the origin, including XSS payloads. Never store authentication tokens or sensitive data here.

### sessionStorage
Slightly better than localStorage because it's scoped to the tab. Still vulnerable to XSS within the tab.

### IndexedDB
Same origin policy applies. Larger storage but same XSS vulnerability as localStorage.

## Recommendations

- **Authentication tokens**: HttpOnly, Secure, SameSite cookies
- **Non-sensitive application state**: localStorage or IndexedDB
- **Temporary sensitive data**: sessionStorage (with acceptance of XSS risk)
- **Nothing sensitive**: should touch localStorage or IndexedDB

## Clearing Data

Implement a proper logout flow that clears all stored data:

```javascript
localStorage.clear();
sessionStorage.clear();
indexedDB.deleteDatabase('app-db');
// Cookies cleared by server setting Max-Age=0
```
