# Batch Endpoints

## When to Use

When clients need to perform multiple operations in one round trip. Reduces HTTP overhead and enables server-side transaction batching.

## Request Shape

```json
{
  "operations": [
    { "method": "POST", "path": "/users", "body": { "name": "Alice" } },
    { "method": "POST", "path": "/users", "body": { "name": "Bob" } },
    { "method": "PATCH", "path": "/users/u_001", "body": { "name": "Carol" } }
  ]
}
```

## Response Shape

```json
{
  "results": [
    { "status": 201, "body": { "id": "u_100", "name": "Alice" } },
    { "status": 201, "body": { "id": "u_101", "name": "Bob" } },
    { "status": 200, "body": { "id": "u_001", "name": "Carol" } }
  ]
}
```

## Design Decisions

- **Atomic or independent?** Google's batch API runs each sub-request independently. Some APIs offer an `atomic: true` flag that wraps all in a transaction.
- **Max batch size:** 100 operations is a common cap.
- **Rate limiting:** count each sub-request against the rate limit, not just the batch request.

Individual failures don't fail the batch — each result has its own status code.
