# Message Brokers vs Database Queues

## The Spectrum

From simplest to most complex:

### 1. Database Queue
Use a table with a status column. Workers poll for unclaimed rows.

```sql
UPDATE tasks SET status = 'processing', worker_id = $1
WHERE id = (
  SELECT id FROM tasks
  WHERE status = 'pending'
  ORDER BY created_at
  LIMIT 1
  FOR UPDATE SKIP LOCKED
)
RETURNING *;
```

Works for low-throughput scenarios (<1000 tasks/minute). No extra infrastructure needed.

### 2. Redis Streams
Durable, consumer-group-based messaging with moderate throughput. See [[db-038]].

### 3. RabbitMQ
Full-featured AMQP broker with routing, dead-letter queues, and acknowledgments. Good for complex routing patterns.

### 4. Apache Kafka
Distributed log with high throughput and replay capability. Essential for event sourcing and stream processing at scale.

## Decision Framework

- **<1K msgs/min + simple**: database queue
- **1K-100K msgs/min + durability**: Redis Streams or RabbitMQ
- **>100K msgs/min OR replay needed**: Kafka
- **Multi-datacenter**: Kafka (with MirrorMaker) or Pulsar

## Don't Over-Engineer

Most applications start fine with a database queue. Add a real broker when the queue table becomes a performance bottleneck.
