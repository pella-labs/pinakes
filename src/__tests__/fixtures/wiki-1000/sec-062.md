# Secure Logging Practices

**Logging** is essential for security monitoring and incident response, but logs themselves can become a security liability if they contain sensitive data.

## What to Never Log

- Passwords or password hashes
- API keys and secrets
- Credit card numbers
- Social Security Numbers
- Full authentication tokens
- Personal health information (PHI)

## Data Masking

When context requires logging sensitive-adjacent data:

```typescript
function maskEmail(email: string): string {
  const [local, domain] = email.split('@');
  return `${local[0]}***@${domain}`;
}

function maskToken(token: string): string {
  return `${token.slice(0, 8)}...${token.slice(-4)}`;
}

logger.info('Authentication attempt', {
  email: maskEmail(user.email),
  token: maskToken(bearerToken),
});
```

## Structured Logging

Use structured (JSON) logging for security events. It enables automated analysis and SIEM integration:

```typescript
logger.warn({
  event: 'authentication_failure',
  actor: { ip: req.ip, userAgent: req.headers['user-agent'] },
  target: { email: maskEmail(email) },
  reason: 'invalid_password',
  requestId: req.id,
});
```

## Log Integrity

Protect logs from tampering:

- Ship logs to a central, append-only store in real-time
- Use TLS for log transport
- Implement log signing or hash chaining for high-assurance environments

See [[sec-014]] for audit logging specifics and [[sec-043]] for security event monitoring.
