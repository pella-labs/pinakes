# Content Security Policy

**CSP** is an HTTP response header that tells the browser which sources of content are allowed to load on your page. It is the most effective defense against **XSS** attacks.

## Basic Policy

```
Content-Security-Policy:
  default-src 'self';
  script-src 'self' https://cdn.example.com;
  style-src 'self' 'unsafe-inline';
  img-src 'self' data: https:;
  connect-src 'self' https://api.example.com;
  frame-ancestors 'none';
  base-uri 'self';
  form-action 'self';
```

## Key Directives

- **default-src**: fallback for all resource types
- **script-src**: where JavaScript can load from (most critical)
- **frame-ancestors**: prevents clickjacking (replaces X-Frame-Options)
- **base-uri**: prevents base tag injection attacks

## Deployment Strategy

1. Start with `Content-Security-Policy-Report-Only` to collect violations without breaking anything
2. Review reports, adjust policy
3. Switch to enforcing mode
4. Monitor ongoing violations

## Nonce-Based Script Loading

Instead of `unsafe-inline`, use **nonces**:

```html
<script nonce="abc123">
  // inline script allowed because nonce matches CSP header
</script>
```

The server generates a fresh nonce per request and includes it in both the CSP header and script tags. See [[sec-009]] for XSS prevention strategies that complement CSP.
