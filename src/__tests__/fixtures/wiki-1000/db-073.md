# Optimistic vs Pessimistic Locking

## Pessimistic Locking

Lock the row when you read it. No one else can modify it until you're done.

```sql
BEGIN;
SELECT * FROM inventory WHERE product_id = 42 FOR UPDATE;
-- Row is locked, other transactions wait
UPDATE inventory SET quantity = quantity - 1 WHERE product_id = 42;
COMMIT;
```

Good when conflicts are frequent. The cost is reduced concurrency — transactions queue up.

## Optimistic Locking

Don't lock anything. Instead, check that nothing changed between your read and write using a version number or timestamp.

```sql
-- Read
SELECT id, quantity, version FROM inventory WHERE product_id = 42;
-- Returns: quantity=10, version=5

-- Update with version check
UPDATE inventory
SET quantity = 9, version = 6
WHERE product_id = 42 AND version = 5;
-- If 0 rows updated → conflict, retry
```

Good when conflicts are rare. The cost is retry logic in the application.

## Application-Level Pattern

```typescript
async function decrementInventory(productId: number): Promise<void> {
  for (let attempt = 0; attempt < 3; attempt++) {
    const item = await db.query('SELECT * FROM inventory WHERE product_id = $1', [productId]);
    const result = await db.query(
      'UPDATE inventory SET quantity = quantity - 1, version = version + 1 WHERE product_id = $1 AND version = $2',
      [productId, item.version]
    );
    if (result.rowCount > 0) return;
  }
  throw new Error('Conflict: too many retries');
}
```
