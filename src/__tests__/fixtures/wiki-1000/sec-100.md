# Runtime Application Self-Protection

**RASP** monitors an application from within the runtime, detecting and blocking attacks in real-time. Unlike WAFs (see [[sec-029]]) that inspect traffic at the network level, RASP has full context of the application state.

## How RASP Works

RASP instruments the application runtime to intercept security-relevant operations:

- SQL query construction (detect injection)
- File system access (detect path traversal)
- Command execution (detect OS command injection)
- Deserialization (detect malicious payloads — see [[sec-090]])
- Outbound network connections (detect SSRF — see [[sec-058]])

## Advantages Over WAF

- **Context-aware**: knows the application logic, not just HTTP patterns
- **Lower false positives**: understands whether a suspicious string is actually used dangerously
- **Post-transformation**: inspects data after the application has processed it

## Disadvantages

- **Performance overhead**: runtime instrumentation has a cost
- **Language-specific**: each runtime needs its own agent
- **Deployment complexity**: tightly coupled with the application

## Practical Usage

RASP is most valuable when:

- Legacy applications that can't be easily refactored for security
- High-risk applications handling sensitive data
- As an additional defense layer behind WAF

## Open-Source Options

- **Sqreen** (now part of Datadog)
- **OpenRASP** (Baidu, for Java/PHP)
- Custom instrumentation via Node.js --require hooks

Don't rely on RASP as your only defense. It's one layer in defense in depth (see [[sec-064]]).
