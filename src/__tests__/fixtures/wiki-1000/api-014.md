# DR: Default Pagination Strategy

## Context

We need a consistent pagination approach across all list endpoints. Currently three services use three different patterns: offset, cursor, and page-number.

## Decision

All new endpoints use **cursor-based pagination** with keyset implementation. Existing offset-based endpoints get a compatibility shim that internally converts to cursor queries.

## Rationale

- Offset pagination performs O(n) skips in the database
- Cursor pagination is O(1) seek via indexed WHERE clause
- Concurrent writes don't cause duplicate/skipped results with cursors
- The GraphQL connection pattern ([[api-002]]) already assumes cursors

## Default Parameters

| Parameter | Default | Max |
|---|---|---|
| `first` / `limit` | 25 | 100 |
| `after` / `cursor` | null (start) | — |

## Consequences

- Client libraries need updating to handle cursor-style responses
- Admin dashboards lose "jump to page N" — mitigated by adding a search/filter instead
