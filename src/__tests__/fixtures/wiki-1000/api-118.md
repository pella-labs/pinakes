# Server-Sent Events vs WebSockets

## Feature Comparison

| Feature | SSE | WebSocket |
|---|---|---|
| Direction | Server → Client | Bidirectional |
| Protocol | HTTP | WS (upgrade from HTTP) |
| Auto-reconnect | Built-in (EventSource) | Manual |
| Binary data | No (text only) | Yes |
| Browser support | All modern | All modern |
| Connection limit | 6 per domain (HTTP/1.1) | No hard limit |
| Load balancer | Standard HTTP | Needs sticky sessions |
| Compression | Standard HTTP gzip | Per-message deflate |

## Choose SSE When

- Server pushes data, client only listens
- You want simple HTTP infrastructure
- You need auto-reconnect with last-event-ID
- Data is text/JSON

## Choose WebSockets When

- Client sends data frequently (chat, gaming)
- You need binary frames
- Sub-100ms latency matters
- You need >6 concurrent streams

## Can't Decide?

Start with SSE. It's simpler, uses standard HTTP, and works behind most proxies. Upgrade to WebSockets only if you hit a wall.
