# HTTP Parameter Pollution

**HTTP Parameter Pollution** (HPP) exploits how web applications handle duplicate HTTP parameters. Different frameworks parse duplicates differently, creating inconsistencies that attackers can exploit.

## The Problem

Consider this URL: `https://api.example.com/transfer?to=alice&to=mallory`

Different frameworks handle this differently:
- Express (Node.js): returns an array `['alice', 'mallory']`
- PHP: uses the last value `mallory`
- Python Flask: uses the first value `alice`
- ASP.NET: concatenates `alice,mallory`

## Attack Scenarios

### WAF Bypass

A WAF checks the first parameter value and allows it. The backend uses the last value:

```
?admin=false&admin=true
```

WAF sees `admin=false` (safe), backend uses `admin=true` (escalated).

### Logic Bypass

If an application checks one parameter value but uses another:

```
?amount=100&amount=10000
```

Validation checks `100`, processing uses `10000`.

## Defenses

- Explicitly handle duplicate parameters — reject or use only the first occurrence
- Ensure WAFs and backends parse parameters identically
- Test with duplicate parameters in your security testing
- Use schema validation that rejects unexpected array values for singular fields (see [[sec-036]])
