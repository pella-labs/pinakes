# Handling Concurrent Updates

## Lost Update Problem

1. Client A reads order (version 1)
2. Client B reads order (version 1)
3. Client A updates order → version 2
4. Client B updates order → overwrites A's changes (version 3, but A's changes gone)

## Optimistic Concurrency

Include a version or ETag in every response. Require it on updates.

```typescript
app.put('/orders/:id', async (req, res) => {
  const { version, ...data } = req.body;
  
  const result = await db.query(
    `UPDATE orders SET data = $1, version = version + 1 
     WHERE id = $2 AND version = $3
     RETURNING *`,
    [data, req.params.id, version]
  );

  if (result.rowCount === 0) {
    return res.status(409).json({
      error: {
        code: 'CONFLICT',
        message: 'Order was modified by another request. Refetch and retry.',
      }
    });
  }

  res.json(result.rows[0]);
});
```

## Pessimistic Locking

Use `SELECT FOR UPDATE` in the database. Blocks concurrent writes. Only use for short critical sections — holding locks too long causes deadlocks.

See [[api-032]] for ETag-based conditional requests.
