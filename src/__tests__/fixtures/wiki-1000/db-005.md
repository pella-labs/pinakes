# MongoDB Document Model

## Documents and Collections

MongoDB stores data as **BSON documents** — binary JSON with additional types like Date, ObjectId, and Decimal128. Documents are grouped into collections, which are analogous to tables but schema-free by default.

## Schema Design Philosophy

Unlike relational databases, MongoDB encourages **denormalization**. Instead of joining across normalized tables, you embed related data within a single document. This trades storage efficiency for read performance.

The decision of when to embed vs. reference depends on:

- Access patterns (do you always read the related data together?)
- Document size (16MB hard limit per document)
- Update frequency (embedded data that changes often causes write amplification)

## Aggregation Pipeline

The aggregation framework is MongoDB's answer to complex queries:

```javascript
db.orders.aggregate([
  { $match: { status: "completed" } },
  { $group: {
    _id: "$customerId",
    totalSpent: { $sum: "$amount" },
    orderCount: { $sum: 1 }
  }},
  { $sort: { totalSpent: -1 } },
  { $limit: 10 }
]);
```

## Indexing

MongoDB supports B-tree indexes, compound indexes, text indexes, geospatial indexes, and wildcard indexes. Without proper indexes, queries trigger a **collection scan** — the document-store equivalent of a full table scan.

See [[db-020]] for indexing strategies that apply across database engines.
