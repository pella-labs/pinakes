# Mutual TLS Authentication

**Mutual TLS** (mTLS) requires both the client and server to present certificates during the TLS handshake. This provides strong authentication in both directions, going beyond standard TLS where only the server is authenticated.

## How mTLS Works

1. Client connects to server
2. Server presents its certificate (standard TLS)
3. Server requests client certificate
4. Client presents its certificate
5. Both sides verify certificates against their trust stores
6. Encrypted channel established with mutual authentication

## Use Cases

- Service mesh communication (Istio, Linkerd)
- Zero trust network architectures
- API authentication for high-security partners
- Internal microservice communication

## Certificate Management

The operational burden of mTLS is in **certificate lifecycle management**. You need:

- A **Certificate Authority** (internal CA or managed service)
- Automated certificate provisioning and renewal
- Short-lived certificates (hours or days, not years)
- Revocation checking (CRL or OCSP)

Tools like **cert-manager** (Kubernetes) or **SPIFFE/SPIRE** automate much of this. See [[sec-017]] for TLS configuration details and [[sec-010]] for certificate pinning.
