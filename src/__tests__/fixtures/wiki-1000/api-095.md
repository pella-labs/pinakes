# API Load Balancing Strategies

## Round Robin

Each request goes to the next server in sequence. Simple, works when all servers are equal.

## Least Connections

Route to the server with the fewest active connections. Better for uneven request durations.

## Weighted

Assign weights based on server capacity. A 4-core server gets weight 4, a 2-core gets weight 2.

## Consistent Hashing

Hash the request key (user ID, session ID) to a server. The same user always hits the same server. Good for **WebSocket connections** and **session affinity**.

```
hash(userId) → server_3
```

## Health Checks

The load balancer must actively probe backends:

```yaml
health_check:
  path: /health/live
  interval: 10s
  timeout: 3s
  unhealthy_threshold: 3
  healthy_threshold: 2
```

Remove unhealthy servers from the pool. Re-add after they pass `healthy_threshold` consecutive checks. See [[api-029]] for health endpoint design.

## Layer 4 vs Layer 7

- **L4** (TCP): fast, protocol-agnostic, no request inspection
- **L7** (HTTP): can route by path, header, cookie. Needed for API gateway features
