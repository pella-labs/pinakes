# Content-Type Validation

Improperly validated **Content-Type** headers can lead to security vulnerabilities including CSRF, XSS, and deserialization attacks.

## Server-Side Validation

Always validate the Content-Type of incoming requests:

```typescript
app.use((req, res, next) => {
  if (['POST', 'PUT', 'PATCH'].includes(req.method)) {
    const contentType = req.headers['content-type'];
    if (!contentType || !contentType.startsWith('application/json')) {
      return res.status(415).json({
        error: 'Content-Type must be application/json',
      });
    }
  }
  next();
});
```

## Why It Matters

### CSRF Prevention

Browsers can send cross-origin requests with `Content-Type: application/x-www-form-urlencoded` without a preflight CORS check. By requiring `application/json`, you force a preflight that the browser will block if CORS isn't configured (see [[sec-012]]).

### MIME Sniffing

Without the `X-Content-Type-Options: nosniff` header, browsers may interpret response bodies differently than intended, potentially executing JavaScript from a JSON response.

### Deserialization Safety

The Content-Type tells the server how to parse the body. If the server accepts any Content-Type, an attacker might send XML to a JSON endpoint, triggering an XML parser with XXE vulnerabilities.

## Response Content-Type

Always set the correct Content-Type on responses:

```
Content-Type: application/json; charset=utf-8
```

Combined with `X-Content-Type-Options: nosniff`, this prevents the browser from reinterpreting the response.
