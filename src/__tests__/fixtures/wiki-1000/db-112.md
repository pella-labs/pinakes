# Event Sourcing

## The Core Idea

Instead of storing current state, store the **sequence of events** that produced the state. The current state is derived by replaying events from the beginning.

## Key Components

### Event Store
An append-only log of events. See [[db-070]] for event store schema design.

### Aggregates
Domain objects that process commands and emit events. Each aggregate has a stream of events identified by a stream ID.

### Projections
Read models built by processing events. A projection subscribes to events and updates a denormalized view.

### Snapshots
Periodic checkpoints of aggregate state to avoid replaying the entire event history.

## Example: Order Lifecycle

```typescript
// Events
type OrderEvent =
  | { type: 'OrderCreated'; customerId: string; items: Item[] }
  | { type: 'ItemAdded'; productId: string; quantity: number }
  | { type: 'OrderShipped'; trackingNumber: string }
  | { type: 'OrderDelivered'; deliveredAt: Date };

// Build state from events
function buildOrderState(events: OrderEvent[]): Order {
  return events.reduce((state, event) => {
    switch (event.type) {
      case 'OrderCreated': return { ...state, status: 'created', items: event.items };
      case 'ItemAdded': return { ...state, items: [...state.items, event] };
      case 'OrderShipped': return { ...state, status: 'shipped' };
      case 'OrderDelivered': return { ...state, status: 'delivered' };
    }
  }, {} as Order);
}
```

## Benefits and Tradeoffs

Benefits: complete audit trail, temporal queries, event replay for debugging, natural fit for CQRS ([[db-108]])

Tradeoffs: complexity, eventual consistency, event schema evolution, storage growth
