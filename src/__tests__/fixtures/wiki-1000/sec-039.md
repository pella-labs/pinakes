# Secure File Upload Handling

File uploads are a high-risk feature. Improper handling can lead to remote code execution, denial of service, and data exfiltration.

## Validation Checklist

Before accepting any uploaded file:

- **File size**: enforce a maximum (e.g., 10 MB)
- **File type**: validate against an allowlist of MIME types
- **File extension**: validate against an allowlist
- **Content inspection**: verify the file's magic bytes match its claimed type
- **Filename sanitization**: strip path separators, null bytes, and special characters

## Storage

- **Never store uploads in the webroot** — they could be served as executable content
- Use a separate storage service (S3, GCS) with no execute permissions
- Generate a random filename — never use the user-provided name for storage
- Store metadata (original name, uploader, timestamp) in the database

## Virus Scanning

For user-facing applications, scan uploads with ClamAV or a commercial scanner before making them available. Quarantine suspicious files.

## Image Processing Pitfalls

Image processing libraries have a history of vulnerabilities:

- Process images in a sandboxed environment
- Strip EXIF metadata (may contain GPS coordinates and other PII)
- Re-encode images rather than serving originals
- Set memory limits on image processing to prevent decompression bombs

## Content-Disposition

When serving uploaded files, always set:

```
Content-Disposition: attachment; filename="safe-name.pdf"
Content-Type: application/octet-stream
X-Content-Type-Options: nosniff
```

This prevents the browser from rendering uploaded content as HTML.
