# Building Accessible Components

## Accessible Buttons

A `<button>` element is inherently accessible. Avoid using `<div onClick>` as a button substitute—it lacks keyboard support, focus styling, and semantic meaning.

```tsx
// Bad - not accessible
<div className="btn" onClick={handleClick}>Submit</div>

// Good - semantically correct
<button type="submit" onClick={handleClick}>Submit</button>

// If you must use a non-button element
<div
  role="button"
  tabIndex={0}
  onClick={handleClick}
  onKeyDown={(e) => {
    if (e.key === 'Enter' || e.key === ' ') {
      e.preventDefault();
      handleClick();
    }
  }}
>
  Submit
</div>
```

## Form Accessibility

Every form input needs an associated label. The `htmlFor` attribute connects them programmatically.

- Always pair `<label>` with `<input>` via `htmlFor`/`id`
- Use `aria-describedby` for supplemental help text
- Use `aria-invalid` and `aria-errormessage` for validation errors
- Group related inputs with `<fieldset>` and `<legend>`

## Live Regions

For dynamic content updates (toast notifications, form submission results), use **aria-live regions** to announce changes without moving focus.

```tsx
<div aria-live="polite" aria-atomic="true">
  {statusMessage && <p>{statusMessage}</p>}
</div>
```
