# Database Indexing Internals

## B-Tree Structure

A **B-tree** (actually B+ tree in most databases) is a balanced tree where:

- Internal nodes contain keys and pointers to child nodes
- Leaf nodes contain keys and pointers to heap tuples (rows)
- Leaf nodes are linked as a doubly-linked list for range scans
- The tree is always balanced — every leaf is at the same depth

## Index Lookup Flow

1. Start at the root node
2. Binary search for the key in the current node
3. Follow the pointer to the next level
4. Repeat until reaching a leaf node
5. Follow the heap pointer to read the actual row

For a table with 10 million rows, a B-tree with a branching factor of 500 has depth 3-4. That's 3-4 page reads to find any row.

## Index-Only Scans

If the index contains all columns needed by the query, PostgreSQL can skip the heap access entirely. This is called an **index-only scan** and is a major performance win.

```sql
-- This can be an index-only scan if we have:
-- CREATE INDEX ON orders (customer_id) INCLUDE (total)
SELECT customer_id, total FROM orders WHERE customer_id = 42;
```

## Visibility Map

PostgreSQL's MVCC means the index doesn't know if a row is visible to the current transaction. The **visibility map** tracks which heap pages have only visible tuples, enabling index-only scans to skip heap checks for those pages.
