# Database Replication Topologies

## Single Leader

The most common setup. One **primary** node accepts writes; one or more **replicas** (standbys) receive a stream of changes. PostgreSQL uses streaming replication over WAL. MySQL uses binlog replication.

Replicas serve read traffic and provide failover capability. The tradeoff is **replication lag** — reads from replicas may return stale data.

## Multi-Leader

Multiple nodes accept writes. Conflicts are resolved asynchronously. This is harder to get right and usually only justified for:

- Multi-datacenter deployments where latency to a single leader is unacceptable
- Offline-first applications (CRDTs, operational transforms)

CockroachDB and Spanner use consensus protocols instead of traditional multi-leader replication.

## Leaderless

Systems like Cassandra and DynamoDB use **quorum-based replication**. Writes go to W replicas, reads go to R replicas, and as long as W + R > N, you're guaranteed to read the latest write.

- N=3, W=2, R=2 is a common configuration
- Tunable consistency: lower W for faster writes, lower R for faster reads

## Synchronous vs Asynchronous

- **Synchronous**: the primary waits for at least one replica to confirm. Slower but no data loss on failover.
- **Asynchronous**: the primary doesn't wait. Faster but replicas can fall behind.
- **Semi-synchronous**: wait for one replica, send asynchronously to the rest.
