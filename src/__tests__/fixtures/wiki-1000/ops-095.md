# Kubernetes Resource Management

## Requests vs Limits

Understanding the difference between **requests** and **limits** is critical for Kubernetes resource management:

- **Requests**: guaranteed resources. Used by the scheduler for placement. Affects QoS class.
- **Limits**: maximum resources. Container is throttled (CPU) or killed (memory) if exceeded.

## QoS Classes

Kubernetes assigns QoS classes based on requests and limits:

| Class | Condition | Eviction Priority |
|-------|-----------|-------------------|
| **Guaranteed** | requests == limits for all containers | Last to be evicted |
| **Burstable** | At least one request or limit set | Medium priority |
| **BestEffort** | No requests or limits | First to be evicted |

## Right-Sizing

1. Deploy with VPA in recommendation mode
2. Observe actual usage for 7 days
3. Set requests to P95 usage
4. Set limits to 2x requests (or use no CPU limit)

## No CPU Limits?

A growing consensus advocates for **no CPU limits**, only requests. CPU limits cause throttling even when the node has spare capacity. Memory limits are still necessary to prevent OOM conditions.

```yaml
resources:
  requests:
    cpu: "250m"
    memory: "256Mi"
  limits:
    memory: "512Mi"
    # No CPU limit — let the pod burst
```
