# Secure WebSocket Communication

**WebSocket** connections have different security characteristics than HTTP. They maintain long-lived connections, bypass some traditional HTTP protections, and require specific security measures.

## Authentication

Authenticate before upgrading to WebSocket:

```typescript
// Validate token during upgrade
server.on('upgrade', (request, socket, head) => {
  const token = new URL(request.url, 'http://localhost').searchParams.get('token');
  if (!verifyToken(token)) {
    socket.write('HTTP/1.1 401 Unauthorized\r\n\r\n');
    socket.destroy();
    return;
  }
  wss.handleUpgrade(request, socket, head, (ws) => {
    wss.emit('connection', ws, request);
  });
});
```

## Origin Validation

Validate the `Origin` header during the handshake to prevent cross-site WebSocket hijacking:

```typescript
if (!ALLOWED_ORIGINS.includes(request.headers.origin)) {
  socket.destroy();
  return;
}
```

## Message Validation

Every message received over WebSocket must be validated:

- Parse and validate against a schema
- Enforce message size limits
- Rate limit messages per connection
- Never evaluate received messages as code

## TLS

Always use `wss://` (WebSocket Secure). Never use plain `ws://` in production. The TLS configuration follows the same rules as HTTPS (see [[sec-017]]).

## Connection Management

- Set idle timeouts for inactive connections
- Implement heartbeat/ping to detect dead connections
- Limit concurrent connections per user
- Log connection lifecycle events for audit (see [[sec-014]])
