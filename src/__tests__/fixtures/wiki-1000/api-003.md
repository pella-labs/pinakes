# API Versioning Strategies

Three main approaches. Pick one and stick with it.

## URL Path Versioning

`/v1/users`, `/v2/users`

**Pros:** Explicit, easy to route, easy to deprecate. **Cons:** Breaks URI stability, complicates client migrations.

This is what most teams pick. It works. See [[api-001]] for how versioning interacts with resource naming.

## Header Versioning

`Accept: application/vnd.myapi.v2+json`

Cleaner URIs but harder to test in a browser. Requires middleware to parse and route. Good for internal APIs where clients are controlled.

## Query Parameter Versioning

`/users?version=2`

Rarely recommended. Pollutes the query string and caches poorly.

## Our Decision

We use **URL path versioning** with a compatibility window of 2 major versions. When v3 ships, v1 gets a 90-day sunset notice. Deprecation headers are added to v1 responses immediately.
