# Kubernetes RBAC

**Kubernetes RBAC** controls who can perform what actions on which resources within a cluster. It's the primary authorization mechanism for the Kubernetes API.

## Core Objects

- **Role**: grants permissions within a namespace
- **ClusterRole**: grants permissions cluster-wide or across namespaces
- **RoleBinding**: binds a Role to subjects in a namespace
- **ClusterRoleBinding**: binds a ClusterRole to subjects cluster-wide

## Principle of Least Privilege

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: my-app
rules:
  - apiGroups: [""]
    resources: ["pods"]
    verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: my-app
subjects:
  - kind: ServiceAccount
    name: my-app-sa
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

## Common Mistakes

- Granting `cluster-admin` to service accounts
- Using ClusterRoleBindings when RoleBindings suffice
- Granting wildcard verbs (`*`) instead of specific verbs
- Not auditing unused RBAC rules

## Service Account Security

- Create dedicated service accounts per workload
- Don't mount service account tokens automatically:

```yaml
automountServiceAccountToken: false
```

- Only mount tokens when the pod actually needs API access

See [[sec-004]] for general RBAC concepts and [[sec-046]] for container security.
