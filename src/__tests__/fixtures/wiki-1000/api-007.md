# Idempotency Keys

## The Problem

Network failures cause retries. Without idempotency, a retried `POST /payments` creates a duplicate charge. This is the single most common payment bug.

## Implementation

The client sends an `Idempotency-Key` header (UUID) with every mutating request. The server:

1. Checks if the key exists in the idempotency store
2. If yes, returns the stored response (no side effects)
3. If no, processes the request, stores the response keyed by the idempotency key

```typescript
async function handleWithIdempotency(
  key: string,
  handler: () => Promise<Response>
): Promise<Response> {
  const cached = await idempotencyStore.get(key);
  if (cached) return cached;

  const response = await handler();
  await idempotencyStore.set(key, response, { ttl: 86400 });
  return response;
}
```

## Storage

Use Redis with a 24-hour TTL. The key is the idempotency key, the value is the serialized response. For distributed systems, use a compare-and-swap operation to prevent race conditions between concurrent retries.

## Which Endpoints Need It

- **All** payment/billing endpoints
- Any endpoint that triggers external side effects (emails, webhooks)
- `POST` endpoints that create resources

`GET`, `PUT`, `DELETE` are naturally idempotent and don't need this mechanism. See [[api-024]] for retry strategies on the client side.
