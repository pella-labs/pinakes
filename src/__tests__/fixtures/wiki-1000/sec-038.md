# Permission Middleware Design

A well-designed **permission middleware** centralizes authorization checks, preventing the scatter of ad-hoc permission logic throughout the codebase.

## Architecture

```
Request → Auth Middleware → Permission Middleware → Route Handler
              ↓                    ↓
        Verify identity     Check permissions
        (who are you?)      (are you allowed?)
```

## Implementation

```typescript
function requirePermission(...permissions: string[]) {
  return (req: Request, res: Response, next: NextFunction) => {
    const user = req.user;
    if (!user) {
      return res.status(401).json({ error: 'Authentication required' });
    }

    const hasPermission = permissions.every(
      perm => user.permissions.includes(perm)
    );

    if (!hasPermission) {
      auditLog({
        actor: user.id,
        action: 'authorization_denied',
        resource: req.path,
        permissions_required: permissions,
      });
      return res.status(403).json({ error: 'Insufficient permissions' });
    }

    next();
  };
}

// Usage
app.delete('/api/documents/:id',
  requirePermission('documents:delete'),
  deleteDocumentHandler
);
```

## Design Principles

- **Deny by default**: if no permission is explicitly granted, access is denied
- **Fail closed**: errors in the permission system should deny access, not grant it
- **Log denials**: every authorization denial should be audit-logged (see [[sec-014]])
- **Separate from business logic**: handlers should not contain permission checks

## Resource-Level Permissions

For permissions that depend on the specific resource (e.g., "can edit their own documents"), use a resource loader middleware that fetches the resource before the permission check.

See [[sec-004]] for RBAC and [[sec-005]] for ABAC patterns.
