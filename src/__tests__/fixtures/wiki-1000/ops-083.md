# Rate Limiting Strategies

## Why Rate Limit

**Rate limiting** protects your services from abuse, prevents cascading failures, and ensures fair resource allocation among clients.

## Algorithms

### Token Bucket

Tokens are added at a fixed rate. Each request consumes a token. Allows bursts up to the bucket size while maintaining a long-term average rate.

### Sliding Window

Counts requests in a rolling time window. More accurate than fixed windows but uses more memory.

### Leaky Bucket

Requests queue and are processed at a fixed rate. Smooths out bursts but adds latency.

## Implementation Layers

Rate limit at multiple layers:

1. **CDN/WAF**: coarse IP-based limits (see [[ops-072]])
2. **API Gateway**: per-client or per-API-key limits
3. **Application**: fine-grained business logic limits
4. **Database**: connection limits and query timeouts

## Redis-Based Rate Limiting

```lua
-- Sliding window rate limiter in Redis
local key = KEYS[1]
local window = tonumber(ARGV[1])
local limit = tonumber(ARGV[2])
local now = tonumber(ARGV[3])

redis.call('ZREMRANGEBYSCORE', key, 0, now - window)
local count = redis.call('ZCARD', key)

if count < limit then
  redis.call('ZADD', key, now, now .. math.random())
  redis.call('EXPIRE', key, window)
  return 1
end
return 0
```

## Response Headers

Always include rate limit headers so clients can self-regulate: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`.
