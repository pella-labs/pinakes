# Data Migration Patterns

## The Expand-Contract Pattern

Safely evolve schemas in production without downtime:

### Phase 1: Expand
Add the new column or table alongside the old one. Deploy code that writes to both.

### Phase 2: Migrate
Backfill existing data from old to new. Verify consistency.

### Phase 3: Contract
Remove reads from the old column. Then remove writes. Then drop the column.

This pattern works with any database and any deployment strategy. It's slower than a single ALTER TABLE but never breaks running applications.

## Backfill Strategies

For large tables, batch the backfill:

```sql
-- Backfill in chunks to avoid long locks
UPDATE users SET normalized_email = lower(email)
WHERE id BETWEEN 1 AND 10000 AND normalized_email IS NULL;

-- Repeat for next batch...
```

Use a script that processes batches with short pauses between them to avoid overwhelming the database.

## Zero-Downtime Renames

Never rename a column directly in production. Instead:
1. Add new column
2. Write to both columns
3. Backfill old → new
4. Switch reads to new column
5. Stop writing to old column
6. Drop old column

Each step is a separate deployment. See [[db-016]] for migration tooling.
