# Encryption at Rest

**Encryption at rest** protects data stored on disk from unauthorized physical or logical access. It is a requirement for most compliance frameworks.

## Layers of Encryption

### Full Disk Encryption
Operating system level (LUKS, BitLocker, FileVault). Protects against physical theft of storage media but doesn't protect against a compromised OS.

### Database Encryption
Transparent Data Encryption (TDE) in PostgreSQL, SQL Server, Oracle. Encrypts data files automatically. Protects against direct file access but not SQL-level access.

### Application-Level Encryption
Encrypt specific fields before storing them. Provides the strongest protection because the database never sees plaintext.

```typescript
import { createCipheriv, createDecipheriv, randomBytes } from 'crypto';

function encrypt(plaintext: string, key: Buffer): string {
  const iv = randomBytes(12);
  const cipher = createCipheriv('aes-256-gcm', key, iv);
  const encrypted = Buffer.concat([cipher.update(plaintext, 'utf8'), cipher.final()]);
  const tag = cipher.getAuthTag();
  return `${iv.toString('base64')}:${tag.toString('base64')}:${encrypted.toString('base64')}`;
}
```

## Key Management

Application-level encryption is only as strong as the key management:

- Never store encryption keys alongside encrypted data
- Use **envelope encryption**: encrypt the data key with a master key stored in a KMS
- Rotate encryption keys regularly (see [[sec-016]])
- Maintain key access audit logs

## What to Encrypt

Prioritize encrypting:

- Credentials and secrets
- Personal Identifiable Information (PII)
- Financial data
- Health records (PHI)
- Any data classified as confidential
