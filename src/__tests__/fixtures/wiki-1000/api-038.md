# DR: Standardized Error Response Format

## Context

Three different error shapes across our APIs. Mobile team parsing errors differently per service. Support tickets about "unknown error" because the client couldn't parse the response.

## Decision

All services adopt the RFC 7807 Problem Details shape, extended with our `request_id`:

```json
{
  "type": "https://api.example.com/errors/validation-error",
  "title": "Validation Error",
  "status": 400,
  "detail": "The 'email' field must be a valid email address",
  "instance": "/orders/ord_123",
  "request_id": "req_xyz",
  "errors": [
    { "field": "email", "message": "Invalid email format" }
  ]
}
```

## Consequences

- All services need migration (estimated 2 sprints across 4 services)
- Client SDKs need a single error parser update
- `type` URI provides machine-readable error categorization
- `detail` provides human-readable explanation

Reference: [[api-012]] for the full error taxonomy.
