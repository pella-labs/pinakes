# Database Normalization

## The Normal Forms

**Normalization** is the process of organizing relational data to reduce redundancy and improve integrity. The main normal forms are:

### First Normal Form (1NF)

Every column contains atomic values. No repeating groups or arrays. Each row is unique.

### Second Normal Form (2NF)

Must be in 1NF. Every non-key attribute depends on the entire primary key, not just part of it. This matters for composite keys.

### Third Normal Form (3NF)

Must be in 2NF. No **transitive dependencies** — non-key attributes must depend only on the primary key, not on other non-key attributes.

### Boyce-Codd Normal Form (BCNF)

A stricter version of 3NF. Every determinant must be a candidate key. In practice, most schemas that satisfy 3NF also satisfy BCNF.

## When to Denormalize

Normalization optimizes for write consistency. But read-heavy workloads often benefit from **strategic denormalization** — duplicating data to avoid expensive joins.

Common denormalization patterns:
- Materialized views for dashboard queries
- Precomputed aggregates (counts, sums)
- Embedding frequently-accessed related data
- Caching derived fields

The key insight: normalize first, denormalize selectively based on measured query patterns. See [[db-005]] for how MongoDB takes a denormalization-first approach.
