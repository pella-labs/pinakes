# REST Resource Naming Conventions

## Plural Nouns for Collections

Always use plural nouns for collection endpoints. The resource name should describe **what** the collection contains, not what you do with it.

- `/users` — not `/getUsers` or `/user-list`
- `/orders/{id}/items` — nested when ownership is clear
- `/search/users` — action-style only for non-CRUD operations

## Hierarchical Nesting

Limit nesting to two levels maximum. Deeply nested URLs become fragile and hard to cache. If you need `/users/{id}/orders/{oid}/items/{iid}/variants`, flatten to `/order-items/{iid}/variants` with a filter param instead.

See [[api-003]] for versioning strategies that affect path structure.

## Naming Rules

1. Use **kebab-case** for multi-word resources: `/line-items`, not `/lineItems`
2. Never expose database table names directly
3. Use consistent pluralization — don't mix `/user` and `/accounts`
4. Resource identifiers should be **opaque** — UUIDs preferred over sequential IDs
