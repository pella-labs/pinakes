# Event Store Design

## What Is an Event Store?

An **event store** is a database optimized for append-only event streams. Every state change is stored as an immutable event. The current state is derived by replaying events.

## Schema

```sql
CREATE TABLE events (
  id BIGSERIAL PRIMARY KEY,
  stream_id TEXT NOT NULL,
  version INT NOT NULL,
  event_type TEXT NOT NULL,
  data JSONB NOT NULL,
  metadata JSONB DEFAULT '{}',
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (stream_id, version)
);

CREATE INDEX idx_events_stream ON events (stream_id, version);
CREATE INDEX idx_events_type ON events (event_type, created_at);
```

## Optimistic Concurrency

The `(stream_id, version)` unique constraint provides optimistic concurrency control. To append an event, specify the expected version:

```sql
INSERT INTO events (stream_id, version, event_type, data)
VALUES ('order-123', 5, 'ItemAdded', '{"product_id": 42, "quantity": 1}');
-- Fails if version 5 already exists
```

## Snapshots

Replaying thousands of events for every read is slow. Periodically snapshot the current state:

```sql
CREATE TABLE snapshots (
  stream_id TEXT PRIMARY KEY,
  version INT NOT NULL,
  state JSONB NOT NULL,
  created_at TIMESTAMPTZ NOT NULL
);
```

Load from snapshot, replay only newer events. See [[db-112]] for the full event sourcing architecture.
