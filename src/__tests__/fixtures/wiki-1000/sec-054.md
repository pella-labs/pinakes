# Security Incident Response

A well-rehearsed **incident response** process minimizes damage and recovery time. The worst time to create a process is during an active incident.

## Phases

### 1. Preparation
- Define roles: Incident Commander, Communications, Technical Lead
- Maintain contact lists and escalation paths
- Prepare runbooks for common incident types
- Ensure logging and monitoring are in place (see [[sec-043]])

### 2. Identification
- Triage the alert: is this a real incident or a false positive?
- Determine scope: what systems and data are affected?
- Classify severity: critical, high, medium, low
- Assign an Incident Commander

### 3. Containment
- Short-term: isolate affected systems, revoke compromised credentials
- Long-term: patch the vulnerability, deploy mitigations
- Preserve evidence for forensics

### 4. Eradication
- Remove the attacker's access and artifacts
- Patch the root cause vulnerability
- Verify the fix with the security team

### 5. Recovery
- Restore systems from clean backups if needed
- Monitor for re-compromise
- Gradually return to normal operations

### 6. Lessons Learned
- Conduct a blameless post-mortem within 72 hours
- Document the timeline, impact, and response actions
- Identify and track improvements
- Update runbooks based on what was learned

## Communication

Have templates ready for:
- Internal stakeholder updates
- Customer notifications
- Regulatory notifications (GDPR 72-hour requirement)
- Public disclosure (if required)
