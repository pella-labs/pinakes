# Certificate Pinning

**Certificate pinning** restricts which certificates a client will accept for a given server, even if the certificate is signed by a trusted CA. This defends against compromised or rogue CAs.

## How It Works

The client stores a hash of the expected certificate (or its public key) and compares it during the TLS handshake. If the presented certificate doesn't match the pin, the connection is rejected.

## Pinning Strategies

- **Leaf pinning**: pin the exact server certificate. Most specific but requires pin updates on every certificate rotation.
- **Intermediate pinning**: pin the intermediate CA certificate. More resilient to leaf rotation.
- **Public key pinning**: pin the SPKI hash. Survives certificate renewal as long as the same key pair is used.

## Implementation

For mobile apps, include pins in the app bundle:

```kotlin
// Android with OkHttp
val client = OkHttpClient.Builder()
    .certificatePinner(
        CertificatePinner.Builder()
            .add("api.example.com", "sha256/AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=")
            .add("api.example.com", "sha256/BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB=")
            .build()
    )
    .build()
```

Always include a **backup pin** for a different key to avoid bricking the app if the primary key is compromised.

## Risks

Pinning is dangerous if mismanaged. A botched pin update can make your app unable to connect at all. Consider using shorter-lived pins with automated rotation. See [[sec-006]] for mutual TLS as an alternative strong authentication mechanism.
