# ETag and Conditional Requests

## How ETags Work

The server generates a **fingerprint** of the response (hash of content or version number) and sends it as the `ETag` header.

```
HTTP/1.1 200 OK
ETag: "a1b2c3d4"
Content-Type: application/json

{"id":"ord_123","status":"shipped"}
```

## Conditional GET (Cache Validation)

Client sends `If-None-Match: "a1b2c3d4"`. If the ETag matches, server returns `304 Not Modified` with no body. Saves bandwidth.

## Conditional PUT (Optimistic Locking)

Client sends `If-Match: "a1b2c3d4"` with the update. If the ETag no longer matches (someone else modified the resource), server returns `412 Precondition Failed`. Prevents lost updates.

```typescript
app.put('/orders/:id', async (req, res) => {
  const current = await db.getOrder(req.params.id);
  const etag = generateETag(current);

  if (req.headers['if-match'] && req.headers['if-match'] !== etag) {
    return res.status(412).json({
      error: { code: 'PRECONDITION_FAILED', message: 'Resource was modified' }
    });
  }

  const updated = await db.updateOrder(req.params.id, req.body);
  res.set('ETag', generateETag(updated)).json(updated);
});
```

## Weak vs Strong ETags

- Strong: `"abc"` — byte-for-byte identical
- Weak: `W/"abc"` — semantically equivalent (field reordering OK)

Use weak ETags for JSON APIs where serialization order may vary.
