# Session Management

**Session management** determines how your application tracks authenticated users across requests. Weaknesses here directly translate to account takeover vulnerabilities.

## Session ID Requirements

- Generate with a **CSPRNG** (cryptographically secure pseudorandom number generator)
- Minimum **128 bits of entropy** (typically 32 hex characters)
- Use your framework's built-in session ID generator — don't roll your own
- Regenerate the session ID after login (prevents session fixation)

## Server-Side vs Token-Based Sessions

### Server-Side Sessions
Session data stored on the server (Redis, database). The client holds only a session ID in a cookie.

Advantages:
- Easy to invalidate
- Session data not exposed to client
- Can store large amounts of state

### Token-Based Sessions
Session data encoded in a JWT. The client holds the complete session.

Advantages:
- No server-side storage needed
- Easy horizontal scaling

Disadvantages:
- Cannot be instantly invalidated without a blocklist
- Token size grows with claims

## Session Lifecycle

- **Creation**: after successful authentication
- **Validation**: on every request
- **Renewal**: extend expiration on activity (sliding window)
- **Invalidation**: on logout, password change, or security event
- **Cleanup**: expired sessions removed from storage

## Security Controls

- Set **idle timeout** (15-30 minutes for sensitive apps)
- Set **absolute timeout** (8-24 hours regardless of activity)
- Limit **concurrent sessions** per user
- Invalidate all sessions on password change
- Bind sessions to user-agent and IP range when practical

See [[sec-034]] for cookie configuration and [[sec-003]] for JWT design.
