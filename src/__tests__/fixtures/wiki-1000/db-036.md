# Data Warehousing Concepts

## OLTP vs OLAP

**OLTP** (Online Transaction Processing) databases handle high volumes of short transactions — inserts, updates, point lookups. They're row-oriented and optimized for write throughput.

**OLAP** (Online Analytical Processing) databases handle complex queries over large datasets — aggregations, joins across fact and dimension tables. They're column-oriented and optimized for scan throughput.

## Star Schema

The **star schema** is the foundational data warehouse design:

- **Fact tables** store measurements (revenue, clicks, events) with foreign keys to dimension tables
- **Dimension tables** store descriptive attributes (customer details, product catalog, time hierarchy)

```sql
CREATE TABLE fact_sales (
  sale_id BIGINT PRIMARY KEY,
  customer_key INT REFERENCES dim_customer,
  product_key INT REFERENCES dim_product,
  date_key INT REFERENCES dim_date,
  quantity INT,
  revenue DECIMAL(12,2)
);
```

## Snowflake Schema

Normalizes dimension tables further. More storage-efficient but requires more joins. Most modern warehouses prefer star schemas — their query engines handle the denormalization efficiently.

## Modern Warehouses

- **Snowflake** — separation of compute and storage
- **BigQuery** — serverless, pay-per-query
- **Redshift** — AWS-native, columnar
- **DuckDB** — embedded OLAP for local analysis
