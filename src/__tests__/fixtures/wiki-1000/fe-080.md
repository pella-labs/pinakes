# Package Boundary Enforcement in Monorepos

## Why Boundaries Matter

Without enforced boundaries, a monorepo devolves into a tangled dependency graph where any package can import from any other. This defeats the purpose of having separate packages.

## Eslint Rules

Use `eslint-plugin-import` or `@nx/enforce-module-boundaries` to prevent illegal imports:

```json
{
  "rules": {
    "@nx/enforce-module-boundaries": [
      "error",
      {
        "depConstraints": [
          { "sourceTag": "scope:ui", "onlyDependOnLibsWithTags": ["scope:ui", "scope:shared"] },
          { "sourceTag": "scope:feature", "onlyDependOnLibsWithTags": ["scope:ui", "scope:shared", "scope:data"] },
          { "sourceTag": "scope:data", "onlyDependOnLibsWithTags": ["scope:shared"] }
        ]
      }
    ]
  }
}
```

## Dependency Graph

Maintain a clear dependency direction:

```
apps → features → ui → shared
         ↓
        data → shared
```

No upward or circular dependencies. Shared packages depend on nothing within the monorepo.

## API Surface

Each package should have a single `index.ts` barrel file that exports its public API. Internal modules are not importable from outside the package.

## CI Enforcement

Run boundary checks in CI. A PR that introduces an illegal import must be fixed before merge.
