# Auto-Scaling Strategies

## Horizontal Pod Autoscaler

The **HPA** scales pods based on CPU, memory, or custom metrics. It's the bread and butter of Kubernetes autoscaling.

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-server
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-server
  minReplicas: 3
  maxReplicas: 50
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
    - type: Pods
      pods:
        metric:
          name: http_requests_per_second
        target:
          type: AverageValue
          averageValue: "1000"
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300
```

## Cluster Autoscaler

The **Cluster Autoscaler** adds or removes nodes when pods can't be scheduled or nodes are underutilized. Configure it alongside the HPA for full elastic scaling.

## KEDA

**KEDA** (Kubernetes Event-Driven Autoscaling) scales based on external event sources like SQS queue depth, Kafka lag, or cron schedules. It extends HPA with richer trigger types.

See [[ops-004]] for K8s fundamentals and [[ops-040]] for cost optimization.
