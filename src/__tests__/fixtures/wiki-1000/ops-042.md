# GitHub Actions Self-Hosted Runners

## Why Self-Hosted

GitHub-hosted runners are convenient but limited: fixed specs, no GPU, no access to internal networks, and per-minute billing adds up for large teams.

## Architecture

We run self-hosted runners on EKS using the **Actions Runner Controller** (ARC):

- Runners scale to zero when idle
- Each job gets a fresh pod (ephemeral runners)
- Docker-in-Docker or Kaniko for container builds
- Access to internal registries and caches

## Setup

```yaml
apiVersion: actions.summerwind.dev/v1alpha1
kind: RunnerDeployment
metadata:
  name: ci-runners
spec:
  template:
    spec:
      repository: myorg/myrepo
      labels:
        - self-hosted
        - linux
        - x64
      resources:
        requests:
          cpu: "2"
          memory: "4Gi"
        limits:
          cpu: "4"
          memory: "8Gi"
---
apiVersion: actions.summerwind.dev/v1alpha1
kind: HorizontalRunnerAutoscaler
metadata:
  name: ci-runners-autoscaler
spec:
  scaleTargetRef:
    name: ci-runners
  minReplicas: 0
  maxReplicas: 20
  scaleUpTriggers:
    - githubEvent:
        workflowJob: {}
      duration: "30m"
```

## Security

Self-hosted runners execute arbitrary code from PRs. Require **approval for first-time contributors** and use ephemeral runners that are destroyed after each job.
