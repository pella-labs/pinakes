# PostgreSQL VACUUM FULL vs Regular VACUUM

## Regular VACUUM

Marks dead tuples as reusable but doesn't return space to the OS. The table file stays the same size, but new rows can use the reclaimed space.

- Does NOT lock the table
- Runs concurrently with reads and writes
- Updates the visibility map and free space map
- Should run frequently via autovacuum

## VACUUM FULL

Rewrites the entire table, compacting it. Returns space to the OS.

- Acquires an **ACCESS EXCLUSIVE** lock — blocks ALL access
- Requires enough disk space for a complete copy of the table
- Takes a long time on large tables
- Should be used sparingly

## When to VACUUM FULL

Almost never. Consider it when:
- A table has extreme bloat (>50% dead space)
- After a massive DELETE that won't be followed by new inserts
- Disk space is critically low

## Alternatives to VACUUM FULL

- **pg_repack**: online repacking without exclusive locks. The best alternative for most cases.
- **CLUSTER**: rewrites the table ordered by an index. Also takes an exclusive lock.
- **CREATE TABLE AS** + swap: create a new table from a SELECT, then rename.

Monitor table bloat with:

```sql
SELECT
  relname,
  n_dead_tup,
  n_live_tup,
  round(n_dead_tup::numeric / NULLIF(n_live_tup, 0) * 100, 1) AS dead_pct
FROM pg_stat_user_tables
ORDER BY n_dead_tup DESC;
```
