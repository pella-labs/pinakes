# Kubernetes Init Containers

## Purpose

**Init containers** run before application containers start. They're useful for setup tasks like database migration checks, configuration fetching, or waiting for dependencies.

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
spec:
  template:
    spec:
      initContainers:
        - name: wait-for-db
          image: busybox:1.36
          command: ['sh', '-c', 'until nc -z database 5432; do sleep 2; done']
        - name: run-migrations
          image: registry.example.com/api-server:latest
          command: ['node', 'dist/migrate.js']
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: db-credentials
                  key: url
      containers:
        - name: api-server
          image: registry.example.com/api-server:latest
```

## Init Container Behavior

- Init containers run **sequentially**, one at a time
- Each must complete successfully before the next starts
- If an init container fails, the pod restarts (subject to `restartPolicy`)
- They don't count toward the pod's resource requests for scheduling

## Common Use Cases

- Wait for a dependent service to be available
- Run database migrations before the app starts
- Fetch configuration or secrets from an external system
- Set up file permissions on shared volumes

See [[ops-015]] for deployment patterns.
