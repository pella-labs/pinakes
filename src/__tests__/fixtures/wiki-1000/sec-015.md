# HashiCorp Vault Patterns

**HashiCorp Vault** is the most widely adopted secrets management platform. It provides dynamic secrets, encryption as a service, and fine-grained access control.

## Core Concepts

- **Secrets Engine**: a backend that stores or generates secrets (KV, database, PKI, transit)
- **Auth Method**: how clients authenticate to Vault (token, AppRole, Kubernetes, OIDC)
- **Policy**: defines what paths a token can access and what operations are allowed
- **Lease**: secrets have a TTL and must be renewed or they expire

## Common Patterns

### Dynamic Database Credentials

Instead of a static database password, Vault generates short-lived credentials on demand:

```bash
vault read database/creds/my-role
# Returns: username=v-my-role-xyz, password=A1B2C3, ttl=1h
```

When the lease expires, Vault automatically revokes the credentials.

### Transit Encryption

Use Vault's **transit engine** for encryption without managing keys:

```bash
vault write transit/encrypt/my-key plaintext=$(base64 <<< "sensitive data")
# Returns: ciphertext=vault:v1:abc123...
```

The application never sees the encryption key.

### AppRole Authentication

For automated systems, AppRole provides a two-factor auth method:

- **Role ID**: like a username (less sensitive)
- **Secret ID**: like a password (highly sensitive, single-use)

## Operational Notes

Always run Vault in **HA mode** with a proper storage backend (Consul, Raft). Never run a single Vault instance in production. See [[sec-016]] for key rotation using Vault's transit engine.
