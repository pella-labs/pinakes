# ETL Pipeline Design

## Extract, Transform, Load

**ETL** is the classic pattern for moving data from operational systems to analytical systems. Each stage has distinct concerns:

### Extract
Pull data from source systems. Sources might be databases, APIs, files, or event streams. Key challenges:
- Incremental vs full extraction
- Handling schema changes in sources
- Rate limiting and backpressure

### Transform
Clean, validate, and reshape data for the target schema. Transformations include:
- Type casting and normalization
- Deduplication
- Joining data from multiple sources
- Computing derived fields and aggregates

### Load
Write transformed data to the destination (data warehouse, data lake, or downstream database).

## ELT: The Modern Twist

With powerful warehouses like BigQuery, Snowflake, and DuckDB, the trend is **ELT** — load raw data first, transform inside the warehouse. Tools like **dbt** manage the transformation layer as SQL models with dependency tracking.

```yaml
# dbt model definition
models:
  - name: daily_revenue
    description: "Revenue aggregated by day"
    columns:
      - name: date
        tests: [not_null, unique]
      - name: total_revenue
        tests: [not_null]
```

## Pipeline Orchestration

Tools like Airflow, Dagster, and Prefect manage dependencies between ETL stages. The key abstraction is a **DAG** (directed acyclic graph) of tasks. See [[db-100]] for data warehouse architecture.
