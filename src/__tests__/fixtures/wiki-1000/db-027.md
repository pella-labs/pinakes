# Time-Series Data

## What Makes Time-Series Special

Time-series data is append-heavy, time-ordered, and rarely updated after insertion. Examples: metrics, IoT sensor readings, financial ticks, application logs. Standard OLTP databases can store time-series data, but specialized systems optimize for the access patterns.

## TimescaleDB

Built as a PostgreSQL extension, **TimescaleDB** automatically partitions data into time-based chunks called **hypertables**:

```sql
SELECT create_hypertable('metrics', 'time');

-- Insert like normal PostgreSQL
INSERT INTO metrics (time, device_id, temperature, humidity)
VALUES (NOW(), 'sensor-42', 22.5, 65.3);

-- Time-bucket aggregation
SELECT
  time_bucket('1 hour', time) AS bucket,
  device_id,
  avg(temperature) AS avg_temp
FROM metrics
WHERE time > NOW() - INTERVAL '24 hours'
GROUP BY bucket, device_id
ORDER BY bucket DESC;
```

## Compression and Retention

Time-series databases compress older data aggressively. TimescaleDB's native compression achieves 90-95% reduction. Retention policies automatically drop data older than a threshold:

```sql
SELECT add_retention_policy('metrics', INTERVAL '90 days');
SELECT add_compression_policy('metrics', INTERVAL '7 days');
```

## Alternatives

- **InfluxDB** — purpose-built, Flux query language
- **QuestDB** — SQL-native, optimized for high-ingest scenarios
- **ClickHouse** — columnar OLAP, excellent for time-series analytics
- **PostgreSQL + partitioning** — viable for moderate scale (see [[db-024]])
