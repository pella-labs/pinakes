# SAML 2.0 Integration

**SAML** (Security Assertion Markup Language) is an XML-based protocol for exchanging authentication and authorization data. It is the standard for enterprise SSO, particularly in organizations using Active Directory.

## SP-Initiated Flow

The most common flow:

1. User visits Service Provider (your app)
2. SP generates a **SAML AuthnRequest** and redirects to IdP
3. User authenticates at IdP
4. IdP generates a **SAML Response** containing assertions
5. IdP POSTs the response to SP's **Assertion Consumer Service (ACS)** URL
6. SP validates the response and creates a session

## Critical Validations

Every SAML response must be validated:

- **Signature verification** on both the response and assertion
- **Destination** matches your ACS URL
- **Audience** matches your entity ID
- **NotBefore / NotOnOrAfter** conditions are within acceptable clock skew
- **InResponseTo** matches your original request ID (prevents replay)

Failing to validate any of these opens your application to assertion forgery attacks.

## Common Pitfalls

- XML signature wrapping attacks — use a well-tested SAML library, not custom XML parsing
- Not validating the full certificate chain of the IdP's signing certificate
- Accepting unsigned assertions when the IdP is configured to sign them
- Large clock skew tolerance enabling replay attacks

## Metadata Exchange

Use **SAML metadata** XML documents to configure trust between IdP and SP automatically. This reduces manual configuration errors. See [[sec-018]] for the broader SSO architecture.
