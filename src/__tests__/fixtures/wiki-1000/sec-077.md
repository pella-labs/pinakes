# Privilege Escalation Prevention

**Privilege escalation** occurs when an attacker gains higher-level access than intended, either vertically (user to admin) or horizontally (accessing another user's data).

## Vertical Escalation

Moving from a lower privilege to a higher one:

- Exploiting RBAC misconfigurations (see [[sec-004]])
- Manipulating JWTs to add admin claims
- Bypassing frontend-only authorization checks
- Exploiting race conditions in permission changes

## Horizontal Escalation

Accessing resources belonging to another user at the same privilege level:

- IDOR vulnerabilities (see [[sec-068]])
- Session fixation or hijacking
- Predictable resource identifiers

## Prevention Strategies

### Server-Side Authorization

Never rely on client-side authorization. The server must verify permissions for every operation.

### Separation of Duties

Critical operations should require multiple actors:

```typescript
// Require approval from a different user than the requester
if (approver.id === requester.id) {
  throw new ForbiddenError('Cannot approve your own request');
}
```

### Least Privilege

Grant the minimum permissions necessary. Review and revoke unused permissions regularly.

### Audit Trail

Log all privilege-related events: role assignments, permission changes, administrative actions. See [[sec-014]] for audit logging patterns.

### Rate Limiting on Sensitive Operations

Limit the rate of administrative operations to slow down automated escalation attempts (see [[sec-026]]).
