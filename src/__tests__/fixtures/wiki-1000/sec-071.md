# Token Revocation

Revoking tokens is necessary when users log out, change passwords, or when a token is suspected to be compromised. Token revocation is straightforward for opaque tokens but challenging for JWTs.

## Opaque Token Revocation

Opaque tokens are validated by looking them up in a server-side store. Revocation is simply deleting the token from the store. Immediate and reliable.

## JWT Revocation Challenges

JWTs are self-contained and stateless. The server doesn't need to look them up, which is their advantage — but it also means there's no central place to revoke them.

## Revocation Strategies

### Short Expiration + Refresh Token Rotation

Keep access tokens very short-lived (5-15 minutes). Revoke the refresh token. The access token becomes useless after expiration. See [[sec-032]] for refresh patterns.

### Token Blocklist

Maintain a set of revoked token IDs (`jti` claims):

```typescript
const revokedTokens = new Set<string>(); // Use Redis in production

function isRevoked(tokenId: string): boolean {
  return revokedTokens.has(tokenId);
}

// Add to validation
if (isRevoked(decoded.jti)) {
  throw new UnauthorizedError('Token has been revoked');
}
```

The blocklist only needs to hold entries until the token would naturally expire.

### Token Versioning

Store a version counter per user. Include the version in the token. On revocation, increment the version. Tokens with old versions are rejected.

## Logout

On logout:
- Revoke the refresh token
- If using blocklists, add the access token
- Clear all client-side token storage
- Invalidate the server-side session if applicable
