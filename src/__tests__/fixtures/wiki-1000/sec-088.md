# Timing Attack Prevention

**Timing attacks** exploit measurable differences in execution time to extract secrets. Even nanosecond differences can be amplified with enough samples.

## Vulnerable Pattern

Standard string comparison short-circuits on the first different character:

```typescript
// VULNERABLE — timing leaks password length and content
if (userInput === storedSecret) { ... }
```

An attacker can determine the secret character by character by measuring response time.

## Constant-Time Comparison

Use timing-safe comparison for all security-critical comparisons:

```typescript
import { timingSafeEqual } from 'crypto';

function secureCompare(a: string, b: string): boolean {
  if (a.length !== b.length) return false; // Length check is acceptable
  return timingSafeEqual(Buffer.from(a), Buffer.from(b));
}
```

## Where to Apply

Use constant-time comparison for:

- HMAC verification (webhook signatures — see [[sec-078]])
- API key comparison
- CSRF token validation
- Password hash comparison (handled by bcrypt/argon2 internally)
- Any secret comparison on the server

## Beyond String Comparison

Other timing attack vectors:

- **Database lookups**: `WHERE username = ?` may take longer for existing users. Mitigate by always running the full auth flow.
- **Conditional logic**: branches that only execute for valid users. Ensure consistent execution paths.
- **Early returns**: returning errors before completing expensive operations. Add artificial delays to normalize response times.
