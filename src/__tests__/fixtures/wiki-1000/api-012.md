# Error Handling Patterns

## Standard Error Envelope

Every error response follows this shape:

```json
{
  "error": {
    "code": "RESOURCE_NOT_FOUND",
    "message": "Order ord_abc123 not found",
    "request_id": "req_xyz789",
    "details": null
  }
}
```

## Error Code Taxonomy

| HTTP Status | Code | When |
|---|---|---|
| 400 | `VALIDATION_ERROR` | Schema validation failed |
| 400 | `INVALID_ARGUMENT` | Business rule violated |
| 401 | `UNAUTHENTICATED` | No valid credentials |
| 403 | `PERMISSION_DENIED` | Valid creds, wrong permissions |
| 404 | `RESOURCE_NOT_FOUND` | Entity doesn't exist |
| 409 | `CONFLICT` | Duplicate or state conflict |
| 422 | `UNPROCESSABLE` | Syntactically valid but semantically wrong |
| 429 | `RATE_LIMITED` | Too many requests |
| 500 | `INTERNAL_ERROR` | Unexpected server failure |

## Don't Leak Internals

Never expose stack traces, SQL queries, or internal service names in error responses. Log them server-side with the `request_id`, then return only the safe error shape to the client.

See [[api-008]] for validation-specific error details.
