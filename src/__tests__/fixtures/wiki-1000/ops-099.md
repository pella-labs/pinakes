# Kubernetes Pod Disruption Budgets

## Purpose

A **Pod Disruption Budget** (PDB) limits how many pods in a set can be voluntarily disrupted at the same time. This protects availability during node drains, cluster upgrades, and autoscaler scale-downs.

## Configuration

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: api-server-pdb
spec:
  minAvailable: 2
  selector:
    matchLabels:
      app: api-server
```

Or using `maxUnavailable`:

```yaml
spec:
  maxUnavailable: 1
  selector:
    matchLabels:
      app: api-server
```

## minAvailable vs maxUnavailable

- **minAvailable**: guarantee at least N pods are always running. Better for critical services.
- **maxUnavailable**: allow at most N pods to be down. Better for large deployments where absolute numbers matter less.

For a 3-replica service, `minAvailable: 2` and `maxUnavailable: 1` are equivalent.

## Impact on Operations

PDBs affect:
- `kubectl drain` — will wait if draining would violate the PDB
- Cluster autoscaler — won't remove nodes if it would violate PDBs
- Node upgrades — proceeds one node at a time respecting PDBs

## Common Pitfall

Setting `minAvailable` equal to the replica count blocks all voluntary disruptions. Never do this — node drains and upgrades will hang indefinitely.

See [[ops-015]] for deployment patterns and [[ops-084]] for K8s upgrades.
