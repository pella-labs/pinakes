# Secure Search Implementation

Search functionality can expose sensitive data if not implemented with security in mind.

## Authorization in Search

Search results must respect access controls. A user should never find documents they don't have permission to view.

```typescript
async function search(query: string, userId: string) {
  // Apply access filter in the search query itself
  const results = await searchIndex.query({
    text: query,
    filter: {
      $or: [
        { visibility: 'public' },
        { ownerId: userId },
        { sharedWith: userId },
      ],
    },
  });
  return results;
}
```

## Information Leakage via Search

### Snippet Leakage
Search snippets can reveal content the user shouldn't see if access control is applied after snippet generation.

### Existence Leakage
"0 results" vs. "access denied" reveals whether restricted content exists. Consider returning uniform responses regardless of the reason for empty results.

### Autocomplete Leakage
Autocomplete suggestions derived from all users' data can reveal other users' queries or content.

## Search Injection

If using full-text search engines (Elasticsearch, Solr):

- Escape special query syntax characters
- Use parameterized queries where available
- Limit query complexity to prevent resource exhaustion
- Apply rate limiting to search endpoints (see [[sec-026]])

## Indexing Security

- Don't index secrets, tokens, or passwords
- Encrypt the search index at rest if it contains sensitive data
- Apply the same retention policies as the source data
- Control who can manage and configure the search index
