# Analyzing and Optimizing JavaScript Bundles

## Why Bundle Size Matters

Every kilobyte of JavaScript must be downloaded, parsed, compiled, and executed. On mobile networks and devices, this translates directly to slower **Time to Interactive**.

## Analysis Tools

- **webpack-bundle-analyzer**: treemap visualization of bundle contents
- **source-map-explorer**: analyzes production source maps
- **bundlephobia**: checks npm package size before installing
- **Import Cost** (VS Code extension): shows import size inline

## Common Culprits

- **Moment.js**: 300KB+ including locales. Replace with **day.js** (2KB) or **date-fns** (tree-shakeable)
- **Lodash**: import specific functions (`lodash/debounce`) instead of the full library
- **Icon libraries**: import individual icons, not the entire set
- **Polyfills**: use `browserslist` to only polyfill what's needed

## Tree Shaking

ES modules enable tree shaking—dead code elimination by the bundler. For it to work:

- Use `import { specific }` not `import *`
- Ensure libraries ship ES module builds
- Mark packages as `"sideEffects": false` in `package.json`
- Avoid barrel files (`index.ts` that re-export everything) when possible

## Dynamic Imports

Move large, non-critical dependencies behind dynamic imports:

```typescript
const { marked } = await import('marked');
```

This creates a separate chunk loaded only when needed. See [[fe-016]] for code splitting and [[fe-010]] for performance budgets.
