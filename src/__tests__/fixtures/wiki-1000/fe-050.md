# State Machines for UI Logic

## Why State Machines

Complex UI interactions—multi-step forms, async operations, animations—are often implemented with scattered boolean flags (`isLoading`, `isError`, `isOpen`). This leads to **impossible states** and bugs.

**State machines** make valid states and transitions explicit.

## XState

```typescript
import { createMachine, assign } from 'xstate';

const fetchMachine = createMachine({
  id: 'fetch',
  initial: 'idle',
  context: { data: null, error: null },
  states: {
    idle: {
      on: { FETCH: 'loading' },
    },
    loading: {
      invoke: {
        src: 'fetchData',
        onDone: {
          target: 'success',
          actions: assign({ data: (_, event) => event.data }),
        },
        onError: {
          target: 'failure',
          actions: assign({ error: (_, event) => event.data }),
        },
      },
    },
    success: {
      on: { REFRESH: 'loading' },
    },
    failure: {
      on: { RETRY: 'loading' },
    },
  },
});
```

## useReducer as a Simple State Machine

For simpler cases, `useReducer` with a discriminated union state type provides similar benefits without a library:

```typescript
type State =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: Data }
  | { status: 'error'; error: Error };
```

## When to Use

State machines shine for wizard flows, payment processes, drag-and-drop, and any interaction with more than three states. For simple toggle states, a boolean is fine. See [[fe-035]] for TypeScript patterns that complement state machines.
