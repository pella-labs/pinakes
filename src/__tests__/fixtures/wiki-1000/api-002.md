# GraphQL Schema Design Principles

## Type-First Modeling

Start with your **domain types**, not your database schema. GraphQL types should represent business concepts.

```graphql
type Order {
  id: ID!
  customer: Customer!
  items: [OrderItem!]!
  status: OrderStatus!
  totalAmount: Money!
  placedAt: DateTime!
}

type Money {
  amount: Int!
  currency: Currency!
}
```

## Nullable vs Non-Null

Default to **non-null** (`!`) and relax only when there's a real reason a field might be absent. This makes the schema self-documenting and reduces null checks on the client.

## Connection Pattern for Pagination

Use the Relay **connection pattern** for any list that could grow unbounded:

```graphql
type Query {
  orders(first: Int, after: String): OrderConnection!
}

type OrderConnection {
  edges: [OrderEdge!]!
  pageInfo: PageInfo!
  totalCount: Int!
}
```

See [[api-015]] for cursor-based pagination implementation details.

## Input Types

Always use dedicated `input` types for mutations. Never reuse output types as inputs — they evolve differently.
