# OAuth2 Scope Design

Well-designed **OAuth2 scopes** control what actions a token can perform. Overly broad scopes grant unnecessary access; overly narrow scopes frustrate developers and lead to scope sprawl.

## Naming Convention

Use a consistent `resource:action` pattern:

- `documents:read` — read documents
- `documents:write` — create and update documents
- `documents:delete` — delete documents
- `users:admin` — manage user accounts

## Granularity Guidelines

Strike a balance between security and usability:

- Too coarse: `admin` scope grants access to everything
- Too fine: 200 individual scopes nobody can keep track of
- Right: 10-30 scopes grouped by resource and action level

## Scope Validation

Validate scopes at multiple levels:

1. **Authorization server**: ensures the client is allowed to request the scope
2. **API gateway**: validates the token contains required scopes
3. **Service layer**: verifies scope before executing the operation

```typescript
function requireScope(scope: string) {
  return (req: Request, res: Response, next: NextFunction) => {
    if (!req.token.scopes.includes(scope)) {
      return res.status(403).json({ error: `Requires scope: ${scope}` });
    }
    next();
  };
}
```

## Scope Consent

For user-facing OAuth2, display scopes in human-readable form during consent. Users should understand what they're granting. See [[sec-001]] for the authorization code flow where consent happens.
