# OpenAPI Specification Best Practices

## Schema Organization

Keep your OpenAPI spec modular. Split into:

- `paths/` — one file per resource
- `schemas/` — shared data models
- `parameters/` — reusable query/path params
- `responses/` — standard error shapes

Use `$ref` aggressively. A 5000-line monolith YAML file is unmaintainable.

## Generating From Code vs Code From Spec

**Spec-first** is almost always better. Write the OpenAPI doc, get stakeholder buy-in, then generate server stubs and client SDKs. Code-first leads to accidental API design driven by implementation convenience.

## Required Fields

```yaml
openapi: 3.1.0
info:
  title: Orders API
  version: 2.1.0
  description: Manage customer orders
paths:
  /orders:
    get:
      operationId: listOrders
      summary: List all orders
      parameters:
        - $ref: '#/components/parameters/PageCursor'
        - $ref: '#/components/parameters/PageSize'
      responses:
        '200':
          description: Paginated list of orders
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/OrderList'
```

Every operation needs an `operationId`. Client generators use it as the method name. Make it meaningful.
