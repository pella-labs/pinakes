# Idempotent Database Operations

## Why Idempotency Matters

In distributed systems, operations can be retried due to timeouts, network failures, or duplicate messages. An **idempotent** operation produces the same result whether executed once or many times.

## Techniques

### Upserts

The most common idempotent pattern. See [[db-060]] for implementation details.

```sql
INSERT INTO events (id, type, payload)
VALUES ($1, $2, $3)
ON CONFLICT (id) DO NOTHING;
```

### Idempotency Keys

Store a unique key with each operation. Reject duplicates:

```sql
CREATE TABLE processed_events (
  idempotency_key TEXT PRIMARY KEY,
  processed_at TIMESTAMPTZ DEFAULT now()
);

-- In a transaction:
INSERT INTO processed_events (idempotency_key) VALUES ($1);
-- If this succeeds, process the event
-- If it fails (duplicate key), skip
```

### Deterministic IDs

Generate IDs from content rather than randomly:

```typescript
import { createHash } from 'crypto';

function deterministicId(scope: string, uri: string, section: string): string {
  return createHash('sha1').update(`${scope}:${uri}:${section}`).digest('hex');
}
```

Re-ingesting the same content produces the same ID, making the operation naturally idempotent.

## Testing Idempotency

Run every write operation twice in your tests. The second execution should produce the same state as the first.
