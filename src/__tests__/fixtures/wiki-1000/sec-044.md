# Security in Microservices

Microservices architectures introduce unique security challenges due to increased network communication, distributed authentication, and expanded attack surface.

## Service-to-Service Authentication

Every service call must be authenticated. Options:

- **mTLS** (see [[sec-006]]): strongest, infrastructure-level
- **JWT propagation**: pass the user's token between services
- **Service tokens**: services authenticate as themselves with [[sec-002]] client credentials

## API Gateway Security

The API gateway is the edge of your microservice network:

- Terminate TLS at the gateway
- Validate and decode JWTs
- Enforce rate limits (see [[sec-026]])
- Apply WAF rules (see [[sec-029]])
- Strip sensitive headers before forwarding

## Secrets Distribution

Each service needs credentials for its dependencies. Use:

- Kubernetes Secrets (with encryption at rest)
- HashiCorp Vault with dynamic credentials (see [[sec-015]])
- Cloud provider secret managers

Never bake secrets into container images.

## Network Policies

In Kubernetes, use **NetworkPolicy** resources to restrict pod-to-pod communication:

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-ingress
spec:
  podSelector:
    matchLabels:
      app: api
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: gateway
      ports:
        - port: 8080
```

## Distributed Tracing for Security

Include security context in distributed traces so that a single suspicious request can be tracked across all services it touched.
