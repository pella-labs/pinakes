# API Security Versioning

When security requirements evolve, you need a strategy for upgrading API security without breaking existing clients.

## Version Security Policies

Each API version should have documented security requirements:

- **v1**: API key authentication, TLS 1.2+
- **v2**: OAuth2 bearer tokens, TLS 1.2+, rate limiting
- **v3**: OAuth2 with PKCE, TLS 1.3, signed requests

## Deprecation Timeline

When upgrading security:

1. **Announce**: 90 days notice minimum for security-breaking changes
2. **Dual-run**: both old and new security accepted during migration
3. **Warn**: add deprecation headers to responses using old security
4. **Enforce**: reject requests not meeting new requirements

```
Deprecation: true
Sunset: Sat, 01 Jun 2026 00:00:00 GMT
Link: <https://api.example.com/docs/migration>; rel="sunset"
```

## Security Headers for API Versioning

Include version-specific security requirements in responses so clients can adapt programmatically.

## Breaking Security Changes

Some security upgrades are worth the breaking change:

- Disabling insecure authentication methods
- Requiring encrypted connections
- Removing endpoints with unfixable vulnerabilities

Communicate clearly, provide migration guides, and offer support during the transition.
