# SQLite for Application Development

SQLite is not a toy database. It handles more transactions per day than all other databases combined — it runs on every smartphone, every browser, and inside countless desktop applications.

## When SQLite Is the Right Choice

- Single-user or single-writer applications
- Embedded systems and mobile apps
- Local-first software
- Test databases
- Small to medium web sites (up to ~100K requests/day)
- Application file formats (replacing custom binary formats)

## When SQLite Is Wrong

- High write concurrency from multiple processes
- Multi-server deployments (no network access)
- Very large datasets (>1TB, though it technically supports 281TB)
- Workloads requiring fine-grained user permissions

## Performance Tips

```sql
PRAGMA journal_mode = WAL;        -- essential for concurrent reads
PRAGMA synchronous = NORMAL;      -- safe with WAL, much faster
PRAGMA cache_size = -20000;       -- 20MB cache
PRAGMA temp_store = MEMORY;       -- temp tables in RAM
PRAGMA mmap_size = 268435456;     -- 256MB memory-mapped I/O
```

These pragmas can make SQLite 10x faster for read-heavy workloads. See [[db-003]] for WAL details and [[db-042]] for better-sqlite3 usage in Node.js.
