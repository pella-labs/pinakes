# Attribute-Based Access Control

**ABAC** evaluates access decisions based on attributes of the subject, resource, action, and environment. It provides fine-grained control that RBAC alone cannot achieve.

## How ABAC Works

Instead of checking role membership, ABAC evaluates a **policy** against a set of attributes:

- **Subject attributes**: user department, clearance level, location
- **Resource attributes**: classification, owner, creation date
- **Action attributes**: read, write, delete, approve
- **Environment attributes**: time of day, IP address, device type

## Policy Example

```
PERMIT if:
  subject.department == resource.department
  AND subject.clearance >= resource.classification
  AND environment.time BETWEEN "08:00" AND "18:00"
  AND action IN ["read", "list"]
```

## When to Choose ABAC Over RBAC

ABAC shines when the number of distinct access scenarios would require an unmanageable number of RBAC roles. A common pattern is to use **RBAC for coarse-grained access** and ABAC for fine-grained decisions within that boundary.

## Implementation Considerations

ABAC policies can become complex and hard to audit. Keep policies simple, well-documented, and testable. Use a policy engine like **Open Policy Agent** or **Cedar** rather than hand-rolling evaluation logic. See [[sec-004]] for the simpler RBAC approach.
