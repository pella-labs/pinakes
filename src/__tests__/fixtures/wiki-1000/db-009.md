# EXPLAIN ANALYZE Deep Dive

Understanding query plans is the single most important skill for database performance tuning. Every major RDBMS provides an **EXPLAIN** command, but PostgreSQL's `EXPLAIN ANALYZE` is the gold standard.

## Reading a Query Plan

```sql
EXPLAIN ANALYZE
SELECT u.name, count(o.id) as order_count
FROM users u
JOIN orders o ON o.user_id = u.id
WHERE u.created_at > '2024-01-01'
GROUP BY u.name
ORDER BY order_count DESC
LIMIT 10;
```

The output shows a tree of plan nodes. Read from the innermost node outward. Key things to look for:

- **Seq Scan** vs **Index Scan** — sequential scans on large tables are red flags
- **Nested Loop** vs **Hash Join** vs **Merge Join** — the join strategy matters
- **Actual rows** vs **estimated rows** — large discrepancies indicate stale statistics
- **Buffers: shared hit** vs **shared read** — cache hit ratio

## Common Fixes

When EXPLAIN shows a problem:

1. Add a missing index (see [[db-020]])
2. Run `ANALYZE` to update statistics
3. Increase `work_mem` for sort/hash-heavy queries
4. Rewrite the query to help the planner
5. Consider a materialized view for complex aggregations

## MySQL's EXPLAIN

MySQL's EXPLAIN output is less detailed but follows similar principles. The `type` column is key: `ALL` (full scan) < `index` < `range` < `ref` < `eq_ref` < `const`. Aim for `ref` or better on your critical queries.
